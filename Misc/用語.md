# Windowsシステム

## プロセス・スレッド・プロシージャ

プロシージャはプログラム中の「処理のかたまり」で、命令の集合。

スレッド（日本語で糸）はプロセスを細分化したもの。

- プロセス詳細：[プロセス](../Cheatsheet/🪟Windows/Windowsの基礎・前提知識/Windows%20Internal.md#プロセス)
- スレッド詳細：[スレッド](../Cheatsheet/🪟Windows/Windowsの基礎・前提知識/Windows%20Internal.md#スレッド)

|項目|プロセス|スレッド|
|---|---|---|
|定義|実行中のアプリケーション単位|プロセス内の実行単位（命令の流れ）|
|メモリ|独立したメモリ空間を持つ|プロセス内でメモリ空間を共有する|
|オーバーヘッド|大きい（プロセス切り替えは重い）|小さい（スレッド切り替えは軽い）|
|実行単位|通常は1つ以上のスレッドを持つ|単体では存在できない（プロセスの一部）|

## サービス

Windowsがバックグラウンドで実行する、特定の用途に設計された特殊なプロセス（Unixでいう「デーモン」と同じ考え）。

|項目|サービス|プロセス|
|---|---|---|
|概要|ユーザ（ネットワーク経由を含む）に奉仕（サービス）する仕組み。実態はプロセス|OS上で動く単位。原則として実行ファイル（exe）に対応し、プロセス内ではスレッドが動作する|
|例|Oracleサービス、IISサービス、FTPサービス、リモートデスクトップサービス、Postfixサービス|Wordのプロセス（WINWORD.EXE）、Excelのプロセス（EXCEL.EXE）、サービス用のプロセス（svchost.exe）|
|管理|SCM（サービスコントロールマネージャ）によって管理され、開始・停止・再起動などが制御される|OSによって自動的に管理される|
|操作|SCMからの指示（自動起動、手動起動、再起動など）で動作する|ユーザの操作（クリック、キーボード入力など）によって動作するのが一般的|
|画面（GUI）|原則としてGUIは持たない（バックグラウンド実行）|通常はGUIを持つ（アプリケーション）が、持たないプロセス（バックグラウンド処理）も存在する|

### 仮想サービスアカウント

実際のユーザーがログオンしたりパスワードを管理したりする必要がない特別なアカウント。

- 自動でランダムで複雑に設定されるため、パスワードクラックは実質不可能
- `NT SERVICE\サービス名` または `NT SERVICE\インスタンス名$名前`
    - mimikatzで`$`がついたユーザーのパスワードハッシュがダンプされることがよくある
- 具体例：`nt authority\mysql$sqlexpress`

## DLL（Dynamic Link Library）

UnixでいうShared Object。通常のDLLとリソースのみDLLがある。

- [DLL](../Cheatsheet/🪟Windows/Windowsの基礎・前提知識/Windows%20Internal.md#DLL)

通常のDLLは関数・クラス・データが定義されている。

### リソースのみDLL

リソースのみDLLにはUI要素のみが定義されている（文字列・アイコン・画像・メニュー名など）。

- 内部にIDがあり、例えばIDに対応する文字列が格納されている（ID: 3000 → Document 等）
- 同じUI要素であれば同じIDで定義されているため、多言語対応が可能（nameタグでIDを指定するとユーザー環境にあわせて切り替わる）
- 近年は表示言語用のライブラリを分離するため、EXE・DLL・リソースのみDLL/MUI（Multilingual User Interface）を分けている

## CIM・WMI

### CIM（Common Information Model）

「情報」の表現方法を共通化した業界標準規格。目的は、異なるベンダーの製品でも同じ方法で情報を取得・管理できるようにすること。

### WMI（Windows Management Instrumentation）

CIMをWindows用に実装したもの。Windowsシステムの情報を取得・管理するための統一的な仕組みで、Windows管理の中核であり重要な攻撃ベクターでもある。カテゴリごとに様々なクラスがある。

|カテゴリ|WMIクラス|取得できる情報|
|---|---|---|
|プロセス|Win32_Process|実行中のプロセス一覧、PID、メモリ使用量|
|サービス|Win32_Service|サービスの状態、起動方法|
|ハードウェア|Win32_BIOS|BIOS情報、バージョン|
|ネットワーク|Win32_NetworkAdapter|ネットワークアダプタ情報|
|OS|Win32_OperatingSystem|OSバージョン、インストール日時|
|ディスク|Win32_LogicalDisk|ドライブ容量、空き容量|
|ユーザー|Win32_UserAccount|ローカルユーザーアカウント一覧|

```powershell
Win32_Process クラス
│
├─ プロパティ（データ）
│   ├─ Name (String)           例: "chrome.exe"
│   ├─ ProcessId (UInt32)      例: 1234
│   ├─ SessionId (UInt32)      例: 1
│   ├─ Handle (String)         例: "1234"
│   ├─ CommandLine (String)    例: "chrome.exe --flag"
│   └─ CreationDate (DateTime) 例: 2025-10-15 10:30:00
│
└─ メソッド（操作）
    ├─ Create()                新しいプロセスを作成
    ├─ Terminate()             プロセスを終了
    ├─ GetOwner()              プロセスの所有者を取得
    └─ SetPriority()           プロセスの優先度を設定
```

## MMC（Microsoft Management Console）

Windowsの管理ツールを表示するための枠組み。MMCがスナップイン（.msc）を読み込んで以下のような管理を可能にする。

具体例：コンピュータの管理・ディスクの管理・イベントビューアー・サービス管理・デバイスマネージャー・証明書管理・グループポリシー管理（gpedit.msc）・Active Directoryユーザーとコンピューター

## VSS（Volume Shadow Copy Service）

- 🔗[Microsoft documentation page](https://docs.microsoft.com/en-us/windows-server/storage/file-server/volume-shadow-copy-service)

Windowsのストレージ管理機能で、指定されたボリュームの内容を自動的に複製し専用の領域に保管する機能。特定の時点におけるファイルのコピーを自動的に作成することで、ファイルの誤削除や上書き時の復旧が可能になる。

**ボリュームとは：** OSによって区画付けされた、ひとつながりの連続した記憶領域の管理単位。HDDやSSDの全体を1ボリュームとする場合と、複数のパーティションに区切ってそれぞれをボリュームとする場合がある。

## manifest

Windowsアプリケーションに関する情報を含むXML形式のテキストファイル。アプリケーションの属性や要求を指定し、UACプロンプトの表示やアプリケーションの自動昇格などの動作を制御する。

## 名前付きパイプ

パイプとは、プロセス間の入出力をつなげる仕組み。名前がついていないパイプは `|` で使われる。

```zsh
ls | grep xxx
```

名前付きパイプは `mkfifo` コマンド等で作られ、ファイルシステム上に存在する（ただしファイルエクスプローラーでは見えない）。データはDiskに保存されず、メモリのバッファ上を経由してプロセス間で直接やり取りされる。ファイルシステム上の名前は、プロセス同士が通信相手を見つけるための「待ち合わせ場所」の役割を果たす。

```zsh
# 名前付きパイプを作成
mkfifo mypipe

# ターミナル1で読み取り待機
cat < mypipe

# ターミナル2でデータ送信
echo "Hello" > mypipe
```

名前付きパイプを作成する側（パイプサーバー）とパイプを利用する側（クライアント）は同じシステム上になくてもよく、リモート間でもやり取り可能。

# Windows認証・セキュリティ

## プリンシパル

ほぼ直訳すると「セキュリティ主体」の意。認証・認可を受けるエンティティのことで、SIDが割り当てられる。

## SID・RID

SID（Security Identifier）とRID（Relative Identifier）は、Windowsのセキュリティモデルにおけるプリンシパルの識別に使われる重要な概念。LSASSが発行する。

### SID

Windows内でユーザーやグループ、コンピューターアカウントを**一意に**識別するための識別子。一度生成されると変更されない。🔗[well-known SID](https://learn.microsoft.com/en-us/windows/win32/secauthz/well-known-sids)がある。

```bash
# S-R-X-Y
S-1-5-21-3623811015-3361044348-30300820-500
```

- `S`：SIDであることを示す
- `R`（`1`）：版数を表す。常に1
- `X`（`5`）：SIDを発行したIdentifier Authority（5は"NT Authority"）
- `Y`（`21-3623811015-3361044348-30300820`）：ドメインまたはコンピューターを識別する部分（サブAuthority）。Built-in（組込）プリンシパルは`21`の部分が必ず`32`になる
- `500`：RID

### RID（Relative Identifier）

SIDの末尾の番号で、特定のアカウントやグループを識別する。作成されたユーザーは1000から順に割り当てられる（例：1001は2番目に作成されたユーザー）。

|RID|アカウント名|説明|
|---|---|---|
|500|Administrator|Windowsの管理者アカウント|
|501|Guest|ゲストアカウント|
|512|Domain Admins|ドメイン管理者グループ|
|513|Domain Users|ドメインユーザーグループ|
|514|Domain Guests|ドメインゲストグループ|
|1000以上|通常のユーザー|作成されたローカルユーザーやドメインユーザー|

ペンテストでは、SMBやLDAPを使ってSIDを取得し、RIDを列挙することでターゲットのユーザー情報を収集する。

## SAM・SYSTEM・LSA（認証ハイブ）

### SAM（Security Account Manager）

- 保存場所：`C:\Windows\System32\config\SAM`
- Windowsにおけるローカルアカウントに関する情報を保存するデータベース
- ==ローカルのユーザー名、パスワードハッシュ、セキュリティID（SID）==などの情報が含まれる
- SYSTEM権限/BackupPrivでのみアクセス可

### SYSTEM

- 保存場所：`C:\Windows\System32\config\SYSTEM`
- OSのハードウェア・ソフトウェア構成情報
- 暗号鍵（BootKey）を含む。このキーがSAMに保存されているハッシュを復号するために必要

### LSA（Local Security Authority）

Windowsのセキュリティポリシー・ログオン認証・アクセス制御などを管理するセキュリティサブシステム（プロセス名：`lsass.exe`）。SIDの生成も行う。

- 保存ファイル：SECURITYハイブ（`C:\Windows\System32\config\SECURITY`）
- 「LSA Secrets」として、サービスアカウントのパスワードやドメインユーザーのキャッシュ認証情報などが含まれる
    - ※ドメインアカウントの管理自体はActive Directory側で行われる

## ACL・DACL・ACE

ACLはアクセス制御のリストであり、DACLはそのリストの一種であり、ACEはACL内の個々のエントリ。

![ 300](../Images/Pasted%20image%2020230526163901.png)

$$アクセス制御の見た目$$

### ACL（Access Control List）

リソースへのアクセスを制御するために使用されるセキュリティメカニズム全体。

1. クライアントが対象のオブジェクトを使いたいとき、オブジェクトに対しアクセストークンを送る
2. オブジェクトはアクセストークンをACLと突合し、許可/却下する

DACLとSACL（System Access Control List）で構成され、SACLは自由に変更できない。

### DACL（Discretionary Access Control Lists）

ACLの中身。日本語訳は任意アクセス制御リスト。所有者によって自由に変更可能。

![ 400](../Images/Pasted%20image%2020230526164041.png)

### ACE（Access Control Entry）

ACLの中身で、「どのプリンシパルに」「どんな操作が」「許可/拒否されているのか」を定義する。

# Windowsスクリプト技術

## COM・DCOM・ActiveX

### COM（Component Object Model）

Microsoft社が提唱した、ソフトウェアの機能を部品化して外部から呼び出して利用する仕組みを定めた技術仕様。主にWindowsシリーズのOSやその上で動作するソフトウェアで利用される。COMの仕様に則って開発されたソフトウェア間は容易に連携して動作させることができる。特定のOSやプログラミング言語に依存しない仕様になっており、異なる言語やOSで開発されたプログラムを連携させることができる。

### DCOM（Distributed COM）

COMがローカル上のソフトウェア間で連携するための仕様であるのに対し、DCOMはネットワークを跨いでソフトウェア間で連携できるようにしたCOMの拡張版。通信はRPCを介して行われるため、TCP 135が使用される。

### ActiveX

Windows専用の技術で、アプリ間で機能やデータをやり取りするための仕組み。COMという仕組みを使ったコンポーネントの1つ。WordやExcel、Internet Explorerなど、Windowsアプリケーション同士がお互いの機能を使えるようにする技術。

主な用途：ファイル操作（読み書き・削除など）・レジストリの操作・Shell（コマンド実行）・ブラウザ制御・ネットワーク通信（XMLHTTP等）

## マクロ・VBA・VBScript・WSH

### マクロ・VBA

マクロは一連の操作を自動化する仕組み。Officeソフトでは、VBA（Visual Basic for Applications）がほぼ唯一の正式なマクロ言語。VBAマクロはActiveXオブジェクトとWindows Script Host（WSH）へのフルアクセスが可能であり、システムファイルの操作やネットワーク通信、プロセスの実行などが可能。命名規則はキャメルケースが多い（`HogeFoo`）。

#### VBファミリーにおけるイベント

JavaScriptのイベントと同義。何かされたときに自動で呼ばれるトリガー。命名規則はキャメル・スネークケース（`Hoge_Foo`）。

例：`Document_Open()` はドキュメントが開かれたときにトリガーし、ボディ部の処理を実行する。

```vb
Sub Document_Open()
	MyMacro
End Sub
```

### VBScript

VBAと同じくVisual Basicファミリー。VBAはOfficeソフト上で実行されるのに対し、VBScriptはデスクトップ上で実行される。VBAもActiveXやCOMオブジェクトを通じてWSHにアクセスできる（＝Shellの操作も可能）。

|比較項目|VBA（Visual Basic for Applications）|VBScript（VBS）|
|---|---|---|
|用途|Officeアプリ（Excel, Word, Outlookなど）の自動化|Windows全体でのスクリプト実行|
|実行環境|Officeアプリ内のエディタ（Alt + F11）|Windows Script Host（`.vbs` ファイル）|
|機能|豊富（UI、フォーム、イベント、Officeオブジェクト、ActiveX操作）|限定的（基本的なスクリプトとActiveX操作）|
|UIの作成|可能（UserFormなど）|不可能（フォームなし）|
|オブジェクトアクセス|ExcelのセルやWordのドキュメントなどを操作可能|それらを直接操作するにはOfficeを呼び出す必要あり|
|セキュリティ|マクロ設定で制限・無効化が可能|`.vbs`はダブルクリック即実行されるためマルウェアに悪用される|
|拡張子|`.xlsm`, `.docm`, `.pptm`などに内包|`.vbs` ファイル単体|

VBAでメモ帳を起動する場合：

```vb
Sub MyMacro()
	CreateObject("WScript.Shell").Run "notepad.exe"
End Sub
```

`WScript.Shell`＝WindowsシステムにアクセスするためのCOM（ActiveX）オブジェクト。`CreateObject`はグローバルオブジェクト。

VBSでメモ帳を起動する場合：

```vbscript
CreateObject("WScript.Shell").Run "notepad.exe"
```

ファイルに保存後、ダブルクリックで実行。

### WSH（Windows Script Host）

Windows OSに組み込まれているスクリプト実行環境。WSH用のスクリプトである `.vbs` や `.js` ファイルをダブルクリックで実行可能にする。

### WScript・CScript

どちらもWSHスクリプト（vbs, js）で使えるグローバルオブジェクト（≒インスタンス）。WScript・CScriptを使用してShellにアクセスすることなどが可能。実行時に `wscript.exe` や `cscript.exe` が動く。

|メソッド|説明|
|---|---|
|`WScript.Echo`|メッセージを出力（GUIまたはコンソール）|
|`WScript.Sleep(ms)`|指定したミリ秒だけスリープする|
|`WScript.Quit(code)`|指定したコードでスクリプトを終了する|
|`WScript.Shell`|Shellにアクセスする|

使用例：コンソールへの文字列表示

```vbscript
WScript.Echo "Hello, world!"
```

`test.vbs` で保存し、`cscript test.vbs` を実行するとコンソールに「Hello, world!」と表示。

|項目|WScript.exe|CScript.exe|
|---|---|---|
|名称|Windows Script Host（GUIモード）|Console Script Host（CUIモード）|
|動作モード|GUIベース（メッセージボックスなど）|コンソールベース（標準出力に表示）|
|適している用途|ユーザーとのGUI対話（MsgBoxなど）|バッチ処理やログ出力、スクリプトの自動実行向け|
|出力|`MsgBox` などウィンドウ表示|`Echo` はコマンドラインに出力される|
|既定のホスト|拡張子 `.vbs` や `.js` をダブルクリックで実行時に使われる|コマンドライン実行時に手動指定可能|

# ネットワーク・プロトコル

## msrpc・microsoft-ds・NetBIOS・mc-nmf

### MS-RPC（Port 135）

分散コンピューティング環境において、リモートのコンピュータ上のプログラム（手続き）を呼び出すために開発されたRPC（Remote Procedure Call）という機能を、Windows OS上に実装したもの。==SMB==やCIFS上でも利用されている。

### microsoft-ds（Port 445）

Windows 2000以降のWindows系OSで、NetBIOSを使用せずにファイルやプリンタを共有できる。

### NetBIOS

コンピューター同士が名前で互いを識別し、ファイル共有や通信を行うための仕組み。過去のWindowsシステムで使用されていたネットワーク通信プロトコルで、現在はSMBが主流。

### mc-nmf

mc-nmf（正確にはMC‑NMF / NMF）は「ADWS（Active Directory Web Services）向けのSOAPメッセージをフレーミングするためのトランスポート層」。ペンテストにおいて有用な情報を入手したりエクスプロイトにつながるものではないため、優先度は低い。

## ピボットとトンネリング

|技術|対象となるネットワーク|目的|
|---|---|---|
|ピボット（Pivoting）|同じネットワークセグメント内のPC|内部の他のPCやサーバーへ移動する|
|トンネリング（Tunneling）|異なるネットワークセグメント間|ファイアウォールやルーターを越えて通信する|

## トポロジー

ネットワークやシステムの構造や配置を表す用語。Active Directory（AD）の場合、トポロジーはドメインコントローラ（DC）間の接続と情報の同期方法を指す。

## REST API

Representational State Transfer API。state（状態）をXML・JSONなどで表現（representation）してやり取り（transfer）する。

特徴として、サーバーが前回のリクエストの内容を覚えていない（ステートレス）。そのため、クライアントは都度認証情報（JWTなど）を送る必要がある。なお、ステートレスの対義語はステートフルで、こちらはログインセッションなどの情報をサーバーが保持している。

## xmlnsにおけるリンク

URNであり、ロケーション（L）を示すのではなく、あくまでも識別子として使われる。そのため、実際にアクセスできないことも多い。

```xml
<?xml version="1.0" encoding="UTF-8"?>
<libraryDescription xmlns="http://schemas.microsoft.com/windows/2009/library">
</libraryDescription>
```

# Linux・OS共通

## SUID bit・SGID bit

- **SUID**：ファイルを作成したユーザーの権限でファイルを実行できるよう設定するフラグ。通常のユーザーがファイル所有者（通常はroot）の権限でファイルを実行できる。PrivEscのきっかけになりやすい
- **SGID**：Groupオーナーの許可を得て実行する。同じディレクトリに作成されたファイルには同じGroupオーナーが設定される

## UID・GID

|UID範囲|用途|例|
|---|---|---|
|0|スーパーユーザー（root）|root|
|1–99|伝統的なシステムユーザー|daemon, bin, sys|
|100–499（※または1–999）|OSが自動作成するサービスユーザー（非ログイン）|www-data(33), sshd, mail, mysql|
|500–999|（ディストロにより）追加のシステムユーザー|nginx, redis, postgres|
|1000–60000|一般ユーザー（ログイン可能）|user1(1000), user2(1001)|
|65534|nobody（制限付きユーザー）|nobody(65534)|

## WindowsファイルのLinuxからの指定ルール

Windowsでは階層の区切りとして `\`（バックスラッシュ）を使用するが、Linuxではバックスラッシュ自体がエスケープ処理をする意味の特殊文字なので、LinuxからWindowsのファイルやディレクトリを指定する場合は `\\` としてバックスラッシュをエスケープ処理する。

# 攻撃手法

## Exploit・Payload・Shellcode

|用語|定義|
|---|---|
|Exploit|脆弱性を悪用するトリガーコード全体。脆弱性を突いてターゲットに不正な操作をさせる|
|Payload|Exploitが注入・実行させる「目的の動作をするコード」。シェルを起動したり、逆接続したりする|
|Shellcode|Payloadの一種。一般的に「シェルを起動するバイナリコード（x86/ARMなど）」を指す|
