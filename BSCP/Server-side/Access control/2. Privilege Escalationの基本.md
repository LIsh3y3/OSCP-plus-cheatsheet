## プラットフォームの誤設定によるアクセス制御の不具合

### プラットフォームの誤設定を悪用する方法

#### ①URL-based access controlのバイパス

1. 権限上許可されないHTTPリクエストを確認する
```http
GET /admin HTTP/2
```
- "DENY"

2. `X-Original-URL` や `X-Rewrite-URL`を試す。成功したらPrivEsc
```http
GET / HTTP/2
...
X-Original-URL: /admin
```

- *＊*：パラメタ部分は本来のリクエスト行に記載する。`X-Original-URL`でパラメタを指定すると"Missing parameter"エラーが表示されるので。
- [Lab: URL-based access control can be circumvented](https://portswigger.net/web-security/access-control/lab-url-based-access-control-can-be-circumvented)

#### ②Method-based access controlのバイパス

1. [2. Privilege Escalationの基本](2.%20Privilege%20Escalationの基本.md#①URL-based%20access%20controlのバイパス)のステップ2までは同じことをする
2. 高権限のアカウントでログインし、高権限垢で可能なアプリの機能を調べる

```http
POST /admin-roles HTTP/2
...
username=carlos&action=upgrade
```

3. 元のリクエストメソッドと異なるメソッドを選択しリクエストしてみる

- 元がPOSTの場合
```http
GET /admin-roles?username=wiener&action=upgrade
```

- [LAB: Method-based access control can be circumvented](https://portswigger.net/web-security/access-control/lab-method-based-access-control-can-be-circumvented)

###### プラットフォームの誤設定の原理

- プラットフォームとは、OSやDBMS、フレームワークなどWebアプリケーション開発に使用するもの
- プラットフォームのアクセス制限で以下のように脆弱な設定をしていることがある
```
DENY: POST, /admin/deleteUser, managers
```
- これは`/admin/deleteUser`へのPOSTリクエストを`managers`は不可能にする設定

- `X-Original-URL` や `X-Rewrite-URL`といいった非標準のHTTPヘッダは元々のリクエストのURLを上書きできる
	- 処理順としては①X-Original-URL → ②元々のリクエスト行
	- よって、①`/admin/delete` + ② `/?username=carlos` ＝ `/admin/delete/?username=carlos`となる

## URLマッチングの不一致によるアクセス制御の破綻

### URLマッチングの不一致を悪用する方法

- 元のPathを`admin/deleteUser`とする

- Caseの変更
	- `/ADMIN/DELETEUSER`

- 無意味な拡張子の付与
	- `/admin/deleteUser.anything`

- 末尾へスラッシュの付与
	- `/admin/deleteUser/`

###### URLマッチングの不一致でアクセス制御をバイパスできる原因

- 結局同じエンドポイントへアクセスするのに、アクセス制御が異なるエンドポイントと解釈し適用されない。
- フレームワークによっては勝手にパスを変更・抽出することがある
	- 例えばJavaのSpringフレームワークでは、`useSuffixPatternMatch`オプションが有効だと、任意のファイル拡張子を持つパスでも、拡張子のない同等のエンドポイントへ誘導するようになる
	- これはSpring5.3以前ではデフォルトで有効
