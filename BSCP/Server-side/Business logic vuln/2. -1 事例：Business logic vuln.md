## クライアント側の制御に対する過度の信頼

###### クライアント側の制御に対する過度の信頼を悪用する方法

- [3. 多要素認証の脆弱性](../Authentication/3.%20多要素認証の脆弱性.md#ログイン状態のみアクセスできるページへの直接アクセス)
- クライアント側から送信される値をProxyで改変して送信する

###### クライアント側の制御に対する過度な信頼の何が問題なのか

- データの流れは、クライアント(ブラウザなど)-> Proxy -> サーバ
- Proxyでリクエストをキャプチャして編集すればクライアント側での制御は簡単にやぶれる
- 本当はサーバ側で検証しないといけない。

## 予想してない入力を処理できない


### 整数型のオーバーフローを起こす

###### オーバーフローの確認

1. 限界の入力値（99まで選択できる場合は99を選択）したPOSTリクエストを、Payload type: Null payloadsで、"Continue indefinitely"を選択してIntruderを走らす
```http
POST /cart HTTP/2
...
productId=1&redir=PRODUCT&quantity=99§§
```

2. `/cart`へ移動し、送信している間ページを定期的にリロードし、オーバーフローすることを確認する
	- 価格がマイナスになるなどの異常値が表示されること

###### オーバーフローを悪用した目的達成

3. 一旦カートの中身をクリアする
4. 目的商品の価格がマイナスの状態になるよう調整するために必要なリクエスト数を割り出す

- ==整数型オーバーフロー時の遷移==：
```
0 → 2,147,483,647 > 2147483647 + 1 = - 2147483647 > -2147483647 → 0
```
- LABで323とあるが、これはオーバーフローを起こす値が2,147,483,647 + 1なので、2,147,483,647 ➗ (商品価格：133700 ✖︎ 99)で約162~163。- 2,147,483,647からは0に戻っていくので、なるべく0に近いマイナスの値にする。162✖︎2= 324でまた正の数になるので、そのぎりぎり手前の323

5. ステップ1のリクエストに対して以下に設定してIntruderを実行する
	- Payload type: Null paylaods
	- Generate 323 payloads
	- Resource pool -> Maximum concurrent request 1
	- (Pro のLive taskは停止しておく)

4. 仕上げに数量を調整してPOSTし、他の商品もcartに加えるなどして、なるべく0に近い+の値にする

- [Lab: Low-level logic flaw](https://portswigger.net/web-security/logic-flaws/examples/lab-logic-flaws-low-level)

### トランケーション(切り捨て)を起こす

###### Registerフォームでのメールアドレスのトランケーション例

- mail: `username@subdomain.domain`

1. `username`を異常に長い文字列にしてRegisterし、ログインする
2. ログイン後、自身のメールアドレスが切り捨てられていることを確認する
3. `very-long-string@right-domain.attacker-domain`とし、切り捨て後が`very-long-string@right-domain`となるようにする
- `right-domain`は`attacker-domain`のサブドメインとなるので、攻撃者のドメインにはメールが届く。

- [Lab: Inconsistent handling of exceptional input](https://portswigger.net/web-security/logic-flaws/examples/lab-logic-flaws-inconsistent-handling-of-exceptional-input)

###### トランケーションが脆弱性となる原理

- とても長い文字列を入力すると一部が切り捨てられる
- 切り捨て前は異なる値でも、適切な箇所で切り捨てられればまったく同じ値をバックエンドのDBに登録させることができる

###### ⭐️レスポンスで確認すること

- 入力するデータに課される制限はなにかあるか？
- 制限を超えるとどうなるか？
- 入力に対して何らかの変換や正規化が行われているか？

###### どのような場合に予想してない入力を処理できないのか

- 予想してない入力を安全に扱えないフォームが１つでもあれば他のフォームにも同じ問題があると考える。
- ユーザが入力するであろう値のあらゆる可能性を考慮できてない
	- 送金額に負の値を入力するなど
- これがクライアント側の制御によってブロックされていると、ロジックの欠陥を見落としてしまう
- 他にも、数値が32bitの上限を超えると*オーバーフロー*で挙動がおかしくなり、マイナスからカウントを始めることがある
