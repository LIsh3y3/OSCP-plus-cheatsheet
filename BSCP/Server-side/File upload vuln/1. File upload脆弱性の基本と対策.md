### File upload脆弱性のImpact

- ❌ファイルの拡張子が検証されない → サーバ上に[1. File upload脆弱性の基本と対策](1.%20File%20upload脆弱性の基本と対策.md#Web%20Shell)をuploadされ、RCE
- ❌ファイル名が検証されない → 重要なファイル(SSHの`authorized_keys`等)を上書きされる
- ❌[Path traversal](../Path%20traversal/Path%20traversal.md)にも脆弱　→ 予期しない場所へのファイルアップロードの誘発
- ❌ファイルサイズが検証されない　→ DoS攻撃

### 事前知識：サーバは静的なファイルをどう処理するのか

1. リクエストのパスを解析する
2. 事前に用意した拡張子とMIMEタイプのリストによって、比較する
3. 以降のサーバの動きはファイルタイプとサーバの設定による
	- non-executable(imageなど) → ファイルの内容をHTTPレスポンスで返すだけ
	- executable(phpなど) + ファイルを実行する設定 → スクリプト実行前にHTTPリクエストのヘッダとパラメータに基づいて変数を割り当てる。結果をHTTPレスポンスで返す
	- executable + ファイルを実行しない設定 → 通常はエラー。しかし、ファイルの内容をplain textで応答＝🚨[1. 情報開示の基本と対策](../Information%20disclosure/1.%20情報開示の基本と対策.md)

### 事前知識：file uploadにて重要なヘッダの基本

#### Content-Type in Request

- fileタイプの検証に使用される。サーバに送信されたデータのMIMEタイプを知らせる
- 単純なテキストのみのHTMLフォーム　→　`application/x-www-form-url-encoded`
- imageやPDFなど大きなサイズのバイナリ → ⭐️`multipart/form-data`
- 💡クライアントが送信したデータで、ブラウザが認識したデータの形式（サーバが意図しているデータの形式ではない）
#### Content-Disposition

- 入力フィールドに関する基本的な情報。データの取り扱い方法を指示するヘッダ。
- このヘッダに注目することで、どの入力フィールドを指しているのかがわかる
	- `name="..."`：どの入力フィールドか、を指している
	- `Content-Type: ....`：

###### 主な使い方

◎フォームデータの送信 (`form-data`)
- クライアントがサーバーにデータを送信する際、どのフィールドに該当するデータなのかを示す。(↓例)
```
Content-Disposition: form-data; name="username"
```
- これにより、`username` というフィールドに入力されたデータがサーバーに送信されることが分かる

⚪︎レスポンス時の添付ファイル指定 (`attachment`)
- サーバーがブラウザに対して、ファイルをダウンロードさせたい場合に使用される(↓例)
```
Content-Disposition: attachment; filename="example.txt"
```
- これにより、クライアント側で `example.txt` という名前でファイルがダウンロードされる。        

#### Content-Type in Responce

- サーバがどのタイプのファイルを期待しているかがわかるかも。
- Webアプリのコードにより明示的に設定されている場合はそのContent-Typeが表示
- 明示的に設定されてない場合、事前に設定された拡張子とMIMEタイプのリストが返される。

### 事前知識：Webアプリのユーザーについて

- セキュリティ上、Webのユーザーは権限が低く設定されていることが多い（が、たまにゆるいことがある。）

- Debian, Nginxなどは`www-data`。
- Apacheは`apache`
- WindowsのWebサーバー`IIS`(Internet Information Service)においては、以下2つがアカウント
	- Network Service（旧）
	- IIS Application Pool Identity（新）（※）
		- (※：Application Pool＝Webアプリを独立して管理する仕組みで、安定性・セキュリティ向上に役立つ)
		- 例：`IIS APPPOOL\DefaultAppPool` のような形式で、ACLを使ってアプリごとに適切な権限が付与される

- その他、"独自のWebサーバを含むプログラミング言語"では、面倒な権限設定を避けるため、アプリをroot/administratorで実行してしまうことがある。
	- Python : Flask
	- PHP : built-in server ... 等
- →常に、ファイルアップロードの脆弱性で root/Administrator 権限を利用できるかを確認すべき

### Web Shell

- Web shellは悪意のあるスクリプトで、攻撃者は適切なエンドポイントにHTTPリクエストを送るだけでRCEが可能になる

- ファイルの読み出し
```php
<?php echo file_get_contents('/[path to target file]'); ?>
```

- 汎用性(高)：任意のコマンドを実行させる
```php
<?php echo system($_GET['command']); ?>
```
- このスクリプトをアップロードし、BurpもしくはURLのパスを編集する
```http
GET /example/exploit.php?command=[system command] HTTP/1.1
```

- 📝：PHPに限らず、ASP等でも同様の方法でWebshellを獲得できる。
	- Classic ASP : `asp`
	- ASP.NET: `aspx`

### 🛡️File upload vulnの対策

- [ ] 悪意のあるファイルをuploadさせない
	- `filename`の拡張子でフィルタリングする。❌ブラックリスト⭕️ホワイトリスト
	- MIMEタイプを比較する
	- `Content-Type`
- [ ] `filename`パラメータの値をランダム化し、表示しない
	- 悪意あるファイルをアップロードされても実行されない
	- ファイルが上書きされることも防ぐ
- [ ] ユーザがアップロードできるディレクトリを制限するためにアクセス制限をする
- [ ] ファイル名に[Path traversal](../Path%20traversal/Path%20traversal.md#Traversal%20sequences)が含まれないように検証する
- [ ] アップロードされたファイルがサーバに永久に保存される前に検証する
	- アップロードされたファイルはサンドボックス化したtmpディレクトリなどに一旦保存
	- ファイルタイプ、マルウェア、コンテンツなど異常がないか検証
	- 検証できたら本番ディレクトリへと移行
- [ ] ファイルアップロードのためのフレームワークを利用する
