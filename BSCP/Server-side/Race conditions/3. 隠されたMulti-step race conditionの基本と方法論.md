- Multi-step = サブステートの悪用

## 基本

- 1つのリクエストがバックエンド(≒隠された場所)でマルチステップシーケンス全体を開始し、リクエスト処理が完了する前にアプリケーションを複数の隠されたステートに移行させるか、または終了させる。
- これらを "サブステート "と呼ぶ。([📕](../../Misc/📕.md#ステート))

###### 悪用の概要

1. サブステートを経由してそうな、一つまたは複数のHTTPリクエストを見つける

	 - これらのHTTPリクエストが同一のデータにアクセスし、同時に一連のサブステートを経由すると、システムの内部状態が時間的に変化することがある

2. 時間的な変化や途中の状態を攻撃者が利用する

	 - サブステートを悪用することで、(multi-stepのロジックの欠陥で一般的な)"time-sensetive"な検証を突破し、[2. Limit overrun](2.%20Limit%20overrun.md)よりも重大なインパクトを与えることが可能


- 以下は典型的な隠されたmulti-step
```python
session['userid'] = user.userid 
if user.mfa_enabled: #親ステート
	session['enforce_mfa'] = True 
	# MFAコードを生成しユーザに送信する処理(サブステート)
	# MFAコード入力フォームにリダイレクトする処理(サブステート)
```
- ユーザが一時的に有効なログインセッションを持つが、MFAがまだ実施されていないサブステートを通過している
- MFA認証完了後にアクセスできるエンドポイントへのリクエストと、同時にログインリクエストを送信することで、MFA認証を突破できるかも。

### ① 衝突の可能性を予測する

- すべてのエンドポイントをテストすることは現実的ではない。
- 対象サイトを探索した後、以下の質問を自問自答する

#### テストすべきエンドポイントの選定

1. このエンドポイントのセキュリティは重要か?
	- 多くのエンドポイントは重要な機能に触れていないので、テストする価値はない。

2. 衝突の可能性はあるか？
	- 例えば、Cookieなどの値(sessionIDやuserIDを含む）を削除して確認する。

###### 衝突が成功する要件

- 衝突を成功させるには、通常、同じレコードに対する操作をトリガーする2つ以上のリクエストが必要。
- 例えば、パスワードリセットの実装の以下のバリエーションを考えてみる：
![[Pasted image 20231214215829.png | 400]]

- 最初の例では、2つの異なるユーザーに対して並行してパスワードのリセットを要求しても、2つの異なるレコードに変更を加えることになるため、衝突が発生する可能性は低い。
- しかし、2番目の実装では、2つの異なるユーザーに対するリクエストなのに同じレコードを編集しているので衝突の可能性が高い
	- これは、異なるレコードが同じidによって(上記ではsessionidが共通)一つのレコードになっていることが要件

### ②手がかりを探る

1. Burp Repeaterですべてのリクエストをグループ化し、[Sending over separate connections](https://portswigger.net/burp/documentation/desktop/tools/repeater/send-group#sending-requests-in-sequence:~:text=Sending%20over%20separate%20connections)を選択
	- まずはエンドポイントの「通常」の挙動をベンチマークする。
	- これはmulti-stepプロセスを必要とする脆弱性のテストを簡単にする目的で使用される

2. Sending over single connectionsを選択しリクエストごとにJitter(遅延の揺らぎ）がないか確かめる
	- もしあれば、[4. Multi-endpoint race condtion](4.%20Multi-endpoint%20race%20condtion.md#①接続をあたためる)を試し、Jitterが少なくなるのを確認する
	- 少なくならなければ、[4. Multi-endpoint race condtion](4.%20Multi-endpoint%20race%20condtion.md#②レートやリソース制限の乱用)

3. Single packet attack(Send group in parallel)を行う
4. ベンチマーク中に観察されたものから何らかの形で逸脱していないか探す
	- どんな違いでも手掛かりになる
	- 1つまたは複数のレスポンスの変化
	- メールの内容の違いやアプリケーションの動作の眼に見える変化などの2次的な影響も
