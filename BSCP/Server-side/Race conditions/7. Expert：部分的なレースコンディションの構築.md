
- アプリケーションがユーザ登録などのオブジェクトを作成するとき、ほとんどがmulti-step
- multi-step = 悪用可能な状態が存在すると考える
- トークンやAPI keyが生成されてからDBに格納されるまでの一瞬の遅延が悪用できる

#### 部分的なレースコンディションの構築方法方法

- 難：[Lab: Partial construction race conditions](https://portswigger.net/web-security/race-conditions/lab-race-conditions-partial-construction)

- 基本は[3. 隠されたMulti-step race conditionの基本と方法論](3.%20隠されたMulti-step%20race%20conditionの基本と方法論.md)に従う

1. オブジェクトが作成されそうなエンドポイントを見つける
	- ユーザ登録など
	- [JavaScriptとJSON](../../Misc/JavaScriptとJSON.md)ファイルでconfirmフォームを動的に生成している場合もある。(興味を引くリクエストがあれば必ずチェックすること)

2. 未設定の値が存在しそうなフォームやリクエストを探す
	- API keyやtokenなど
	- JSファイルからフォームが動的に生成される場合、フォームの構成を読み取ることができる
	- フォームの構成がわかれば、そのフォームから発されるリクエストも見当がつく

3. フォームから発されるリクエストを予測＆操作し、挙動をベンチマークする
```http
POST /confirm?token=1
	or
POST /confirm?
	or
POST /confirm?token=
	or
POST /confirm?token[]=
```

4. 3でわかった挙動からrace windowや構造を推察する
	- `Forbidden`エラー：開発者が脆弱性対策としてアクセス禁止にしたのかもしれない＝狙い目

5. ユーザ登録と確認の2つのリクエストをRepeaterでグループ化し、Send group系で挙動を見る
```http
POST /register HTTP/2
...

username=test&email=test%40ginandjuice.shop&password=test
```
```http
POST /confirm?token=1 HTTP/2
...
```
- このとき、基本的なエラーは回避しつつリクエストする(ユーザ名を都度変更するなど)
- ==レスポンスタイムを必ず確認すること==
	- どのリクエストが先に処理が終わるかわかる。
	- もしユーザ登録よりも先にconfirmの処理が速いのであれば、未設定状態のパラメータが存在するかも。

6. 処理時間が異なる場合、Race windowを揃える必要がある
![[Pasted image 20231216202149.png | 400]]
- Turbo Intruderの`race-single-packet-attack.py`を改変する

- 今回は/confirmの処理が早かったので、[4. Multi-endpoint race condtion](4.%20Multi-endpoint%20race%20condtion.md#②レートやリソース制限の乱用)より、早い方のリクエスト(confirm)に遅延を起こす
- 今回は登録済みユーザはエラーが表示されるので、毎度異なるユーザ名としている
- パスワードはアカウント作成成功時のため、固定にしておくこと。

###### この脆弱性が見られるケース

- ユーザ登録などのAPIキー
- ユーザ登録などのパスワード
	- ⚠️：パスワードはハッシュ化されているので、攻撃者は未初期化のパスワードハッシュ値と一致するハッシュ値を送信する必要がある

###### この脆弱性の原因

- このため、オブジェクトが悪用可能な状態が発生する可能性がある
#### 具体例

- 新規ユーザーを登録するケース
- ユーザ作成と[📕](../../Misc/📕.md#API%20key)の設定が別々のSQL文で行われているとする
- ユーザは作成されているが、APIキーがまだ設定されていない間(race window)を突く

- APIキーが未設定　＝　空文字 or NULL
- この未設定のAPIキーと一致する入力を引き渡す。

- アプリケーションはvictimユーザがAPIキーを未設定の状態のとき、攻撃者が送信した空文字 or NULLと一致すると判断する

- 例えば、`api-key[]=`というパラメータを付けてリクエストを送る。これにより、`api-key`パラメータが空の配列（つまり未初期化）と認識される可能性がある
```http
GET /api/user/info?user=victim&api-key[]= 
HTTP/2 Host: vulnerable-website.com
```
 
- システム側では、通常はAPIキーが未設定の状況を認めず、適切に設定されたAPIキーを必要とする。
- しかし、このrace conditionを利用したリクエストにより、システムが誤ってそのリクエストを「APIキーが設定されている（ただし、値は空）」と誤認し、そのリクエストを認証してしまう

###### なぜ`api-key[]= `がいいのか

- PHPの特殊なオブジェクトの文法に関係する
	- `param[]=foo` は`param = ['foo']`と同じ
	- `param[]=foo&param[]=bar` は `param = ['foo', 'bar']`と同じ
	- `param[]` は `param = []`と同じ

- つまり、`api-key[]=(空白) `で`api-key['(空白)']`と設定できる。
- 加えてURLの形式的にもこれがいい

- Ruby on Railsでも同じようなことができる
- クエリパラメータもしくはPOSTのパラメータにキーは指定するが値は指定しないことで、以下のようなオブジェクトが作成される
```ruby
params = {"param"=>{"key"=>null}}
```
