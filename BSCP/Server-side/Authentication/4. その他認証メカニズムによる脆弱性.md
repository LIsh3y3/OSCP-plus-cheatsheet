## ログイン状態を保持する

- ”Keep me logged in" / "Remember me"等

- "Remember me"トークンを生成することで実装される
- トークンは永続的なCookieに保存される

- このCookieを獲得することで、ログインプロセスをバイパスできる。

### Cookie生成の脆弱性とその悪用方法概要

#### 攻撃者がアカウントを所持している場合

###### 脆弱性

1. Cookieが推測できてしまう
	- CookieがusernameやTimestamp、あるいはpasswordなどの「静的」な値の予測可能な連結によって生成される場合、Cookie生成の計算式が推測可能

2. Cookieの単純な暗号化
	- Base64のような単純な双方向暗号化
	- ソルト&ストレッチングなしのハッシュ

##### 悪用ステップ

1. 攻撃者自身のCookieを研究し、Cookie生成の計算式を推測する
2. ターゲットユーザのCookieをブルートフォースし、(Cookieからクレデンシャルを入手するなどして)アクセスする
	- ソルト&ストレッチングなしの単純なハッシュ化の場合は単にWordlistをハッシュ化してブルートフォースすればよい

#### インターネットに公開のハッシュを使用している

- よく知られたパスワードをハッシュ化したリストがインターネットにある。
- ハッシュ化されたクッキーの中にパスワードがあれば、インターネットで検索するだけでパスワードがわかる


#### 可能性低: 攻撃者がアカウントを所持できない場合

- オープンソースで試験は出題されない

###### 脆弱性

1. Cookie生成の詳細がドキュメントとして外部に公開されている
	- Webサイトがオープンソースのフレームワークを使って構築されている場合など

2. XSSの脆弱性がある

###### 悪用方法

- ドキュメントからCookie生成方法を確認し、ターゲットユーザが使用しているCookieを生成する
- [6. XSSの悪用による3つの目標達成](../../Client-side/XSS/6.%20XSSの悪用による3つの目標達成.md#Cookieの窃取)

### Remember me Cookieのブルートフォース実践

1. 攻撃者自身のアカウントでログイン→"Stay-logged-in"ボタンにチェックを入れ、Cookieを発行
2. CookieをInspectorでDecode
3. Cookieの暗号方式を確認
	- この場合はHashとBase64

4. IntruderのPayload processingでpayloadに対するルールを設定してブルートフォース

## パスワードをリセットする

### URLを使用してパスワードをリセットするケース

```http
POST /forgot-password?temp-forgot-password-token=TOKEN HTTP/2
...
temp-forgot-password-token=TOKEN&username=carlos&new-password-1=...
```

###### リセット用URLを動的に生成している場合はHHI攻撃

- [Password reset poisoning](../../Advanced/HTTP%20Host%20header%20attacks/Password%20reset%20poisoning.md)

###### パスワードリセットのPOSTリクエスト送信時にトークンを検証しない

1. 攻撃者が自分のアカウントへログインし、パスワードリセットを実行する
2. トークンの値を削除してもパスワードリセット機能が動作することを確認
```http
POST /forgot-password?temp-forgot-password-token=TOKEN HTTP/2
...
temp-forgot-password-token=TOKEN&username=carlos&new-password-1=...
```
	↓
```http
POST /forgot-password?temp-forgot-password-token= HTTP/2
...
temp-forgot-password-token=&username=carlos&...
```

3. リセット対象を被害者へ変更

## パスワードを変更する

###### パスワード変更の基本

- パスワードの変更には、現在のパスワードと新しいパスワードを2回入力することが多い
- ユーザ名と現在のパスワードが一致するかどうかをチェックする通常のログインページと同じプロセス
- 従って、これらのページは他のログインバイパステクニックに対して脆弱である可能性がある
### Password brute-force via password change

- 例えば、ユーザ名が`hidden`フィールドにある場合、ユーザ名を任意のユーザに編集できる可能性がある
- [2. パスワードログインの脆弱性](2.%20パスワードログインの脆弱性.md#Usernameの列挙)をし、
- これを悪用してユーザ名を列挙し、パスワードを総当たりできる

###### 仕様

- ログアウトするのは誤ったパスワード + 変更後のパスワードと再確認のパスワードが一致する時
- 変更後のパスワードと再確認のパスワードが一致していると、誤ったパスワードを入力した時点でログアウトしてしまうのでIntruderが機能しない。
- 変更後のパスワードと再確認のパスワードが異なるとき、誤ったパスワードを入力するとログアウトせずにエラーメッセージが表示される仕組み。
	- このとき、"New passwords do not mutch"と表示されるので、これをIntruderのsettings-> Grep Matchに入力
