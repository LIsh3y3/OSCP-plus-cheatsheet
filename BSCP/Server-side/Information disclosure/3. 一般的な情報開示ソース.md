## ディレクトリ一覧

###### ディレクトリ一覧をみる方法

- ディレクトリを探索し、アクセスしてみる。インデックスページが表示されずにディレクトリ一覧が表示されてないか確認する。
###### ディレクトリ一覧が表示される原理

- Webサーバは特定のディレクトリにインデックスページ(`index.html`や`index.php`)が存在しない場合、そのディレクトリのファイル一覧を表示する設定がある。
- 開発者が誤ってtempファイルやクラッシュダンプ(システムエラー時に出力される、問題解析用の詳細な情報）などの機密情報をそのディレクトリに保存した場合に情報開示となる
- アクセスコントロールがされてない場合は特に危険

###### 方法２

以下の関数をConsoleにペーストして実行することで、ページ内のコメント全て取得
```js
(function() {
    var comments = [];
    var treeWalker = document.createTreeWalker(document, NodeFilter.SHOW_COMMENT, null, false);
    while (treeWalker.nextNode()) comments.push(treeWalker.currentNode.nodeValue);
    console.log(comments);
})();
```

###### エラーメッセージ表示方法

- 無効なパラメータの入力

## デバッグのデータ

###### デバッグデータ閲覧方法

- デバッグ情報が記録されたファイルにアクセスする。
###### デバッグデータの活用方法

- Webアプリを操作するために有効なPayloadを構築する手かがりとする
- セッション変数の値を操作し、セッションを乗っ取ったり、特権昇格を行う
- 暗号化を復号する
###### デバッグデータからわかること　

- ユーザの入力で変更可能な主要なセッション変数の値
	- 権限レベルをセッション変数としていることもある

- バックエンドを構成しているホストの名前と認証情報
- サーバ上のファイル名とディレクトリ名
- クライアント経由で送信されるデータの暗号化キー
###### 具体例

- デバッグログが"User input "test" was rejected by our anti-SQL injection filter"だとする
- 攻撃者は"test"をSQLiの一部として処理しているとわかる

## ユーザのアカウントページ

###### ユーザのアカウントページへアクセスする方法

- パラメータをユーザごとに割り当てられたものに変更してアクセスする。
- アカウントページか、もしくはメールアドレスなど個々の値がvictimユーザのものだと確認する
- 例
```http
GET /user/personal-info?user=carlos
```
###### ユーザのアカウントページから情報開示へとつながる原理

- このように単純にアカウントページを表示させることはなくても、メールアドレスなど個々の値を表示してしまうことがある。
- アクセス制御がきちんとされてないとき
- ロジックの欠陥

## バックアップファイルによるソースコード開示

###### バックアップファイルを探す方法

- シンプルにバックアップファイルを探すか、[3. 一般的な情報開示ソース](3.%20一般的な情報開示ソース.md#エラーメッセージ)で構成情報を明らかにし、オープンソースがあればソースコードの一部を確認できる

OR

1. ソースコードファイルを参照している部分を探す

2. バックアップファイル名と予測されるものをGETリクエストする
	- `example.php~`
	- `example.php.bak`
	- etc
###### バックアップファイルが閲覧できる原理

- テキストエディタはオリジナルファイルの編集中に一時的なバックアップファイルを作成する
	- `~`の付与： `index.php~`
	- 別の拡張子の付与 : `index.php.bak`

- 脆弱なテキストエディタはバックアップを外部からアクセス可能なディレクトリに保存する
- 本来、ソースコードが保存されたファイルへの参照をしてもソースコードそのものが帰ってくるのではなく、処理後の結果がHTMLで返される。しかし、バックアップならソースコードがそのまま返される。
- 脆弱なWebサーバはバックアップへのアクセス制御をしていないし、定期的にバックアップを削除することもない。

###### ソースコードの重要性

- アプリケーションの動作を理解し、インパクトの大きい攻撃を構築できる。[1. 安全でないデシリアライズの基本と対策](../../Advanced/Insecure%20deserialization/1.%20安全でないデシリアライズの基本と対策.md)へ展開するなど
- 機密データがハードコードされていることがある
	- バックエンドのホストにアクセスするためのAPIキーや認証情報など

## 安全でない設定

###### 安全でない設定から情報を閲覧する方法

- TRACEメソッド(例)
```http
TRACE /admin HTTP/2
```
- [3. 一般的な情報開示ソース](3.%20一般的な情報開示ソース.md#デバッグデータからわかること)にもあるように、権限レベルをユーザが操作可能なセッション変数で操作していることがわかれば、そこを操作する
or
- エラーメッセージ等で明らかにした構成情報から、サードパーティの技術の中で脆弱性につながりそうな設定を探す

###### 安全でない設定が脆弱性として表出してくる原理

- サードパーティの技術を使用していて、膨大な設定項目を理解していない場合
- HTTP `TRACE`メソッドなどのデバッグオプションを無効にし忘れているとき

## バージョン管理履歴

###### バージョン管理履歴の閲覧方法

1. `/.git`にアクセス可能か調べる
2. `.git`ディレクトリ全体をダウンロード
```zsh
wget -r https://example.com/.git
```

3. ターミナルからGit guiを開き、でダウンロードしたディレクトリを開く
```zsh
git gui
```
![[Pasted image 20231205212418.png | 400]]

4. 最新コミットを訂正をするなど、情報を暴いていく
![[Pasted image 20231205212834.png | 400]]

###### バージョン管理履歴へアクセスできてしまう原理

- 全てのWebサイトはGitのようなバージョン管理システムを使用している
- Gitの場合、デフォルトではGitプロジェクトのすべてのデータを`.git`というフォルダに保存しているが、たまにWebサイトが本番環境でこのディレクトリを公開することがある
- ソースコード全体にアクセスすることはできないかもだが、diffを比較することで、コードの小さな断片を読むことができる
