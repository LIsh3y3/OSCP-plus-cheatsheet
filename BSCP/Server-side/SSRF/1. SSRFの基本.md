###### SSRFとは

- Server-side request forgery = サーバからのリクエストですよと偽造すること
- システム設計と「暗黙の信頼」が原因
　
![[Pasted image 20231130105100.png | 600]]
###### SSRFの被害

- 外部公開サーバ上の脆弱なWebアプリを介して、アプリからしか本来はアクセスを許可されていない内部のサーバなどにアクセスするされる
- 外部公開サーバを操作して攻撃者側が用意したサーバに通信させ機密情報を窃取される
- 攻撃者に任意のコマンドを実行させられる
- 「暗黙の信頼」を悪用し他のシステムへ攻撃を展開される

---
## SSRF脆弱性の探し方

#### リクエストにURLの一部分だけがある

- ホスト名やURLのパスの一部だけをリクエストに組み込んでいる
- サーバはリクエストの内容を受け取り、完全なURLを生成する。
- ==URL全体を操作できるわけではないので、悪用の範囲は限られる==
###### 例

- 通常
```http
GET /fetchUserProfile?userId=1234 HTTP/1.1
Host: application.com
```
- サーバ側で入力をURLへ組み込み、完全なURLにて送信する
	- `http://application.com/user/1234`

- 悪用
```http
GET /fetchUserProfile?userId=evil-net HTTP/1.1
Host: application.com
```
- サーバ側で入力をURLへ組み込み、攻撃者の意図する宛先へアクセスしてしまう
	- `http://application.com/user/evil-net`

#### データフォーマット内のURL

- 詳細は：[2. XXE攻撃によるファイル取得とSSRF攻撃](../XXE%20injection/2.%20XXE攻撃によるファイル取得とSSRF攻撃.md#XXEによるSSRF攻撃の実行)

#### RefererヘッダーによるSSRF

- 実際の例については[3.  Blind SSRF](3.%20%20Blind%20SSRF.md)を参照。事実上Blind SSRF。結果を見ることはできない。
- WebアプリによってはSEOのため、サーバサイドの分析ソフトウェアがReferer URLを訪問することがある。
- robots.txtの設定が緩いことがある

### サーバ自身に対する攻撃

#### サーバ自身に対する攻撃の方法

1. URLを`http://localhost`に変更して反応があるかを確認
2. Administratorなどjuicyな情報が存在するか確認するため、サブディレクトリのブルートフォースを行う
	- `http://localhost/§example§`

3. レスポンスが表示されたらそのディレクトリに対して操作を試す
4. Unauthorizedエラーが表示されたら、REST操作のURLを入手し、ステップ1のURLに追加する
	- REST操作のURL例 :`https://example/admin/delete?username=carlos`
	- →`http://localhost/admin/delete?username=carlos`

###### 脆弱性の原理

- アクセス制御のチェックポイントが別のコンポーネントに実装されている。サーバからの要求はチェックしない設計になっている場合
- 災害復旧のために、アプリケーションは管理者権限でのアクセスにログインを求めない設計になっている場合。これは「暗黙の信頼」

###### 具体例

- 例えば、ある商品が特定の店舗に在庫があるかどうかをユーザーに表示させるショッピング・アプリケーションがあるとする
- アプリケーションは様々なバックエンドREST APIに問い合わせる。
- ユーザーが商品の在庫状況を見るとき、ブラウザは次のようなリクエストを行う
```HTTP
POST /product/stock HTTP/1.0 
Content-Type: application/x-www-form-urlencoded 
Content-Length: 118 stockApi=http://stock.weliketoshop.net:8080/product/stock/check%3FproductId%3D6%26storeId%3D1
```
- これにより、サーバーは指定されたURLにリクエストを行い、在庫状況を取得し、これをユーザーに返す。  
  
- この例では、攻撃者はサーバにローカルなURLを指定するようにリクエストを変更することができる：
```
POST /product/stock HTTP/1.0 
Content-Type: application/x-www-form-urlencoded 
Content-Length: 118 
stockApi=http://localhost/admin
```
- サーバーは/admin URLの内容を取得し、ユーザーに返す。  
  
- もともと、攻撃者は/admin URLにアクセスできるが、通常認証されたユーザーしかアクセスできない機能がある。
- しかし、/admin URL へのリクエストがローカルマシンから来た場合は、通常のアクセス制御はバイパスされてしまう。なぜなら、リクエストは信頼された場所から発信されたように見えるので

### 外部からは直接アクセス不可能なプライベートIPアドレスをもつサーバへの攻撃

- 👉 プライベートIPアドレスを指定するだけ
#### プライベートIPアドレスを指定してSSRFを行う方法

1. ==プライベートIPアドレスを含むURL==がアプリケーションのリクエストにあることを確認する
	- プライベートIPアドレスが`192.168.0.x`だとする
2. 管理者ページを発見するために、`x`の部分をIntruderで1~255で総当たり
```
http://192.168.0.§x§/admin
```
3. 実行可能なREST APIの操作URLをコピーし、サーバがバックエンドシステムにリクエストを送っているパラメータに入力
