## Web messageがソースとなる理由

- ==Originを正しく検証しない==などの、安全でない方法で受信したWeb messageを処理するとき、Web messageが受信ページのDOM based vulnを突くためのソースとして扱われる

- このとき、イベントリスナーによって呼び出されるプロパティや関数がシンクになる可能性がある。
- つまり、==悪意のあるデータをシンクに伝播させるためのソースとして==、Web messageを使用できる

## Web messageをソースとして使用した攻撃の構築基本

###### 基本原理の具体例

- messageイベントをリッスンし、受信したデータをeval関数で実行する
```js
<script>
window.addEventListener('message', function(e) {
  eval(e.data);
});
</script>
```
- 送信元のオリジンを検証していないので、安全でない

- 攻撃者は`Window.postMessage()`メソッドを使用して、ペイロードをメッセージとして渡す
```html
<iframe src="target" onload="this.contentWindow.postMessage('print()','*')"></iframe>
```
- `this`: 現在のオブジェクト。ここでは`iframe`のこと。
- `contentWindow`: プロパティ。iframeや内部DOMにアクセスする。Windowオブジェクトを取得する。ここではiframeのコンテンツのwindowオブジェクトを取得する
	- *windowオブジェクト*とは：画面上に表示されているすべてのオブジェクトの親となるオブジェクトで、JSのオブジェクト階層の最上位に位置する。ウィンドウに関する情報の取得や、ウィンドウを設定・操作する。
- [Window.postMessage()](https://developer.mozilla.org/ja/docs/Web/API/Window/postMessage)：windowオブジェクト間でのメッセージのやり取りを可能にする
- `*`: どのオリジンからのメッセージか伝える(ターゲットのWebはこれを検証しないのでメッセージを受信する)

- つまり、iframeのウィンドウからターゲットのwebアプリに`print()`を送信するということ

###### 方法1

1. [5. DOM based XSS](../XSS/5.%20DOM%20based%20XSS.md#手動での検出とテスト)でDOM XSS脆弱性を発見する
2. [5. DOM based XSS](../XSS/5.%20DOM%20based%20XSS.md#それぞれのソースとシンクでのDOM%20XSS実施)や[DOM XSS wiki](https://github.com/wisec/domxsswiki/wiki)を参考にしたペイロードを`postMessage()`の第一引数に代入し、攻撃者のサーバにホストし被害者にDeliver
```html
<iframe src="target" onload="this.contentWindow.postMessage('<img src=1 onerror=print()>','*')">
```

## Originが検証される時

#### 検証関数の脆弱性をつく

- 💡`indexOf('')`: 指定した文字列がURLに含まれるかどうか検証するだけ
- `startWith('')` or `endWith('')`：それぞれ、前方一致、後方一致の検証だけ
- 参考記事 [JavaScriptの文字列マッチングまとめ](https://qiita.com/aqril_1132/items/9f69575bfbcf24bdf7b5)
