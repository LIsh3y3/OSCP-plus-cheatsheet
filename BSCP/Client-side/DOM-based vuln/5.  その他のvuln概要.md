## JavaScript injection

- [0. sinkとsourceのリスト](0.%20sinkとsourceのリスト.md#JavaScript%20injection)

###### 影響

- session tokenやクレデンシャルの窃取
- 任意のアクションの実行
- キーロギング
###### 対策

- 信頼できないソースからのデータをJavaScriptとして実行させない
###### 方法概要

- 攻撃者が制御可能なデータでJavaScriptを実行する場合
1. 攻撃者はこの脆弱性を利用してURLを作成する。
2. 被害者がそのURLにアクセスする
3. 任意のJSが被害者のブラウザで実行される

- 詳細は[5. DOM based XSS](../XSS/5.%20DOM%20based%20XSS.md#反射型DOM%20XSS)]

## document-domain manipulation

- [0. sinkとsourceのリスト](0.%20sinkとsourceのリスト.md#document-domain%20manipulation)

###### 影響

- `document.domain`は[2. 同一オリジンポリシー(SOP)](../CORS/2.%20同一オリジンポリシー(SOP).md)を実施する
- 異なるオリジンが同じ`document.domain`値を明示すると、無制限にリソースを共有できる
- 攻撃者のWebサイトのページで設定した`document.domain`値と同じ値をターゲットのWebサイトに設定することで、攻撃者のWebサイトからターゲットのWebサイトの完全なコントロールが可能になるのでXSSと同じくらいクリティカル

###### 対策

- 信頼できないソースからのデータが動的にdocument.domainプロパティを設定することを避ける

###### 方法概要

- 攻撃者が制御可能なデータで`document.domain`プロパティを設定する場合
1. 被害者がアクセスしたときに応答ページで任意の`document.domain`プロパティの値を設定させるURLを構築する
2. 攻撃者のWebサイトのページで設定した`document.domain`値と同じ値をターゲットのWebサイトに設定する
	- 値に割り当てられるものに制限を貸していても、制限の弱いサブドメインなどをターゲットにすることができる

3. 攻撃者のWebサイトからターゲットのWebサイトの完全なコントロールが可能になる

## WebSocket-URL poisoning

- [0. sinkとsourceのリスト](0.%20sinkとsourceのリスト.md#WebSocket-URL%20poisoning)

###### 影響

- ユーザーのブラウザが攻撃者の制御下にあるURLへのWebSocket接続を開くようにする
- WebサイトがユーザのブラウザからWebSocketでつながったサーバに機密データを送信する場合、攻撃者はこのデータをキャプチャできる
- もしくはWebSocketでつながったサーバからデータを読み込んで何らかの処理をする場合、攻撃者から悪意あるデータを送りつけてロジックを破壊したりClient-sideの攻撃を実行できる

###### 対策

- 信頼できないソースからのデータが WebSocket 接続のターゲット URL を動的に設定することを許可しない

###### 方法概要

- スクリプトがWebSocket接続のターゲットURLとして攻撃者の望むデータを書き込む場合
1. 制御可能なデータを使用し、WebSocket 接続のターゲット URL を攻撃者の管理するサーバのURLにする
2. 機密データのキャプチャや、悪意あるclient-sideのスクリプトを送りつける

## link-manipulation

- [0. sinkとsourceのリスト](0.%20sinkとsourceのリスト.md#link-manipulation)
###### 影響

- ユーザを任意の外部URLにリダイレクトさせ、フィッシング攻撃を容易にする。
- 攻撃者がコントロールするサーバに対して、ユーザの機密性の高いフォームデータを送信させる
- リンクに関連付けられたファイルやクエリ文字列を変更することで、ユーザがアプリケーション内で意図しないアクションを実行する。
- XSSエクスプロイトを含むオンサイトリンクを注入することで、ブラウザのアンチXSS防御を回避する。
	- XSS対策はサイト内のリンクには実行していないことが多い

###### 対策

- 信頼できないソースからのデータがリンクやフォームのターゲットURLを動的に設定することを避ける

###### 方法概要

- クリック可能なリンクやフォームの送信URLのような、現在のページ内のナビゲーションターゲットに攻撃者が制御可能なデータを書き込める場合
1. レスポンス内のリンク先を変更するような URL を構築する

## Ajax request-header manipulation

- [0. sinkとsourceのリスト](0.%20sinkとsourceのリスト.md#Ajax%20request-header%20manipulation)

###### 影響

- 攻撃者はこの脆弱性を利用して、他のユーザがアクセスした場合に、その後のAjaxリクエストに任意のヘッダを設定するURLを作成する
- 他の種類の攻撃を連鎖させるための出発点として使用する

###### 対策

- 信頼できないソースからのデータが動的にAjaxリクエストヘッダを設定することを避ける

###### 方法概要

- `XHR`オブジェクトを使用して発行されたAjaxリクエストのリクエストヘッダに対し、スクリプトが、攻撃者の望むデータを書き込んだ場合
1. 攻撃者はXHRオブジェクトのリクエストヘッダに任意のヘッダを追加する

## local file-path manipulation

- [0. sinkとsourceのリスト](0.%20sinkとsourceのリスト.md#local%20file-path%20manipulation)

###### 影響

- 被害者がアクセスした場合に、そのユーザのブラウザに任意のローカルファイルを開かせるような URL を構築できる
- Webが機密ファイルに特定のデータを書き込むと、攻撃者はそのファイルに自分のデータを書き込むことができる。OSのconfigなど

###### 対策

- 信頼できないソースからのデータがファイルハンドリング API に動的にファイル名を渡すことを避ける

###### 方法概要

- スクリプトが、攻撃者の望むデータをファイルハンドリングAPIに`filename`パラメータとして渡す場合
1. `filename`パラメタに任意のファイルを渡す

## client-side SQLi

- [0. sinkとsourceのリスト](0.%20sinkとsourceのリスト.md#client-side%20SQLi)

###### 影響

- DBに機密情報が格納されていたら窃取できる
- クエリによるアクションを被害者のコンテキストで実行できる

###### 対策

- プリペアドステートメントの使用(`?`にデータを渡す)
- プレースホルダーごとにコンテンツの内容を指定する

###### 方法概要

- スクリプトが、攻撃者が制御可能なデータを、安全でない方法でclient-side SQLクエリに組み込む場合
- 被害者がアクセスした場合に、そのユーザのブラウザのローカルSQL DB内で任意のSQLクエリを実行するようなURLを構築

## HTML5-storage manipulation

- [0. sinkとsourceのリスト](0.%20sinkとsourceのリスト.md#HTML5-storage%20manipulation)
###### 影響

- データを保存させるだけでは対して影響はないが、もしアプリケーションがストレージからデータを読み戻し、安全でない方法で処理するなら、XSSや[5.  その他のvuln概要](5.%20%20その他のvuln概要.md#JavaScript%20injection)につながる

###### 対策

- 信頼できないソースからのデータをHTML5のストレージに置かない

###### 方法概要

- スクリプトが攻撃者が制御可能なデータをWebブラウザのHTML5ストレージ（`localStorage`または`sessionStorage`）に保存する場合
- 他のユーザがアクセスした場合に、そのユーザのブラウザに攻撃者が制御可能なデータを保存させる URL を作成

## DOM-data manipulation

- [0. sinkとsourceのリスト](0.%20sinkとsourceのリスト.md#DOM-data%20manipulation)

###### 影響

- 特定のページに表示されるテキストや画像を変更するなど、Webサイトを仮想的に改ざんする
	- 攻撃者が要素の src プロパティを変更できた場合、悪意のある JavaScript ファイルをインポートすることで、ユーザに意図しないアクションを実行させる

###### 対策

- 信頼できないソースからのデータがDOM-dataフィールドに動的に書き込まれることを避ける
- ⚠️Burp Scannerはこの脆弱性を自動的に特定するが、実際には悪用できない==誤検知==の場合もある。

###### 方法概要

- スクリプトが、攻撃者が制御可能なデータをDOM内のフィールドに書き込み、それが目に見えるUIやクライアント側のロジックで使用される場合
- 他のユーザがアクセスした場合に、クライアント側のUIの外観や動作を変更するようなURLを構築する
- 反射型、持続型どちらのタイプでも悪用される可能性がある

## DoS

- [0. sinkとsourceのリスト](0.%20sinkとsourceのリスト.md#DoS)

###### 対策

- 信頼できないソースからのデータを問題のあるプラットフォームAPIに動的に渡すことを避ける

###### 方法概要

- スクリプトが、攻撃者が制御可能なデータを安全でない方法で、問題のあるプラットフォームAPI（例えば、その呼び出しによってユーザのコンピュータが過剰な量のCPUやディスクを消費するようなAPI）に渡す場合
	- 例えば`localStorage`にデータを保存しようとする試みを拒否したり、ビジー状態のスクリプトを強制終了したりすることで、ブラウザがWebサイトの機能を制限した場合に過剰な量のCPUを消費する
