###### 脆弱性の種類

- サーバに送信されたユーザの入力が安全でない方法で処理される場合、[1. SQLi基本と対策](../../Server-side/SQLi/1.%20SQLi基本と対策.md)や[1. XXE injectionの基本と対策](../../Server-side/XXE%20injection/1.%20XXE%20injectionの基本と対策.md)などの脆弱性につながる。

### 脆弱性を悪用するためのWebSocketメッセージの操作

###### 基本概念と具体例

- WebSocketに影響を与える入力ベースの脆弱性の大部分は、WebSocketメッセージの内容を改ざんすることによって発見し、悪用する
  
- 例えば、あるチャット・アプリケーションが WebSocket を使ってブラウザとサーバ間でチャット・メッセージを送信している。
- ユーザがチャット・メッセージを入力すると、次のような WebSocket メッセージがサーバに送信される：
```js
{"message":"Hello Carlos"}
```
- メッセージの内容は、再びWebSocketを介して別のチャットユーザーに送信され、ユーザのブラウザで次のようにレンダリングされる
```html
<td>Hello Carlos</td>
```
- この状況では、他の検証や防御をしていない場合、攻撃者は次のようなWebSocketメッセージを送信することで、XSS攻撃を実行できる：
```js
{"message":"<img src=1 onerror='alert(1)'>"}
```

###### WebSocketメッセージの操作による悪用方法

1. WebSocketのhistoryからメッセージの形式を明らかにする
2. RepeaterなどでXSSペイロードなどを注入する(client-sideの検証を突破するため)
```js
{"message":"<img src=1 onerror='alert(1)'>"}
```

### 脆弱性を悪用するためのWebSocketハンドシェイクの操作

###### 基本概念と具体例

- ハンドシェイクを操作することによってのみ発見され、悪用可能なものがある
- ハンドシェイクについては[1. WebSocketsの基本と安全化](1.%20WebSocketsの基本と安全化.md#WebSocketコネクションの確立の流れ)を参照

###### 原因となる設計上の欠陥

- `X-Forwarded-For`ヘッダーのような、悪用されやすいHTTPヘッダーの信頼
- セッション処理メカニズムの欠陥
	- ハンドシェイク時に確立されたセッションが、以後のWebSocket通信全体のセッションとなる
	- 欠陥があるとセッションIDを推測または窃取し、被害者になりすます。もしくは機密情報の窃取

- アプリケーションによって異なるカスタム HTTP ヘッダーによる攻撃ベクターがある

#### 悪用方法

1. [2. WebSocketの脆弱性と悪用](#WebSocketメッセージの操作による悪用方法)を試すと、WebSocket historyにて攻撃がブロック&接続が遮断されていることが確認できる
2. "Reconnect"で再接続を試み、Responceを確認して接続拒否の理由を確認。IPアドレスがブロックされている等
3. `X-Forwarded-For`でIPスプーフィングして接続するとブロックを回避できる
```http
X-Forwarded-For: 1.1.1.1
```

4. XSSペイロードの難読化[XSS contexts](../../../OSCP/Cheatsheet/Web%20App/XSS%20contexts.md#難読化)を試して再度悪用を試みる
```js
{"message":"<img src=0 OnErRor=alert`1`>"}
```
