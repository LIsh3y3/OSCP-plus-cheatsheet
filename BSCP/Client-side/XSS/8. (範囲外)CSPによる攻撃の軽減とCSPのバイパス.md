
## Content Security Policyとは

- XSSやその他の攻撃を軽減するためのセキュリティの仕組み
- ページがロードできるリソース（スクリプトや画像など）を制限する
- どのページが他のページによって`<iframe>`タグで埋め込まれるか（フレーム内に表示されるか）制限する
- `<script>`をよくブロックするが、`<img>`での外部リクエストをブロックしてないことが多い。

### CSPを有効にするには

- レスポンスに`Content-Security-Policy: [対象] [policy value]`を含める
- `;`で区切った複数の指示(ディレクティブ)で構成される
- ↓例
```
Content-Security-Policy: default-src 'self'; img-src https://*.example.com
```

# 攻撃の軽減

## CSP使用によるXSS軽減方法

### 特定のドメインからのスクリプトロードを許可する

- スクリプトがそのページと同じオリジンからのみロードを許可する
```
Content-Security-Policy: script-src 'self'
```
- 特定のドメインからのみスクリプトのロードを許可する
```
Content-Security-Policy: script-src https://scripts.normal-website.com
```

#### nonceとハッシュ値による信頼できるリソースの指定

- どうしても外部リソースのロードが必要な場合で、自分のドメインでリソースをホストできない場合
	- 信頼できるドメインとして特定のドメインをホワイトリストに登録し、そこからリソースをロードしたとしても、攻撃者はそのドメインを悪用して悪意あるスクリプトを読み込ませることが可能
###### nonce(ランダムな値)

- Webサーバにより、ページがロードされるたびにnonceが生成される
- 生成されたnonceはその都度CSPにセットされる

###### ハッシュ値

- 信頼できるスクリプトのハッシュ値をCSPにセットする
	- 膨大なスクリプトでも一意で固定長のハッシュ値

##### ⚠️：外部ドメインからのスクリプトロードを許可するときの注意点

- 攻撃者が外部ドメインから配信されるコンテンツをコントロールし、XSSを可能にするかも

- 攻撃者がCDN上のリソースを利用してXSSを実施するなど
	-  特に注意が必要なのは、カスタマーごとに特有のURLを提供しないCDN
	- たとえば `ajax.googleapis.com`(CDN) のように、多くの異なるユーザーが共通のURLを通じてリソースをCDNからロードする場合、攻撃者によってそのドメインにコンテンツがアップロードされ、それに悪意がある場合がある

## CSPによるDangling markup injectionの軽減

- 画像がそのページと同じオリジンからのみロードを許可する
```
Content-Security-Policy: img-src 'self'
```
- 特定のドメインからの画像の読み込みを許可する
```
Content-Security-Policy: img-src https://images.normal-website.com
```

- Dangling markup injectionには、no user ineteractionでXSSを実行する`<img>`が使われることが多い。
- Chromeはエンコードされていない改行や角括弧(`<>`)など、特定の文字をURLに含むリクエストをブロックする軽減機能を内蔵している

###### 防げないこと

- href属性のdangling markup injection。

## CSPによるクリックジャッキング対策

- フレーム化(frameおよびiframeでの参照)をそのページと同じオリジンからのみ許可する
```
Content-Security-Policy: frame-ancestors 'self'
```

- 完全にフレーム化を禁止する
```
Content-Security-Policy: frame-ancestors 'none'
```

### [X-Frame-Options](https://developer.mozilla.org/ja/docs/Web/HTTP/Headers/X-Frame-Options)との違い

- X-Frame-OptionsはCSPをサポートしていない古いブラウザでも有効なので、CSPと併用する
- CSPは全てのフレームを検証するが、X-Frame-Optionsは最上位フレームのみを検証するので脆弱
	- 最上位フレームとは、複数のフレームやiframeが組み込まれている状況で一番外側のフレームのこと。
- CSPでは複数のドメインを指定したりワイルドカードを使用可能でXFOより柔軟
```
frame-ancestors 'self' https://normal-website.com https://*.robust-website.com
```

# CSPのバイパス

###### 前提知識

- `report-uri`: ユーザの入力がCSPに反映される
- `script-src-elem`: `script-src`ディレクティブを上書きする(すべてのscriptに適用される)。Chromeのみ有効。

###### CSPバイパス方法

1. `Content-Security-Policy: report-uri /〇〇?token`等の記述を見つける
2. 操作可能なパラメータ(`token`)からCSPを上書きし、ペイロードを注入する
```
https://example.com/?search=<script>alert(1)</script>&token=;script-src-elem 'unsafe-inline'
```
- `/?search`：クエリパラメタはなんでもいい
- `unsafe-inline`: cspの一種。記事参照。
