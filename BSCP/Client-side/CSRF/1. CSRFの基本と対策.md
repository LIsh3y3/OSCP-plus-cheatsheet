- [同一オリジンポリシー](https://developer.mozilla.org/ja/docs/Web/Security/Same-origin_policy)(異なるオリジンのリソースへのアクセスを制限する）を回避できる

## ⭐️CSRFが動作する条件：第一に確認すること

###### 攻撃者にとって魅力的なアクションが1つ以上のHTTPリクエストで可能

- パスワード変更やEmail変更など

###### Cookieのみでセッションを管理している

- セッションの追跡やアクション実行のまえの検証がない

###### 予測不可能なリクエストパラメータがない

- パスワード変更の際に既存のパスワードの入力を求められるならこの条件には当てはまらない。

- [CSRFの動作例](https://portswigger.net/web-security/csrf#:~:text=For%20example%2C%20suppose%20an%20application)

## CSRFペイロードの生成

###### ⭐️Generate CSRF PoC(pro)

- Options->Include auto-submit scriptでアクセスと同時にsubmitされるスクリプトを組み込む

##### CSRF PoC Generator(OSS)を使用する

- https://tools.nakanosec.com/csrf/
- ⭐️生成されたHTMLのbodyに以下のスクリプトを追加で注入し、攻撃者が用意したサーバのページにアクセスすると同時にsubmitされるようにする
```html
<script>
	history.pushState('', '', '/')
	document.forms[0].submit();
</script>
```
- `history.pushState('', '', '/')`：CSRF隠蔽。リロードやリダイレクトしてないのにURLを変更する。

###### ⚠️
- URLエンコードされているvalue値はデコードし、HTMLエンコードし直すこと
- (VScodeで`memo.html`でもしてペーストしてフォーマットを整える)

## XSSとCSRFの共通点と違い

###### 共通点

- 基本的に[2. 反射型XSS](../XSS/2.%20反射型XSS.md)と一緒で、被害者に攻撃者の用意する悪意あるサーバにアクセスしてもらう必要がある
- つまり、攻撃者のサイトに悪意あるHTMLをホストし、被害者にDeliverする

###### 違い

- XSSはユーザが実行可能なすべてのアクションが可能。対してCSRFはユーザが実行可能なアクションのサブセットのみ。
- XSSはリクエストに対するレスポンスも攻撃者が閲覧可能。対してCSRFはリクエストを送らせることだけが可能

###### CSRF tokenによるXSSの防御

- 単純なXSSはCSRF tokenにより防御できるかも
```
https://insecure-website.com/status?message=<script>/*+Bad+stuff+here...+*/</script>
```
	↓
```
https://insecure-website.com/status?csrf-token=[token]&message=<script>/*+Bad+stuff+here...+*/</script>
```

###### ⚠️
- CSRF tokenで保護されていない関数でのアクションは通常通りXSS攻撃される
- XSS脆弱性によりCSRF tokenを窃取し、防御していたアクションも実行されるかも
- CSRF tokenは[3.持続型XSS](../XSS/3.持続型XSS.md)には無効

## 一般的なCSRF対策とそのバイパス

### CSRF token

- [2. CSRF token検証のバイパス](2.%20CSRF%20token検証のバイパス.md)

- CSRF トークンは、サーバ側のアプリケーションによって生成され、クライアントと共有される、一意で、秘密で、予測不可能な値。
- フォームの送信など、機密性の高いアクションを実行しようとする場合、クライアントは正しい CSRF トークンをリクエストに含める。
	- ==アクション実行ごとにtokenを発行するかも==
- 攻撃者が被害者として有効なリクエストを作成することは非常に困難になる。
- 一般的にはhiddenフィールドに格納されている
```html
<input required type="hidden" name="csrf" value="50FaWgdOhi9M9wyna8taR1k3ODOR8d6u">
```

### SameSite cookie

- [3. SameSite cookie制限のバイパス](3.%20SameSite%20cookie制限のバイパス.md)

- SameSite Cookieとは、Cookieの属性の一つでブラウザがCookieをサーバに送信する際の条件を指定する
- 攻撃者がクロスsiteでアクションを実行することを制限する
- Set-CookieレスポンスヘッダにSameSite属性を付与するだけ
```http
Set-Cookie: session=abcd; SameSite=Strict
```

##### ⚠️SiteとOriginの違い

- siteはTLD(comなど)+1(1階層下のサブドメイン)を含むので、TLD+1と表される。スキームも含む

![[Pasted image 20240105164103.png|400]]

#### SameSite属性の種類

###### Strict
- CookieはそのCookieが設定されたオリジンでのリクエストにのみ送信される。つまり、ユーザが直接そのサイトを訪れた場合にのみCookieが送信され、他のサイト（クロスサイト）からのリクエストには含まれない
- 認証されたユーザーのみが利用できる特定のページへのアクセスなどで、機密性の高いアクションを実行できるようにするCookieを設定する場合に推奨
- もっとも安全だが、UXに悪影響を与える恐れあり

###### Lax
- 同じオリジンのリクエストには`Strict`と同様にCookieが送信される。
- 加えて、クロスサイトからのナビゲーションであっても、以下の条件に当てはまるときはCookieを送信する
	- `GET`メソッドのリクエスト
	- リンクをクリックするなどの最上位レベルのナビゲーション(新しいページを開いたり、現在のページが別のページに全面的に置き換わること。iframeなどバックグラウンドのリクエストは最上位レベルとは言わない)
- `SameSite`属性が指定されない場合は、Chrome側で自動で`Lax`に設定される

###### None
- Cookieはすべてのコンテキストで送信されるが、この場合はCookieが`Secure`属性も同時に設定されている必要がある。そうしないとブラウザはCookieを拒否する
- Chrome以外では、`SameSite`属性が設定されないとこれに設定される

## 🛡️CSRF対策

### CSRF tokenの使用(最も堅牢)

###### tokenのあるべき姿

- 高エントロピー。予測できない
- ユーザのsessionと結びついている (→[2. CSRF token検証のバイパス](2.%20CSRF%20token検証のバイパス.md#CSRF%20tokenがユーザのセッションと結びついていない))
- アクション実行前にどのようなケースであってもtokenを検証する(→[2. CSRF token検証のバイパス](2.%20CSRF%20token検証のバイパス.md#リクエストメソッドによってはCSRF%20token検証がされない))

###### tokenはどのようにして生成されるべきか

- CSPRNGを使い、token生成時のTimestampと静的な秘密を使ってシードにする
	- シード値：疑似乱数を作る時に使う、元となる値

- さらに強度を確保したい場合は、出力にユーザ固有のエントロピーと連結して個々のトークンを生成し、全体のハッシュ値を算出してtokenとする

- 参考 : [各言語での、本当に安全な乱数の作り方](https://qiita.com/gakuri/items/27cca8f0fa28b78ddeca)

###### tokenはどのようにして渡されるべきか

- 基本はPOSTメソッド & hiddenフィールド
- さらなる安全性のために、CSRF tokenを含むhiddenフィールドは、HTMLドキュメントのできるだけ最初の方に配置する。できればユーザが制御可能なデータがHTMLに埋め込まれる場所の前に
	- これにより、攻撃者がHTMLドキュメントを操作してtokenを抽出するなどのさまざまな攻撃への対策となる

- Cookieで渡さない 

### SameSite属性の使用

- 基本は`SameSite=Strict`
- どうしてもCookieをクロスサイトで送信する必要がある場合のみ、`Lax`を

###### SameSite属性の注意点

- クロスorigin、かつ、同じsiteの攻撃には意味ない。
- 可能であれば、ユーザがアップロードしたファイルのような安全でないコンテンツは、機密性の高い機能やデータとは別のサイトに隔離すること。
- サイトをテストする際には、兄弟ドメインも含め、同じサイトに属する利用可能なすべての攻撃対象領域を徹底的に監査する　(→[3. SameSite cookie制限のバイパス](3.%20SameSite%20cookie制限のバイパス.md#脆弱な兄弟ドメインによるSameSite制限の回避)
