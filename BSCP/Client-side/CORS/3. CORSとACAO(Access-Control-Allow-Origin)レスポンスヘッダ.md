## ACAOレスポンスヘッダとは

- あるWebサイトが別のWebサイトから発信したリクエストに対するレスポンスに含まる
- リクエストが許可された発信元を識別する。
- ブラウザは`Access-Control-Allow-Origin`とリクエスト元のWebサイトのオリジンを比較し、一致した場合にレスポンスへのアクセスを許可する。
- 一致しない場合はJavaScriptがレスポンスにアクセスできない

### ヘッダの値の基本

- 値には、ドメイン、ワイルドカード(`*`)、`null`がある
- 複数のオリジンはサポートせず、ワイルドカードの使用にも制限がある

###### 具体例

- クロスドメインのリクエスト(`Origin`ヘッダを追加し、自身のFQDNを記載)
```http
GET /data HTTP/1.1
Host: robust-website.com
Origin : https://normal-website.com
```
- レスポンス
```http
HTTP/1.1 200 OK
...
Access-Control-Allow-Origin: https://normal-website.com
```

- このとき、リクエスト発行側のJavaScritはレスポンスにアクセスできる

## クレデンシャルを付与したクロスオリジンリソースリクエスト

- レスポンス側で以下を設定している必要がある
```http
Access-Control-Allow-Credentials: true
```

- 上記ヘッダが設定されている、かつ、`Access-Control-Allow-Origin`で許可されたドメインであるとき、リソースにアクセスできる

###### 具体例

- リクエスト(from `https://normal-website.com`)
```http
GET /data HTTP/1.1
Host: robust-website.com
...
Origin: https://normal-website.com
Cookie: JSESSIONID=<value>
```

- レスポンス
```http
HTTP/1.1 200 OK
...
Access-Control-Allow-Origin: https://normal-website.com
Access-Control-Allow-Credentials: true
```
- 条件に合致するので、`https://normal-website.com`のJSはレスポンスにアクセスできる

## ワイルドカードを使用したCORSの緩和(脆弱)

- `Access-Control-Allow-Origin`ヘッダを動的に作成する。
- これはCORS制約の緩和策だが、安全ではない。→

### 許可されていないこと

###### ワイルドカードは値の中に組み込めない

- 🙆 
```http
Access-Control-Allow-Origin: *
```
- 🙅
```http
Access-Control-Allow-Origin: https://*.normal-website.com
```

###### ワイルドカードとクレデンシャルの付与は一緒に使えない

- 🙅
```http
Access-Control-Allow-Origin: *
Access-Control-Allow-Credentials: true
```

## Pre-flight check

- [3. CORSとACAO(Access-Control-Allow-Origin)レスポンスヘッダ](3.%20CORSとACAO(Access-Control-Allow-Origin)レスポンスヘッダ.md#「シンプルなリクエスト」)の条件を満たさないときにメインのリクエストに先んじて送られるリクエスト
- 別名Pre-flight request

#### プリフライトでやり取りするヘッダ

| 要求の種類 | リクエスト | レスポンス |
| ---- | ---- | ---- |
| メソッドに対する許可 | `Access-Control-Request-Method` | `Access-Control-Allow-Methods` |
| ヘッダに対する許可 | `Access-Control-Request-Headers` | `Access-Control-Allow-Headers` |
| オリジンに対する許可 | `Origin` | `Access-Control-Allow-Origin` |
- その他レスポンスヘッダ：`Access-Control-Max-Age`：プリフライトレスポンスを再利用するためのキャッシュの最大時間枠

###### 具体例

- プリフライトリクエスト(既存のリソースに影響を与えないように`OPTIONS`メソッドを使用)
```http
OPTIONS /data HTTP/1.1
Host: <some website>
...
Origin: https://normal-website.com
Access-Control-Request-Method: PUT
Access-Control-Request-Headers: Special-Request-Header
```

- プリフライトレスポンス
```http
HTTP/1.1 204 No Content
...
Access-Control-Allow-Origin: https://normal-website.com
Access-Control-Allow-Methods: PUT, POST, OPTIONS
Access-Control-Allow-Headers: Special-Request-Header
Access-Control-Allow-Credentials: true
Access-Control-Max-Age: 240
```

#### 「シンプルなリクエスト」

- 以下の条件を満たすものをシンプルなリクエストと呼ぶ
###### method

- GET, HEAD, POST

###### `XHR`オブジェクトの`setRequestHeader`メソッドで設定するリクエストヘッダ

- Accept
- Accept-Language
- Content-Language
- Content-Type

###### Content-Typeヘッダに指定するもの

- `application/x-www-form-urlencoded`
- `multipart/form-data`
- `text/plain`
