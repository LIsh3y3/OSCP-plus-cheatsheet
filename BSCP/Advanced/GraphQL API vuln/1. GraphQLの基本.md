
## GraphQLとは

- Web APIのクエリ言語
- つまり、どういうふうにサーバからデータを取得するか・変更するかをまとめた言語
- 結果はJSONで
- 大規模なWebアプリ向け(小規模には向いてない。小規模ならREST APIでいい)
- REST APIの問題点を解決するために開発

###### REST APIの問題点

- [📕](../../Misc/📕.md#REST%20API)
- Webアプリが大きくなるにつれ、エンドポイントが増えていく。
 ![[Pasted image 20240114113456.png | 190]]
- 帯域を余計に使用する。
- 処理が複雑。

### GraphQLの特徴

![[Pasted image 20240114113640.png | 400]]

- 主にPOSTリクエスト（RESTはさまざまなHTTPメソッド）

### GraphQLの動作

###### 可能なこと

- クエリ：　データの取得 [1. GraphQLの基本](1.%20GraphQLの基本.md#GraphQL%20クエリとは)
- ミューテーション：　データの操作（追加・変更など） [1. GraphQLの基本](1.%20GraphQLの基本.md#GraphQLミューテーションとは)
- サブスクリプション：　クライアント側からサーバに「常に接続し、データを監視し、変化があればそれをクエリで取得して」というもの。SNSなどで使われる。 [1. GraphQLの基本](1.%20GraphQLの基本.md#サブスクリプション)

## GraphQL schema

- フロントエンドとバックエンド間の契約のこと
- 人間が読めるスキーマ定義言語を使用して、一連の型として利用可能なデータを定義する
- ほとんどの型はオブジェクト型である
- `field`名: `データ型`という形式
	- フィールドにはスカラー型（通常のデータ型）、enum型、union型、interface型、カスタムの型が設定できる
	- =="kind": "OBJECT"==のような形式

- 以下はProduct型のスキーマの定義
```GraphQL
   #Example schema definition

    type Product {
        id: ID!
        name: String!
        description: String!
        price: Int
        //field : type　の形式
    }
```
- `!`: は必須である意味

## GraphQL クエリとは

- データの取得用(REST APIのGETリクエストと似てる)

```
query [query名] {
	[データ構造](1つ以上のargument){
		field
	}
}
```
- `query`、`query名`はオプションだが、推奨
- `データ構造`：レスポンスで何を返して欲しいか定義する
- `1つ以上のargument`はオプション。特定のオブジェクトの詳細を返すクエリを作成するときに使用

- 以下の例は、myGetProductQuery というクエリで、id が 123 の商品の name フィールドと description フィールドを要求
```
    #Example query

    query myGetProductQuery {
        getProduct(id: 123) {
            name
            description
        }
    }
```
- 実際には、Produc型にもっと多くのフィールドが存在したが、必要なフィールドだけ取得できる。これはGraphQLの強み

## GraphQLミューテーションとは

- データの操作用(これはREST APIのPOST, PUT, DELETEと似てる)
- `データ構造`の引数に必ずなんらかの型の入力を取る
- インライン値も可能だが、一般的には変数
```
mutation {
	データ型(name: "Kosi", descreption: "men"){ //name: $name, description: $des..も可
		id
		name
		descreption
	}
}
```

- 例として、新商品を作成するためのミューテーションと、そのレスポンスを示す。この場合、サービスは自動的に新しい商品にIDを割り当てるように設定されており、リクエスト通りに返されている
```
#Example mutation request

    mutation {
        createProduct(name: "Flamin' Cocktail Glasses", listed: "yes") {
            id
            name
            listed
        }
    }
```

```
  #Example mutation response

    {
        "data": {
            "createProduct": {
                "id": 123,
                "name": "Flamin' Cocktail Glasses",
                "listed": "yes"
            }
        }
    }
```

###### スキーマ上の見た目

```
{
            "kind": "OBJECT", //⭐️
            "name": "mutation", //⭐️
            "description": null,
            "fields": [
              {
                "name": "deleteOrganizationUser",
                "description": null,
                "args": [
                  {
                    "name": "input",
                    "description": null,
                    "type": {
                      "kind": "INPUT_OBJECT",
                      "name": "DeleteOrganizationUserInput",
                      "ofType": null
                    },
```

## クエリとミューテーションの構成要素

### field

- 以下の例だと、`id`、`name.firstname`、`name.lastname`がフィールド
```
    #Request

    query myGetEmployeeQuery {
        getEmployees {
            id
            name {
                firstname
                lastname
            }
        }
    }
```

### Argument

- [1. GraphQLの基本](1.%20GraphQLの基本.md#GraphQL%20schema)であらかじめ定義しておく(`id: ID!`)
- 条件として使用できたりする(idが1のデータ)
```
...
getEmployees(id:1) {
...
```

- 🚨：ユーザが制御可能な値をArgumentとして使用し、オブジェクトにアクセスする場合、IDOR([3. Lateral movement(IDOR)の基本](../../Server-side/Access%20control/3.%20Lateral%20movement(IDOR)の基本.md))につながるかもしれない

### Variable

- 変数をあらかじめ別のJSONで定義しておく(変数辞書)
- 変数と型を宣言しておく必要がある
- `変数：カスタムのデータ型!`とすることで、必須の値であることを使用前に定義しておく
```
 #Example query with variable

    query getEmployeeWithVariable($id: ID!) {
        getEmployees(id:$id) {
            name {
                firstname
                lastname
            }
         }
    }

    Variables:
    {
        "id": 1
    }
```

### Aliase(別名)

- 1つのオブジェクトに同じ名前のフィールドを複数含むことはできない
- 下記例は、`product`型を2回含んでいるので不可
```
 #Invalid query

    query getProductDetails {
        getProduct(id: 1) {
            id
            name
        }
        getProduct(id: 2) {
            id
            name
        }
    }
```
- ↓同一のprocutフィールド(= プロパティ)が2個返ってしまう
```
{
  "data": {
    "product": {
      "name": "Kosi"
    },
    "product": {
      "name": "Nagi"
    }
  }
}
```

- しかし、返ってくるフィールドに別名をつけることができる
```
 #Valid query using aliases

    query getProductDetails {
        product1: getProduct(id: "1") {
            id
            name
        }
        product2: getProduct(id: "2") {
            id
            name
        }
    }
```

```
   #Response to query

    {
        "data": {
            "product1": {
                "id": 1,
                "name": "Juice Extractor"
             },
            "product2": {
                "id": 2,
                "name": "Fruit Overlays"
            }
        }
    }
```

- [5. Aliaseを使用した総当たり攻撃に対するrate制限のバイパス](5.%20Aliaseを使用した総当たり攻撃に対するrate制限のバイパス.md)

### Fragment

- 再利用可能な部分のこと
- 一度定義すると、クエリやミューテーションに含められる。
- 変更はフラグメントを呼び出すすべてのクエリやミューテーションに反映される

- 以下の例は、`Product`型において`productInfo `フラグメントに商品の詳細を定義し、` getProduct `クエリ内でフラグメントを再利用している
```
    #フラグメント例

    fragment productInfo on Product {
        id
        name
        listed
    }
```

```
    #フラグメントの呼び出し

    query {
        getProduct(id: 1) {
            ...productInfo
            stock
        }
    }

```

```
    #フラグメントで定義したフィールドを含むレスポンス

    {
        "data": {
            "getProduct": {
                "id": 1,
                "name": "Juice Extractor",
                "listed": "no",
                "stock": 5
            }
        }
    }
```

## サブスクリプション

- リアルタイムの更新をクライアントにプッシュできる。
- サブスクリプションは主に、大きなオブジェクト内の小さな変更や、小さなリアルタイム更新を必要とする機能（チャットシステムや共同編集など）に便利
- 一般的にWS([1. WebSocketsの基本と安全化](../../Client-side/WebSockets/1.%20WebSocketsの基本と安全化.md))を使って実装される

## イントロスペクション

- 基本的に無効にするべき。Information disclosure([1. 情報開示の基本と対策](../../Server-side/Information%20disclosure/1.%20情報開示の基本と対策.md))へとつながる
- GraphQLのスキーマに関する情報を問い合わせることができる組み込み機能

- GraphQLサーバーのスキーマについて、どんなタイプが存在し、どんなフィールドやクエリ、ミューテーションが有効かなど、APIの構造に関する詳細を取得
