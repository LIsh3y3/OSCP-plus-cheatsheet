### 一般的な対策

- APIを公開しない場合は、APIイントロスペクションを無効にする。方法は[this blog post](https://www.apollographql.com/blog/graphql/security/why-you-should-disable-graphql-introspection-in-production/#turning-off-introspection-in-production)
- 公開する場合は不必要なfieldが露呈していないかスキーマを見直す
- サジェスチョン機能を無効にする。
	- Apolloで直接無効にできないので、[this GitHub thread](https://github.com/apollographql/apollo-server/issues/3919#issuecomment-836503305)を参照

- APIのスキーマが、メールアドレスやユーザーIDなどのプライベート・ユーザー・フィールドを公開していないことを確認する

### GraphQLブルートフォース攻撃対策

###### 目的
- [5. Aliaseを使用した総当たり攻撃に対するrate制限のバイパス](5.%20Aliaseを使用した総当たり攻撃に対するrate制限のバイパス.md)でバイパスされないように
- DoS対策にもなる

###### 対策

- APIのクエリの深さを制限する。
	- 「クエリの深さ」とは、クエリ内の入れ子のレベルの数を指す。
	- 深いネストのクエリはパフォーマンスに重大な影響を及ぼす可能性があり、それが受け入れられるとDoS攻撃の機会を提供する可能性がある。

- Operation(ミューテーションorサブスクリプション）の制限を設定する。
	- 操作の制限を設定することで、APIが受け付ける一意のフィールド、エイリアス、ルート・フィールドの最大数を設定できる

- クエリが含むことができる最大バイト量を設定する。

- APIにコスト分析を実装することを検討する。
	- コスト分析とは、ライブラリアプリケーションが受信したクエリの実行に関連するリソースコストを特定するプロセスである。クエリの実行が複雑すぎる場合、APIはそのクエリを削除する。

###### 実施方法

- Apolloでの上記の対策の実効方法は[this blog post](https://www.apollographql.com/blog/graphql/security/securing-your-graphql-api-from-malicious-queries/)

### GraphQL CSRF対策
  
- API は JSON エンコードされた POST でのみクエリーを受け付けるよう設定する
  - 提供されたコンテンツが提供されたContent-Typeと一致するか検証する
  - 安全なCSRF tokenを実装する
