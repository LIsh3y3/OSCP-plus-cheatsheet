## GraphQLエンドポイントの見つけ方

- 1つのエンドポイントにリソースを集約しているので非常に重要な情報

- 💡：Scannerで自動的にエンドポイントを検出できる。見つかると"GraphQL endpoint found"となる

### 1. 一般的なエンドポイント名を探すためにブルートフォースする

- 何も兆候がなければ、リクエストメソッドを変更してみる
- 200だけでなく、405など他のリクエストと異なるStatusでも見逃さない

###### 一般的なエンドポイント名wordlist

- ⚠️：Intruderで"URL-encode these characters"のチェックを外す
```wordlist
/graphql
/api
/api/graphql
/graphql/api
/graphql/graphql
/graphql/v1
/api/v1
/api/graphql/v1
/graphql/api/v1
/graphql/graphql/v1
```

### 2. 1で何も兆候がなければ、Universalクエリを試す

1. IntruderにてGraphQLエンドポイント候補をブルートフォースしながら、以下のクエリを送る(universalクエリ)

- POSTの場合
```
query{__typename}
```
- GETの場合(必要であればURLエンコード)
```
?query=query{__typename}
```

2. 以下のレスポンスが返れば、そのエンドポイントはGraphQLに対応
```
{
  "data": {
    "__typename": "query"
  }
}
```

###### 機能する要因

- すべてのGraphQLエンドポイントに`__typename`という予約フィールドがあり、クエリーされたオブジェクトの型を文字列として返すから

###### Note:

- GraphQLサービスは多くの場合、GraphQL以外のリクエストに対して"query not present"などのエラーで応答する

### 3.  2がうまくいかない時、リクエストメソッドを変更して再送してみる

- 元がPOSTリクエストで、Content-Typeが`application/json`のときで、リクエストのBodyが以下のようだとする
```json
{"query":"\n    mutation changeEmail($input: ChangeEmailInput!) {\n        changeEmail(input: $input) {\n            email\n        }\n    }\n","operationName":"changeEmail","variables":{"input":{"email":"test@normal-user.net"}}}
```

###### 1. 以下の2つをコピーしてメモしておく

- body部のクエリに当たる値すべて
- GraphQLタブで閲覧できるクエリ、変数も忘れずに
![[Pasted image 20240116160335.png | 150]]

###### 2. Repeaterの"Change request method"で以下の順に遷移するので、それぞれのリクエストメソッドを試す

| リクエストメソッド | Content-Type |
| ---- | ---- |
| GET | ヘッダなし |
| POST | `application/x-www-form-urlencoded` |

1. ==`{"query":"\n` は　`query=`に置き換える==こと。これでGraphQLタブから編集できるようになる
- まず何もせずGETでリクエストを送りMethodが許可されているか確かめる。それから、GETの場合、URLパラメタで
```http
?query=
```

- POST✖︎ `application/x-www-form-urlencoded`の場合、body部に
```http
query=
```

2. 置き換えた後、ステップ1でコピーしたクエリをGraphQLタブに貼り付ける
	- 自動でURLエンコードされる

3. 送信し、適切な値がレスポンスで返るか確認する

- ==⚠️==：元々のリクエストのbody部には記載してあっても、GraphQLタブで閲覧できない値がある。例えば元のリクエストbodyに記載してある、以下の値は閲覧できない。
```json
,"operationName":"changeEmail",variables
```
- このとき、この値も元の位置に組み込むこと(🚨`,`→`&`、`:`→`=`、`"`→削除)
```http
&operationName=changeEmail&variables
```

###### 機能する理由

- CSRF 脆弱性からの保護のため、本番用 GraphQL エンドポイントでは、`content-type` が `application/json `の POSTリクエストのみを受け付けることがベストプラクティス。
- ただし、エンドポイントによっては、Content-type が `x-www-form-urlencoded` のPOSTや、GET リクエストなど、別のメソッドを受け付ける場合もあるから
