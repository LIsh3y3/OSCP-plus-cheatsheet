###### 署名検証の欠陥について

- JWTのトークンはクライアント側に保存される。
- つまりサーバは元の値や署名について知らない。
- サーバが署名について適切に検証しないケースが脆弱

---
## 署名が検証されない欠陥

###### 基本概念

- JWTのライブラリではトークンを検証するものと、単純にデコードするものがある
- 例えばNode.jsのライブラリ`jsonwebtoken`では、`veryfy()`と`decode()`がある
- このとき、アプリケーションが`decode()`にトークンを渡すのあれば、署名が全く検証されないということになる
	- [📕](../../Misc/📕.md#検証(JWT))

### 悪用ステップ

1. JWTは認証・認可等で発行されるので、ログイン等の操作を実行する
2. Json Web tokenタブでペイロードを目的の値に変更する(もしくは追加する)
```json
{  
    "iss": "portswigger",  
    "sub": "wiener",  
    "exp": 1706861370  
}
```
	↓
```json
	"sub": "administrator"
```

3. リクエストをSend

## 署名なしのtokenを受け入れる欠陥

###### 基本概念

- JWTではヘッダで`alg`パラメタを使用している
- この値をサーバは暗黙的に信頼するしかない
- "unsecured JWT"と呼ばれる

- `"alg": "none"`(署名なしのトークン)はサーバに拒否される事が多いが、`none`という文字列でフィルタリングして拒否している事が多い
- フィルタリングは難読化のテクニックを使用してバイパスできることもある

### 悪用ステップ

1. JWTは認証・認可等で発行されるので、ログイン等の操作を実行する
2. Json Web tokenタブでペイロードを目的の値に編集する
3. 左下の"Attack" -> "none" Signing Algorithmを選択
4. リクエストをSend

- もし`"alg": "none"`がダメなら以下のような難読化を試す
	- nOnE (大文字小文字ミックス)
	- noneをbase64urlencode以外の方法でエンコードして組み込む
