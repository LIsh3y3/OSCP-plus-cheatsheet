
## Server-side template injectionとは

- テンプレート構文を利用して悪意のあるペイロードをテンプレートに注入し、サーバーサイドで実行させること
- ==ユーザの入力がデータとして渡されるのではなく、テンプレートに直接連結される場合==に発生する可能性がある
	- 「データとして」：変数などを介して安全にデータとしてテンプレートに渡される
	- 「連結される」：入力された値がプログラムによってそのままテンプレートの一部として埋め込まれ、サーバー側でテンプレートとして評価される

#### template engineとは

- テンプレート＝雛形

- テンプレートに動的な入出力を記述することで一定の処理を行う仕組み
- 基本的に、テンプレートは静的にし、アプリケーションの実行中には変更せず、ユーザ名のような動的に変更したい部分にデータを流し込む形で使用
- 基本テンプレートエンジンにはサンドボックスが実装されていて、好きなコマンドを自由に実行できるとは限らない。しかし、サンドボックスの欠陥をつくことは可能。

###### ↓テンプレートエンジンの見た目

```html
<html>
	<head>
		<title>{{ title }}</title>
	</head>
	<body>
		<h1>Hello {{ username　}}</h1>
	</body>
</html>
```

- 簡単なSSTI例　 7\*7の結果である49が出力されたらSSTIかも
```html
<html>
	<head>
		<title>{{ title }}</title>
	</head>
	<body>
		<h1>Hello {{ 7*7　}}</h1>
	</body>
</html>
```

## SSTI脆弱性の発生原因

###### 発生する条件

- 外部入力に基づいてテンプレートを作成している
- 外部入力をテンプレートに直接連結している
	- つまり、入力された値がプログラムによってそのままテンプレートの一部として埋め込まれ、サーバ側でテンプレートとして評価される

- 管理者やコンテンツ編集者など、特定の権限のあるユーザに意図的にカスタムテンプレートの編集や投稿を許可している時、攻撃者がこのようなアカウントを侵害し、外部入力をテンプレートに連結するカスタムテンプレートを作成するケースもある。

###### 具体例

- 動的コンテンツがレンダリングされるプレースホルダを提供するだけの静的テンプレートは、一般的にSSTIに対して脆弱ではない。
- 例えば、Twig テンプレートから抜粋した、各ユーザに名前で挨拶するメール：
```php
$output = $twig->render("Dear {first_name},", array("first_name" => $user.first_name) );
```

- しかし、値を動的に編集しようとレンダリング前にユーザ入力を直接テンプレートに連結すると脆弱になる
```php
$output = $twig->render("Dear " . $_GET['name']);
```

- このような場合、以下のようにして悪意あるペイロードをテンプレートに組み込むと、サーバ側でテンプレートとしてペイロードが評価され、実行される
```
http://vulnerable-website.com/?name={{bad-stuff-here}}
```

- SQLi([1. SQLi基本と対策](../../Server-side/SQLi/1.%20SQLi基本と対策.md))と似ている

## SSTIの影響

- RCE
- 機密情報開示
- XSS

---
## SSTI対策

###### 根本対策

- ユーザの入力値をテンプレートエンジンに連結しない(入力された値をそのままテンプレートの一部として埋め込まない)

###### 軽減策（ユーザの入力データをテンプレートエンジンに連結する必要がある時）

- logic-lessなテンプレートエンジンを使用する
	- Mustacheなど

- サンドボックス環境でのユーザのコードの実行： 
	- ユーザーが提供するコードは危険なモジュールや関数を削除した安全な環境（サンドボックス）で実行すべき
	- しかし、信頼されていないコードのサンドボックス化は難しく、回避される可能性がある

- サンドボックス化されたコンテナ環境を利用する：
	- 例えば、任意のコード実行がほぼ避けられないと理解した上で、Dockerコンテナを使用してテンプレート環境を展開し、独自の隔離策を講じること
