- HTTP/1でもHTTP/2でも悪用可能なテクニック
###### Response queue poisoningとは

- フロントエンドサーバがバックエンドからのレスポンスを間違ったリクエストにマッピングし始めるという、強力なRequest smuggling
- つまり、同じ接続を使用する全てのユーザが、(半永久的に)ある特定のユーザへの応答を見ることができる

- ==完全なる==1つのリクエストを密輸し、2つのレスポンスをバックエンドから引き出す。
	- 通常のHRSは不完全なリクエストを密輸し、次のリクエストの開始として処理させてい

---
###### Response queue poisoningの影響

- 攻撃者によって、他のユーザへのレスポンスを傍受できる
- 機密データが含まれているので、被害者のアクセスを完全に侵害できる
- 攻撃者以外の他のユーザからすると、レススポンスがランダムな値を返すので、WebアプリのDoSともなる

---
## Response queue poisoningの構築

###### Response queue poisoningの前提条件

1. フロントエンドとバックエンド間のTCP接続が何回か再利用されること
2. バックエンドが、通常のリクエストだけでなく、密輸した完全なるリクエストに対してもレスポンスを返すこと
3. サーバに「これは無効なリクエストだ」と判断されてTCP接続が遮断されることがないようにすること
	- フロントエンドとバックエンドのどちらのサーバにも有効なリクエストとして認識される必要がある

---
#### Request smugglingの余波を理解しておくこと

- HRSでは通常、部分的なリクエストの密輸を行う
- サーバは密輸されたリクエストを接続上の次のリクエストの先頭に追加する。

- 密輸されたリクエストの内容が、リクエストを密輸した後の接続の状態に影響を及ぼす可能性がある

###### 影響

- 密輸したリクエストにbodyを含むと、後続のリクエストは密輸リクエストのbodyとして見なされる。
- 密輸リクエストの`CL`の長さによって、後続のリクエストはbodyから切り離され単なるバイト列のゴミになり、==無効なリクエスト==として処理される
	- 通常のHRSは後続のリクエストをゴミとし、密輸したリクエストを正規のものとしてバックエンドに解釈させる狙いがあったのでこれで良かった

- 結果、接続が遮断される。[3.  Response queue poisoning](3.%20%20Response%20queue%20poisoning.md#Response%20queue%20poisoningの前提条件)の3を満たさなくなる

###### 具体例

- 例えば、攻撃者が密輸したリクエストがリクエストラインとヘッダだけの場合
```http
...
0

GET / HTTP/1.1
Host: example.com
```
 - バックエンドサーバは、後続のリクエストと密輸されたリクエストを2つの完全に別なものと解釈する

- しかし、密輸したリクエストにbodyを含む場合
```http
...
0

GET / HTTP/1.1
Host: example.com
Content-Length: 20
Content-Type: application/javascript

anything
```
- バックエンドサーバは後続のリクエストが、密輸されたリクエストの「コンテンツの余り」として解釈するので、後続のリクエストは単なるバイト列と見なされてしまう。

---
### 完全なリクエストを密輸する

- [3.  Response queue poisoning](3.%20%20Response%20queue%20poisoning.md#Response%20queue%20poisoningの前提条件)と[3.  Response queue poisoning](3.%20%20Response%20queue%20poisoning.md#Request%20smugglingの余波を理解しておくこと)を踏まえて、この攻撃では完全(正常)なリクエストを密輸する必要がある
- 1つのリクエストで正確に2つのリクエストを送ると、後続のリクエストも有効なリクエストとして解釈される
- つまり、TCP接続が維持される

- ↓HTTP/1のCL.TEの場合
###### フロントエンド(CL)
```http

Content-Type: x-www-form-urlencoded\r\n

\r\n
0\r\n
\r\n
GET /anything HTTP/1.1\r\n
Host: vulnerable-website.com\r\n

```

###### バックエンド(TE)

#### レスポンスqueueの非同期化

![[Pasted image 20240130145320.png | 400]]
- 完全なリクエストを密輸すると、バックエンドは2つのレスポンスを生成する
- 1つ目のResponse1は1つめのリクエストにマッピング(対応付)されて返される
- しかし、2つ目のResponse2は応答待ちのリクエストが無いため、==接続上のキューに保持==される
- こうなると、後続のRequest3に対してResponse2が返り、Response3はキューに保持 -> Request4に対しResponse3が返り、Response4はキューに保持....というように、リクエストとレスポンスの==対応付けにズレが生じる==

### 他のユーザへのレスポンスを盗む
![[Pasted image 20240130145938.png | 300]]

- 一旦Request queue posioningが成功すると、攻撃者は被害者がリクエストした後に==正常なリクエストを送るだけ==で、他のユーザのレスポンスを入手できる
- ⚠️：どのレスポンスを受け取るかはわからないしコントロールできない。

- しかし、Intruderで正常なリクエストを送信し続けている間に、被害者によるリクエストが行われれば、レスポンスに興味深い情報が返るかもしれない
- TCP接続の再利用上限回数(一般的に100回)に到達するまでレスポンスを盗み続けられる
- 接続が閉じられたら、また[3.  Response queue poisoning](3.%20%20Response%20queue%20poisoning.md#完全なリクエストを密輸する)

###### レスポンス窃取のテクニック

- 盗んだレスポンスと攻撃者自身のリクエストに対するレスポンスとの違いを区別しやすくするため、通常のリクエストと密輸リクエストの両方で存在しないパスを指定する
- 結果、攻撃者のリクエストに対するレスポンスは404が返るので、被害者のレスポンスと違いがわかりやすい

## Response queue poisoningステップ

- H2.TEの場合(HTTP/2)

1. 完全な1つのリクエストを密輸する
	- [3.  Response queue poisoning](3.%20%20Response%20queue%20poisoning.md#レスポンス窃取のテクニック)にあるように常に404を返すようなリクエストを送る
```http
POST /x HTTP/2
Host: [target-hostname]
Content-Type: application/x-www-form-urlencoded
Transfer-Encoding: chunked

0

GET /x HTTP/1.1
Host: YOUR-LAB-ID.web-security-academy.net
```
- ==⚠️== 密輸リクエストの後に、`\r\n`を追加すること
	- 一つの完全なリクエストとして認識してもらうには、1つの空白行が必要
 ![[Pasted image 20240130155107.png | 300]]

2. 一度リクエストをし、5秒ほど待つ。またリクエストし5秒待つ。繰り返して、被害者のリクエストに対するレスポンスと思われる、404以外のレスポンスStatusが返るまでリクエストを送り続ける
3. 被害者のレスポンスから機密情報(session)などを窃取する

- うまくいかない場合、通常のリクエスト(`GET /`)を10回程送ることでTCP接続を意図的に遮断させる
