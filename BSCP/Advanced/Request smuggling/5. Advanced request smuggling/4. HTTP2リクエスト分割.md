- [3.  Response queue poisoning](3.%20%20Response%20queue%20poisoning.md)のように、body部を使ってリクエストを分割し完全なるリクエストを密輸できる
- しかし、HTTP/1へダウングレードするのであれば、==ヘッダを使用しての分割==も可能

## ヘッダを使用したリクエストの分割

###### メリット

- body部を含めないGETメソッドなどでも使用可能。汎用性が高い
- `CL`の長さが検証されたり、バックエンドが`TE`をサポートしていなくても使える

## フロントエンドによる書き換えを考慮したリクエストの分割

###### 基本概念

- リクエストをヘッダで分割する際には、リクエストがフロントエンドサーバによってどのように書き換えられるかを理解し、==書き換えを考慮して適切な箇所にHTTP/1.1ヘッダを追加する==必要がある

###### ヘッダによる分割でのResonse queue poisoning具体例

- フロントエンドサーバは`:authority`を取り除き、代わりにHTTP/1.1 `Host`ヘッダに置き換える
- この方法にはさまざまな手段がある

- 例えば以下のようなリクエスト分割を行うとする
![[Pasted image 20240130183613.png | 250]]
- ここで、フロントエンドサーバが新しい`Host`ヘッダを現在のヘッダリストの最後に追加する仕様だとする
- そうすると、以下のようなリクエストになってしまう
```http
GET / HTTP/2
foo: bar

GET /admin HTTP/1.1
Host: vulnerable-website.com
Host: vulnerable-website.com //フロントエンドにより追加されたヘッダ
```
- こうなると、最初のリクエストが完全なリクエストで無くなる

- 対策として、フロントエンドの書き換えを考慮し、以下のように密輸する
![[Pasted image 20240130183917.png | 280]]
↓
```http
GET / HTTP/2
foo: bar
Host: vulnerable-website.com

GET /admin HTTP/1.1
Host: vulnerable-website.com　//フロントエンドにより追加されたヘッダ
```

###### ヘッダによるリクエスト分割のTip

- Response queue poisoningだけでなく、通常のHRSにも使えるテクニック。
- その場合は、[1. HTTP2 request smuggling](1.%20HTTP2%20request%20smuggling.md#H2.CLのTip)にあるように、`Content-Length`を密輸したリクエストのbody部よりも少しだけ大きくすることで、密輸リクエスト後続のリクエストにより攻撃を邪魔されるのを防ぐ
