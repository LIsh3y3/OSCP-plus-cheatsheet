
## シリアライゼーションとは

![[Pasted image 20240107144048 1.png | 500]]
- (↑シリアライズ・デシリアライズのイメージ)

#### シリアライズ

- オブジェクトやそのフィールドのような複雑なデータ構造を、連続したバイトストリームとして、送受信可能な「平坦な」フォーマットに変換するプロセスのこと
	- フィールド：個々のインスタンスが持つ変数、メンバ変数等の変数データのこと

- シリアライズしてもオブジェクトの属性や値は保持される

###### 可能になること

- 複雑なデータの書き込み →　プロセス間メモリ。ファイル。DB。
	- プロセス間メモリ：複数のプロセスで共有するメモリ

- 複雑なデータの送信 →ネットワーク経由で。アプリケーション内で異なるコンポーネントに。APIコールで。

#### デシリアライズ

- シリアライズで変換されたバイト・ストリームを、まったく同じ状態で、元のオブジェクトとして機能するレプリカに復元するプロセス。
- Webサイトは他のオブジェクトと同じように、このデシリアライズされたオブジェクトとも対話できる
![[Pasted image 20240110123342.png | 500]]

#### シリアライゼーションの注意

- オブジェクトのシリアライズ形式はプログラミング言語によって異なる
	- バイナリ形式、文字列形式など

- 元のオブジェクトの属性は、private fieldも含めてすべてシリアライズされたデータ・ストリームに格納される
	- 「private field」とは、オブジェクト指向プログラミングにおけるクラス内で宣言された変数のうち、クラス外部から直接アクセスできないものを指す。
	- クラスのカプセル化を守るために使われ、クラスの内部状態を隠蔽する役割を果たす
	- `private`修飾子を使って宣言されたフィールド(対義：`public`)

- そのため、フィールドがシリアライズされないようにするには、クラス宣言で`transient`も付与する(↓例)
```java
class MyClass implements Serializable {
  private String value;
  private transient int tmp;    // シリアライズ対象外
}
```

###### 言語ごとのシリアライゼーションの呼び方の違い

- Ruby：　マーシャリング
- Python：　ピックリングと呼ぶ

## Insecure deserializationとは

![[Pasted image 20240110124346.png | 400]]

- ==ユーザが制御可能なデータがWebサイトによってデシリアライズされること==
- 攻撃者はシリアライズされたオブジェクトを操作し、悪意あるデータをアプリケーションコードに渡すことが可能
	- シリアライズされたオブジェクトをまったく別のクラスのオブジェクトに置き換えることも可能。
	- 🚨Webサイトが利用可能なあらゆるクラスのオブジェクトが、どのクラスが期待されていたかに関係なくデシリアライズされ、インスタンス化される。
	- [📕](../../Misc/📕.md#クラス、インスタンス、オブジェクト)
- オブジェクトインジェクションとも呼ばれる

- 多くのデシリアライズベースの攻撃は、デシリアライズが終了する**前**に目的達成される。つまり、WEbサイト自身の機能が悪意のあるオブジェクトと直接やりとりしていなくても、デシリアライズの処理自体が攻撃を開始する可能性がある。

## 安全でないデシリアライズの脆弱性はどのようにして生じるのか？

- ユーザの入力がデシリアライズされるから
- ユーザの入力をデシリアライズすることによる危険性の認識が欠けているから
- デシリアライズされた**後**のデータのチェックをしているので安全と思っているから

- バイナリ形式でシリアライズ化されたものは悪用できないと考えているから
	- 実際には、(大変だけど)文字列形式と同様に悪用可能

- ほとんどのWebサイトが多くの異なるライブラリを実装し、それぞれに依存関係があり、クラスやメソッドが大量に存在するので、悪意のあるデータがどのメソッドを呼び出すかを予測するのは事実上不可能。
- 攻撃者が想定外のメソッド呼び出しを長い連鎖で組み合わせて、最初のデータソースとは全く関係のないSinksにデータを渡すことができる。そのため、悪意のあるデータの流れを予測し、全ての潜在的な穴を塞ぐことはほとんど不可能
	- 連鎖：クラスやメソッド同士でデータを受け渡していくこと

## 安全でないデシリアライズの影響

- RCE
- PrivEsc
- 任意のファイルアクセス
- DoS

## 安全でないデシリアライズへの対策

###### 確実

- 絶対に必要な場合を除いて、ユーザの入力をデシリアライズしないこと。悪用される危険性がユーザ入力のデシリアライズによる利点を上回ることがほとんど。

###### もしユーザの入力をデシリアライズしないといけないなら...

- デシリアライズ処理を開始する**前**にデータ改ざん防止のための対策を取る
	- デジタル署名など

- 標準機能のデシリアライズは使わない。機密情報を含むおそれのあるprivate fieldもシリアライズするから。
- 代わりにクラス固有のシリアライズメソッドを自作し、どのフィールドが公開されるかを制御する
```php
class MyClass implements Serializable {
  private String value;
  private transient int tmp;    // シリアライズ対象外
}
```

###### 勘違いしないようにすること

- gadget chainの存在が問題なわけではない。gadget chainはライブラリの依存関係でほぼ確実に存在するもので、すべてのgadget chainを取り除き対策することは不可能
- 問題なのはユーザの入力をデシリアライズしていること

- gadget chainが見当たらなくても、安全でないデシリアライゼーション脆弱性を使って悪用できるメモリ破壊の方法がインターネットに転がっている。RCEにつながる。
