###### HTTP Host headerとは？

- HTTP/1.1では必須のリクエストヘッダ
- クライアントがアクセスしたいドメイン名を指定
- リクエストがリバースプロキシ等によって転送される場合等のケースでは、`Host`の値は意図したバックエンドに到達する前に変更される可能性がある

---
## HTTP host headerの存在目的

- クライアントがどのバックエンドコンポーネントと通信したいかを特定するため
- IPv4の枯渇もあり、同じIPアドレスで複数のアプリケーションを運営する方法が増えている

#### 同じIPアドレスで複数のアプリケーションを運営するときの課題

###### 仮想ホスティングの課題

- 複数のWebサイトやアプリケーションが単一のWebサーバーでホストされている
- それぞれのWebサイトは異なるドメイン名を持つが、共通のIPアドレスを使用する
- Webサイトが単一のサーバーでホストされているため、ユーザにとっては個別のWebサイトと専用のサーバでホストされたWebサイトの区別がつきにくい。

###### 中間システム(LBやリバースプロキシ)を介したトラフィックのルーティングの課題：

- Webサイトが独立したバックエンドサーバにホストされているが、クライアントとサーバの間のすべてのトラフィックが中間システムを経由してルーティングされるため、バックエンドへの適切なリクエストのルーティングが必要。
- Webサイトは別々のバックエンドサーバにホストされているが、すべてのドメイン名が中間システムの単一のIPアドレスに名前解決されるため、仮想ホスティングと同様の課題が発生する

#### Hostヘッダによる課題の解決

- 上記の2つの課題で、`Host`ヘッダは意図された受信者を指定するために使用される。
- ブラウザがリクエストを送信する際、対象のURLは特定のサーバのIPアドレスに解決される。
- サーバはリクエストを受け取った際、`Host`ヘッダを参照して適切なバックエンドを判断し、リクエストを転送することで課題を解決する

## HTTP Host header攻撃とは

![[Pasted image 20240125115324.png | 400]]
- Host header injection(*HHI*)とも呼ばれる
- サーバがHostヘッダを暗黙的に信頼し、検証やエスケープを適切に行わない場合に起きる

- しかも、URLを動的に生成するために`Host`ヘッダを使用していると以下の攻撃ベクターとなり得る
	- Web cache poisoning 
	- 特定の機能におけるビジネスロジックの欠陥
	- Routing-based SSRF 
	- SQLiなどのサーバサイドの脆弱性

- 詳細は[3. HHI脆弱性の悪用](3.%20HHI脆弱性の悪用.md)

## HHIの対策

###### Hostヘッダの使用を避ける

- サーバサイドのコードで`Host`ヘッダを完全に使わないようにする。
- 絶対URLが本当に必要かを再検討し、可能なら相対URLを使用する。
	- Webキャッシュポイズニング対策になる

###### 絶対URLを保護する
    
- 絶対URLを使用する場合は、現在のドメインを設定ファイルに手動で指定し、`Host`ヘッダの代わりにこの値を参照する。

###### Hostヘッダを検証する
    
- `Host`ヘッダを使用する必要がある場合は、許可されたドメインのホワイトリストと照合し、リストにないホストへのリクエストを拒否またはリダイレクトする。
- 例えば、Djangoフレームワークは設定ファイルに`ALLOWED_HOSTS`オプションがある

###### ホストを上書きできるヘッダをサポートしない
    
- `X-Forwarded-Host`などを禁止する。サードパーティではデフォルトで使用可能な場合がある

###### 許可されたドメインのホワイトリストを作成する
    
- 内部インフラに対するルーティングベースの攻撃を防ぐため、LBやリバースプロキシを設定して、許可されたドメインのホワイトリストにのみリクエストを転送するようにする。

###### 内部専用の仮想ホストに注意する
    
- Virtual hostを使用する場合は、内部専用のWebサイトやアプリケーションをホストするサーバと公開コンテンツをホストするサーバとを切り分ける
- 同じサーバ上でホストしてしまうと、攻撃者が`Host`ヘッダの操作を通じて内部ドメインにアクセスできる可能性がある。
