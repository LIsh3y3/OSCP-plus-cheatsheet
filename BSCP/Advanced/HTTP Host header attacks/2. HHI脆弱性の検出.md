## ドメイン検証の欠陥を探す

- ドメイン名のみチェックしポートをチェックしないケース
```http
GET /example HTTP/1.1
Host: vulnerable-website.com:[bad-stuff-here]
```

- ホワイトリストで後方一致のケース(侵害済みのサブドメインを指定する)
```http
GET /example HTTP/1.1
Host: pwned-subdomain.vulnerable-website.com
```

## あいまいなリクエストを送る

###### 基本概念

- ホスト名を検証するコードと、そのホスト名を使用して処理を実行するコードが異なるコンポーネントにある場合
- `Host`ヘッダの値の解釈に違いが出る
- システムによっては、同じリクエストでも異なるホストとして認識されることがある。
- 曖昧なリクエストにより、あるシステムでは正当なホスト、別のシステムでは異なるホストとして認識させる

#### 2つのHostヘッダの注入

- どちらか片方に正当なドメイン名を、もう片方にペイロードを注入
```http
GET /example HTTP/1.1
Host: vuln.com
Host: bad-stuff-here　// 順不同
```
- エラー表示で次へ
###### 有効な理由

- もしフロントエンド(リバースプロキシやLB等)が最初のHostを優先しバックエンドサーバが2番目のHostを優先する場合
- 最初のHostでターゲットサイトへルーティングし、2番目のHostの値をサーバサイドのコードに渡す

#### Hostヘッダの前にスペースを追加する

```http
GET /example HTTP/1.1
    Host: bad-stuff-here
Host: vuln.com //順不同
```

###### 有効な理由

- ヘッダ前のスペースについて解釈は2通りある
	1. 折り返し行と解釈し、先行のヘッダの値の一部として扱う
	2. スペース付きのヘッダは完全に無視する

- フロントエンド(LB等）がスペース付きヘッダを無視すると、リクエストは`vuln.com`へ普通にリクエストする。
- ここで、バックエンドが先頭のスペースを無視し、重複のヘッダは最初のヘッダを優先するとする
- この不一致は、「ラップされた」Hostヘッダを通して任意の値を渡せるかも

#### リクエスト行に絶対URLを指定する

- リクエスト行には絶対URLを指定 (`http` or `https`)
- Hostヘッダにはペイロードを注入
```http
GET https://vuln.com/ HTTP/1.1
Host: bad-stuff-here
```
- Routing-based SSRF

###### 有効な理由

- リクエスト行は通常相対パス(`GET /`)
- 絶対URLとHostヘッダの両方を指定するとあいまいさが生じる。
- 異なるシステム間の解釈の不一致が起きる
