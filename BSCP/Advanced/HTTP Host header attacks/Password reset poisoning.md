## Password reset poisoningとは

![[Pasted image 20231127212818.png | 500]]

- Password resete poisoningは、==パスワードリセットトークンを盗む==方法

###### パスワードリセットトークンの前提知識

- パスワードのリセットは、ユーザにトークンを発行しそれを含んだURLをユーザに送り、Webサイト側でトークンを検証するという仕組み
- トークンがユーザ本人しかアクセスできないという前提で成り立つ。

###### X-Forwarded-Hostヘッダとは

- HTTP の Hostリクエストヘッダー内でクライアントから要求された元のホストを特定する
- つまり、クライアントが本来要求したのはこのホストですよ、な情報

- 例えば:
```http
X-Forwarded-Host: example.com
```
- クライアントが `example.com` にリクエストを送り、そのリクエストがロードバランサーによって `server1.example.com` に転送された場合
- その後続くサーバは、クライアントが元々 `example.com` をリクエストしたという情報を取得することができる 

- これを悪用すると、「`HOST`ヘッダに指定した値にアクセスしようとしているけど、元々は`X-Forwarded-Host`に指定した値のドメインにアクセスする予定だったんだよ」とWebサイトに理解させ、Webサイトが==X-Forwarded-Hostのドメインへリダイレクトする==よう仕向ける

### 攻撃可能なシチュエーション

- `Host` / `X-Forwarded-Host`の値がユーザ制御可能である
- `Host` / `X-Forwarded-Host`ヘッダの値を検証していない
- `Host` / `X-Forwarded-Host`ヘッダの値で動的にURLを生成している

## 攻撃手法

- パスワードリセットリクエストの`Host`もしくは`X-Forwarded-Host`ヘッダに攻撃者のサーバhostnameを指定する
	- エラーが起きた場合：`Referer`ヘッダの値も攻撃者のサーバの"URL"に変更する

###### 攻撃の具体的な流れ

1. ターゲットユーザのユーザ名・メールアドレスを使用してパスワードリセットリクエストを送信する。リクエストをinterceptし、`HOST`ヘッダを攻撃者のサーバhostnameに変更する

2. 攻撃者のサーバでリセットトークンを確認する
3. webサイトへアクセスし、パスワードをリセットしてアカウントを侵害する

- パスワードリセットリンクを制御できなくても、Hostヘッダを使って機密メールにHTMLを注入できる場合がある。
