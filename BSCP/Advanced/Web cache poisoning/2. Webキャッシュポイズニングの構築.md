
## ①unkeyの特定

###### なぜUnkeyを特定するのか

- Webキャッシュポイズニングの基本は、Cache keyとして評価されない、かつ、サーバがサポートしている"Unkey"を特定・操作し、ペイロードを注入することだから
	- [1. Web cache poisoningの基本と対策](1.%20Web%20cache%20poisoningの基本と対策.md#Cache%20key)

### 1. Param Minerを使用したUnkeyの特定

1. リクエストを右クリック-> Extensions -> Param Miner -> Guess Params -> ”Guess headers"
	- 本来のリクエストヘッダに存在しないヘッダがUnkeyのときもある
	- ==⚠️==: 1度のParam Minerで1つのUnkeyヘッダしか検出できない場合もある。つまり、複数ヘッダが悪用に必要なら、必要なヘッダの数だけ、それぞれ異なるリクエストにParam Minerをかける必要があるときもある([3. キャッシュ設計の欠陥の悪用](3.%20キャッシュ設計の欠陥の悪用.md#複数のヘッダを使用したWebキャッシュポイズニング))

2. Param Minerで注入したヘッダーがレスポンスに影響を与えたリクエストを確認する
	- Pro: "Issues"ペインから
	- Community : Extensions -> Installed -> Param Miner -> Output

##### 注意
- Param Minerを使用しない検出方法では、テスト中に生成したレスポンスを実際のユーザに返してしまうことがある。このため、Cache keyが一意となる(どれとも被らない)様にキャッシュバスターを追加する
```http
GET /?cb=1234
```

- Param Minerでは自動でキャッシュバスターが付与されているので心配なし

### 2. Param Minerで見つけられない場合

- リクエストに固有の値があり、それがレスポンスに反映されている箇所を探す

## ②バックエンドサーバから有害なレスポンスを引き出す

### 1. 特定したUnkeyへの入力を、Webサイトがどのように処理しているかを理解する

- ⚠️：==テスト時はキャッシュバスターを追加する==。これにより入力値がすぐさまキャッシュされ、入力値の反映箇所を確認しやすい。2回リクエストし、キャッシュされているかを[2. 実装の欠陥の探索](4.%20キャッシュ実装の欠陥の悪用/2.%20実装の欠陥の探索.md#キャッシュされたレスポンス%20or%20サーバからのレスポンス?)で確認する
```http
GET /?cb=1234
```

###### 確認事項

- どこに入力値が反映される？
- 入力がサニタイズされずにレスポンスに反映されていないか？
- 他のデータを動的に生成するために入力値が使用されていないか？
	- ユーザ制御可能な値を動的に組み込むのは危険

### 2. 悪用するペイロードを注入する

- このときはキャッシュバスターを削除すること。本来のキャッシュを汚染するため。

## ③レスポンスをキャッシュさせる

- キャッシュされるかどうかは、以下の様なあらゆる要因に依存する。
	- ファイル拡張子
	- Content-Type
	- URLパス
	- Status Code
	- レスポンスヘッダなど。

- このレスポンスはキャッシュする値があるとキャッシュに思わせる

- いろいろなページでリクエストをしてみて、キャッシュがどのように動作するかを研究する
