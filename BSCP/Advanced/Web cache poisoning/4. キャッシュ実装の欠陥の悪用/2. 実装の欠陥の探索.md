## 1. 適切なCache oracleの特定

###### Cacheオラクルとは

- キャッシュされたかどうかで反応が何か異なるページやエンドポイントのこと

#### キャッシュされたレスポンス or サーバからのレスポンス?

###### Cacheされている兆候

- キャッシュがヒットしたかどうかを示すHTTPヘッダ(`X-Cache: hit`等)
- 動的に生成されるコンテンツの変更点
- レスポンスの応答時間の違い

###### サードパーティ製のキャッシュ(CDN等)が使用されている場合、何がCache keyかを明らかにする方法

1. 特殊なヘッダを見つける。
2. 該当サードパーティのドキュメントを参照し、Cache keyを直接確認できる機能などの便利な情報がないか探す

- 例えばAkamai(CDNサービス提供)をベースとしたWebサイトなら`Pragma: akamai-x-get-cache-key`というヘッダがCache keyを直接確認するために使える
```http
GET /?param=1 HTTP/1.1
Host: innocent-website.com
Pragma: akamai-x-get-cache-key //🚨: Burpでは独自にx-get-cache-keyとなる
```
```http
HTTP/1.1 200 OK
X-Cache-Key: innocent-website.com/?param=1 //どれがCache keyか示す
```

## 2.Cache keyの処理方法の調査

### 調査すること

1. 何がCache Key?
2. Cache Keyの中で変換処理により何がUnkeyとなる？
	[1. Cache keyの欠陥](1.%20Cache%20keyの欠陥.md#Cache%20keyのコンポーネントに対する変換処理)

###### ↑の意味

- Cache keyに渡されるデータとアプリケーションコードに渡されるデータの不一致を探す
- データの不一致がある-> ==「通常の」Cache keyを 保持したまま、アプリケーションにペイロードを渡す==ことが出来そう

### 方法

- Cache Keyと思われるコンポーネントを操作し、==何かしらの変換==が行われたかを確かめる。
- その際、「2つ」の同じようなリクエストを送り、レスポンスを比較する

1. Cache keyコンポーネントに検証用ペイロードを加えたもの。キャッシュされるまでリクエストする(↓例)
```http
...
Host: example.com:1337
```
2. 「通常の」Cache keyでのリクエスト
```http
...
Host: example.com
```

3. ステップ2で、ステップ1でキャッシュされたレスポンスが返ってくるかを確かめる。
	- つまり、この例ではHostヘッダがCache Keyだが、ポート番号が変換処理によりUnkeyということ
