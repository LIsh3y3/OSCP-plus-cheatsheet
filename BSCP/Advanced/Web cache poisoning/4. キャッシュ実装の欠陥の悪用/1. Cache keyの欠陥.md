- このディレクトリでは、Cacheingシステムの特定の実装の癖を悪用することで、Webキャッシュポイズニングのより大きな攻撃対象へのアクセス方法を示す

###### Cache keyが悪用できないかもと思われていたわけ(Unkeyのみだった)

- WebサイトはURLパスとクエリ文字列を使用して情報を得る
- URLパスとクエリ文字列は他の攻撃手法ではよく使われるが、Webキャッシュポイズニングにおいてはキャッシュバスターとして機能してしまい、悪意あるペイロードを被害者に配布するのに使用できないと考えられていた。
	- そのため、Unkeyを悪用してきた([3. キャッシュ設計の欠陥の悪用](../3.%20キャッシュ設計の欠陥の悪用.md))

## Cache keyのコンポーネントに対する変換処理

- しかし、実際には多くのWebサイトやCDNは、Cache keyのコンポーネントに対してさまざまな変換を行う：
	- クエリ文字列の排除
	- 特定のクエリパラメタのフィルタリング
	- Cache keyコンポーネントに対する入力の正規化

### 変換がもたらすおかしな挙動

- ==Cache keyに書き込まれるデータと、アプリケーション・コードに渡されるデータの不一致==を引き起こす。
- 変換されることで、Cache Keyコンポーネントは他のユーザと同じもの。しかし、アプリケーションコードに渡した悪意あるペイロードのレスポンスをキャッシュさせることが可能になる

- ==つまり、「通常の」Cache keyを 保持したまま、アプリケーションにペイロードを渡すことを可能にする==

###### 具体例

- Cache keyのコンポーネントの中に、特定の文字列を排除するものがあるとする。　例えばポート番号を排除する例
```http
Host: example.com:1337
```
- この中で、Cache Keyは`example.com`のみ。`1337`は排除するとする。このリクエストに対するレスポンスをキャッシュさせる
```http
HTTP/1.1 302 Moved Permanently
Location: example.com:1337/en
Cache-Status: hit
```

- 他のユーザが以下のリクエストを送る
```http
GET / HTTP/1.1
Host: example.com
```
- Cache keyが`Host: example.com`のみで`1337`を排除しているが、キャッシュ時にアプリケーションコードには`example.com:1337`が渡されていたので、以下のレスポンスが返る
```http
HTTP/1.1 302 Moved Permanently
Location: https://example.com:1337/en
Cache-Status: hit
```

- つまり、Cache Keyが`Host: example.com`でポート番号を排除するので、`Host: example.com`にアクセスする他のユーザは、汚染された`example.com:1337`がレスポンスに返ってしまう
