
## Webキャッシュポイズニングとは

- Webサーバとキャッシュの挙動を悪用することで、有害なHTTPレスポンスを他のユーザに「配布」する攻撃

###### Webキャッシュポイズニングの2段階のフェーズ

1. 攻撃者はバックエンドのサーバから、何らかの危険なペイロード（悪意のあるデータ）を含むレスポンスを導き出す方法を見つける
2. そのレスポンスがキャッシュされ、意図した犠牲者に提供されるようにする

### Webキャッシュの基本

- キャッシュはサーバとユーザの間にある
	- (このコンテキストでのキャッシュはサーバ側NWのリバースプロキシとする）

- 特定のリクエストに対するレスポンスを一定時間保存する。
- 別のユーザが同じようなリクエストを送信した場合、キャッシュはバックエンドサーバと通信することなく、キャッシュしたレスポンスのコピーを直接ユーザに提供する。
- これにより、サーバが処理しなければならない重複リクエストの数を減らすことで、サーバの負荷を大幅に軽減する

#### Cache key

- キャッシュしたレスポンスを返すべきリクエストかどうか判断するためのキー
- キャッシュキーは一般的にリクエスト行(`GET~`）と`HOST`ヘッダを結合した値
	- それ以外のコンポーネント(他のヘッダ等)は=="Unkey"==と呼ばれる

- ⚠️：Unkeyなコンポーネントをキャッシュサーバはすべて無視する

## Webキャッシュポイズニングの影響

- XSS、JavaScript injection、オープンリダイレクトなどの脆弱性を悪用し、多くの異なる攻撃を分散させる
- ＊キャッシュの有効期限はこの攻撃において問題でない。攻撃はキャッシュを無期限に再ポイズニングするような方法でスクリプト化できるので。

#### 影響の大きさを決める２つの要因

###### 攻撃者によってバックエンドサーバに何をキャッシュさせられるか

- ポイズニングされたキャッシュは、単独の攻撃というよりも配布の手段である
- 影響度は「注入されたペイロードがどれだけ有害であるか」ということと関係が深い。
- (他の種類の攻撃と同様に、Webキャッシュポイズニングも他の攻撃と組み合わせて使用することで、影響をさらに大きくできる）

###### 影響を受けたページの人気度

- ポイズニングされたレスポンスは、キャッシュがポイズニングされている間に影響を受けたページにアクセスしたユーザにのみ提供される。
- そのため、アクセスするユーザが増えれば増えるほど影響は大きい

## Webキャッシュポイズニングの対策

###### 根本対策

- キャッシュを完全に無効にする: 
	- これは最も確実な方法。

###### 軽減策

- 静的レスポンスのみをキャッシュする
	- 何をもって「静的」となすのかが重要。攻撃者が静的リソースの脆弱なバージョンを取得させることがある。

- サードパーティ技術のセキュリティレベルを理解する
	- 一番弱い場所があなたのセキュリティレベル。サードパーティの技術を利用する場合は、そのセキュリティの影響を完全に理解すること

- 不要なヘッダーを無効にする
	- サイトの機能に必要でないヘッダーは無効にすべきです。

- パフォーマンス向上のためにCache keyから除外しない
	- 代わりにリクエストを書き換える

- fat GETサポートをしない
	-  一部のサードパーティ技術ではデフォルトでこれが許可されている

- クライアントサイドの脆弱性を修正する
	- 悪用できないように見えるものでも、キャッシュの予測不可能な挙動によって悪用可能になることがある
