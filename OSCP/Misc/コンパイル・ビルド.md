# クロスコンパイル環境の準備・実行

- Windows環境で実行することが想定されているものは、Linux上で通常の方法ではコンパイル・実行はできないので、クロスコンパイルをする必要がある
- 目的はバイナリの動作検証

1. mingw-w64（クロスコンパイル用のツール）のインストール
```zsh
sudo apt install mingw-w64
```

2. [wine](https://www.winehq.org/)のインストール
```zsh
sudo apt install wine
```
```zsh
# i386(x86)を有効化→パッケージ更新→wine32のインストール
sudo dpkg --add-architecture i386 && apt update &&
apt install wine32
```
- wineとは、Linuxなどの環境でWindows用ファイルを実行できるもの。検証に用いられる。

3. クロスコンパイル（以下は32ビット）
	- 64ビットの場合は、`x86_64-w64-mingw32-gcc`
```zsh
i686-w64-mingw32-gcc [example.c] -o [example.exe] -lws2_32
```
- エラーが出たら調べて対策
- `-lws2_32`：ws2_32DLLの*l*ibraryを静的リンクすることで、`/usr/bin/i686-w64-mingw32-ld: /tmp/cchs0xza.o:42341.c:(.text+0x97): undefined reference to '_imp__WSAStartup@8'`エラーを解消
	- ライブラリがリンクされていない、というリンクエラー

4. ファイル実行時にWindowsへソケット接続が正常に開始することを確認するため、Wiresharkでインターフェースをキャプチャする（VPN接続時はtun0インターフェースをキャプチャする）：[🦈Wireshark](../Tools/🦈Wireshark.md)

5. Windows環境で動作が想定されている実行可能ファイルを実行する
```zsh
wine [example.exe]
```
- エラー："wine: could not load kernel32.dll, status c0000135"の場合、
	- `rm -r ~/.wine`
	- `winecfg`
	- ポップアップでそのままOK
	- 再度実行
	- [Source](https://forum.manjaro.org/t/wine-could-not-load-kernel32-dll-status-c0000135/69811)

# Windows CLIからのコンパイル

## C

Microsoft Visual C++（MSVC）もしくはMinGWを使用する
```powershell
# MSVC
'C:\Program Files\Microsoft Visual Studio\<version>\VC\Tools\MSVC\<ver>\bin\Hostx64\x64\cl.exe' <example.c> /Fe:<example.exe>

# MinGW
C:\MinGW\bin\gcc.exe <example.c> -o <example.exe>
```

## C++

MinGWもしくはMSVCを利用
	MSVCは[コンパイル・ビルド](#C言語)と同じ方法で
```powershell
# MinGW g++
'C:\MinGW\bin\g++.exe' <example.cpp> -o <example.exe>
```

## C\#

.NET Frameworkもしくは.NET SDKを利用
```powershell
# .NET Framework（古い）
C:\Windows\Microsoft.Net\Framework\<version>\csc.exe <example.cs>

# .NET SDK（新しい）
C:\Program Files\dotnet\dotnet.exe <example.cs>
```

# Python Package Management (pip)

## 基本概念

- pip とは
    - Pythonのパッケージ管理ツール
    - 標準ライブラリに含まれない外部プログラム（パッケージ）を [PyPI](https://pypi.org/) から自動ダウンロード・インストールする

- pipx と pip の使い分け
    - pipx: 「独立したコマンド（ツール）」としてインストールする場合（例: `impacket`, `bloodhound`）        
    - pip: 「ライブラリ（Pythonコード内で `import` するもの）」としてインストールする場合（例: `aerospike`, `requests`）

## ⚠️ 制約：PEP 668 (externally-managed-environment)

- 現在の Kali や Ubuntu 等の最新 OS では、システム環境の破壊を防ぐため、システム全体への `pip install` が制限されている
- 理由：`sudo pip install` でシステム全体のライブラリを上書きすると、OS自体が動作に使用している Python 環境が壊れるリスクがあるため
    
- 回避策： プロジェクトごとに 仮想環境（venv） を作成してその中でインストールする

# 構築手順（具体例：Aerospike の導入）

公開エクスプロイト等で `import aerospike` が必要な場合のセットアップ手順
```python
# requires aerospike package from pip
import aerospike
# if this isn't installing, make sure os dependencies are met
# sudo apt-get install python-dev
# sudo apt-get install libssl-dev
# sudo apt-get install python-pip
# sudo apt-get install zlib1g-dev
```

1. OS側の依存関係（ビルドツール）をインストールする
```zsh
sudo apt update
sudo apt install python3-dev libssl-dev zlib1g-dev
```

2. システム環境から隔離された仮想環境（venv）を作成
```zsj
python -m venv <venv_name>
```

3. 作成した仮想環境を現在のシェルに適用して有効化
```zsh
source <venv_name>/bin/activate
```
- →有効化されるとプロンプトの先頭に (`<venv_name>`) が表示される

4. 仮想環境内でライブラリをインストール
	- 仮想環境が有効化された状態で `pip` を実行すると、システムを汚さずにインストールが可能
```zsh
pip install aerospike
```

5. スクリプトの実行
```zsh
python <exploit_script.py>
```

6. 仮想環境を抜ける
```zsh
deactivate
# 再度使う場合は、ステップ3のsource...を実行
# もういらないなら、仮想環境を削除
rm -rf <venv_name>
```

# エラー集

## GLIBC_xxx not found

- ターゲット環境でコンパイルする
- ツールなら、古いバージョンを使用する
	- 以下のコマンドで表示されたバージョンがreleaseされた日と近い日にrelaseされたバージョンを探す
```zsh
ldd --version
# GLIBC xxx
```

## `socket.gaierror: [Errno -2] Name or service not known`：

- gai : Get Address Infoの略
- 名前解決に失敗したときに出るエラー

| 原因         | 具体例                  | 解説                                       |
| ---------- | -------------------- | ---------------------------------------- |
| 書式のミス      | `192.168.1.1:8080`   | `HTTPConnection` 等にポート番号まで含めて渡してしまった。    |
| プロトコルの混入   | `http://192.168.1.1` | IPアドレスを入れる場所に `http://` などのスキームを含めてしまった。 |
| タイポ（打ち間違い） | `192.168.1.1500`     | 存在しない数字や、ドメイン名のスペルミス。                    |
| 名前解決不可     | `target.local`       | `/etc/hosts` に登録していない名前を指定した。            |
