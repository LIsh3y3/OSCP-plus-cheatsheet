##### マクロに関する前提知識

- マクロは一連の操作を自動化する仕組みのこと
- Officeソフトでは、VBA（Visual Basic for Applications）がほぼ唯一の正式なマクロ言語
- VBAマクロは、ActiveXオブジェクトとWindows Script Host(WSH)へのフルアクセスが可能
	- 💥つまり、システムファイルの操作やネットワーク通信、プロセスの実行などが可能
- 命名規則はキャメルケースが多い（`HogeFoo`）

###### VBファミリーにおけるイベントとは

- JavaScriptのイベントと同義
- 何かされたときに自動で呼ばれるトリガー
- 命名規則はキャメル・スネークケースで命名される（`Hoge_Foo`)
- 例`Document_Open()`：ドキュメントが開かれたときにトリガーし、ボディ部の処理を実行
```vb
Sub Document_Open()

End Sub
```

##### ActiveXオブジェクト

- Windows専用の技術で、アプリ間で機能やデータをやり取りするための仕組みのこと
- WordやExcel、Internet Explorerなど、Windowsアプリケーション同士がお互いの機能を使えるようにする技術
- COM(Component Object Model)という仕組みを使ったコンポーネントの１つ

###### 主な用途：

- ファイル操作（読み書き、削除など）
- レジストリの操作
- *Shell*（コマンド実行）
- ブラウザ制御（IEの自動化）
- ネットワーク通信（XMLHTTP など）

##### COM・DCOM

###### COM

- Component Object Modelの略で、Microsoft社が提唱した、ソフトウェアの機能を部品化して外部から呼び出して利用する仕組みを定めた技術仕様の一つ
- 主にWindowsシリーズのOSやその上で動作するソフトウェアで利用されている
- プログラムの機能を外部から呼び出して利用するための手順やデータ形式などの標準を定めており、<u>COMの仕様に則って開発されたソフトウェア間は容易に連携して動作</u>させることができる
- 特定のOSやプログラミング言語に<u>依存しない仕様</u>になっており、異なる言語やOSで開発されたプログラムを連携させることができる

###### DCOM

- COMがローカル上のソフトウェア間で連携するための仕様であるのに対し、DCOM (Distributed COM) はネットワークを跨いでソフトウェア間で連携できるようにしたCOMの拡張版
- 通信はRPCを介して行われるため、TCP 135が使用される

##### Windows Script Host(WSH)

- Windows OSに組み込まれているスクリプト実行環境のこと
- WSH用のスクリプトである`.vbs`や`.js`ファイルをダブルクリックで実行可能にしてくれる

###### VBScript

- VBAと同じくVisual Basicファイミリー
- VBAはOfficeソフト上で実行されるのに対し、VBScriptはデスクトップ上で実行される
	- ※ 以下比較表ではVBAマクロはOfficeアプリの自動化を用途としているが、VBAもActiveXやCOMオブジェクトを通じて、WSHにアクセスできる（＝つまりShellの操作も可能）

| 比較項目          | VBA（Visual Basic for Applications）          | VBScript（VBS）                     |
| ------------- | ------------------------------------------- | --------------------------------- |
| ✅ 用途          | Officeアプリ（Excel, Word, Outlookなど）の自動化       | Windows全体でのスクリプト実行                |
| 🧩 実行環境       | Officeアプリ内のエディタ（Alt + F11）                  | Windows Script Host（`.vbs` ファイル）  |
| 🔧 機能         | 豊富（UI、フォーム、イベント、Officeオブジェクト、**ActiveX操作**） | 限定的（基本的なスクリプトと**ActiveX操作**）      |
| 🖼 UIの作成      | 可能（UserFormなど）                              | 不可能（フォームなし）                       |
| 📦 オブジェクトアクセス | ExcelのセルやWordのドキュメントなどを操作可能                 | それらを直接操作するにはOfficeを呼び出す必要あり       |
| 🛡 セキュリティ     | マクロ設定で制限・無効化が可能                             | `.vbs`はダブルクリック即実行されるためマルウェアに悪用される |
| 📁 拡張子        | `.xlsm`, `.docm`, `.pptm`などに内包              | `.vbs` ファイル単体                     |

- VBAでメモ帳を起動する場合：
```vb
Sub MyMacro()
	CreateObject("WScript.Shell").Run "notepad.exe"
End Sub
```
	`WScript.Shell`＝WindowsシステムにアクセスするためのCOM（Active X）オブジェクト
	CreateObjectはグローバルオブジェクト

- VBSでメモ帳を起動する場合：
```vbscript
CreateObject("WScript.Shell").Run "notepad.exe"
```
	ファイルに保存後、ダブルクリックで実行

###### WScript・CScriptの違い

| 項目      | WScript.exe                          | CScript.exe                 |
| ------- | ------------------------------------ | --------------------------- |
| 名称      | Windows Script Host（GUIモード）          | Console Script Host（CUIモード） |

| 既定のホスト  | 拡張子 `.vbs` や `.js` をダブルクリックで実行時に使われる | コマンドライン実行時に手動指定可能           |

##### MetasploitのContextって？

- Metasploitの「context」とは、攻撃者が侵入しようとしているシステムに関する情報のことです。たとえば、攻撃者がどのような攻撃手法を使用しているか、どのような情報を収集しているか、攻撃が成功した場合にどのようなリソースにアクセスすることができるかなどです。これらの情報は、攻撃者が攻撃を行う上での判断材料となり、次のステップを決定するために重要な役割を果たします。
- 例えば、簡単なシチュエーションを考えてみましょう。あなたが宝物を盗むために敵の城に侵入しているとします。あなたが現在いる場所、どのようなツールを使用しているか、あなたを追跡する敵の数、敵の城の地形、宝物がある場所など、あなたが持っている情報はすべて「context」として考えることができます。この情報を使って、あなたはどこに行って、どのような行動を取るかを決定することができます。
- Metasploitの場合、攻撃者が目標システムに侵入しようとしているとき、Metasploitは攻撃者に関する情報を提供し、「context」を作成します。攻撃者はこの情報を使用して、次のステップを決定し、成功に向けた戦略を立てることができます。

##### SUID bit, SGID bit

- SUID:　ファイルを作成したユーザの権限でファイルを実行できるよう設定する。SUIDビットは実行時にファイル所有者じゃなくてもファイルの所有者が持つ権限を使うことを可能にするフラグ。SUIDがあるファイルは通常のユーザがファイルを所有するユーザーの権限（通常はroot権限）でファイルを実行できる。Priv Escのキッカケになりやすい
- SGID:　Groupオーナーの許可を得て実行する。同じディレクトリに作成されたファイルには同じGroupオーナーが設定される

##### SID, RID

- SID (Security Identifier) と RID (Relative Identifier) は、Windows のセキュリティモデルにおける[用語](#プリンシパル)の識別に使われる重要な概念
- LSASSが発行

###### SID

- Windows 内でユーザーやグループ、コンピューターアカウントを**一意に**識別するための識別子。一度生成されると変更されない。
- [well-known SID](https://learn.microsoft.com/en-us/windows/win32/secauthz/well-known-sids)がある

```bash
# S-R-X-Y
S-1-5-21-3623811015-3361044348-30300820-500
```
- `S`：SIDであることを示す
- `R`（`1`）：版数を表す。常に1。将来SIDの構造が変わると2になる可能性。
- `X`（`5`）：SIDを発行したIdentifier Authoriy（5 は "NT Authority"）
- `Y`（`21` + `-3623811015-3361044348-30300820`）： ドメインまたはコンピューターを識別する部分（サブAuthority）。必ず一意になるようランダムな値。Built-in(組込)プリンシパルについては、`21`の部分が必ず`32`になる。
- `500` ：RID（[用語](#RID%20(Relative%20Identifier))）

###### RID (Relative Identifier) 

- RID は、SID の末尾の番号で、特定のアカウントやグループを識別するために使われる。同じドメインやローカルシステム内では、この RID の部分が異なることで、それぞれのアカウントを区別できる。
- 作成されたユーザーは1000から順に割り当てられる（例:1001は2番目に作成されたユーザー）

**代表的な RID の一覧：**

| RID     | アカウント名        | 説明                     |
| ------- | ------------- | ---------------------- |
| 500     | Administrator | Windows の管理者アカウント      |
| 501     | Guest         | ゲストアカウント               |
| 512     | Domain Admins | ドメイン管理者グループ            |
| 513     | Domain Users  | ドメインユーザーグループ           |
| 514     | Domain Guests | ドメインゲストグループ            |
| 1000 以上 | 通常のユーザー       | 作成されたローカルユーザーやドメインユーザー |

##### SAM(Security Account Manager)

- 保存場所： `C:\Windows\System32\config\SAM`
- Windowsにおけるローカルアカウントに関する情報を保存するデータベース
- ==ローカルのユーザー名、パスワードハッシュ、セキュリティID（SID）==などの情報が含まれる
- SYSTEM権限/BackupPrivでのみアクセス可

##### SYSTEM

- 保存場所：`C:\Windows\System32\config\SYSTEM`
- OS のハードウェア・ソフトウェア構成情報
- 暗号鍵（BootKey）を含む
- このキーが SAM に保存されているハッシュを復号するために必要

##### LSA

- Local Security Authority
- Windowsのセキュリティポリシー・ログオン認証・アクセス制御などを管理するセキュリティサブシステム（プロセス名：`lsass.exe`）。
	- SIDの生成もする
- 保存ファイル：SECURITYハイブ（`C:\Windows\System32\config\SECURITY`）
	- LSA ⊃ SECURITYハイブ
- 「LSA Secrets」として、サービスアカウントのパスワードや、<u>ドメインユーザーのキャッシュ認証情報</u>などが含まれる。
	- ※ドメインアカウントの管理自体は Active Directory 側で行われる。

##### ACL, DACL, ACE

ACLはアクセス制御のリストであり、DACLはそのリストの一種であり、ACEはACL内の個々のエントリ



$$アクセス制御の見た目$$

###### ACL(Access Control List):

- リソースへのアクセスを制御するために使用されるセキュリティメカニズム全体のことをいう
1. クライアントが対象のオブジェクトを使いたいというとき、オブジェクトに対しアクセストークンを送る
2. オブジェクトはアクセストークンをACLと突合し、許可/却下する
- DACLとSACL(System Access Control List)で構成され、SACLは自由に変更できない

###### DACL(Discretionary Access Control Lists):

- ACLの中身。日本語訳は任意アクセス制御リスト
- 所有者によって自由自在に変更可能


###### ACE(Access Control Entry):  

- ACLの中身で、「どのプリンシパルに」「どんな操作が」「許可/ 拒否されているのか」を定義する

##### トポロジー

- トポロジーは、ネットワークやシステムの構造や配置を表す用語です。Active Directory（AD）の場合、トポロジーはドメインコントローラ（Domain Controller、DC）間の接続と情報の同期方法を指す

##### VSS (volume shadow copy service)

###### volumeとは

- コンピュータのストレージ（外部記憶装置）の管理単位の一つで、OSによって区画付けされた、ひとつながりの連続した記憶領域をボリュームという。OSによって識別名を付けられることがあり、これをボリューム名あるいはボリュームラベルという。
- HDやSSDなどの場合、全体を一つのボリュームとする場合と、複数のパーティションに区切ってそれぞれをボリュームとする場合がある。

###### VSSとは

- [Microsoft documentation page](https://docs.microsoft.com/en-us/windows-server/storage/file-server/volume-shadow-copy-service)
- Windowsのストレージ管理機能の一つで、指定されたボリュームの内容を自動的に複製し、専用の領域に保管する機能。
- 特定の時点におけるファイルのコピーを自動的に作成し、専用の記憶領域で管理する。
- これにより、ファイルをうっかり消してしまったり上書きしてしまったときに、保管してあるものを取り出して復旧することができる

##### NTLM周りの用語

| 用語                  | 同義語                 | 用途・使用場面               | 取得方法                   | 攻撃での使い方              |
| ------------------- | ------------------- | --------------------- | ---------------------- | -------------------- |
| LM ハッシュ             | LAN Manager ハッシュ    | ローカルの古い Windows 認証    | SAM ダンプ、LSASS、NTDS.dit | オフラインクラック可、PtH にも使用可 |
| NT ハッシュ             | NTLM ハッシュ           | 現代 Windows 認証のパスワード保管 | SAM ダンプ、LSASS、NTDS.dit | オフラインクラック可、PtH に使用可  |
| NTLM                | NTLM = プロトコル全体      | ログオンやファイル共有等の認証プロトコル  | -                      | 帳票として区別せず使用される       |
| NTLMv1 / Net‑NTLMv1 | NTLMv1 = Net‑NTLMv1 | ネットワーク越しの認証に使用        | ネットワークキャプチャ（Responder） | レスポンスをクラック／リレー攻撃に使用可 |
| NTLMv2 / Net‑NTLMv2 | NTLMv2 = Net‑NTLMv2 | 改良版ネットワーク認証方式         | ネットワークキャプチャ（Responder） | レスポンスをクラック／リレー攻撃可    |

###### NTLMのハッシュの読み方



- 3番目のフィールドがLMハッシュ
- 4番目のフィールドがNTハッシュ

##### manifest

- manifest（マニフェスト）は、Windowsアプリケーションに関する情報を含むXML形式のテキストファイルです。アプリケーションマニフェストは、アプリケーションの属性や要求を指定し、UACプロンプトの表示やアプリケーションの自動昇格などの動作を制御します。

##### 名前付きパイプ

- パイプとは、プロセス間の入出力をつなげる仕組み
- 名前がついていないパイプは、`|`で使われる
```zsh
ls | grep xxx
```

- 名前付きパイプは`mkfifo`コマンド等で作られ、ファイルシステム上に存在する
	- ただしファイルエクスプローラーで見ることはできない
- ファイルのように機能するが、データはDiskに保存されず、メモリのバッファ上を経由して直接プロセス間でやり取りされる
- ファイルシステム上の名前は、プロセス同士が通信相手を見つけるための「待ち合わせ場所」の役割を果たしている
```zsh
# 名前付きパイプを作成
mkfifo mypipe

# ターミナル1で読み取り待機
cat < mypipe

# ターミナル2でデータ送信
echo "Hello" > mypipe
```

- 名前付きパイプを作成する側（パイプサーバー）とパイプを利用する側（クライアント）は同じシステム上になくてもOK
	- →リモート間でもやり取り可能

##### プロバイダー

- Windows認証を実際に処理するモジュールのこと（`.dll`）
- ローカルログオンやドメイン環境ログオンなどで使い分けられる

| プロバイダー   | 役割                | メモリに保持する情報             |
| -------- | ----------------- | ---------------------- |
| MSV1_0   | NTLM認証            | NTLMハッシュ、平文パスワード（条件付き） |
| Kerberos | Kerberos認証        | チケット、鍵、パスワード           |
| WDigest  | Digest認証          | 平文パスワード（古いOS）          |
| TsPkg    | ターミナルサービス認証       | 資格情報                   |
| LiveSSP  | Microsoft アカウント認証 | トークン                   |
| CloudAP  | Azure AD認証        | トークン                   |

```

│ TsPkg.dll       │ ← RDP認証担当

└─────────────────┘
```
