# CPU

- アドレス：データ、命令などの場所を指す
- 命令＝オペコード + オペランド
	- オペコード：実行する処理

### スタック

- 関数や変数の値を一時的に格納するためのメモリ領域
- バッファ：確保したメモリの空き容量
- ヒープと異なり、サイズは固定
- LIFO（Last-In First-Out）：最後に格納された値が最初に取り出される
	- イメージ例：満員電車のドア付近の人は最後に乗車するが最初に降りる
- 高位のアドレスのほうが古く、低位のアドレスのほうが新しい
	- (便宜上アドレスを数字で表す)
```c
1　// 最新
2 // 1より古い
3　// 最古
```

- スタックのための領域（ここまで使っていいよ）はあらかじめメモリ領域に確保されている
- スタックのための領域を使用するために、sp（スタックポインタ）を下方に移動させ、spまでを使用中と宣言する

![](../画像ファイル/Pasted%20image%2020250605123530.png)

$$スタックポインタの遷移図$$

※それぞれの関数のための領域を"スタックフレーム"という

#### 補足：スタックが生まれた背景

- 以下の要望があった：
	- 関数呼び出しを行うたびに、任意のサイズの領域を確保したい
	- 関数呼び出しはネストしたい
	- 呼び出し元の関数の領域にアクセスできる必要はない
	- 関数呼び出しは基本処理で頻繁に使うので限りなくシンプルな処理にしたい

- 以下のようなデータ構造（スタック）とすることで、ポインタ1個で簡単に要望をかなえることができる
	- ある程度のサイズのメモリ領域を確保
	- 「メモリ領域のどこまでを使っているか？」を指し示すポインタを1個用意する
	- 関数呼び出しで自動変数の領域が必要な場合には、ポインタを移動させる。すると、移動した分の領域は「使用中」の意味になる
	- 関数呼び出しから戻ったら、ポインタを元の位置に戻す。すると、その分の領域は「未使用」の意味になる。
	- 関数呼び出しがネストする場合には、どんどんポインタを移動させていく。

つまり、アセンブラにおいては、==スタックポインタの加減算==でメモリの確保を表している。

##### なぜ下方伸長なのか

- 下方伸長：高位アドレス→低位アドレスにスタックが成長すること

- 理由①：スタックポインタから加算することで、データにアクセスできるから
	- もし下方伸長でないならば、減算でアクセスしなければならなくなる
	- ([アセンブラ](#アセンブラ))
```c
addiu sp,sp,-8 //スタックの確保（減算）・・・①
li v0,254
sw v0,0(sp)
li v0,255
sw v0,4(sp) //加算でspから+4のアドレスにアクセス
addiu sp,sp,-16 //スタックの確保（減算）・・・②
```

- 理由②：開きメモリ領域の終端にspを合わせておけば、あとはヒープ領域などでメモリが"加算方向"に利用されたときに、spにぶつかるまでメモリを利用できるから、ヒープ領域などに固定サイズを割り当てる必要がない。つまり、メモリ領域をスタックとヒープで効率的に利用できるから。

### ヒープ

- 動的に生成した変数や構造体を格納
- ユーザー側で動的にメモリを使用可能
- 都度、使う前にメモリの確保と、使った後のメモリ解放の処理が必要
- malloc：メモリ確保（memory allocate）
- free：メモリ解放
- アドレスが小さい順に割り当てられる
	- (便宜上数字で表す)
```c

2 // 1より新しい
3 // 最新
```

### リターンアドレス(ra)とは

- 1つの関数呼び出しが終わった後に実行されるべき次の命令のアドレスのこと
```c
void foo() {
    bar();   // ← 呼び出しポイント（ここでリターンアドレスが記録される）
    do_something();  // ← bar() 終了後にここに戻る（ここがリターンアドレス）
}
```

- 関数の先でさらに関数を呼び出す場合は、呼び出し先のraで元のraは自動的に上書きされる
- raが上書きされる場合は元のraを退避させ、呼び出した関数の処理が終わった後に、再び元のraで上書きする（↓MIPS）
```c
<call_function1>:
	addiu sp,sp,-24 //スタック割当
	sw ra,16(sp) //ra退避
	jal fe1460 <call_function2> // 関数の先で関数呼び出し
	li a0,254
	addiu v0,v0
	lw ra,16(sp) //ra復元
	nop
	jr ra //ra先に戻る
	addiu sp,sp,24 //スタック解放
```

- スタックの確保、raの退避をあわせてプロローグといい、反対のスタックの解放、raの復元をエピローグという

## レジスタ

- ポインタもそれ専用のレジスタがある（そうでないCPUもある）
- 不揮発性、揮発性のレジスタがある
	- 揮発性レジスタ（volatile）：呼び出された関数が自由に上書きしてOKなレジスタ。呼び出し元が中身の保持を期待してはいけない。上書きされたくないなら呼び出し元の方で退避する必要がある
	- 不揮発性レジスタ（non-volatile）：関数をまたいでも中身が保持されるべきレジスタ。関数がこのレジスタを使いたい場合は、退避してから使う（保存・復元の責任が関数側にある）。

| lr       | Link register                         | リターンアドレスを保持するレジスタ                                                                                                                                                                                                                                                                                     |
| ra       | return address                        | リターンアドレスを保持するレジスタ                                                                                                                                                                                                                                                                                     |

## スタックの確保について

- sp(スタックポインタ)が以下のようにして確保する
```c
addiu sp,sp,-8
```
- つまり、spをより低位のアドレス側にスライドさせることで、メモリを確保する。
- メモリを確保し、"同じ"関数内の変数は第一引数から"順に"格納する(基本的に)
- 次の関数はより低位のアドレス側にスライドさせてから格納するので、LIFOの原則は崩れない

## アドレッシングについて

狭義では、CPUがメモリにアクセスする際のアドレス指定方式のこと
（広義では、メモリだけでなく、即値やレジスタへのアクセス方法も指す）

|アドレス指定方式|説明|例（C風）|アセンブリ的表現|特徴|
|---|---|---|---|---|
|即値指定（イミディエイト）|値そのものを使う|`x = 5;`|`MOV r0, #5`|値を直接埋め込む|
|直接アドレス指定|メモリアドレスを直接指定|`x = *0x1000;`|`MOV r0, [0x1000]`|決まったアドレス|
|間接アドレス指定|レジスタが指すメモリアドレスにアクセス|`x = *p;`|`MOV r0, [r1]`|レジスタの中身をアドレスとして使う|
|相対アドレス指定|プログラムカウンタからの相対位置|`jmp label;`|`B +4`|主にジャンプ命令|
|指標アドレス指定（インデックス）|基準レジスタ + インデックスレジスタ|`x = arr[i];`|`MOV r0, [r1 + r2]`|配列アクセス向け|
|基底アドレス指定（ベース）|基底（ベース）レジスタ + オフセット|`x = struct_ptr->member;`|`MOV r0, [r1 + #4]`|構造体アクセス向け|

| 用語                 | 意味                                     |
| ------------------ | -------------------------------------- |
| レジスタ間接             | レジスタに入っているアドレスをそのまま使う（例：`0(r3)`）       |
| ディスプレースメント付きレジスタ間接 | レジスタのアドレスに定数（オフセット）を足してアクセス（例：`8(r3)`） |
| インデックス付きレジスタ間接     | 2つのレジスタを加算してアドレスにする（例：`r3 + r4`）       |
| ディスプレースメント         | 加える定数（オフセット）のこと（例：`8(r3)` の「8」）        |

## アーキテクチャ（RISC/CISC)

- 明確に区別はできないようになってきている（元々CISC的なアーキテクチャが、RISCの考えを取り入れたり、プログラム上はCISC的にみえても、内部ではRISCのアーキテクチャで動いていたりするから。）
- 適材適所

#### RISC

- Reduced Instruction Set Computer：固定長命令のこと。
- 命令の実行をパイプライン化して、パイプラインを効率的に動作させることを突き詰めたアーキテクチャ。
- 現在の主流
- スマホ、組込みシステムのCPUとして使われることが多い。
- [コンピュータ・アセンブラ・バイナリの基礎知識](#サンプルプログラム)の機械語は4バイトで固定長となっているのでRISC。

##### RISCの特徴

- パイプライン動作：
	- 複数の命令をステージと呼ばれる単位に分割し、並行して処理すること
- キャッシュメモリの搭載：
	- メモリからの読み・書きを早くするため、オペランドをより高速なメモリであるキャッシュメモリに退避する。
- 多数の汎用レジスタ（16〜32本）：
	- データ処理はメモリでなくレジスタ上で行うことを主体として処理速度を予測可能にし、コンパイラによる最適化をしやすくする
	- レジスタは高速、メモリは低速とはっきり区別して、極力レジスタを使うようにする
- ロードストアアーキテクチャ：
	- キャッシュメモリにアクセスするステージはロード・ストア処理のみに限る。
	- なぜなら、パイプライン処理のさまざまなステージにキャッシュメモリを接続する必要性をなくすため。
- レジスタ間コピーとロード・ストアは別命令
- 遅延分岐を使うことがある

#### CISC

- Complex Instruction Set Computer：可変長命令のこと。
- 昔はメモリが高価で低速だったので、命令のサイズを小さく、速度を早くすることを突き詰めたアーキテクチャ。（[CISCが生まれた背景](https://yuyargon.hatenablog.com/entry/2022/03/06/134107)）
- 直交性が高い(≒１つのニーモックで即値、レジスタ、メモリを扱える)

| 操作          | PowerPC(RISC) | MIPS(RISC) | **H8(CISC)** |
| ----------- | ------------- | ---------- | ------------ |
| レジスタへの即値の代入 | li            | li         | mov.w        |
| レジスタ間の値のコピー | mr            | move       | mov.w        |
| メモリからの値のロード | lwz           | lw         | mov.w        |
| メモリへの値のストア  | stw           | sw         | mov.w        |

- 1つの命令が複雑な処理を行える。
- 過去の主流。
- レジスタ間コピーとロード・ストアは同一命令（movで完結）

## バイナリファイルでの機械語コードの位置の求め方

### 前提事項

- アセンブラに記載されているアドレスはメモリ上の「仮想アドレス」で、実際のファイル上のアドレスとは異なるため、計算する必要がある
- バイナリはセクションにわかれていて、一般的に「.text」というセクションに記載されている
	- [コンピュータ・アセンブラ・バイナリの基礎知識](#メモリの領域)
- readelfを使い、「.text」セクションを閲覧すれば、どのoffsetにプログラムがあるかを計算できる

### 手順

1. アセンブラ出力から、目的の命令の「仮想メモリアドレス」を得る
```zsh
objdump -d sample.x
```
	↓
```c#
 00fe1468 <add>:
    fe1468:   7c 63 22 14    add r3, r3, r4
    fe146c:   4e 80 00 20    blr
```
- `00fe1468`：add "命令"の メモリアドレス
- `7c 63 22 14`：この"命令"の バイナリ

2. セクションヘッダ情報を得る
```zsh
readelf -a sample.x
```
	↓
```less
Section Headers:
 [Nr] Name      Type       Addr      Off     Size    ES  Flg Lk Inf Al
 [ 1] .text     PROGBITS   00fe1400  001400  0002b0  00  AX  0  0   16
```
- `.text` セクションの開始位置＝機械語コードの先頭位置：
    - 機械語コードの先頭のメモリアドレス（`Addr`）：`0x00fe1400`        
    - 機械語コードの先頭のファイル相対位置(offset)（`Off`）：`0x001400`
    - [コンピュータ・アセンブラ・バイナリの基礎知識](#セクションヘッダ%20AddrとOffの違い)

3. メモリの差を求める
	「目的の命令(add)の機械語コードのメモリアドレス」（`00fe1468`） - 「`.text`セクション（機械語コード先頭）のメモリアドレス」（`0x00fe1400`）　= `0x68`
	
4. ファイル上の位置を求める
	「機械語コードの先頭のファイル相対位置」（`0x001400`） - 「3で求めたメモリの差」（`0x68`） = `0x001468`

#### 補足情報

##### セクションヘッダ AddrとOffの違い

- Addr：実行時にメモリ上で .text セクションが配置される仮想アドレス
- Off：実行可能ファイル（sample.x）内で、.text セクションが始まるバイト相対位置
```text
【実行ファイル】
--------------------------
| ファイル先頭           | ← オフセット 0x000000
| ヘッダーなど           |
| .text セクション       | ← オフセット 0x001400（Off）
|                       |
--------------------------

【プログラムが実行され、仮想メモリにロードされる】
仮想メモリ：
--------------------------
0x00fe1400: ← .text セクションがロードされるアドレス（Addr）
0x00fe1404: 命令
0x00fe1408: 命令
--------------------------
```

##### `.text`セクションヘッダのカラムとその意味

| 略称   | 意味         | 説明                                                   |
| ---- | ---------- | ---------------------------------------------------- |
| Type |            | セクションの種類（例：`PROGBITS`、`NOBITS`）                      |
| Addr | Address    | セクションがメモリにロードされる仮想アドレス                               |
| Off  | Offset     | セクションがファイル内で始まる位置（オフセット）                             |
| ES   | Entry Size | セクション内の各エントリのサイズ（例えばシンボルテーブルなど）                      |
| Flg  | Flags      | セクションの属性。例：`A`=Alloc（メモリに読み込まれる）、`X`=実行可能、`W`=書き込み可能 |
| Lk   | Link       | 他セクションとの関連（例えばシンボルテーブルがどの文字列テーブルとリンクしているか）           |
| Inf  | Info       | セクションの追加情報（セクションによって意味が異なる）                          |
| Al   | Alignment  | アライメント制約（例：16バイト境界に揃える）                              |
