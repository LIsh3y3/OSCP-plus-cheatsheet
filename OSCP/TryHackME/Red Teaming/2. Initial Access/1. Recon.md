### Intro

- ターゲットのインフラや人員について知っていればいるほど、より良い攻撃を仕掛けることができる
- レッドチームの活動では、まず会社名だけで、そこからターゲットの情報を収集する
- そこで登場するのがRecon。ターゲットに警戒心を抱かせることなく、事前に調査や観察を行うこと

---
### Reconの分類

[[Passive & Active Reconnaissance]]

- Active Reconをさらに分類
	- 外部偵察：ターゲットのネットワークの外で行われ、インターネットから評価可能な外部に面した資産に焦点を当てる。例えば、社内ネットワークの外からNiktoを実行するような場合
	- 内部偵察：ターゲット企業のネットワーク内から実施される。つまり、物理的に会社の建物内にいる。このシナリオでは、ターゲットのネットワーク上の悪用されたホストを使用する。例えば、Nessusを使用して、ターゲットのコンピュータの1つを使用して内部ネットワークをスキャンするなど

## Built-In ツール

- `whois`
- `dig`, `nslookup`, `host`
- `traceroute`/`tracert`

### whois

- ドメインレジストラは、貸し出しているドメイン名のWHOISレコードを管理する責任があり、whois serverがある。これはTCP 43ポートで待ち受ける。
- ドメイン名だけで、多くの貴重な情報を得ることが可能。
- whois検索の結果、運が良ければ、名前、電子メールアドレス、郵便番号、電話番号に加え、その他の技術情報を見つけることができるかも。
- whoisクエリの最後には、ドメインの権威サーバを見つけることが可能
###### 提供する情報
- レジストラのWHOISサーバー  
- レジストラのURL  
- レコード作成日  
- レコードの更新日  
- 登録者の連絡先と住所（プライバシーのため非公開の場合を除く）  
- 管理者の連絡先と住所（"）  
- 技術者の連絡先と住所（"）

### nslookup, dig, host

- デフォルトのDNSサーバーを使用して、ドメインに関連するAレコードとAAAAレコードを取得する
#### nslookup

- 全部で使えるがオプションが少ない。
- 古いやつなので非推奨
- ドメインに関連するAレコードとAAAAレコードを取得する

#### dig

- Unix系システムでよく使われる
- 新しいやつなので推奨
- Domain Information Groper (dig)の略でdig。
- digには多くの問い合わせオプションがある。
- 別のDNSサーバーを指定することも可能。

#### host

- digを簡潔にして見やすくしたもの

### traceroute(unix) / tracert(windows)

- システムからターゲットホストまでのパケットが通ったルートをトレース
- システムに接続するルーター（ホップ）が表示される。
- ただし、tracerouteが送信したパケットに応答しないルーターもあり、その結果、IPアドレスが表示されないことがある

## Advanced Searching

### 検索エンジンで機能する検索修飾語リスト

シンボル・文法 | 機能
-- | --
`"[検索用語]"` | 確実にマッチするものを表示
`[検索用語] filetype:[filetype]` | 特定の用語に関連する指定したファイルタイプのファイルを検索
`[検索用語] site:[FQDN]` | 特定のサイトのみに結果を限定する
`[検索用語] -site:[FQDN]` | 特定のサイトを除外する
`[検索用語] intitle:[タイトル名]` | 指定したタイトルを含みかつ検索用語を含む結果
`[検索用語] inurl:[url内文字]` | 指定した文字を含むURLで検索用語を含む結果
- filetype：pdf, doc, docx, ppt, pptx, xls, xlsxなど

### 機密情報について

- 検索エンジンは、新しいWebページやファイルをインデックスするために、wwwをクロールしている。時には、これが機密情報のインデックス化につながることがある。

#### 機密情報の例

- 社内で使用する文書  
- ユーザ名、メールアドレス、パスワードまで記載された機密性の高いスプレッドシート  
- ユーザー名を含むファイル 
- 機密性の高いディレクトリ 
- サービスのバージョン番号（脆弱性があり、パッチが適用されていないかも）  
- エラーメッセージ

#### GHDB (Google Hacking Database)

- [Google Hacking Database](https://www.exploit-db.com/google-hacking-database)
- Googleの高度な検索と特定の用語を組み合わせることで、機密情報を含む文書や脆弱なウェブサーバを見つけることができる
- GHDBは、このような検索用語を収集し、公開している
- 不用意にアクセスすると法的に罰せられるかもしれないので注意
###### 例

- Footholds ([GHDB-ID: 6364](https://www.exploit-db.com/ghdb/6364)): Nginxのログを発見し、悪用可能なサーバーの誤設定を明らかにするかも
- ユーザ名を含むファイル([GHDB-ID: 7047](https://www.exploit-db.com/ghdb/7047)): 
- 機密ディレクトリ([GHDB-ID: 6768](https://www.exploit-db.com/ghdb/6768)): RSA秘密鍵が公開されているかどうかを調べる
- Web Server検出([GHDB-ID: 6876](https://www.exploit-db.com/ghdb/6876)): GlassFish Serverを探す
- 脆弱なファイル ([GHDB-ID: 7786](https://www.exploit-db.com/ghdb/7786)): PHPファイルの位置を特定
- 脆弱なサーバ([GHDB-ID: 6728](https://www.exploit-db.com/ghdb/6728)) : SolarWinds Orionウェブコンソールを発見
- エラーメッセージ([GHDB-ID: 5963](https://www.exploit-db.com/ghdb/5963))：エラーに関連するログファイルを検索

### SNS, 求人情報

#### SNS

- SNSは、個人的な使用だけでなく、企業からもよく使われる。
- SNSの中には、ターゲットに関する大量の情報を明らかにしているものもある。
- 企業の従業員の名前を簡単に収集できる。
- 特定の情報を知ることで、パスワード回復のための質問の答えや、ターゲットのためのwordlistに含めるアイデアを得ることができることがある。
- 技術スタッフからの投稿では、その会社のシステムやベンダーに関する詳細がわかるかも。例えば、最近Juniperの認定を受けたネットワークエンジニアがいるということは、職場でJuniperのネットワークインフラが使用されている可能性があることを示す

#### 求人

- 名前やメールアドレスがわかるだけでなく、技術職の求人広告を見れば、対象企業のシステムやインフラを知ることができるかも。

## 特化型Search

- whoisコマンドやdigなどのDNSのコマンドの他に、そのためのWebサイトがある
- Whois Historyというwebサイトは有料だが、whoisの結果の履歴をとっているので、ドメイン登録者がドメイン登録時にWHOISプライバシー（非公開にする設定）を使用しなかった場合に便利

### ViewDNS.info

- [ViewDNS.info](https://viewdns.info/)
- ViewDNS.infoは、リバースIPルックアップを提供する。
- 昔は各Webサーバーは1つまたは複数のIPアドレスを使用していた
- 最近では、共有ホスティングサーバーを使用するのが一般的。
- 共有ホスティングでは、1つのIPアドレスが、異なるドメイン名を持つ異なるWebサーバで共有されている。
- IP逆引きでは、ドメイン名またはIPアドレスから始めて、特定のIPアドレスを使用している他のドメイン名を検索することができる
- 下の図では、cafe.thmredteam.comが使用する同じIPアドレスを共有する他のサーバーを見つけるために、IP逆引きを使用した。
- つまり、IPアドレスが必ずしも1つのWebサイトにつながるわけではないことに注意する

##  Recon-ng

- [Recon-ng](https://github.com/lanmaster53/recon-ng)
- Recon-ngは作戦やOSINTタスクに役立つ様々な断片的な情報を見つけるために使用する
- 収集されたデータはすべて、ワークスペースに関連するDBに自動保存される。
- 例えば、後で行うポートスキャンのためのホストアドレスを発見したり、フィッシング攻撃用の連絡先メールアドレスを収集したりする

- Recon-ngを起動する
```zsh
recon-ng
```
- プロンプトが`[recon-ng][default] >`で成功

- ここから、使用するインストール済みモジュールを選択する。ただし、recon-ngを初めて実行する場合は、必要なモジュール（複数可）をインストールする必要がある

###### 流れ

1. プロジェクト用のワークスペースを作成する  
2. データベースに開始情報を挿入する  
3. マーケットプレイスでモジュールを検索し、インストールする前にそのモジュールについて学習する
4. インストールされているモジュールをリストアップし、1つロードする  
5. ロードしたモジュールを実行する

#### ①ワークスペースの作成

- 作成
```zsh
workspaces create [Workspace name]
```
- 作成したワークスペースのプロンプトに自動で変わる

- 起動時に作成したワークスペースを使用する
```zsh
recon-ng -w [Workspace name]
```

#### ②DBにデータを挿入する

- Reconでは、1つの情報から始めて、それを新たな情報に発展させていくことになる。
- 例えば、企業名から調査を開始し、それをもとにドメイン名、連絡先、プロフィールを発見する。

- ターゲットのドメイン名わかっていて、それを現在アクティブのワークスペースのDBに挿入したい場合。
- DBのテーブル名を確認する
```recon-ng
db schema
```
![[Pasted image 20230609151806.png | 100]]
- テーブルの中身を閲覧する
```recon-ng
db query select * from domains
```
- データを挿入する
```recon-ng
db insert domains
domain (TEXT): [domain name]
notes (TEXT):
```

#### ③Recon-ng Marketplaceでモジュールの検索と理解

- ドメイン名を入手している状態だとして、次のステップは、ドメインを他のタイプの情報に変換するモジュールを検索すること
- マーケットプレイスから適切なモジュールを検索する

##### マーケットプレイスの使い方に関連するコマンド

- `marketplace search`: すべての利用可能なモジュールを表示する
- `marketplace search [Keyword]`: キーワードで利用可能なモジュールを検索
- `marketplace info [Module]`: 当該モジュールに関する情報を入手
- `marketplace install [Module]`: 指定されたモジュールをRecon-ngにインストール
- `marketplace remove [Module]`:

#####  モジュールについて

- discovery, import, recon そして reportingなどのカテゴリがあり、さらに情報変換の種類によってサブカテゴリがある。
- サブカテゴリがどのような情報を入手するか示している。
- `whoxy_whoisn`のように、=="K"の列に`*`が付いているものはキーが必要==で、関連するサービスを利用するためのキーがない限り、このモジュールは使用できない。詳細は→[1. Recon](1.%20Recon.md#Keys)
- =="D"列に`*`が付いているものはサードパーティのPythonライブラリが必要==。依存関係があるので
 ![[Pasted image 20230609152847.png]]
 ![[Pasted image 20230609153050.png]]

##### 手順

- モジュールを検索
```recon-ng
marketplace search [keyword]
```
- モジュールについて調べる
```recon-ng
marketplace info [Module]
```
- モジュールをインストール
```recon-ng
marketplace install [Module]
```
- モジュールはフルパスではなくても、最下層の名前だけ指定すればいい。上の画像だと、`cencys_companiese`だけでいい。

#### ④モジュールをロードし実行する

- インストールされているすべてのモジュールをリストアップ
```recon-ng
modules search
```
- 特定のモジュールをメモリにロードする
```recon-ng
modules load [Module]
```
 ![[Pasted image 20230609154912.png | 300]]
- ロードしたモジュールの詳細を表示
```recon-ng
list
```
- ロードしたモジュールに設定できるオプションをリストアップ
```recon-ng
options list
```
- オプションの値を設定
```recon-ng
options set [option] [value]
```
- 実行する
```recon-ng
run
```
- Ctrl+Cでロードしたモジュールをunloadする

#### Keys

- "K"列に`*`がある一部のモジュールは、各サービスAPIのキーがないと使用できない。
- keyをリストアップ
```recon-ng
keys list
```
- keyの追加
```recon-ng
keys add [KEY_NAME] [KEY_VALUE]
```
- keyの削除
```recon-ng
keys remove [KEY_NAME]
```

## Maltego

### 基本

- [Maltego](https://www.maltego.com/)
- 用途はRecon-ngと一緒だがより視覚的
- マインドマップとOSINTを融合させたアプリケーション。
- 最初に入手した情報、例えば、ドメイン名、会社名、人名、メールアドレスなどから始める。
- そして、この情報の断片をトランスフォームしていく
- Maltegoで収集した情報は、後のステージで利用することができる。例えば、収集した会社情報、連絡先名、メールアドレスは、非常に合法的に見えるフィッシングメールを作成するために使用することができる
- Maltegoのグラフ上の各ブロックは、エンティティとして考える。エンティティは、それを説明する値を持つことができる。
- Maltegoの用語では、トランスフォームとは、特定のエンティティに関連する情報を取得するためにAPIに問い合わせるようなコードの一部のこと。
- ![[Pasted image 20230609170039.png]]
- ==Maltegoで利用可能なトランスフォームの中には、ターゲットシステムに積極的に接触するものもある==ので、Passive Reconにとどめたい場合は、そのトランスフォームがどのように機能するかを事前に知っておく
- すべてのトランスフォームは、いくつかの新しい値を導き出すかも。

### 手順例

  ![[Pasted image 20230609165839.png | 400]]

- New > Entity Paletteに"DNS name"と入力し、エンティティをダブルクリックしてcafe.thmredteam.comと入力する
- 使用するトランスフォームに基づいて新しい種類のエンティティを取得する。
- たとえば、エンティティを右クリック > All Transformsをクリック > 検索タブに"IP"と入力 > To IP Address \[DNS\]をクリックする
 ![[Pasted image 20230609170510.png | 300]]
 - IPアドレスを表示してくれる。
 ![[Pasted image 20230609170554.png | 300]]

- IPアドレスの一つを右クリックしてTo DNS Name from passive DNS (Robtex)というトランスフォームを選択する
- ![[Pasted image 20230609171313.png | 300]]

- このようにして、ターゲットマシンの情報収集を効率的にわかりやすく行える。
