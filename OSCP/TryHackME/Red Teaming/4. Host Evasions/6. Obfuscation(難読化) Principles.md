
ツールに依存しないソフトウェアの難読化手法を活用し、悪意のある機能を隠蔽して独自のコードを作成する

## Intro
- 難読化は、悪意のあるソフトウェアの検出回避手法や解析の妨げとなる重要な要素

## 静的検出回避のための難読化機能

- AVとEDどちらにも、静的シグネチャと呼ばれる既知のシグネチャの広範なデータベースと、アプリケーションの動作を考慮したヒューリスティックシグネチャを活用
- シグネチャを回避するために難読化を実施する。正規のアプリケーションに重要な識別情報を隠すデータ難読化を悪用することで実現される
  
- [Layered Obfuscation Taxonomy](https://cybersecurity.springeropen.com/articles/10.1186/s42400-020-00049-3)では、Code-Element Layerの下でこれらのプラクティスをうまく要約している。以下は、Obfuscating Data サブレイヤーにおいて、タクソノミーがカバーする手法の表

## Object Concatenation

- ==Data Splitting/Merging==
- 連結は、2つの別々のオブジェクトを文字列などの1つのオブジェクトに結合する、一般的なプログラミングの概念である。  
  
- あらかじめ定義された演算子によって、2つの独立したオブジェクトを結合するための連結が行われる場所が定義される。以下は、Pythonにおける文字列の連結の一般的な例
```python
>>> A = "Hello "
>>> B = "THM"
>>> C = A + B
>>> print(C)
Hello THM
>>>
```
- プログラムで使用される言語によっては、連結に使用できる演算子が異なるか、複数定義されている場合がある。以下は、一般的な言語とそれに対応する定義済み演算子の表

|Language  |Concatenation Operator|
|---|---|
|Python|“**+**”|
|PowerShell|“**+**”, ”**,**”, ”**$**”, or no operator at all|
|C#|“**+**”, “**String.Join**”, “**String.Concat**”|
|C|“**strcat**”|
|C++|“**+**”, “**append**”|

- マルウェアで使用される連結の最も一般的な例は、[7. Signature Evasion](7.%20Signature%20Evasion.md)にあるように、標的となる静的シグネチャーを破ること。
- また、攻撃者は、プログラムのすべてのオブジェクトを分割し、すべてのシグネチャを一度に削除しようとするためにこれを使用する。
  
- 以下では、静的なYaraルールを観察し、連結を使って静的なシグネチャを回避することを試す
```powershell
rule ExampleRule
{
    strings:
        $text_string = "AmsiScanBuffer"
        $hex_string = { B8 57 00 07 80 C3 }

    condition:
        $my_text_string or $my_hex_string
}
```
- コンパイルされたバイナリをYaraでスキャンすると、定義された文字列が存在する場合、警告/検出が行われる。連結を使用すると、文字列は機能的に同じであっても、スキャン時に2つの独立した文字列として表示されるため、アラートが発生ない。
![[Pasted image 20230621113935.png | 400]]

- 攻撃者は非解釈文字を使用して静的なシグネチャを混乱させることもできる。これらは、シグネチャの強度や実装に応じて、単独で、または連結と一緒に使用することができる。以下は、一般的な非解釈文字の表

|Character|Purpose|Example|
|---|---|---|
|Breaks|1つの文字列を複数のサブ文字列に分割し、それらを結合する|`('co'+'ffe'+'e')`|
|Reorders|文字列の構成要素の並び替え|`('{1}{0}'-f'ffee','co')`|
|Whitespace|解釈されない空白を含む|`.( 'Ne' +'w-Ob' + 'ject')`|
|Ticks|解釈されないticksを含む|``d`own`LoAd`Stri`ng``|
|Random Case|トークンは一般的に大文字と小文字を区別しないので、任意の大文字と小文字を使用する|`dOwnLoAdsTRing`|

## 難読化のAnalysis騙しのための機能

- 悪意のあるコードの基本的な機能を難読化した後、ソフトウェアの検出をパスすることができるかもしれないが、人間からは分析されやすいまま。
- 攻撃者は、高度な論理と数学を活用して、より複雑で理解しにくいコードを作成し、解析とリバースエンジニアリングに対抗する
  
- 難読化レイアウトと難読化コントロールのサブレイヤーにおいて、タクソノミーがカバーする手法の表
 ![[Pasted image 20230621121307.png | 400]]

|Obfuscation Method  |Purpose|
|---|---|
|Junk Code|コードスタブとも呼ばれる、機能しないジャンクな命令を追加する|
|Separation of Related Code|関連するコードやインストラクションを分離して、プログラムの読み取りの難易度を上げる|

## Control Flow and Logic

- 制御フローは、プログラムの実行において重要な要素であり、プログラムが論理的にどのように進行するかを定義するもの。
- 論理はアプリケーションの制御フローを決定する最も重要な要素の1つで、if/else文やforループなど様々な用途がある。
- プログラムは上から下に実行され、論理文に遭遇すると、その文に従って実行を継続する。  
  
- 以下は、制御フローやプログラムロジックを扱う際に遭遇する可能性のある論理文の一覧。

|Logic Statement|Purpose|
|---|---|
|if/else|条件を満たした場合のみ実行し、それ以外の場合は別のコードブロックを実行する|
|try/catch|コードブロックの実行を試み、エラー処理に失敗した場合はキャッチする|
|switch case|スイッチは、if文と同様の条件ロジックに従うが、ブレークまたはデフォルトに解決する前に、ケースでいくつかの異なる可能な条件をチェックする。|
|for/while loop|forループは、条件を満たしている間だけ実行される。whileループは、条件が満たされなくなるまで実行される。|

- このことは、攻撃者にとってどのような意味を持つのか。
- ==control flowのなかをちょっといじって混乱させ悪意のあるコードを難読化する==
- Analystは、制御フローを通じてプログラムの機能を理解しようとする。
- 問題がある一方で、論理と制御フローはほとんど簡単に操作でき、故意に混乱させることができる。
- 制御フローを扱う場合、攻撃者は分析者を混乱させるのに十分な不明瞭なロジックを導入することを目指すが、それ以上の疑念を抱かせたり、プラットフォームが悪意あるものとして検出する可能性があるほどにはしない。

## Arbitrary(任意の) Control Flow Patterns

- 攻撃者がAnalystを混乱させるために使用できるさまざまな制御フローのパターンについて

- 任意の制御フローパターンを作成するために複雑なアルゴリズムを活用し、悪意のある関数に別の制御フローを注入する
- このような複雑な論理や数学のアルゴリズムを作るために、Predicateを活用する。
- Predicateとは、真偽値を返す関数。Predicateは、if文がコードブロックが実行されるかどうかを決定するために使用する「条件」と同じように考えられる

- 不透明なPredicateとは、難読化する側にとっては値がわかっているが、推論が困難なPredicateである。ジャンクコードのような他の難読化手法とシームレスに適用することで、リバースエンジニアリングの試みを困難な作業に変えることができる
- 不透明なPredicateは、プログラムに任意にロジックを追加したり、既存の関数の制御フローをリファクタリングするために使用することができる

### 例
- コラッツ予想は、不透明Predicateの例として使用できる一般的な数学の問題である。
- それは、2つの算術演算を繰り返すと、すべての正整数から1を返す。既知の入力（正の整数）に対して必ず1を出力することがわかるということは、不透明なPredicateとして成立していることを意味する。
- コラッツ予想については、[Collatz Problem](https://mathworld.wolfram.com/CollatzProblem.html)を参照。
- 以下は、PythonでCollatz予想を適用した例
```python
x = 0
while(x > 1):
	if(x%2==1):
		x=x*3+1
	else:
		x=x/2
	if(x==1):
		print("hello!") 
```
- 上のコードでは、コラッツ予想が数学的な演算を行うのは` x > 1 `のときだけで、結果は`1` または `TRUE` となる。コラッツ問題の定義から、正の整数入力に対しては常に1を返すので、xが1以上の正の整数であれば、この文は常に真を返すことになる。

## Protecting and Stripping Identifiable Information

- Meanigless Identifiers
  ![[Pasted image 20230621121307.png | 400]]
- separation of related codeとして知られており、連結と一緒によく見られる
- 識別可能な情報は、Analystが悪意のあるプログラムを分解して理解しようとする際に、最も重要な要素の1つになる。
- Analystが持つ識別可能な情報（変数、関数名など）の量を制限することで、攻撃者が元の機能を再構築できない可能性を高めることがでる
  
- 高度なレベルでは、Object Names(オブジェクト名)、code structure(コード構造)、 file/compilation properties(ファイル/コンパイルプロパティ)という3種類の識別可能なデータを考慮する。

### Object Names
- オブジェクト名は、プログラムの機能に関する最も重要な洞察を提供し、関数の正確な目的を明らかにする。
- Analystは、関数の動作からその目的を分解することはできるはできるけど、関数のコンテキストがない場合、はるかに困難になる

- リテラルオブジェクト名の重要性は、言語がコンパイルされるか解釈されるかによって変わる。
- PythonやPowerShellのような解釈言語が使用されている場合、すべてのオブジェクトが重要であり、修正する必要がある。
- CやC#のようなコンパイルされた言語が使用されている場合、一般的に文字列に現れるオブジェクトのみが重要。
- オブジェクトは、IOオペレーションを生成するあらゆる関数によって、文字列に現れる可能性がある。
- ==とにかく、オブジェクト名をわけわからんやつにしとく==

### Code Structure
- コード構造は、悪意のあるコードのあらゆる側面に対処する際に、見落とされがちで簡単に特定できない厄介な問題とる。
- 解釈言語とコンパイル言語の両方で適切に対処されないと、シグネチャやアナリストからのリバースエンジニアリングを容易にすることにつながる
- ジャンクコードとコードの並び替えは、いずれも解釈型プログラムに複雑さを加える追加措置として広く用いられている。
- プログラムがコンパイルされていないため、アナリストはプログラムに対する洞察力が非常に高く、人為的に複雑さを増していなければ、アプリケーションの悪意のある機能がすぐバレる  
- 関連するコードの分離は、解釈言語とコンパイル言語の両方に影響を与え、識別が困難な隠れたシグネチャをもたらす可能性がある。
- ヒューリスティックシグネチャエンジンは、周囲の関数やAPIコールに基づき、プログラムが悪意あるものであるかどうかを判断することがある。
- これらのシグネチャを回避するために、攻撃者は関連するコードの出現をランダム化し、安全な呼び出しや関数であるとエンジンを欺くことができる

### File & Compilation Properties
- コンパイル方法は分析者を支援する利点につながる可能性がある。
- 例えば、プログラムがデバッグビルドとしてコンパイルされている場合、解析者は利用可能なすべてのグローバル変数とその他のプログラム情報を得ることができる  
- コンパイラは、プログラムをデバッグビルドとしてコンパイルする際に、シンボルファイルを含む。
- シンボルは一般的にバイナリ・イメージのデバッグに役立ち、グローバル変数やローカル変数、関数名、エントリ・ポイントなどを含むことができる。
- 攻撃者は、適切なコンパイルを行い、アナリストに情報が漏れることがないよう、これらの起こりうる問題に注意する必要がある
  
- 攻撃者にとって幸運なことに、シンボルファイルはコンパイラを通じて、あるいはコンパイル後に簡単に削除することができる。
- Visual Studioのようなコンパイラでシンボルを削除するには、コンパイルターゲットをDebugからReleaseに変更するか、mingwのような軽量なコンパイラを使用する
  
- プリコンパイルされたイメージからシンボルを削除する必要がある場合は、コマンドラインユーティリティである`strip`を使用
  
- 詳細は[7. Signature Evasion](7.%20Signature%20Evasion.md#Static%20Property-Based%20Signatures)
