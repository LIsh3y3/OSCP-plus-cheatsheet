## Intro

- Windows APIは、Windows OSの主要コンポーネントと対話するためのネイティブ機能を提供する。
- このAPIは、攻撃者からブルーチームまで幅広く利用されている
- APIはWindowsシステムとシームレスに統合することができ、さまざまな用途で使用できる。
- Win32 APIは、攻撃ツールやマルウェアの開発、EDRエンジニアリング、一般的なソフトウェア・アプリケーションに使用されている。
- APIのすべての使用例に関する詳細については、[Windows API Index](https://docs.microsoft.com/en-us/windows/win32/apiindex/windows-api-list)を参照

## Subsystem and Hardware Interaction

- プログラムはよくWindowsのサブシステムやハードウェアにアクセスしたり変更したりするが、マシンの安定性を維持するために制限されている。
- この問題を解決するために、ユーザモードアプリケーションとカーネルをつなぐライブラリであるWin32 APIをリリースした。
- Windowsでは、ハードウェアへのアクセスをユーザモードとカーネルモードという2つの異なるモードで区別している。
- これらのモードは、アプリケーションやドライバが許可されるハードウェア、カーネル、およびメモリアクセスを決定する。
- APIやシステムコールは、各モード間のインターフェイスとして、カーネルモードで処理される情報をシステムに送信する

|User mode|Kernel Mode|
|---|---|
|直接的なハードウェアアクセスはできない|直接的なハードウェアアクセスが可能|
|「所有されたメモリ位置」へのアクセス|物理メモリ全体へのアクセス|
- メモリマネジメントの詳細 [1. Windows Internal](1.%20Windows%20Internal.md#仮想メモリ)

- User modeまたは"Userland"で開始されたアプリケーションは、System Callが発生するかAPIを介してインターフェースされるまで、そのモードに留まる。システムコールが発生すると、アプリケーションはモードを切り替える。

## Components of the Windows API

- Windows APIとしてより一般的に知られているWin32 APIは、APIの構造と構成を定義するために使用されるいくつかの依存コンポーネントを備えている
- Win32 APIをトップダウンのアプローチで分解する。ここでは、APIを最上位層とし、特定の呼び出しを構成するパラメータを最下位層とする
- 以下の表では、トップダウンの構造を高いレベルで説明し、後でさらに詳細に掘り下げていく

|Layer|説明|
|---|---|
|API|win32 API構造で見られるあらゆる呼び出しを記述するために使用されるトップレベル/一般的な用語または理論|
|Header files or imports|実行時にインポートするライブラリを定義する。ヘッダーファイルまたはライブラリインポートで定義する。関数のアドレスを取得するためにポインターを使用する。|
|Core DLLs|呼び出し構造を定義する4つのDLL群。(KERNEL32、USER32、ADVAPI32）。これらのDLLは、単一のサブシステムに含まれないカーネルとユーザサービスを定義する|
|Supplemental DLLs|Windows APIの一部として定義されたその他のDLL。Windows OSの個別のサブシステムを制御する。その他に定義されたDLLは36個ある。(NTDLL、COM、FVEAPIなど)|
|Call Structures|APIコールそのものとコールのパラメータを定義する。|
|API Calls|プログラム内で使用されるAPIコールで、関数アドレスはポインタから取得|
|In/Out Parameters|Call Structuresで定義されるパラメータ値。|

- [Windows.hについて](https://en.wikipedia.org/wiki/Windows.h)
- User32.dllヘッダーをインポート・定義する

##  OS Libraries

- Win32ライブラリの各API呼び出しは、メモリ内に存在し、メモリアドレスへのポインタが必要。
- ASLR（アドレス空間配置ランダム化）の実装により、これらの関数へのポインタの取得プロセスは隠されている。
- 各言語やパッケージには、ASLRを克服するための独自の手順がある。
- 最も人気のある2つの実装として[P/Invoke](https://docs.microsoft.com/en-us/dotnet/standard/native-interop/pinvoke)と[Windows header file](https://docs.microsoft.com/en-us/windows/win32/winprog/using-the-windows-headers)がある

### Windows Header File
- ASLRの実装に関連する問題の直接的な解決策として、Windowsローダーとしても知られるWindows Header Fileをリリース。
- 大まかに説明すると、ランタイム時にローダーはどの呼び出しが行われているかを判断し、関数のアドレスやポインタを取得するためのthunkテーブルを作成する
- 一旦、`windows.h`ファイルがアンマネージドプログラムのトップにインクルードされれば、どんなWin32関数でも呼び出すことができる。  
- この概念についての詳細 [2. Windows APIのIntro](2.%20Windows%20APIのIntro.md#C%20API%20Implementations)

### P/Invoke
- P/Invokeは、==DllImport属性を使用して宣言されたメソッドによって実現==
- Microsoftは、P/InvokeまたはPlatformインボークを「マネージドコードからアンマネージドライブラリ内の構造体、コールバック、および関数にアクセスできる技術」と説明している

	- マネージドコート:.NET Frameworkや.NET Coreなどのマネージド環境で定義されたメソッドやコード
	- アンマネージドライブラリ: マネージド環境（.NET Frameworkや.NET Coreなど）とは異なるプラットフォームや言語で作成されたライブラリ。これらのライブラリは、ネイティブコード（通常はC/C++など）で書かれており、マネージドランタイムやガベージコレクションなどのマネージド環境の機能を直接利用することはない。システムレベルの機能やハードウェアとの直接的な対話、高速性が求められる場合などに使用される。

- P/Invokeは、マネージドコードからアンマネージド関数またはWin32 APIを呼び出すための全プロセスを処理するツールを提供する。
- アンマネージド関数またはWin32 API呼び出しを含むDLLをインポートすることから始まる。以下は、オプションを指定してDLLをインポートする例
```csharp
using System;
using System.Runtime.InteropServices;

public class Program
{
[DllImport("user32.dll", CharSet = CharSet.Unicode, SetLastError = true)]
...
} 
```
- 上記のコードでは、user32.dllを`DLLImport`属性を使用してインポートしてる

- 注意: セミコロンは含まれていない。
- なぜなら、P/Invoke関数はまだ完全ではないから。次のステップでは、マネージドメソッドを外部メソッドとして定義する必要がある。`extern`キーワードは、以前にインポートされた特定のDLLをランタイムに伝える。以下は、外部メソッドを作成する例
```csharp
using System;
using System.Runtime.InteropServices;

public class Program
{
...
private static extern int MessageBox(IntPtr hWnd, string lpText, string lpCaption, uint uType);
} 
```
- これで、マネージドメソッドとして関数を呼び出すことができますが、アンマネージド関数を呼び出していることになる
- 詳細は[2. Windows APIのIntro](2.%20Windows%20APIのIntro.md#.NET%20and%20PowerShell%20API%20Implementations)

## API Call Structure

- APIコールは、Win32ライブラリの2番目の主要コンポーネント。
- これらの呼び出しは、多くのユースケースを満たすために使用することができる拡張性と柔軟性を提供する。
- ほとんどのWin32 APIコールは、[Windows API documentation](https://docs.microsoft.com/en-us/windows/win32/apiindex/windows-api-list)と[pinvoke.net](http://pinvoke.net/)の下で文章化されている
- APIコールの命名スキームとin/outパラメータについてみていく
- APIコールの機能は、命名スキームを変更し、代表的な文字を追加することで拡張することができる。
- 以下は、Microsoftが命名スキームとしてサポートしている文字の表

|Character|Explanation|
|---|---|
|A|ANSIエンコーディングの8ビット文字集合を表す|
|W|Unicodeエンコーディングを表す|
|Ex|APIコールに拡張機能またはin/outパラメータを提供する|
- 詳細：[Microsoft documentation](https://docs.microsoft.com/en-us/windows/win32/learnwin32/working-with-strings)

- 各APIコールは、その入出力パラメータを定義するために、あらかじめ定義された構造を持っている。これらの構造のほとんどは、[Windows documentation](https://docs.microsoft.com/en-us/windows/win32/apiindex/windows-api-list)の対応するAPIコールのドキュメントページに、各入出力パラメータの説明とともに掲載されている

- 例として、`WriteProcessMemory` APIコールを見てみる。
- 以下は、ここで得られた呼び出しのI/O構造
```cpp
BOOL WriteProcessMemory(
  [in]  HANDLE  hProcess,
  [in]  LPVOID  lpBaseAddress,
  [in]  LPCVOID lpBuffer,
  [in]  SIZE_T  nSize,
  [out] SIZE_T  *lpNumberOfBytesWritten
);
```
- また、Microsoftは各I/Oパラメータについて、その用途、期待される入力または出力、許容される値について説明する  
- 説明を受けても、これらの値を決定することは、特定のコールでは困難な場合がある。
- ==すべてのI/Oパラメータは、Windowsのドキュメントに記載されている==

## C API Implementations

- Microsoftは、CやC++などの低レベルのプログラミング言語に、必要なAPIコールにアクセスするために使用できるライブラリのセットをあらかじめ用意している
- 先ほど説明したwindows.h Header Fileは、Call Structureを定義し、関数ポインタを取得するために使用される。
- windows headerをインクルードするには、CまたはC++のプログラムの前に以下の行を追加する
- `#include <windows.h>`
- APIコールの作成。
- 最初の目的として、タイトルのついたポップアップウィンドウを作成することを目指す
- `CreateWindowExA`を使用して、"Hello THM!"というタイトルのポップアップウィンドウを作成する。
- [2. Windows APIのIntro](2.%20Windows%20APIのIntro.md#API%20Call%20Structure)で説明したことをもう一度確認するために、呼び出しのIn/Out・パラメータを観察する
```cpp
HWND CreateWindowExA(
  [in]           DWORD     dwExStyle, // Optional windows styles
  [in, optional] LPCSTR    lpClassName, // Windows class
  [in, optional] LPCSTR    lpWindowName, // Windows text
  [in]           DWORD     dwStyle, // Windows style
  [in]           int       X, // X position
  [in]           int       Y, // Y position
  [in]           int       nWidth, // Width size
  [in]           int       nHeight, // Height size
  [in, optional] HWND      hWndParent, // Parent windows
  [in, optional] HMENU     hMenu, // Menu
  [in, optional] HINSTANCE hInstance, // Instance handle
  [in, optional] LPVOID    lpParam // Additional application data
);
```
- これらの事前定義されたパラメーターを用いて、値を割り当ててみる。API呼び出しの各パラメーターには、その目的と潜在的な値についての説明がある。
- 以下は、`CreateWindowsExA` の完全な呼び出しの例
```cpp
HWND hwnd = CreateWindowsEx(
	0, 
	CLASS_NAME, 
	L"Hello THM!", 
	WS_OVERLAPPEDWINDOW, 
	CW_USEDEFAULT, CW_USEDEFAULT, CW_USEDEFAULT, CW_USEDEFAULT, 
	NULL, 
	NULL, 
	hInstance, 
	NULL
	);
```
- C言語で最初のAPIコールを定義した。
- あとは、これをアプリケーションに実装して、APIコールの機能を使う
- 以下は、APIを使用して小さな空白のウィンドウを作成するアプリケーションの例
```cpp
BOOL Create(
        PCWSTR lpWindowName,
        DWORD dwStyle,
        DWORD dwExStyle = 0,
        int x = CW_USEDEFAULT,
        int y = CW_USEDEFAULT,
        int nWidth = CW_USEDEFAULT,
        int nHeight = CW_USEDEFAULT,
        HWND hWndParent = 0,
        HMENU hMenu = 0
        )
    {
        WNDCLASS wc = {0};

        wc.lpfnWndProc   = DERIVED_TYPE::WindowProc;
        wc.hInstance     = GetModuleHandle(NULL);
        wc.lpszClassName = ClassName();

        RegisterClass(&wc);

        m_hwnd = CreateWindowEx(
            dwExStyle, ClassName(), lpWindowName, dwStyle, x, y,
            nWidth, nHeight, hWndParent, hMenu, GetModuleHandle(NULL), this
            );

        return (m_hwnd ? TRUE : FALSE);
    }
```

- 低レベルの言語では、APIコールを素早く定義することが非常に簡単。その使いやすさと拡張性から、C言語ベースの言語は、攻撃者の間で最も人気がある

## .NET and PowerShell API Implementations

- P/Invokeでは、DLLのインポートやAPI呼び出しへのポインタの割り当てが可能
- P/Invokeがどのように実装されているかを理解するために、以下の例ですぐに取り掛かり、個々のコンポーネントについてはその後に説明する
```cpp
class Win32 {
	[DllImport("kernel32")]
	public static extern IntPtr GetComputerNameA(StringBuilder lpBuffer, ref uint lpnSize);
}
```

- クラス関数は、定義されたAPIコールと、今後のすべてのメソッドで参照するための定義を保存する
- APIコール構造が格納されているライブラリは、ここで`DllImport`を使用してインポートする必要がある。
- インポートされたDLLは、ヘッダーパッケージと同様の働きをするが、探しているAPIコールを持つ特定のDLLをインポートすることが必要。特定のAPIコールがDLLのどこにあるかは、[API index](https://docs.microsoft.com/en-us/windows/win32/apiindex/windows-api-list)または[pinvoke.net](http://pinvoke.net/)を参照する
- DLLのインポートから、特に`intPtr`で定義された、使用したいAPIコールへの新しいポインタを作成することができる。
- 他の低レベル言語とは異なり、ポインタの中にin/outパラメータ構造を指定する必要がある。
- 必要なAPIコールのin/outパラメータは、Windowsのドキュメントから見つけられる
  
- あとは、定義されたAPIコールをアプリケーションに実装して、その機能を利用することができる。
- 以下は、APIを利用して、実行するデバイスのコンピュータ名などを取得するアプリケーションの例
```csharp
class Win32 {
	[DllImport("kernel32")]
	public static extern IntPtr GetComputerNameA(StringBuilder lpBuffer, ref uint lpnSize);
}

static void Main(string[] args) {
	bool success;
	StringBuilder name = new StringBuilder(260);
	uint size = 260;
	success = GetComputerNameA(name, ref size);
	Console.WriteLine(name.ToString());
}
```
- 成功すれば、プログラムは現在のデバイスのコンピュータ名を返す
  
- 同じ構文をPowerShellで使えるようにする方法を考える
- APIコールの定義は.NETの実装とほぼ同じだが、クラスの代わりにメソッドを作成し、いくつかの演算子を追加する必要がある
```powershell
$MethodDefinition = @"
    [DllImport("kernel32")]
    public static extern IntPtr GetProcAddress(IntPtr hModule, string procName);
    [DllImport("kernel32")]
    public static extern IntPtr GetModuleHandle(string lpModuleName);
    [DllImport("kernel32")]
    public static extern bool VirtualProtect(IntPtr lpAddress, UIntPtr dwSize, uint flNewProtect, out uint lpflOldProtect);
"@;
```
- これで呼び出しは定義されたが、PowerShellでは初期化する前にメソッド定義内の各Win32 DLLのポインタに新しい型を作成する必要がある。
- `Add-Type`という関数で`/temp`ディレクトリに一時ファイルをドロップし、`csc.exe`を使って必要な関数をコンパイルする。以下は、この関数を使用した例
```powershell
$Kernel32 = Add-Type -MemberDefinition $MethodDefinition -Name 'Kernel32' -NameSpace 'Win32' -PassThru;
```
- これで、以下の構文で必要なAPIコールを利用できるようになった
- `[Win32.Kernel32]::<Imported Call>()`

## よく悪用されるAPIコール
- Win32ライブラリ内のいくつかのAPIコールは、悪意のある活動に容易に利用される可能性がある
- [SANs](https://www.sans.org/white-papers/33649/)や[MalAPI.io](http://malapi.io/)などは、==悪意のあるベクトルを持つすべてのAPIコールの文書化と整理を試みている==
  
- 多くのコールが悪用されているが、いくつかの呼び出しは他の呼び出しよりも多く見られるようになっている。
- 以下は、最もよく悪用されるAPIを、サンプルの集まりの中で頻度別に整理した表


|API Call|説明|
|---|---|
|LoadLibraryA|指定されたDLLを呼び出しプロセスのアドレス空間にマッピング|
|GetUserNameA|現在のスレッドに関連するユーザー名を取得|
|GetComputerNameA|ローカルコンピュータのNetBIOS名またはDNS名を取得|
|GetVersionExA|現在動作しているOSのバージョン情報を取得|
|GetModuleFileNameA|指定されたモジュールとプロセスのファイルの完全修飾パスを取得|
|GetStartupInfoA|STARTUPINFO構造体の内容（ウィンドウステーション、デスクトップ、標準ハンドル、プロセスの外観）を取得|
|GetModuleHandle|指定されたモジュールが呼び出し元プロセスのアドレス空間にマッピングされている場合、そのモジュールハンドルを返す|
|GetProcAddress|指定されたエクスポートされたDLL関数のアドレスを返す|
|VirtualProtect|呼び出し元プロセスの仮想アドレス空間内のメモリ領域に対する保護を変更する。|

## Malware Case Study

- Win32ライブラリの基本的な実装とよく使われるAPIコールを理解したところで、2つのマルウェアのサンプルを分解して、それらのコールがどのように相互作用するかを観察する

### Keylogger
- キーロガーの分析を始めるには、どのAPIコールとフックを実装しているかを収集する必要がある。
- キーロガーはC#で書かれているため、P/Invokeを使用して各呼び出しのポインタを取得する必要がある。以下は、マルウェアのサンプルソースコードのp/invoke定義のスニペット
```csharp
[DllImport("user32.dll", CharSet = CharSet.Auto, SetLastError = true)]
private static extern IntPtr SetWindowsHookEx(int idHook, LowLevelKeyboardProc lpfn, IntPtr hMod, uint dwThreadId);
[DllImport("user32.dll", CharSet = CharSet.Auto, SetLastError = true)]
[return: MarshalAs(UnmanagedType.Bool)]
private static extern bool UnhookWindowsHookEx(IntPtr hhk);
[DllImport("kernel32.dll", CharSet = CharSet.Auto, SetLastError = true)]
private static extern IntPtr GetModuleHandle(string lpModuleName);
private static int WHKEYBOARDLL = 13;
[DllImport("kernel32.dll", CharSet = CharSet.Auto, SetLastError = true)]
private static extern IntPtr GetCurrentProcess();
```
- 以下、各APIコールの説明とそれぞれの用途

|API Call|Explanation|
|---|---|
|SetWindowsHookEx|特定のイベントを監視するために、フックチェーンにメモリフックをインストールする|
|UnhookWindowsHookEx|インストールしたフックをフックチェーンから取り外す|
|GetModuleHandle|指定されたモジュールが呼び出し元プロセスのアドレス空間にマッピングされている場合、そのモジュールハンドルを返す|
|GetCurrentProcess|現在のプロセスの疑似ハンドルを取得|

- 以下は、マルウェアサンプルソースコードのフッキングセクションのスニペット
```csharp
public static void Main() {
	_hookID = SetHook(_proc);
	Application.Run();
	UnhookWindowsHookEx(_hookID);
	Application.Exit();
}
private static IntPtr SetHook(LowLevelKeyboardProc proc) {
	using (Process curProcess = Process.GetCurrentProcess()) {
		return SetWindowsHookEx(WHKEYBOARDLL, proc, GetModuleHandle(curProcess.ProcessName), 0);
	}
}
```
- キーロガーの目的と手順を理解した上で、上記のスニペットからそれぞれのAPIコールを割り出す
- [Windows API documentation](https://docs.microsoft.com/en-us/windows/win32/apiindex/windows-api-list)と上記のスニペットのコンテキストを使用して、キーロガーの分析をする

### Shellcode Launcher
- シェルコードランチャーの分析を始めるには、もう一度、どのAPIコールを実装しているかを収集する必要がある。
- 以下は、マルウェアサンプルソースコードのp/invoke定義のスニペット
```csharp
private static UInt32 MEM_COMMIT = 0x1000;
private static UInt32 PAGE_EXECUTE_READWRITE = 0x40;
[DllImport("kernel32")]
private static extern UInt32 VirtualAlloc(UInt32 lpStartAddr, UInt32 size, UInt32 flAllocationType, UInt32 flProtect);
[DllImport("kernel32")]
private static extern UInt32 WaitForSingleObject(IntPtr hHandle, UInt32 dwMilliseconds);
[DllImport("kernel32")]
private static extern IntPtr CreateThread(UInt32 lpThreadAttributes, UInt32 dwStackSize, UInt32 lpStartAddress, IntPtr param, UInt32 dwCreationFlags, ref UInt32 lpThreadId);
```
- 以下、各APIコールの説明とそれぞれの用途について

|API Call|Explanation|
|---|---|
|VirtualAlloc|呼び出し元プロセスの仮想アドレス空間内のページ領域を予約、コミット、または状態変更|
|WaitForSingleObject|指定されたオブジェクトがシグナル状態になるか、タイムアウト時間が経過するまで待機する|
|CreateThread|呼び出し元のプロセスの仮想アドレス空間内で実行するスレッドを作成する。|
- ここでは、シェルコードがどのようにメモリに書き込まれ、メモリから実行されるかを分析
```csharp
UInt32 funcAddr = VirtualAlloc(0, (UInt32)shellcode.Length, MEM_COMMIT, PAGE_EXECUTE_READWRITE);
Marshal.Copy(shellcode, 0, (IntPtr)(funcAddr), shellcode.Length);
```
