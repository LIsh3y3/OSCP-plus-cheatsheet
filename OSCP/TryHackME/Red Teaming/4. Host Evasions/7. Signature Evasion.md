## Intro
- 高度なAVやEDRソリューションに直面したとき、攻撃者は特定の検出を回避するのに苦労することがある。
- [6. Obfuscation(難読化) Principles](6.%20Obfuscation(難読化)%20Principles.md)で説明した一般的な難読化または回避技術を採用しても、悪意のあるファイル内のシグネチャが存在する場合がある    
- シグネチャとは何か、シグネチャを見つける方法は何かを理解し、思考プロセスに従ってシグネチャを破ることを試みる。
- また、ヒューリスティックシグネチャに対抗するため、より高度なコードコンセプトや「マルウェアのベストプラクティス」についても学ぶ

## シグネチャの識別
- [4. Introduction to Antivirus](4.%20Introduction%20to%20Antivirus.md)で説明したように、シグネチャはAVが疑わしいプログラムや悪意のあるプログラムを追跡して特定するために使用される。
- シグネチャが開始する正確なバイトを手動で特定する方法をみる
  
- シグネチャを特定する場合、手動であれ自動であれ、シグネチャが何バイトから始まるかを決定するために反復して探さないといけない。
- コンパイルされたバイナリを再帰的に==半分に分割==してテストすることで、バイト範囲の概算を得ることができる  
- コンパイル済みのバイナリを分割するには、ネイティブユーティリティの`head`、`dd`、`split`を使用できる。

### 方法
- 以下のコマンドプロンプトでは、msfvenomバイナリに存在する最初のシグネチャを見つけるために、`head`を使用する手順を説明する
0. このような状態だとする
```zsh
ls -la
-rw-r--r-- 1 root root 58 Jul 27 23:53 example.exe
```

1. 58はバイトなのでこれを半分にして新たなファイルに格納する
```zsh
head --bytes 29 example.exe > half.exe
```

2. AVのあるマシンにこのファイルを移動し、アラートが表示されるか確かめる。(もしそのマシンでバイトを半分にしたファイルを作成したなら、他のディレクトリに移動を試みる)

- アラートが表示された場合：以下のようにさらに半分のバイトをファイルに格納する
```zsh
head --bytes 15 half.exe > quarter.exe
```

- アラートが表示されない場合：分割した残りの部分をファイルに格納した上で、ファイルを移動し、アラートが表示されるか確かめる。
```zsh
head --bytes 30-58 example.exe > half.exe
```

3. バイナリを正確に分割できなくなった時点で、16進エディタを使用して署名が存在するバイナリの末端を表示する

- めっちゃ手間がかかるので、フリーソフトを使う。

## シグネチャの識別の自動化ツール

### ThreatCheck
- ThreatCheckは、シグネチャーの可能性を特定するために、分割コンパイルされたバイナリに対して複数のAVを活用し、不良バイトが存在すると考えられる場所を報告する  
- ThreatCheckは、コンパイル済みのリリースをしてないのでコンパイルする必要あり。
- ThreatCheckはAMSITriggerがサポートする==PowerShellなどの特定のファイルタイプをサポートしていない==

- 基本的なシンタックス
```powershell
ThreatCheck.exe -f [binary file] -e AMSI
```
- 一応、検知漏れがあるかもしれないことは念頭に

- これを利用して、シグネチャが検出されたら破壊し、その後ThreatCheckをかけ・・・というのを繰り返し、シグネチャを完全に消す。

### AMSITrigger
- [[9. Runtime Detection Evasion(範囲外?)#AMSI Overview]]にあるように、AMSIはランタイムを活用するため、シグネチャの識別と解決が困難になる。そのAMSIのためのシグネチャ識別ツール
- AMSITriggerは、コンパイル済みのリリースをGitHubで提供している


## Static Code-Based Signatures

- シグネチャを特定したら、それにどう対処するか。
- シグネチャの強さや種類によっては、[6. Obfuscation(難読化) Principles](6.%20Obfuscation(難読化)%20Principles.md)で説明したような単純な難読化で解決できる場合もあれば、特別な調査と対処が必要な場合もある。
- ここでは、関数内に存在する静的シグネチャを修正するためのいくつかの解決策を説明する

### 難読化手法

|Obfuscation Method|Purpose|
|---|---|
|Method Proxy|プロキシメソッドまたは置換オブジェクトを作成|
|Method Scattering/Aggregation|複数のメソッドを1つにまとめるか、メソッドを複数に分散させる|
|Method Clone|メソッドの複製を作成し、それぞれをランダムに呼び出す|


### クラスの難読化

|Obfuscation Method|Purpose|
|---|---|
|Class Hierarchy Flattening|インターフェイスを使ってクラスのプロキシを作成する|
|Class Splitting/Coalescing|ローカル変数または命令グループを別のクラスに移す|
|Dropping Modifiers|クラスの修飾子（public、private）を取り除き、すべてのメンバーをpublicにする。|
- どのようなオブジェクトやデータ構造にも適用できる

- Class Splitting/Coalescing（クラスの分割・合体)やMethod Scattering/Aggregation（メソッドの散在・集約）のテクニックは、任意のOOP（オブジェクト指向プログラミング）関数のData Splitting/Mergingという包括的な概念にグループ化することができる。  
  ![[Pasted image 20230621112432.png | 300]]  
- Dropping Modifiers(修飾子の削除)やMethod Cloneなどの他のテクニックは、識別可能な情報を削除したり、不明瞭にするというMeaningless Identifiersの包括的な概念にグループ化することができる
  ![[Pasted image 20230621121307.png | 400]]

### Splitting and Merging Objects
- 以前の機能を維持しながら、シグネチャを破ることができる新しいオブジェクト関数を作る
- より具体的な例を示すために、`GetMessageFormat`文字列に存在するCovenantのよく知られた事例を使う。
- まず、この解決策がどのように実装されたかを見てから、それを分解し、難読化の分類法に適用する。

###### Original String
- 以下は、検出されたオリジナルの文字列
```csharp
string MessageFormat = @"{{""GUID"":""{0}"",""Type"":{1},""Meta"":""{2},""IV"":""{3}"",""EncryptedMessage"":""{4}"",""HMAC"":""{5}""}}";
```

###### Obfuscated Method
- 以下は、文字列の置換と連結に使われる新しいクラス
```csharp
public static string GetMessageFormat // Format the public method
{
    get // Return the property value
    {
        var sb = new StringBuilder(@"{{""GUID"":""{0}"","); // Start the built-in concatenation method
        sb.Append(@"""Type"":{1},"); // Append substrings onto the string
        sb.Append(@"""Meta"":""{2}"",");
        sb.Append(@"""IV"":""{3}"",");
        sb.Append(@"""EncryptedMessage"":""{4}"",");
        sb.Append(@"""HMAC"":""{5}""}}");
        return sb.ToString(); // Return the concatenated string to the class
    }
}

string MessageFormat = GetMessageFormat
```
- このケーススタディをまとめると、クラス分割は、連結するローカル変数のために新しいクラスを作成するために使用される。

### 識別情報の削除と隠蔽
- 識別可能な情報を削除することは、[6. Obfuscation(難読化) Principles](6.%20Obfuscation(難読化)%20Principles.md#Protecting%20and%20Stripping%20Identifiable%20Information)で取り上げたように、変数名を不明瞭にすることに似ている。
- ここでは、特にメソッドやクラスを含むあらゆるオブジェクトの識別されたシグネチャに適用する
  
- この例はMimikatzにあり、`wdigest.dll`という文字列に対してアラートが生成される。これは、文字列を、その文字列のすべてのインスタンスで変更された任意のランダムな識別子で置き換えることで解決できる。これは難読化の方法では、Method Proxyに分類される
  
- これは難読化の原則で説明したものとほとんど変わないが、特定の状況に適用される

- 以下の PowerShell スニペットを難読化し、AmsiTrigger を使用してシグネチャを視覚化してみる

## Static Property-Based Signatures
- 様々な検知エンジンや分析者は、文字列や静的なシグネチャではなく、異なるインジケータを考慮する。
- シグネチャは、ファイルハッシュ、エントロピー、作者、名前、その他の識別可能な情報など、いくつかのファイルプロパティに付加することができ、個別または組み合わせて使用することができる。
- これらのプロパティは、YARAやSigmaのようなルールセットでよく使用される。  

- ここでは、オープンソースとクローズドソースの両方のアプリケーションのファイルハッシュとエントロピーの操作について説明する

### File Hashes
- ファイルハッシュはチェックサムとも呼ばれ、一意のファイルをタグ付け/識別するために使用される。
- 一般的に、ファイルの真正性や既知の目的（悪意があるかどうか）を検証するために使用される。ファイル・ハッシュは一般に任意に変更可能であり、ファイルに変更が加えられると変更される。 
- もしアプリケーションのソースにアクセスできれば、コードの任意の部分を修正し、それを再コンパイルして新しいハッシュを作成することができる。
- だが、コンパイル済み、あるいは署名済みのアプリケーションが必要な場合はどうなるか？
  
- 署名されたアプリケーションやクローズドソースのアプリケーションを扱う場合、ビット反転を使う。
- ビット反転は実行可能なビットを見つけるまで、可能なビットを反転してテストすることで、与えられたアプリケーションを変異させる。
- 実行可能なビットを1つ反転させることで、すべての機能を維持したまま、アプリケーションのシグネチャとハッシュを変更する
  
- 各ビットを反転させ、新しい変異型（~3000 - 200000の変異型）を作成するビット反転リストを作成するスクリプトを使うことができる。

- 以下はpythonによるビット反転の実装例
```python
import sys

orig = list(open(sys.argv[1], "rb").read())

i = 0
while i < len(orig):
	current = list(orig)
	current[i] = chr(ord(current[i]) ^ 0xde)
	path = "%d.exe" % i
	
	output = "".join(str(e) for e in current)
	open(path, "wb").write(output)
	i += 1
	
print("done")
```
- リストが作成されたら、ファイルの固有のプロパティをそのまま検索しなければならない。例えば、`msbuild`をビット反転する場合、`signtool`を使用して、使用可能な証明書を持つファイルを検索する必要がある。これにより、ファイルの機能が破壊されず、アプリケーションがその署名された属性を維持することが保証される。  
  
- ビット反転リストをループし、機能的な変種を検証するスクリプトを活用できる。以下にバッチスクリプトの実装例を示す。
```powershell
FOR /L %%A IN (1,1,10000) DO (
	signtool verify /v /a flipped\\%%A.exe
)
```
- ハッシュが発見されるまでには長い時間がかかり、限られた期間しかないが、このテクニックは非常に有利である。以下は、オリジナルのMSBuildアプリケーションと、ビットを反転させたバリエーションとの比較である。

### Entropy
- ファイルに隠されたデータや不審なスクリプトが含まれているかどうかを判断するために使用されるファイル内のデータのランダム性。
- EDRやその他のスキャナは、疑わしいファイルを特定したりするために、エントロピーを活用することがよくある
- エントロピーは、難読化されたスクリプト、特に変数や関数などの識別可能な情報を不明瞭にする場合に問題となることがある
  
- エントロピーを低下させるには、ランダムな識別子をランダムに選択した英単語に置き換える。たとえば、変数を`q234uf`から`nature`に変更する。  
- 識別子を変更することの有効性を証明するために、[CyberChef](https://gchq.github.io/CyberChef/#recipe=Entropy('Shannon%20scale'))を使ってエントロピーがどのように変化するかを観察することができる。

##  Behavioral Signatures
- 関数やプロパティを難読化することで、最小限の修正で多くのことが実現できる。
- ファイルに添付された静的シグネチャを破った後でも、最新のエンジンはバイナリの動作や機能を観察することができる。
- これは攻撃者にとって、単純な難読化では解決できない多くの問題をもたらす
  
- 最近のAVは動作を検出するために2つの一般的な方法を採用する。
- インポートは最小限の要求で簡単に難読化または修正できるが、フックは複雑な技術を必要とする。
- APIコールは特に普及しているため、これらの関数を観察することは、他の挙動テスト/考慮事項とともに、ファイルが疑わしいかどうかを決定する重要な要因となり得る。  
  
- APIコールが伝統的にどのように利用され、インポートされるかについて。
- APIコールやOS・ネイティブの他の関数は、関数アドレスへのポインタと、それらを利用するための構造体を必要とする。    
- 関数の構造体は単純で、Windowsの関数構造やその他のコア情報を格納する`kernel32`や`ntdll`などのインポートライブラリにある
- 関数のインポートで最も重要な問題は、関数アドレスである。ASLR（アドレス空間レイアウトのランダム化）のため、関数アドレスは動的であり、見つけなければならない。  
  
- 実行時にコードを変更するのではなく、Windowsのローダー`windows.h`を使用。
- 実行時に、ローダーはすべてのモジュールをプロセスのアドレス空間にマッピングし、それぞれの関数をすべてリストアップする。これでモジュールは処理されるが、関数アドレスはどのように割り当てられるのか？  
  
- Windowsローダーの最も重要な機能の一つは、IAT（Import Address Table）である。IATは、関数のポインタを割り当てることができるすべてのインポート関数の関数アドレスを格納する
- IATはPE（Portable Executable）ヘッダーの`IMAGE_OPTIONAL_HEADER`に格納され、実行時にWindowsローダーによって埋められる。
- WindowsローダーはAPIコールまたはサンクテーブルからアクセスされたポインターテーブルから関数アドレス、より正確にはサンクを取得する。  

IAT（Import Address Table）に関数が表示されないようにする方法として、動的ローディングというテクニックがあります。これにより、関数のアドレスをインポートライブラリ自体から取得することができます。以下では、C言語での動的ローディングの手順を4つに分けて説明します。

1. 呼び出しの構造体を定義する: 動的ローディングを開始するために、まずメイン関数の前に呼び出しの構造体を定義する必要があります。この構造体は、呼び出しに必要な入力や出力を定義します。具体的な呼び出しの構造体は、Microsoftのドキュメントなどで見つけることができます。例えば、GetComputerNameAの呼び出しの構造体は、Microsoftのドキュメントで確認できます。以下に、C言語での定義の例を示します。

```c
// 1. 呼び出しの構造体を定義する
typedef BOOL (WINAPI* myNotGetComputerNameA)( 	LPSTR   lpBuffer, 	LPDWORD nSize );
```

2. ライブラリのハンドルを取得する: API呼び出しのアドレスが定義されているモジュールをロードするために、main関数でライブラリのハンドルを取得します。一般的には、Windows API呼び出しの場合、kernel32.dllやntdll.dllなどが使用されます。以下は、ライブラリをモジュールハンドルにロードするための構文の例です。

```c
// 2. ライブラリのハンドルを取得する
HMODULE hkernel32 = LoadLibraryA("kernel32.dll");
```

3. 呼び出しのプロセスアドレスを取得する: ロードされたモジュールを使用して、指定したAPI呼び出しのプロセスアドレスを取得します。これは、LoadLibraryの呼び出しの直後に行います。先ほど定義した構造体と共にキャストして、呼び出しを保持することができます。以下は、API呼び出しのプロセスアドレスを取得するための構文の例です。
```c
// 3. 呼び出しのプロセスアドレスを取得する
myNotGetComputerNameA notGetComputerNameA = (myNotGetComputerNameA) GetProcAddress(hkernel32, "GetComputerNameA");

```
`
この方法によって、IATに関数が表示される問題の多くは解決されますが、いくつかの考慮事項があります。まず、GetProcAddressやLoadLibraryAはまだIATに存在します。これは直接的な指標ではありませんが、疑いのもととなる可能性があります。この問題は、PIC（Position Independent Code）を使用することで解決できます。また、近代的なエージェントは特定の関数をフックし、カーネルのインタラクションを監視する場合があります。これに対処するためには、APIのアンフックが必要です。
