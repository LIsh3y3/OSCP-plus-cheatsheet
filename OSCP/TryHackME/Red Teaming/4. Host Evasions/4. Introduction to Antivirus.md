# AVについての前提知識

## AntiVirusとは？

- AVソフトウェアは、ターゲットとなるOSにおける悪意のあるファイルの実行と拡散を検出し、防止することを目的としたセキュリティソリューション
- ホストベースのアプリケーションで、リアルタイム（バックグラウンド）で実行され、現在および新たにダウンロードされたファイルを監視およびチェックする。
- AVソフトは、ここの後半で取り上げるさまざまなテクニックを使って、ファイルが悪意あるものかどうかを検査し、判断する
- 最近のAVソフトは、コンピュータウイルスだけでなく、その他の有害なファイルや脅威も検出・除去できるようになった

## AVが探すもの

- 従来のAVソフトは、あらかじめ定義された悪意のあるパターンやシグネチャーを持つマルウェアを探す
- マルウェアとは、ターゲットマシーンに損害を与えることを主目的とする有害なソフトウェアで、以下のようなものがあるが、これらに限定されない
	- ターゲットマシンにフルアクセス
	- パスワードなどの機密情報を盗み出す
	- ファイルを暗号化し、ファイルに損害を与える
	- 他の悪意のあるソフトウェアや不要な広告を注入する
	- ボットネット攻撃など、さらなる攻撃を行うために侵害されたマシンを使用する

## AVと他のセキュリティ製品との比較

- AVソフトウェア以外にも、ホストベースのセキュリティソリューションがエンドポイントデバイスにリアルタイムで保護を提供する。
- *EDR*は、行動分析に基づくリアルタイムの保護を提供するセキュリティソリューション。
- AVは、悪意のあるファイルのスキャン、検出、削除を実行する。
- 一方、EDRは、ファイルアクティビティ、メモリ、ネットワーク接続、Windowsレジストリ、プロセスなど、対象マシンのさまざま箇所を監視する
- 最近のAV製品は、従来のAV機能とその他の高度な機能（EDRの機能と同様）を1つの製品に統合し、デジタル脅威に対する包括的な保護を提供するように実装されている。
- ホストベースセキュリティソリューションの詳細：[[1. セキュリティ技術や製品の土地勘 -The Lay of the Land-]]

# AVの機能

## Antivirus Engines

- AVエンジンは、悪意のあるコードやファイルを発見し、削除する役割を担っている。
- 優れたAVは、悪意のあるファイルを正確かつ迅速に分析する効果的で強固なAVコアを実装している。
- また、アーカイブファイルのような、自己解凍してすべての圧縮ファイルを検査できる必要のあるファイルを含め、さまざまなファイルタイプを処理しサポートする必要がある
- ほとんどのAV製品は、同じ共通の機能を共有しているが、以下のような異なる実装がされている：
	- Scanner
	- Detection techniques
	- Compressors and Archives
	- Unpackers
	- Emulators

## Scanner

- スキャナ機能は、ほとんどのAV製品に搭載されている
- AVソフトウェアは、リアルタイムまたはオンデマンドで実行され、スキャンされる。
- この機能は、GUIまたはコマンドプロンプトから利用できる。
- ユーザーは、ファイルやディレクトリをチェックするために必要なときに、いつでもこれを使用することができる。
- スキャン機能は、脅威を検出して除去するために、有名な悪意のあるファイルタイプをサポートする必要がある。
- さらに、脆弱性、電子メール、Windowsメモリ、Windowsレジストリなど、AVソフトによって他の種類のスキャンもサポートすることがある。

## Detection techniques

- AVの検出技術は、悪意のあるファイルを検索して検出するもので、AVエンジン内では、以下のようなさまざまな検出技術を使用する
	- *Static Detection*（静的解析）：ファイル内のあらかじめ定義された悪意のあるパターンやシグネチャーを探す、従来のAV技術
	- *Dynamic Detection*（動的解析）：システムコールやAPIを監視し、隔離された環境でテストと分析を行うことを含む技術（= Befavior-Based Detection）
	- *Heuristic Detection*（ヒューリスティック解析）：疑わしいファイルを分析するためのさまざまな行動メソッドを含む、より高度な技術
	- *Machine Learning Detection*（機械学習による検知）：
		- 例えば、Microsoft Windows Defenderには、①MLモデルとヒューリスティックの作成を担当するクライアントMLエンジンと、②提出されたサンプルをすべての提出されたサンプルから構成されるメタデータベースのモデルに対して分析することができるクラウドMLエンジンがある。クライアントMLエンジンは、あるプログラムが良性かどうかを判断できない場合、クラウドMLエンジンに最終的な回答を求める。
- 優れたAVエンジンは、正確で、誤検出が少なく、悪意のあるファイルを素早く検出する
- しかし、不正確な結果を出し、ファイルを誤って分類してしまうAV製品をあとでいくつか紹介する

## Compressors and Archives

- 圧縮・アーカイブ機能は、あらゆるAVソフトに搭載されるべきもの。
- 圧縮ファイルやアーカイブファイルを含む様々なシステムファイルタイプをサポートし、扱うことができる必要がある（ZIP、TGZ、7z、XAR、RAR等）
- 悪意のあるコードは、圧縮ファイルに隠れることでホストベースのセキュリティソリューションを回避しようとすることがよくある。
- このため、AVソフトは、ユーザがアーカイブ内のファイルを開く前に、すべてのファイルを解凍してスキャンする必要がある

## PE (Portable Executable) Parsing and Unpackers

- マルウェアは、悪意のあるコードをペイロード内で圧縮・暗号化することで隠蔽・梱包する。
- マルウェアは、静的解析を行いにくくするために、実行時に自身を解凍・復号する
- したがって、AVソフトウェアは、静的解析のために、ほとんどの既知のパッカー（UPX、Armadillo、ASPackなど）をランタイムの前に検出し、解凍することができる必要がある
  
- マルウェア開発者は、サイズを縮小し、悪意のあるファイルの構造を変更するために、パッキングなどの様々なテクニックを使用する。
	- パッキングは、元の実行ファイルを圧縮して解析しにくくするもの。
- そのため、AVソフトウェアには、保護された実行ファイルや圧縮された実行ファイルを元のコードに解凍するアンパッカー機能が必要
- また、AVソフトに必要な機能として、Windows Portable Executable (PE) ヘッダーパーサがある。
- 実行ファイルのPEを解析することで、悪意のあるソフトウェアと正規のソフトウェア（.exeファイル）を区別することができる。
- Windows（32ビットおよび64ビット）のPEファイル形式には、オブジェクトコード、DLL、アイコンファイル、フォントファイル、コアダンプなど、さまざまな情報やリソースが含まれている

## Emulators

- エミュレータは、疑わしいファイルに対してさらなる分析を行うアンチウイルス機能。
- エミュレータがリクエストを受信すると、エミュレータは疑わしいファイル（exe、DLL、PDFなど）を仮想化・制御された環境で実行する。
- 実行中の実行ファイルの動作（Windows APIの呼び出し、レジストリ、その他のWindowsファイルなど）を監視する。
- 以下は、エミュレータが収集する可能性のあるアーティファクトの例：
	- API calls
	- Memory dumps
	- Filesystem modifications
	- Log events
	- Running processes
	- Web requests
- エミュレータは、マルウェアを検出するのに十分なアーティファクトが収集されると、ファイルの実行を停止する

# Static detection

## Static detectionの基本

### 1. 静的解析とは

- 静的解析（Static Detection）は、事前に定義されたシグネチャ（署名）と照合することで、悪意のあるファイルを検出する手法
- マルウェアを実行せず、ファイルの内容を直接分析する
- 実行前にファイルが持つ特徴を調べ、既知のパターンと一致するかどうかを判断する
	- パターンマッチングという

- 既知のマルウェアに対して有効だが、新種や難読化・暗号化されたマルウェアの検出には弱い

### 2. パターンマッチング技術の種類

- シグネチャベースの静的解析では、以下のような情報に基づいて検出が行われる。
	- 一意の文字列
	- CRC（チェックサム）
	- バイトコードや16進数のシーケンス
	- 暗号ハッシュ（例: MD5、SHA1）

- これらの情報を、シグネチャが登録されたデータベースと照合することで、対象ファイルが既知の悪意あるものであるかを判断する


![[Pasted image 20230619115411.png]]
$$静的解析の図(TryHackMe)$$

### 3. 静的解析の制限と前提

- 静的解析は既知のシグネチャに基づくため、新しいマルウェアや変更されたコードには対応できない
- シグネチャデータベースは定期的に更新する必要がある
	- ⚠️更新が遅れると、最新のマルウェアは検出されない

### 4. ClamAVによる静的解析の例

#### 4.1 ClamAVの概要とデータベース

- ClamAVはオープンソースのAVで、シグネチャベースの検出を行う
- 初回インストール時に、最新のシグネチャデータベースを取得することが推奨されている

#### 4.2 マルウェアサンプル一覧

検証に使用するサンプルは以下の通り。
TryHackMeの学習モジュールで提供されている。

1. [EICAR](https://www.eicar.org/) 
    AVソフトのテスト用に作成されたファイルで、マルウェアではない。ASCII文字列を含み、意図的に悪意ある挙動を示す。
2. Backdoor 1  
    C#で記述されたリバースシェル型マルウェア。Metasploit Frameworkのシェルコードをプロセスに注入することでシェルを確立する。
3. Backdoor 2  
    XORによる暗号化とプロセスインジェクションを用いて、既存のプロセスにMetasploitシェルコードを注入するリバースシェル型マルウェア。
4. AV-Check  
    C#で記述されたツールで、システム上に存在するAVソフトを列挙する。悪意ある挙動は含まれない。
5. notes.txt  
    テキストファイル。単なるメモ書きであり、悪意はない。

#### 4.3 スキャンの実行と結果

以下のコマンドでClamAVによるスキャンを実行する。
```powershell
clamscan.exe --debug [file_to_scan]
```
![[Pasted image 20230619152027.png]]
$$clamavによるファイルスキャンコマンドの実行結果$$

- Backdoor2が検出されず、スルーされている。

##### 結果の理由

- ClamAVはファイルを分割し、既知のバイト列と照合するアンパッキング手法を用いる
- Backdoor1に使用されたMetasploitのシェルコードは、過去に登録されたシグネチャと一致したため、検出に成功している
- 一方で、Backdoor2はシェルコードをXOR暗号化しているため、シグネチャと一致しないバイト列となっており、検出できなかった

#### 4.4 EICARの検出とシグネチャ確認

ClamAVはEICARファイルをmd5シグネチャで検出する
```powershell
clamscan.exe --debug c:\Users\thm\Desktop\Samples\eicar.com
```
	↓
```

```

このシグネチャが正しいかを確認するために、[sigtool](https://manpages.debian.org/testing/clamav/sigtool.1.en.html)を使用する：
```powershell

```
→出力に`44d88612fea8a8f36de82e1278abb02f`が含まれていれば、シグネチャ一致が確認できる

## ClamAVで独自シグネチャを作成し、検出対象を拡張する

### 概要

ClamAVでは独自のDBを作成できるため、公式DBに含まれないマルウェア（例：上記のBackdoor 2）も検出可能になる。
対象ファイルのMD5シグネチャを生成し、カスタムDB（`.hdb`形式）に追加する。

### 手順

1. `sigtool.exe`を使用して、対象ファイル（例：backdoor2.exe）のMD5ハッシュを生成
```powershell
sigtool.exe --md5 backdoor2.exe
```
↓出力形式は `Hash:Size:FileName` となる
```
75047189991b1d119fdb477fef333ceb:6144:backdoor2.exe
```

2. 上記出力を `.hdb` ファイルに保存（例：thm.hdb）
```powershell
sigtool.exe --md5 backdoor2.exe > thm.hdb
```

3. ClamAVでカスタムDBを使って再スキャン
```powershell
clamscan.exe -d thm.hdb backdoor2.exe
```
→検出されれば `FOUND` と表示される

## Yaraによる静的検知

### 概要

- Yaraはルールベースでマルウェア検知を行うツール。
- ClamAVもYaraルールに対応しており、独自ルールを作成することで新種マルウェアにも対応可能。
- ルールを作成するには、マルウェアを調査・分析する必要がある

### ルール作成の前準備

- `strings` コマンドを使用して、対象ファイルから読める文字列を抽出
	- pdbを含む行だけ表示
```powershell
strings AV-Check.exe | findstr pdb
```
- 上記出力から、マルウェア特有の特徴的な文字列をルールに活用する

#### PDB

- Program Databaseは、Microsoftの開発環境で使用されるデバッグ情報ファイル

- 含まれる情報：
	- シンボル情報（関数名、変数名など）
	- 型情報（データ構造、クラス定義など）
	- ソースコードの行番号とバイナリコードの対応関係
	- デバッグに必要なメタデータ

- Yara Rules での活用： 開発者が残したシンボル名や型情報から、そのプログラムの特徴的な文字列やパターンを特定し、効果的な検出ルールを作成できる

### Yaraルール例

```yara
rule thm_demo_rule {
	meta:
		author = "THM: Intro-to-AV-Room"
		description = "Look at how the Yara rule works with ClamAV"
	strings:
		$a = "C:\\Users\\thm\\source\\repos\\AV-Check\\AV-Check\\obj\\Debug\\AV-Check.pdb"
	condition:
		$a
}
```
- `rule` 名はClamAVが検知名として使用
- `meta` セクションは任意の情報
- `strings` で対象の文字列を定義。Windowsパス内のバックスラッシュはエスケープが必要
- `condition` でマッチ条件を指定

### 運用方法

- 作成したルールは `.yara` 拡張子で保存（例：thm-demo-1.yara）
- ルールを用いて、ClamAVでAV-Check.exeを再スキャンし、悪意あるファイルとして検出
- ただし、notes.txt にも同一文字列が含まれていたため、誤検知が発生

---

### 誤検知を防ぐYaraルールの改良方法

- ファイルの種類を制限する
- 実行ファイルは常に `MZ`（Hex: `4D 5A`）から始まる
- 検知したい実行ファイルをHxDなどのHexエディタで開き、ファイルの先頭2バイトを確認→ファイルの種類をバイナリ制限

# Dynamic Detection

静的検出（ファイル構造やシグネチャに基づく）とは異なり、動的検出はファイルを実際に「実行」して振る舞いを監視し、マルウェアを特定する。
![[Pasted image 20230619162228.png]]
$$動的検知の図（TryHackMe）$$
## 1. Windows API Monitoring

- 実行中のアプリケーションが Windows API をどう使っているかを監視する手法
- たとえば、プロセス生成やファイル書き込み、ネットワーク通信など、特定のAPI呼び出しパターンを検知する
- API呼び出しの監視には Windows Hooks が使われる

### Windows Hooks の解説

- Windows Hooksとは、Windows OSにおける特定のイベント（メッセージや関数の呼び出しなど）を横取り・監視・改変できる仕組み
	- フック：フック船長の手についているやつ
- イメージは、プロセスが流れているときにフックに引っかかり、一度ストップさせられ、中身を改変

主な特徴：
- イベントが発生したときに、自分の関数（フックプロシージャ）を先に呼び出させる
- API: `SetWindowsHookEx()` でフックを設定し、`UnhookWindowsHookEx()` で解除する

利用される用途：
- スクリーンリーダーやマクロソフトのような正当なツール
- 一方で、キーロガーなど悪意あるマルウェアによる悪用も可能

#### フックの例

|フック名|説明|
|---|---|
|WH_KEYBOARD_LL|キーボード入力をグローバルに監視（低レベル）|
|WH_MOUSE_LL|マウス操作を監視|
|WH_CALLWNDPROC|ウィンドウメッセージの受け渡しを傍受|
|WH_GETMESSAGE|メッセージキューからの取得時に傍受|
|WH_CBT|ウィンドウの作成・破棄などの流れを検出|
|WH_DEBUG|デバッグ用の監視|

---

## 2. サンドボックス（Sandboxing）

- 実行対象のプログラムを本物のOSとは隔離された仮想環境で実行
- その中での挙動を観察し、不審な処理を行ったらマルウェアと判断する
- 振る舞い情報から新たなシグネチャや検出ルールを生成し、クラウドベースのAVに反映される

---

## 動的検出の欠点

- 仮想環境では、時間やシステムリソースに制限があるため、すべての動作を観察しきれない可能性がある
- マルウェアが環境を検出して活動を停止するような回避コードを含んでいる場合、検知できない
	- 関連ノート： [3. Sandbox Evasion](../5.%20Network%20Security%20Evasion/3.%20Sandbox%20Evasion.md)

### マルウェアの検出回避テクニック

- 仮想環境やデバッガを検出するコードを入れる
- 特定のプロセス名やファイル構成がなければ動作しないようにする
- 数分間スリープして、短時間しか観察しないサンドボックスの網をくぐる

# Heuristic and Behavioral Detection

現代のマルウェアに対しては単なるシグネチャベースの検出では不十分であり、AVは、より柔軟な手法である「ヒューリスティック検知」や「行動ベース検知」を組み合わせてマルウェアを検出している。

## ヒューリスティック（Heuristic）検知とは

- 既知のマルウェアだけでなく、未知のマルウェアや亜種にも対応できる検知方法
- ルールやパターンに基づく分析を通じて、プログラムの危険性を評価する

## 1. 静的ヒューリスティック解析（Static Heuristic Analysis）

- プログラムを**実行せず**に、ソースコードやバイナリを解析
- 逆コンパイルや逆アセンブルによってコードを抽出し、既知のマルウェアのコードパターンと比較する
- パターンの一致度がある閾値（％）を超えると、マルウェアと判定される

ポイント：
- AV内部にヒューリスティック・データベースがあり、それと照合する
- 静的解析なので誤検出（False Positive）が出やすい場合もある

## 2. 動的ヒューリスティック解析（Dynamic Heuristic Analysis）

- プログラムを**実行した**うえで、その挙動を監視
- サンドボックスなどの仮想環境でマルウェアを観察し、行動ルール（behavioral rules）に合致すればフラグが立てられる
- これは「行動ベース検知（Behavioral Detection）」とも密接に関係する

### 行動ルールの例（実際に検出対象となる行為）

- プロセスが `lsass.exe`（ログオン情報を扱うプロセス）にアクセスしようとする
    - → NTLMハッシュやKerberosチケットを盗む動きとみなされる
- プロセスがネットワークのリスニングポートを開いて待機し、外部C2サーバーから命令を待つ
    - → ボットやバックドアの疑い

# AVエンジンのウイルス検出の流れ

- 最近のAVソフトは、すべてのコンポーネントを含めた1つのユニットとして動作し、さまざまな機能や検出技術を組み合わせてAVエンジンを実装している

AVエンジンの検出までの流れ：
1. foobar.zipというファイルがAVソフトに渡される
2. ファイルがzipであると認識され、unarchiverモジュールが起動し、中のfoobar.exeを取り出す
3. file type recognizerでfoobar.exeのファイル種別を判別する
4. 実行ファイル（PE形式）であると判断されると、PE parserがバイナリ構造を解析する
5. ファイルがパックされている場合は、unpackerがコードを展開する
6. fingerprint scanning and heuristicsが、既知のマルウェアの特徴や不審な振る舞いを検出する
7. 最後にファイルがcleanと判定されるか、たとえば「Infected with W32/Malware.ABC」のような感染判定が出される

## AVエンジンの構成と役割

|エンジン名|主な役割|
|---|---|
|File Engine|ファイルシステム上のファイルをスキャンし、既知のウイルスシグネチャやヒューリスティックルールに基づいて静的検出を行う。通常はリアルタイム保護やオンデマンドスキャンで使用される。|
|Memory Engine|実行中のプロセスやそのメモリ空間を監視・スキャンする。パックされたマルウェアのアンパック後、あるいはファイルレスマルウェアなどの動的挙動を検出する目的がある。|
|Network Engine|ネットワークトラフィック（送受信されるデータやURLアクセス）を監視。C2サーバーへの接続や、不審な通信パターンの検出に使用される。|
|Disassembler|バイナリコードをアセンブリ命令に変換し、静的にコードの流れや悪意ある命令を解析するために使われる。ヒューリスティック検知や機械学習の入力データとしても使われる。|
|Emulator / Sandbox|疑わしいファイルを仮想環境やエミュレーションで一時的に実行し、その挙動からマルウェアかどうかを判断する。静的検知を回避するマルウェアにも有効。|
|Browser Plugin|Webブラウザ内の動作を監視し、悪意のあるスクリプトやフィッシングページなどをブロックする。ダウンロードされるファイルの安全性チェックなども担当。|
|Machine Learning Engine|過去のマルウェアパターンや挙動データを学習し、未知のマルウェアや亜種を高精度で検出するために使われる。ファイル構造、API呼び出しパターン、ネットワーク通信など多様な特徴量を使って判断する。|

# AV Testing and Fingerprinting

## AV Testing Environment

- テスト環境では、疑わしいファイルをさまざまなAV製品でスキャン可能
- VirusTotalのようなサービスを使えば、数十種類のAVエンジンによって即座にスキャン結果が得られる
- 攻撃者視点では、自分のペイロードが検出されるかどうかをここでテストすることもある

### VirusTotalの特徴と制限

- [VirusTotal](https://www.virustotal.com/)は、70以上のAVエンジンでスキャン可能なWebプラットフォーム
- URLやファイルのブラックリストチェック、動作解析、API呼び出しの追跡など多くのチェックを行う
- 隔離環境でファイルを実行し、挙動を分析する機能もある
- ただし、アップロードされた情報はAVベンダーに共有されるポリシーがあるため、機密性の高いペイロードを使うには適さない

### VirusTotalの代替サービス

VirusTotalでは情報が共有されるため、秘密保持が重要なテストには向かない。そのため以下のような代替がある
- [AntiscanMe](https://antiscan.me/)：1日6回まで無料スキャン、共有なし
- [Jotti's Malware Scan](https://virusscan.jotti.org/)：情報を共有しない設計
ただし、これらも無料プランではスキャン数に制限がある。無制限で使用するにはサブスクリプションが必要になる

## AVフィンガープリンティング

### フィンガープリンティングの調査対象について

- ターゲットマシンに侵入した時点では、どのAVソフトが導入されているかはわからない
- そのため、インストールされているAVやセキュリティ製品を特定する「フィンガープリンティング」が重要
- フィンガープリンティングにより、バイパス技術を同じ環境でテストできるようになる

調査対象となるのは以下のような静的アーティファクト
- サービス名
- プロセス名
- ドメイン名
- レジストリキー
- ファイルシステム情報

### 自作プログラムによるAV検出（C#）

- AVソフトを列挙するもう一つの方法は、自分たちのプログラムをコーディングすること。
  
- 以下のプログラムの目的は、定義済みのAVプロセス名と照らし合わせて、対象マシンにAVソフトが存在するかをチェックすること
```csharp
using System;
using System.Management;

internal class Program
{
    static void Main(string[] args)
    {
        var status = false;
        Console.WriteLine("[+] Antivirus check is running .. ");
        string[] AV_Check = { 
            "MsMpEng.exe", "AdAwareService.exe", "afwServ.exe", "avguard.exe", "AVGSvc.exe", 
            "bdagent.exe", "BullGuardCore.exe", "ekrn.exe", "fshoster32.exe", "GDScan.exe", 
            "avp.exe", "K7CrvSvc.exe", "McAPExe.exe", "NortonSecurity.exe", "PavFnSvr.exe", 
            "SavService.exe", "EnterpriseService.exe", "WRSA.exe", "ZAPrivacyService.exe" 
        };
        var searcher = new ManagementObjectSearcher("select * from win32_process");
        var processList = searcher.Get();
        int i = 0;
        foreach (var process in processList)
        {
            int _index = Array.IndexOf(AV_Check, process["Name"].ToString());
            if (_index > -1)
            {
                Console.WriteLine("--AV Found: {0}", process["Name"].ToString());
                status = true;
            }
            i++;
        }
        if (!status) { Console.WriteLine("--AV software is not found!");  }
    }
}
```

処理の流れ：
1. 有名なAVプロセス名を`AV_Check`配列として定義
2. `win32_process`クラスを使って、現在実行中の全プロセスを取得
3. 各プロセスの名前と配列内の名前を比較
4. 一致すればAVが存在すると判断し、結果を出力

使用している技術：
- WMI（Windows Management Instrumentation）経由でプロセス情報を取得
- WMICクエリ：`select * from win32_process`
- AVのフィンガープリンティングと同様、静的な痕跡に依存した検出手法

注意点と制限：
- AV製品によっては、WMIやAPI呼び出しを監視している場合があり、プログラムが誤って「マルウェア」と判定される可能性がある
- C#コードは安全でも、VirusTotalにアップロードするとAVエンジンが反応することがある（偽陽性）
